# 类加载与执行引擎

## 了解类的加载机制吗？ (类加载器、双亲委派)

类加载是JVM将.class文件加载到内存，并生成Class对象的过程。

### 类加载的时机

Java虚拟机规范没有强制约束何时开始加载，但规定了**有且只有**以下5种情况必须立即对类进行"初始化"：

1. **遇到 new、getstatic、putstatic、invokestatic 字节码指令**
   ```java
   new MyClass();           // new指令
   int value = MyClass.staticField;  // getstatic
   MyClass.staticField = 10;         // putstatic
   MyClass.staticMethod();           // invokestatic
   ```

2. **使用 java.lang.reflect 包的方法对类进行反射调用**
   ```java
   Class.forName("com.example.MyClass");
   ```

3. **初始化子类时，发现父类还没有初始化**
   ```java
   class Parent {}
   class Child extends Parent {}
   
   new Child();  // 会先初始化Parent
   ```

4. **虚拟机启动时，用户需要指定一个要执行的主类**
   ```bash
   java com.example.Main  # Main类会被初始化
   ```

5. **使用 JDK 7 的动态语言支持时**
   - MethodHandle等

---

### 类加载的过程

类加载分为三个阶段：**加载** → **链接** → **初始化**

```
┌─────────────────────────────────────┐
│          加载 (Loading)              │
├─────────────────────────────────────┤
│          链接 (Linking)              │
│  ├─ 验证 (Verification)             │
│  ├─ 准备 (Preparation)              │
│  └─ 解析 (Resolution)               │
├─────────────────────────────────────┤
│       初始化 (Initialization)        │
└─────────────────────────────────────┘
```

#### 1. 加载（Loading）

**任务**：
- 通过类的全限定名获取二进制字节流
- 将字节流的静态存储结构转化为方法区的运行时数据结构
- 在内存中生成代表这个类的 `java.lang.Class` 对象

**二进制字节流的来源**：
- 从ZIP包读取（JAR、EAR、WAR）
- 从网络获取（Applet）
- 运行时计算生成（动态代理）
- 由其他文件生成（JSP）
- 从数据库读取
- 从加密文件中获取

---

#### 2. 链接（Linking）

##### （1）验证（Verification）

**目的**：确保Class文件的字节流符合当前虚拟机的要求

**验证内容**：
- **文件格式验证**：是否以魔数0xCAFEBABE开头
- **元数据验证**：是否有父类、是否继承了final类
- **字节码验证**：字节码指令是否能正确执行
- **符号引用验证**：符号引用能否找到对应的直接引用

**重要性**：保证虚拟机的安全，防止恶意代码

---

##### （2）准备（Preparation）

**任务**：为类变量（static变量）分配内存并设置默认值

**注意**：
- 这里只分配静态变量，不包括实例变量
- 设置的是默认值，不是代码中指定的值

**示例**：
```java
public class Example {
    public static int value = 123;  // 准备阶段value = 0
    public static final int CONSTANT = 456;  // 准备阶段CONSTANT = 456
}
```

**特殊情况**：
- 如果字段是 `static final` 常量，准备阶段就会赋值为代码中的值
- 因为final常量在编译期就确定了

**默认值对照表**：

| 数据类型 | 默认值 |
|---------|--------|
| byte | 0 |
| short | 0 |
| int | 0 |
| long | 0L |
| float | 0.0f |
| double | 0.0d |
| char | '\u0000' |
| boolean | false |
| reference | null |

---

##### （3）解析（Resolution）

**任务**：将常量池中的符号引用替换为直接引用

**符号引用**：
- 以一组符号来描述所引用的目标
- 可以是任何形式的字面量
- 与虚拟机实现的内存布局无关

**直接引用**：
- 直接指向目标的指针、相对偏移量或句柄
- 与虚拟机实现的内存布局相关

**解析的内容**：
- 类或接口的解析
- 字段的解析
- 类方法的解析
- 接口方法的解析
- 方法类型的解析
- 方法句柄的解析
- 调用点限定符的解析

---

#### 3. 初始化（Initialization）

**任务**：执行类构造器 `<clinit>()` 方法

**`<clinit>()` 方法**：
- 由编译器自动收集类中的所有类变量的赋值动作和静态代码块合并而成
- 不需要定义，编译器自动生成
- JVM保证 `<clinit>()` 的线程安全性

**示例**：
```java
public class InitExample {
    public static int value = 123;  // ①
    
    static {  // ②
        System.out.println("静态代码块");
        value = 456;
    }
    
    public static void main(String[] args) {
        System.out.println(value);  // 输出456
    }
}

// 编译器生成的<clinit>()大致如下：
// <clinit>() {
//     value = 123;           // ①
//     System.out.println(...); // ②
//     value = 456;            // ②
// }
```

**注意事项**：
1. 静态代码块只能访问定义在它之前的静态变量
2. 父类的 `<clinit>()` 先于子类执行
3. 接口的 `<clinit>()` 不需要先执行父接口的
4. `<clinit>()` 只会执行一次

---

## **类加载器有哪些？** (启动、扩展、应用、自定义)

JVM提供了多层级的类加载器：

### 类加载器层次结构

```
Bootstrap ClassLoader (C++实现)
    ↓ 委派
Extension ClassLoader
    ↓ 委派
Application ClassLoader
    ↓ 委派
Custom ClassLoader (用户自定义)
```

---

### 1. Bootstrap ClassLoader（启动类加载器）

**特点**：
- 用C++实现，是JVM的一部分
- 负责加载核心类库
- 无法被Java程序直接引用

**加载范围**：
- `<JAVA_HOME>/lib` 目录
- 或被 `-Xbootclasspath` 参数指定的路径
- 识别文件名（如 rt.jar、tools.jar）

**示例**：
```java
// String类由Bootstrap ClassLoader加载
System.out.println(String.class.getClassLoader());  // null
```

**注意**：返回null表示由Bootstrap ClassLoader加载

---

### 2. Extension ClassLoader（扩展类加载器）

**特点**：
- Java实现，sun.misc.Launcher$ExtClassLoader
- 继承自 ClassLoader

**加载范围**：
- `<JAVA_HOME>/lib/ext` 目录
- 或被 `java.ext.dirs` 系统变量指定的路径

**示例**：
```java
// 查看扩展目录
System.out.println(System.getProperty("java.ext.dirs"));
```

---

### 3. Application ClassLoader（应用程序类加载器）

**特点**：
- Java实现，sun.misc.Launcher$AppClassLoader
- 继承自 ClassLoader
- 也称为 System ClassLoader

**加载范围**：
- 用户类路径（classpath）
- `-cp` 或 `-classpath` 参数指定的路径

**示例**：
```java
// 我们写的类通常由Application ClassLoader加载
System.out.println(MyClass.class.getClassLoader());
// 输出：sun.misc.Launcher$AppClassLoader@xxx
```

---

### 4. Custom ClassLoader（自定义类加载器）

**为什么需要自定义？**
- 加密的Class文件
- 从网络加载类
- 热部署
- 模块化
- 隔离不同版本的类

**如何自定义？**

```java
public class MyClassLoader extends ClassLoader {
    private String classPath;
    
    public MyClassLoader(String classPath) {
        this.classPath = classPath;
    }
    
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            // 1. 读取class文件
            byte[] data = loadClassData(name);
            
            // 2. 定义类
            return defineClass(name, data, 0, data.length);
        } catch (Exception e) {
            throw new ClassNotFoundException(name);
        }
    }
    
    private byte[] loadClassData(String name) throws IOException {
        // 将类名转换为文件路径
        String path = classPath + "/" + name.replace('.', '/') + ".class";
        
        // 读取文件
        InputStream is = new FileInputStream(path);
        ByteArrayOutputStream baos = new ByteArrayOutputStream();
        
        int b;
        while ((b = is.read()) != -1) {
            baos.write(b);
        }
        
        is.close();
        return baos.toByteArray();
    }
}
```

**使用**：
```java
MyClassLoader loader = new MyClassLoader("/path/to/classes");
Class<?> clazz = loader.loadClass("com.example.MyClass");
Object obj = clazz.newInstance();
```

---

### 类加载器的关系

```java
// 查看类加载器的父子关系
ClassLoader appLoader = MyClass.class.getClassLoader();
ClassLoader extLoader = appLoader.getParent();
ClassLoader bootLoader = extLoader.getParent();

System.out.println(appLoader);   // AppClassLoader
System.out.println(extLoader);   // ExtClassLoader
System.out.println(bootLoader);  // null (Bootstrap)
```

**注意**：
- 这里的"父子"是组合关系，不是继承关系
- ExtClassLoader的parent是null，但实际委派给Bootstrap

## **能说一下类的生命周期吗？** (加载、连接、初始化、使用、卸载)

类的完整生命周期包括7个阶段：

```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
         ↑_______________↑
              链接(Linking)
```

---

### 1. 加载（Loading）

- 查找并加载类的二进制数据
- 生成Class对象

---

### 2. 验证（Verification）

- 确保Class文件符合JVM规范
- 保证虚拟机安全

---

### 3. 准备（Preparation）

- 为静态变量分配内存
- 设置默认值

---

### 4. 解析（Resolution）

- 符号引用 → 直接引用

---

### 5. 初始化（Initialization）

- 执行 `<clinit>()` 方法
- 给静态变量赋正确的值
- 执行静态代码块

---

### 6. 使用（Using）

- 创建对象实例
- 访问静态字段
- 调用静态方法
- 反射调用

---

### 7. 卸载（Unloading）

**类卸载的条件**（必须同时满足）：
1. 该类所有的实例都已经被回收
2. 加载该类的ClassLoader已经被回收
3. 该类对应的 `java.lang.Class` 对象没有在任何地方被引用

**注意**：
- 类卸载不像对象回收那么频繁
- JVM可以选择不卸载类
- 大量动态生成类的场景（如OSGi）会用到

---

### 各阶段的特点

| 阶段 | 是否可控制 | 是否可逆 | 说明 |
|------|-----------|---------|------|
| 加载 | 部分 | 否 | 可通过自定义类加载器控制 |
| 验证 | 否 | 否 | JVM自动完成 |
| 准备 | 否 | 否 | JVM自动完成 |
| 解析 | 部分 | 否 | 可在初始化后继续进行 |
| 初始化 | 部分 | 否 | 可触发，但只执行一次 |
| 使用 | 是 | 否 | 应用程序控制 |
| 卸载 | 部分 | 否 | 满足条件后可能发生 |

## **什么是双亲委派模型？为什么要用它？**

双亲委派模型是Java类加载的核心机制。

### 工作原理

```
当一个类加载器收到类加载请求时：

1. 首先检查是否已经加载过
2. 如果没有，委托给父类加载器
3. 父类加载器再委托给它的父类
4. 直到顶层的Bootstrap ClassLoader
5. 如果父类加载器无法完成，子类才尝试自己加载
```

**示意图**：
```
请求加载 MyClass
       ↓
Application ClassLoader
       ↓ 委派
Extension ClassLoader
       ↓ 委派
Bootstrap ClassLoader
       ↓ 无法加载（不在lib目录）
       ↓ 返回
Extension ClassLoader
       ↓ 无法加载（不在ext目录）
       ↓ 返回
Application ClassLoader
       ↓ 从classpath加载
       ↓ 成功！
```

---

### 代码实现

`ClassLoader` 的 `loadClass()` 方法实现了双亲委派：

```java
protected Class<?> loadClass(String name, boolean resolve)
    throws ClassNotFoundException
{
    synchronized (getClassLoadingLock(name)) {
        // 1. 检查是否已经加载
        Class<?> c = findLoadedClass(name);
        
        if (c == null) {
            long t0 = System.nanoTime();
            try {
                // 2. 委托给父类加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    // 3. 父类为null，委托给Bootstrap
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // 父类无法加载
            }
            
            if (c == null) {
                // 4. 父类无法加载，自己尝试
                long t1 = System.nanoTime();
                c = findClass(name);
                
                // 记录统计信息
                sun.misc.PerfCounter.getParentDelegationTime().addTime(t1 - t0);
                sun.misc.PerfCounter.getFindClassTime().addElapsedTimeFrom(t1);
                sun.misc.PerfCounter.getFindClasses().increment();
            }
        }
        
        if (resolve) {
            resolveClass(c);
        }
        
        return c;
    }
}
```

---

### 为什么要用双亲委派？

#### 1. 安全性

**防止核心API被篡改**：

```java
// 恶意代码：试图替换String类
package java.lang;

public class String {
    // 恶意代码
    public static void stealPassword() {
        // 窃取密码
    }
}
```

**双亲委派的作用**：
- 加载 `java.lang.String` 时
- 委派到Bootstrap ClassLoader
- Bootstrap从rt.jar加载真正的String
- 恶意代码不会被加载

---

#### 2. 避免重复加载

**保证类的唯一性**：

```java
// 如果没有双亲委派
ClassLoader loader1 = new MyClassLoader();
ClassLoader loader2 = new MyClassLoader();

Class<?> clazz1 = loader1.loadClass("MyClass");
Class<?> clazz2 = loader2.loadClass("MyClass");

// clazz1 和 clazz2 是不同的类！
// 会导致类型转换异常
clazz1 != clazz2  // true
```

**有了双亲委派**：
- 无论哪个加载器请求
- 最终都由同一个加载器加载
- 保证类的唯一性

---

#### 3. 保证核心类的稳定性

- 核心类库由Bootstrap加载
- 不会被用户代码影响
- 保证JVM稳定运行

---

### 双亲委派的优点

1. **安全性**：防止核心API被篡改
2. **唯一性**：避免类的重复加载
3. **稳定性**：保证核心类的稳定
4. **层次性**：清晰的加载层次

---

## **如何破坏双亲委派机制？** (SPI、热部署等场景)

虽然双亲委派很好，但有些场景需要打破它。

### 破坏方式

#### 1. 重写 loadClass() 方法

```java
public class BreakDelegateClassLoader extends ClassLoader {
    @Override
    protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException
    {
        // 不委派给父类，直接自己加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            c = findClass(name);  // 直接加载，不委派
        }
        if (resolve) {
            resolveClass(c);
        }
        return c;
    }
}
```

---

#### 2. 使用线程上下文类加载器

这是最常见的"合法"破坏方式。

**场景**：SPI（Service Provider Interface）

**问题**：
```
JDBC驱动由Bootstrap加载（java.sql.Driver接口）
但具体的驱动实现（如MySQL驱动）由Application加载

Bootstrap无法加载Application的类！
```

**解决方案**：线程上下文类加载器

```java
// DriverManager源码（简化）
public class DriverManager {
    public static Connection getConnection(String url) {
        // 获取当前线程的上下文类加载器
        ClassLoader cl = Thread.currentThread().getContextClassLoader();
        
        // 使用它来加载驱动
        ServiceLoader<Driver> loadedDrivers = ServiceLoader.load(Driver.class, cl);
        
        // ...
    }
}
```

**设置上下文类加载器**：
```java
// Tomcat等容器会设置
Thread.currentThread().setContextClassLoader(myClassLoader);
```

---

#### 3. OSGi模块化

**特点**：
- 每个模块有自己的类加载器
- 模块间可以隔离
- 可以加载不同版本的同一个类

**实现**：
- 复杂的类加载器网络
- 不完全遵循双亲委派
- 模块间协商加载

---

#### 4. 热部署

**场景**：
- 不停机更新代码
- 重新加载修改后的类

**实现**：
```java
public class HotDeployClassLoader extends ClassLoader {
    public Class<?> reloadClass(String name) throws Exception {
        // 创建新的类加载器
        HotDeployClassLoader newLoader = new HotDeployClassLoader();
        
        // 重新加载类
        return newLoader.findClass(name);
    }
}
```

**框架**：
- Spring Boot DevTools
- JRebel
- Arthas redefine

---

### SPI示例

**完整的SPI例子**：

1. **定义接口**
   ```java
   // 在API模块中
   package com.example.api;
   
   public interface DatabaseDriver {
       void connect();
   }
   ```

2. **实现接口**
   ```java
   // 在MySQL模块中
   package com.example.mysql;
   
   public class MySQLDriver implements DatabaseDriver {
       @Override
       public void connect() {
           System.out.println("Connect to MySQL");
       }
   }
   ```

3. **配置文件**
   ```
   META-INF/services/com.example.api.DatabaseDriver
   ```
   内容：
   ```
   com.example.mysql.MySQLDriver
   ```

4. **加载服务**
   ```java
   ServiceLoader<DatabaseDriver> loaders = 
       ServiceLoader.load(DatabaseDriver.class);
   
   for (DatabaseDriver driver : loaders) {
       driver.connect();  // Connect to MySQL
   }
   ```

**原理**：
- `ServiceLoader` 使用线程上下文类加载器
- 打破了双亲委派
- 允许API模块加载实现模块的类

---

### 破坏双亲委派的风险

1. **安全性降低**
   - 可能被注入恶意代码
   - 核心类可能被替换

2. **类冲突**
   - 同一个类被多次加载
   - 类型转换异常

3. **调试困难**
   - 类加载逻辑复杂
   - 问题难以定位

**建议**：
- 除非必要，不要破坏双亲委派
- 如果必须破坏，要做好隔离
- 充分测试

---

## **说说解释执行和编译执行的区别？** (JIT编译器原理)

JVM执行字节码有两种方式：解释执行和编译执行。

### 1. 解释执行（Interpretation）

**原理**：
- 逐行读取字节码
- 翻译成机器码执行
- 每次执行都要翻译

**示意图**：
```
字节码: aload_0 → 解释器 → 机器码: mov eax, [ebp+8]
字节码: iconst_1 → 解释器 → 机器码: mov ebx, 1
字节码: iadd     → 解释器 → 机器码: add eax, ebx
```

**优点**：
- 启动快
- 无需编译等待
- 占用内存少

**缺点**：
- 执行慢
- 重复翻译相同代码

---

### 2. 编译执行（Compilation）

**原理**：
- 将字节码编译为本地机器码
- 直接执行机器码
- 编译结果缓存，可重复使用

**示意图**：
```
字节码 → JIT编译器 → 机器码（缓存）→ 直接执行
```

**优点**：
- 执行快
- 可以做深度优化

**缺点**：
- 编译需要时间
- 占用更多内存

---

### 3. 混合模式（Hybrid）

**HotSpot采用混合模式**：
- 开始时解释执行
- 热点代码编译执行
- 兼顾启动速度和执行效率

**热点代码判定**：

**（1）方法计数器**
- 记录方法被调用的次数
- 超过阈值则编译

**（2）回边计数器**
- 记录循环体执行的次数
- 超过阈值则编译

---

### JIT编译器

HotSpot有两个JIT编译器：

#### C1编译器（Client Compiler）

**特点**：
- 快速编译
- 优化较少
- 适合短生命周期应用

**优化**：
- 方法内联
- 空值检查消除
- 范围检查消除

---

#### C2编译器（Server Compiler）

**特点**：
- 慢速编译
- 优化更多
- 适合长生命周期应用

**优化**：
- 逃逸分析
- 锁消除
- 循环展开
- 公共子表达式消除

---

### 分层编译（Tiered Compilation）

**JDK 7引入，JDK 8默认开启**

**5个层次**：

```
Level 0: 解释执行
Level 1: C1编译，不带profiling
Level 2: C1编译，带基本profiling
Level 3: C1编译，带完整profiling
Level 4: C2编译
```

**工作流程**：
```
方法首次调用
    ↓
Level 0: 解释执行（收集profiling信息）
    ↓ 调用次数增加
Level 3: C1编译（带profiling）
    ↓ 成为热点
Level 4: C2编译（最优化的代码）
```

**优点**：
- 结合C1的快速和C2的优化
- 根据运行情况动态调整
- 性能最优

---

### 常见优化技术

#### 1. 方法内联（Method Inlining）

**原理**：
- 将被调用方法的代码复制到调用处
- 消除方法调用开销

**示例**：
```java
// 优化前
int result = add(a, b);

int add(int x, int y) {
    return x + y;
}

// 优化后
int result = a + b;  // 直接内联
```

**条件**：
- 方法足够小（默认<35字节）
- 非虚方法（private、static、final）
- 或者经过CHA分析确定唯一实现

---

#### 2. 逃逸分析（Escape Analysis）

**原理**：
- 分析对象是否逃逸出方法或线程
- 未逃逸的对象可以优化

**优化**：
- 栈上分配
- 标量替换
- 同步消除

**示例**：
```java
public void method() {
    Object obj = new Object();  // 未逃逸
    obj.toString();
}  // 可以在栈上分配，或直接消除
```

---

#### 3. 锁消除（Lock Elimination）

**原理**：
- 如果对象不会逃逸，不可能有竞争
- 可以消除同步锁

**示例**：
```java
public void method() {
    StringBuffer sb = new StringBuffer();
    sb.append("hello");  // StringBuffer是synchronized
    sb.append(" world"); // 但sb未逃逸，锁可以被消除
}
```

---

#### 4. 公共子表达式消除

**原理**：
- 如果一个表达式已经计算过
- 且操作数没有变化
- 可以直接使用之前的结果

**示例**：
```java
// 优化前
int a = b * c + d;
int e = b * c + f;

// 优化后
temp = b * c;
int a = temp + d;
int e = temp + f;
```

---

#### 5. 数组边界检查消除

**原理**：
- 如果能确定索引不会越界
- 可以消除边界检查

**示例**：
```java
// 优化前
for (int i = 0; i < arr.length; i++) {
    sum += arr[i];  // 每次都要检查边界
}

// 优化后
// 编译器知道i不会越界，消除检查
```

### 性能对比

```java
public class PerformanceTest {
    public static void main(String[] args) {
        // 预热（让JIT编译）
        for (int i = 0; i < 10000; i++) {
            test();
        }
        
        // 正式测试
        long start = System.currentTimeMillis();
        for (int i = 0; i < 1000000; i++) {
            test();
        }
        long end = System.currentTimeMillis();
        
        System.out.println("耗时：" + (end - start) + "ms");
    }
    
    public static int test() {
        int sum = 0;
        for (int i = 0; i < 100; i++) {
            sum += i;
        }
        return sum;
    }
}
```

**结果**：
- 第一次运行：较慢（解释执行）
- 预热后：快很多（JIT编译）
- 可能相差10倍以上

## 常见问题

### Q1: 为什么String类的getClassLoader()返回null？

A: String由Bootstrap ClassLoader加载，它是C++实现的，不是Java对象，所以返回null。

---

### Q2: 如何判断一个类是由哪个类加载器加载的？

```java
Class<?> clazz = MyClass.class;
ClassLoader loader = clazz.getClassLoader();
System.out.println(loader);

// null → Bootstrap
// sun.misc.Launcher$ExtClassLoader → Extension
// sun.misc.Launcher$AppClassLoader → Application
```

---

### Q3: 双亲委派可以完全避免类冲突吗？

A: 不能完全避免。如果破坏了双亲委派（如OSGi、热部署），仍然可能出现类冲突。需要做好隔离。

---

### Q4: JIT编译会影响程序的确定性吗？

A: 理论上会。同样的代码，不同次运行可能因为JIT编译的时机不同而性能略有差异。但对于业务逻辑，不影响正确性。

---

### Q5: 如何禁用JIT编译？

```bash
# 强制解释执行
-Djava.compiler=NONE

# 或
-Xint
```

**注意**：仅用于调试，生产环境不要禁用。
