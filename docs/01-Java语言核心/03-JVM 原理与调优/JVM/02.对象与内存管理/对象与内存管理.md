# 对象与内存管理

## **对象创建的过程了解吗？**

当我们在Java代码中执行 `new Object()` 时，JVM内部经历了以下过程：

### 对象创建的5个步骤

#### 1. 类加载检查
- JVM首先检查这个类是否已经被加载、解析和初始化
- 如果没有，先执行类加载过程

```java
// 第一次创建User对象时，会先加载User类
User user = new User();
```

#### 2. 分配内存
- 在堆中为对象分配一块确定大小的内存
- 内存大小在类加载完成后就可以确定

**内存分配的两种方式：**

**（1）指针碰撞（Bump the Pointer）**
- 适用场景：堆内存规整（使用标记-整理算法）
- 原理：用指针指示分界点，分配时指针向空闲区移动对象大小
- 比喻：像排队一样，一个接一个

```
已用内存 |←指针→| 空闲内存
---------|-------|----------
```

**（2）空闲列表（Free List）**
- 适用场景：堆内存不规整（使用标记-清除算法）
- 原理：维护一个列表，记录哪些内存块可用
- 分配时从列表中找到足够大的空间

```
空闲块1(64B) → 空闲块2(128B) → 空闲块3(256B)
```

#### 3. 内存空间初始化
- 将分配的内存空间初始化为零值（不包括对象头）
- 这保证了对象的实例字段可以不赋值就直接使用

```java
public class User {
    private int age;      // 默认值为0
    private String name;  // 默认值为null
}
User user = new User();  // age=0, name=null
```

#### 4. 设置对象头
- 设置对象的哈希码
- 设置GC分代年龄
- 设置锁信息
- 设置类型指针（指向类元数据）

#### 5. 执行 `<init>` 方法
- 按照程序员的意愿对对象进行初始化
- 执行构造方法

```java
public class User {
    private String name;
    
    public User(String name) {
        this.name = name;  // <init> 方法中执行
    }
}
User user = new User("张三");  // 最后一步：执行构造方法
```

## **对象的销毁过程了解吗？**

Java对象的销毁由垃圾收集器（GC）自动管理，程序员无法直接控制。

### 对象销毁的流程

#### 1. 对象变为不可达
- 没有任何引用指向该对象
- GC Roots不可达

```java
Object obj = new Object();  // 对象可达
obj = null;                 // 对象变为不可达，可以被回收
```

#### 2. 第一次标记
- GC发现对象不可达，进行第一次标记
- 判断是否需要执行 `finalize()` 方法

#### 3. 执行 finalize()（如果必要）
- 如果对象重写了finalize()且未被调用过
- JVM会创建一个F-Queue队列，由低优先级线程执行
- **注意**：finalize()最多只会被执行一次

```java
public class FinalizeExample {
    @Override
    protected void finalize() throws Throwable {
        System.out.println("对象被回收前执行");
        // 不推荐在此做资源释放，应该用try-finally或try-with-resources
    }
}
```

#### 4. 第二次标记
- 如果在finalize()中对象重新与引用链关联，则复活
- 否则进行第二次标记，准备回收

#### 5. 内存回收
- 真正回收对象占用的内存

**重要提醒：**
- `finalize()` 方法不推荐使用，已被标记为Deprecated
- 应该使用 try-with-resources 或显式的 close() 方法

## **堆内存是如何分配的？** (指针碰撞、空闲列表)

### 1. 指针碰撞（Bump the Pointer）

**前提条件**：堆内存绝对规整（所有用过的内存在一边，空闲的在另一边）

**工作原理**：
- 用一个指针作为分界点的指示器
- 分配内存时，只需将指针向空闲空间方向移动对象大小的距离

**优点**：
- 速度快，只需要移动指针
- 实现简单

**缺点**：
- 要求内存规整，需要配合标记-整理算法

**示意图**：
```
分配前：
[已用内存][已用内存][已用内存]|指针|[空闲内存][空闲内存]

分配一个32字节对象后：
[已用内存][已用内存][已用内存][新对象32B]|指针|[空闲内存]
```

### 2. 空闲列表（Free List）

**前提条件**：堆内存不规整（已用和空闲内存交错分布）

**工作原理**：
- JVM维护一个列表，记录哪些内存块是可用的
- 分配时从列表中找到一块足够大的空间划分给对象实例
- 更新列表记录

**优点**：
- 可以处理不规整的内存
- 配合标记-清除算法

**缺点**：
- 需要额外的数据结构维护
- 查找合适空间可能较慢

**示意图**：
```
空闲列表：
块1: 地址0x1000, 大小64B
块2: 地址0x2000, 大小128B
块3: 地址0x3000, 大小256B

分配一个100B的对象：
→ 选择块2（128B）
→ 分配100B，剩余28B放回空闲列表
```

### 3. 如何选择分配方式？

取决于使用的垃圾收集算法：
- **标记-整理算法**：内存规整 → 指针碰撞
- **标记-清除算法**：内存不规整 → 空闲列表

**HotSpot默认情况**：
- 新生代使用复制算法，内存规整 → 指针碰撞
- 老年代使用标记-整理，内存规整 → 指针碰撞

## **new 对象时，堆会发生抢占吗？JVM 怎么解决？** (TLAB)

### 问题背景

堆是线程共享的，多个线程同时new对象时，可能会出现并发问题：
- 两个线程分配到同一块内存
- 指针移动的原子性问题

### 解决方案：TLAB（Thread Local Allocation Buffer）

**TLAB是什么？**
- 每个线程在Eden区中分配一小块私有内存区域
- 线程优先在自己的TLAB中分配对象
- 避免多线程竞争

**工作原理**：
```
Eden区
┌─────────────────────────────────────┐
│ TLAB1(线程1) │ TLAB2(线程2) │ 剩余  │
│  私有区域     │  私有区域     │ 空间  │
└─────────────────────────────────────┘
```

**分配流程**：
1. 线程尝试在自己的TLAB中分配对象
2. 如果TLAB空间足够，直接分配（无锁，非常快）
3. 如果TLAB空间不足，同步锁定，重新分配TLAB或在Eden区分配

**优点**：
- 大多数情况下无锁分配，性能高
- 减少线程间的竞争

**配置参数**：
```bash
-XX:+UseTLAB              # 启用TLAB（默认开启）
-XX:TLABSize=64k          # 设置TLAB大小
-XX:+PrintTLAB            # 打印TLAB信息
```

**示例验证**：
```java
// 多线程并发创建对象
public class TLABTest {
    public static void main(String[] args) throws Exception {
        Thread[] threads = new Thread[10];
        for (int i = 0; i < 10; i++) {
            threads[i] = new Thread(() -> {
                for (int j = 0; j < 100000; j++) {
                    new Object();  // 在各自TLAB中分配，无竞争
                }
            });
            threads[i].start();
        }
        
        for (Thread t : threads) {
            t.join();
        }
        System.out.println("完成！");
    }
}
```

## **能说一下对象的内存布局吗？** (对象头、实例数据、对齐填充)

在HotSpot虚拟机中，对象在内存中存储的布局可以分为3块区域：

### 对象内存布局

```
┌──────────────────────────┐
│      对象头（Header）      │ ← 包含Mark Word和类型指针
├──────────────────────────┤
│    实例数据（Data）        │ ← 对象真正的有效信息
├──────────────────────────┤
│    对齐填充（Padding）     │ ← 凑够8字节的整数倍
└──────────────────────────┘
```

### 1. 对象头（Header）

对象头包含两部分信息：

#### （1）Mark Word（标记字段）
存储对象自身的运行时数据，如：
- 哈希码（HashCode）
- GC分代年龄
- 锁状态标志
- 线程持有的锁
- 偏向线程ID
- 偏向时间戳

**Mark Word在不同状态下的内容：**

| 锁状态 | 25bit | 4bit | 1bit是否是偏向锁 | 2bit锁标志位 |
|--------|-------|------|-----------------|-------------|
| 无锁 | 对象的hashCode | 对象分代年龄 | 0 | 01 |
| 偏向锁 | 线程ID | Epoch | 1 | 01 |
| 轻量级锁 | 指向栈中锁记录的指针 | - | - | 00 |
| 重量级锁 | 指向互斥量（重量级锁）的指针 | - | - | 10 |
| GC标记 | - | - | - | 11 |

#### （2）类型指针（Klass Pointer）
- 指向它的类元数据的指针
- JVM通过这个指针确定该对象是哪个类的实例
- **注意**：如果对象是数组，还需要存储数组长度

### 2. 实例数据（Instance Data）

- 对象真正存储的有效信息
- 程序代码中所定义的各种类型的字段内容
- 包括从父类继承下来的和本身定义的字段

**存储顺序策略**（HotSpot默认）：
1. longs/doubles
2. ints
3. shorts/chars
4. bytes/booleans
5. oops (Ordinary Object Pointers)

这样排列可以减少内存占用（对齐填充更少）

### 3. 对齐填充（Padding）

- 不是必然存在的
- HotSpot VM要求对象大小必须是8字节的整数倍
- 如果对象头+实例数据不是8的倍数，需要填充

**示例计算：**
```java
public class ObjectSize {
    private int id;        // 4字节
    private String name;   // 引用4字节（压缩指针）
}
// 对象头：12字节（Mark Word 8 + Klass Pointer 4，64位开启压缩）
// 实例数据：8字节（int 4 + 引用 4）
// 总计：20字节
// 对齐填充：4字节（凑够24字节，8的倍数）
// 最终大小：24字节
```

**查看对象大小工具：**
```xml
<!-- pom.xml 添加依赖 -->
<dependency>
    <groupId>org.openjdk.jol</groupId>
    <artifactId>jol-core</artifactId>
    <version>0.16</version>
</dependency>
```

```java
import org.openjdk.jol.info.ClassLayout;

public class ObjectSizeDemo {
    public static void main(String[] args) {
        Object obj = new Object();
        System.out.println(ClassLayout.parseInstance(obj).toPrintable());
    }
}
```

输出示例：
```
java.lang.Object object internals:
 OFFSET  SIZE   TYPE DESCRIPTION               VALUE
      0     4        (object header)           01 00 00 00
      4     4        (object header)           00 00 00 00
      8     4        (object header)           e5 01 00 f8
     12     4        (loss due to the next object alignment)
Instance size: 16 bytes
```

## **new Object() 对象的内存大小是多少？对象的引用大小呢？**

### new Object() 的大小

在64位JVM上：

**情况1：开启指针压缩（默认开启）**
- Mark Word：8字节
- Klass Pointer：4字节（压缩后）
- 对齐填充：4字节
- **总计：16字节**

**情况2：关闭指针压缩**
- Mark Word：8字节
- Klass Pointer：8字节
- **总计：16字节**（刚好是8的倍数，无需填充）

```bash
# 查看指针压缩是否开启
java -XX:+PrintCommandLineFlags -version
# 看到 -XX:+UseCompressedOops 表示开启
```

### 对象引用的大小

**引用是指向对象的指针，不是对象本身！**

| JVM位数 | 指针压缩 | 引用大小 |
|---------|---------|----------|
| 32位 | - | 4字节 |
| 64位 | 开启（默认） | 4字节 |
| 64位 | 关闭 | 8字节 |

**示例：**
```java
Object obj = new Object();
// obj是引用，占4字节（64位开启压缩）
// new Object()是对象，占16字节
```

### 常见对象大小计算

```java
// 示例1：空对象
new Object()  // 16字节

// 示例2：只有一个int字段
class A {
    int x;  // 4字节
}
// 对象头12 + int 4 = 16字节（无需填充）

// 示例3：两个int字段
class B {
    int x;  // 4字节
    int y;  // 4字节
}
// 对象头12 + int 8 = 20 → 填充4 = 24字节

// 示例4：String对象（简化）
String s = "hello";
// String对象本身约40-50字节（包含char数组引用、hash等）
// char数组另算
```

**使用JOL工具验证：**
```java
import org.openjdk.jol.info.ClassLayout;

public class ObjectSizeCalc {
    static class MyObject {
        int id;
        String name;
    }
    
    public static void main(String[] args) {
        MyObject obj = new MyObject();
        System.out.println(ClassLayout.parseInstance(obj).toPrintable());
    }
}
```

## **JVM 怎么访问对象的？** (句柄 vs 直接指针)

JVM提供了两种对象访问方式：

### 1. 句柄访问（Handle Access）

**原理**：
- Java堆中划分出一块内存作为句柄池
- reference中存储的是对象的句柄地址
- 句柄中包含对象实例数据和类型数据的具体地址

**示意图**：
```
栈中的reference
       ↓
   ┌───────┐
   │ 句柄池 │ ← 句柄地址
   └───┬───┘
       ↓
   ┌───────┐     ┌──────────┐
   │实例数据│     │类型数据  │
   └───────┘     └──────────┘
```

**优点**：
- reference中存储的是稳定的句柄地址
- 对象被移动时（GC时），只需改变句柄中的实例数据指针
- reference本身不需要修改

**缺点**：
- 多了一次指针定位，开销较大

### 2. 直接指针访问（Direct Pointer，HotSpot采用）

**原理**：
- reference中直接存储对象地址
- 对象头中包含类型数据指针

**示意图**：
```
栈中的reference
       ↓
   ┌──────────┐
   │对象实例   │ ← 直接指向对象
   │ ├─类型指针─→ 类型数据
   │ └─实例数据
   └──────────┘
```

**优点**：
- 速度更快，节省了一次指针定位的时间开销
- HotSpot主要使用这种方式

**缺点**：
- 对象被移动时，reference也需要修改

### 对比总结

| 对比项 | 句柄访问 | 直接指针 |
|--------|---------|---------|
| 访问速度 | 较慢（两次定位） | 较快（一次定位） |
| 对象移动 | reference不变 | reference需修改 |
| 稳定性 | 更稳定 | 相对不稳定 |
| 采用者 | 较少 | HotSpot默认 |

**为什么HotSpot选择直接指针？**
- 对象访问非常频繁，性能至关重要
- 现代GC算法（如G1）可以减少对象移动
- 性能优势明显

## **说说 Java 的四种引用？** (强、软、弱、虚)

Java中有四种引用强度，从强到弱依次为：

### 1. 强引用（Strong Reference）

**特点**：
- 最常见的引用形式
- 只要强引用还存在，GC永远不会回收对象
- 即使内存不足，也只会抛出OOM，不会回收强引用对象

**示例**：
```java
Object obj = new Object();  // 强引用
// 只要obj还指向这个对象，就不会被GC
obj = null;  // 断开引用，对象可以被回收
```

### 2. 软引用（Soft Reference）

**特点**：
- 描述还有用但非必需的对象
- 只有在内存不足时才会被回收
- 适合做缓存

**示例**：
```java
import java.lang.ref.SoftReference;

// 创建软引用
Object obj = new Object();
SoftReference<Object> softRef = new SoftReference<>(obj);
obj = null;  // 断开强引用

// 获取对象
Object cached = softRef.get();
if (cached != null) {
    System.out.println("缓存命中");
} else {
    System.out.println("缓存已被回收，需要重新创建");
}
```

**应用场景**：
- 浏览器缓存
- 图片缓存
- 数据库连接池

### 3. 弱引用（Weak Reference）

**特点**：
- 描述非必需对象
- 强度比软引用更弱
- **无论内存是否充足，只要发生GC就会被回收**
- 生命周期很短

**示例**：
```java
import java.lang.ref.WeakReference;

Object obj = new Object();
WeakReference<Object> weakRef = new WeakReference<>(obj);
obj = null;  // 断开强引用

// 下一次GC时，weakRef.get() 将返回 null
System.gc();
Object result = weakRef.get();  // 很可能为null
```

**应用场景**：
- ThreadLocal中的key
- WeakHashMap的key
- 防止内存泄漏

**ThreadLocal内存泄漏案例**：
```java
// ThreadLocalMap的Entry继承自WeakReference
static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value;
    Entry(ThreadLocal<?> k, Object v) {
        super(k);  // key是弱引用
        value = v; // value是强引用
    }
}
// 如果ThreadLocal没有外部强引用，key会被回收
// 但value还是强引用，可能导致内存泄漏
// 所以使用完要调用 remove()
```

### 4. 虚引用（Phantom Reference）

**特点**：
- 最弱的引用关系
- **无法通过虚引用获取对象实例**
- 唯一目的：对象被回收时收到系统通知
- 必须和引用队列（ReferenceQueue）联合使用

**示例**：
```java
import java.lang.ref.PhantomReference;
import java.lang.ref.ReferenceQueue;

Object obj = new Object();
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(obj, queue);

obj = null;  // 断开强引用

// 无法通过phantomRef.get()获取对象（永远返回null）
// 当对象被回收时，phantomRef会被加入queue
```

**应用场景**：
- 对象回收监控
- 直接内存管理（如NIO的DirectByteBuffer）

### 四种引用对比表

| 引用类型 | 回收时机 | 获取对象 | 典型应用 |
|---------|---------|---------|----------|
| 强引用 | 永不回收 | 可以 | 普通对象 |
| 软引用 | 内存不足时 | 可以 | 缓存 |
| 弱引用 | GC时 | 可以 | ThreadLocal、WeakHashMap |
| 虚引用 | GC时 | 不可以 | 对象回收监控 |

### 引用队列（ReferenceQueue）

软引用、弱引用、虚引用都可以关联引用队列：

```java
ReferenceQueue<Object> queue = new ReferenceQueue<>();
SoftReference<Object> softRef = new SoftReference<>(obj, queue);

// 当对象被回收时，softRef会被加入queue
Reference<? extends Object> ref = queue.poll();
if (ref != null) {
    System.out.println("对象已被回收");
}
```

## **Java 堆的内存分区了解吗？** (新生代、老年代、Eden、Survivor)

Java堆是GC的主要区域，为了提高GC效率，将堆分为不同的区域：

### 堆内存结构

```
Java Heap
├── 新生代（Young Generation）  ← 存放新创建的对象
│   ├── Eden区                 ← 新对象首先在这里分配
│   ├── Survivor From (S0)     ← 幸存区1
│   └── Survivor To (S1)       ← 幸存区2
│
└── 老年代（Old Generation）   ← 存放长期存活的对象
    └── （JDK 1.7之前还有永久代，1.8改为元空间）
```

### 1. 新生代（Young Generation）

**特点**：
- 存放新创建的对象
- 大多数对象朝生夕死
- GC频繁，但速度快
- 默认占堆的1/3

**组成**：
- **Eden区**：新对象首先在这里分配，占新生代的8/10
- **Survivor区**：分为From和To两个区域，各占新生代的1/10

**为什么有两个Survivor区？**
- 用于复制算法
- 每次只用一个，另一个空闲
- 避免内存碎片

### 2. 老年代（Old Generation）

**特点**：
- 存放长期存活的对象
- 对象经过多次GC仍然存活
- GC频率低，但耗时长
- 默认占堆的2/3

### 对象在堆中的流转

```
新对象
  ↓
Eden区
  ↓ (第一次Minor GC)
Survivor区 (年龄+1)
  ↓ (每次Minor GC，年龄+1)
Survivor区 (年龄达到阈值，默认15)
  ↓
老年代
```

**详细流程**：
1. 新对象在Eden区分配
2. Eden区满时，触发Minor GC
3. 存活的对象复制到其中一个Survivor区（假设是S0）
4. 下次GC时，Eden和S0中存活的对象复制到S1
5. 如此反复，每次对象年龄+1
6. 年龄达到阈值（-XX:MaxTenuringThreshold，默认15），进入老年代

**特殊情况**：
- **大对象直接进入老年代**：避免在Eden和Survivor之间大量复制
- **长期存活对象进入老年代**：年龄达到阈值
- **动态年龄判定**：如果Survivor空间中相同年龄所有对象大小总和大于Survivor空间的一半，年龄≥该年龄的对象直接进入老年代

### 配置参数

```bash
# 堆大小
-Xms2g                      # 堆初始大小
-Xmx2g                      # 堆最大大小

# 新生代大小
-Xmn512m                    # 新生代大小
-XX:NewRatio=2              # 老年代/新生代 = 2（新生代占1/3）

# Survivor比例
-XX:SurvivorRatio=8         # Eden/Survivor = 8（Eden占8/10）

# 对象年龄阈值
-XX:MaxTenuringThreshold=15 # 默认15

# 大对象阈值
-XX:PretenureSizeThreshold  # 超过此大小直接进老年代（仅Serial/ParNew有效）
```

### 实际例子

```java
public class HeapExample {
    public static void main(String[] args) {
        // 小对象，先在Eden区
        String str = "hello";
        
        // 大对象，可能直接进入老年代
        byte[] bigArray = new byte[1024 * 1024];  // 1MB
        
        // 长期存活的对象，最终进入老年代
        List<String> cache = new ArrayList<>();
        for (int i = 0; i < 10000; i++) {
            cache.add("item" + i);
        }
    }
}
```

## **对象什么时候会进入老年代？** (长期存活、大对象、动态年龄判定)

对象进入老年代主要有以下几种情况：

### 1. 长期存活的对象（年龄达到阈值）

**机制**：
- 对象在Survivor区每熬过一次Minor GC，年龄就+1
- 当年龄达到阈值（默认15），进入老年代

**配置**：
```bash
-XX:MaxTenuringThreshold=15  # 设置年龄阈值
```

**示例**：
```java
// 这个对象如果一直存活，经过15次Minor GC后会进入老年代
Object obj = new Object();
```

### 2. 大对象直接进入老年代

**什么是大对象？**
- 需要大量连续内存空间的对象
- 如很长的字符串、大数组

**为什么要直接进入老年代？**
- 避免在Eden和Survivor之间大量复制
- 提高性能

**配置**：
```bash
-XX:PretenureSizeThreshold=1048576  # 1MB，超过此大小直接进老年代
```

**注意**：此参数只对Serial和ParNew收集器有效，G1等收集器有自己

的判断逻辑。

**示例**：
```java
// 这个大数组可能直接进入老年代
byte[] bigArray = new byte[2 * 1024 * 1024];  // 2MB
```

### 3. 动态年龄判定

**规则**：
- 如果在Survivor空间中，相同年龄所有对象大小的总和大于Survivor空间的一半
- 那么年龄 ≥ 该年龄的对象可以直接进入老年代
- 无需等到年龄阈值

**举例**：
```
Survivor区大小：10MB
年龄1的对象总大小：3MB
年龄2的对象总大小：3MB
年龄3的对象总大小：3MB

年龄1+2+3 = 9MB > 10MB/2 = 5MB
→ 年龄≥2的对象直接进入老年代
```

**好处**：
- 让老年代提前接纳可能长期存活的对象
- 减少Minor GC的次数

### 4. 空间分配担保

**场景**：
- Minor GC前，JVM检查老年代最大可用连续空间
- 如果大于新生代所有对象总空间，Minor GC安全
- 如果小于，检查是否允许担保失败

**流程**：
```
Minor GC前检查：
├─ 老年代可用空间 > 新生代对象总空间？
│  ├─ 是 → Minor GC安全
│  └─ 否 → 检查HandlePromotionFailure
│     ├─ 允许担保 → 检查老年代可用空间 > 历次晋升平均大小
│     │  ├─ 是 → 冒险Minor GC
│     │  └─ 否 → Full GC
│     └─ 不允许担保 → Full GC
```

**配置**：
```bash
-XX:+HandlePromotionFailure  # JDK 6u24之后默认开启，此参数废弃
```

### 5. 其他情况

**（1）Survivor区放不下**
- Minor GC后，存活对象无法全部放入Survivor区
- 部分对象直接进入老年代

**（2）引用变化**
- 对象在新生代，但被老年代对象引用
- 为了便于管理，可能提前晋升

### 总结图

```
对象进入老年代的途径：

1. 年龄达标 ────────────→ 老年代
   (默认15次GC)

2. 大对象 ───────────────→ 老年代
   (超过阈值)

3. 动态年龄判定 ─────────→ 老年代
   (同龄对象总和>S0一半)

4. Survivor放不下 ──────→ 老年代
   (空间不足)

5. 分配担保失败 ─────────→ Full GC → 老年代
```

## **对象一定分配在堆中吗？** (逃逸分析与栈上分配)

**不一定！** JVM通过逃逸分析，可能将对象分配在栈上或进行其他优化。

### 什么是逃逸分析？

**定义**：
- 分析对象的作用域
- 判断对象是否会逃出当前方法或线程

**逃逸状态**：
1. **全局逃逸**：对象可能被其他线程访问
2. **参数逃逸**：对象作为参数传递给其他方法
3. **无逃逸**：对象只在当前方法中使用

### 逃逸分析的优化

如果对象没有逃逸，JVM可以进行以下优化：

#### 1. 栈上分配（Stack Allocation）

**原理**：
- 将对象分配在栈上，而不是堆上
- 方法执行完，栈帧销毁，对象自动回收
- 无需GC，性能更高

**示例**：
```java
public class StackAllocation {
    public void method() {
        // 这个对象没有逃逸，可能在栈上分配
        User user = new User();
        user.setName("张三");
        user.setAge(25);
        System.out.println(user.getName());
    }  // 方法结束，user自动销毁，无需GC
}
```

**注意**：
- HotSpot目前并未完全实现栈上分配
- 主要通过标量替换来实现类似效果

#### 2. 标量替换（Scalar Replacement）

**标量**：不能再分解的数据，如int、long、reference
**聚合量**：可以继续分解的数据，如对象

**原理**：
- 如果对象没有逃逸，将其拆分为成员变量
- 成员变量直接在栈上分配
- 不再创建对象

**示例**：
```java
public class ScalarReplacement {
    static class Point {
        int x;
        int y;
    }
    
    public void method() {
        // 优化前：创建Point对象
        // Point p = new Point();
        // p.x = 10;
        // p.y = 20;
        
        // 优化后：直接在栈上分配x和y
        int x = 10;
        int y = 20;
        System.out.println(x + "," + y);
    }
}
```

**好处**：
- 减少堆内存使用
- 减轻GC压力
- 提高性能

#### 3. 同步消除（Synchronization Elimination）

**原理**：
- 如果对象没有逃逸，不可能被其他线程访问
- 那么这个对象上的同步锁可以消除

**示例**：
```java
public class SyncElimination {
    public void method() {
        StringBuffer sb = new StringBuffer();
        // StringBuffer的方法是synchronized
        // 但sb没有逃逸，不会被其他线程访问
        // JIT编译器会消除这些同步锁
        sb.append("hello");
        sb.append(" world");
        System.out.println(sb.toString());
    }
}
```

**好处**：
- 减少锁的开销
- 提高并发性能

### 配置参数

```bash
-XX:+DoEscapeAnalysis      # 开启逃逸分析（JDK 6u23+默认开启）
-XX:+EliminateAllocations  # 开启标量替换（默认开启）
-XX:+EliminateLocks        # 开启同步消除（默认开启）
```

### 如何判断对象是否逃逸？

**逃逸的情况**：
```java
public class EscapeExample {
    private static List<Object> list = new ArrayList<>();
    
    // 情况1：赋值给静态变量（全局逃逸）
    public void escape1() {
        Object obj = new Object();
        list.add(obj);  // 逃逸到全局
    }
    
    // 情况2：作为返回值（参数逃逸）
    public Object escape2() {
        Object obj = new Object();
        return obj;  // 逃逸到调用方
    }
    
    // 情况3：作为参数传递（参数逃逸）
    public void escape3() {
        Object obj = new Object();
        otherMethod(obj);  // 可能逃逸
    }
    
    // 情况4：无逃逸（可以优化）
    public void noEscape() {
        Object obj = new Object();
        obj.toString();  // 只在方法内使用
    }
}
```

### 实际测试

```java
public class EscapeTest {
    static class User {
        private String name;
        private int age;
        
        public User(String name, int age) {
            this.name = name;
            this.age = age;

        System.out.println("耗时：" + (end - start) + "ms");
    }
}
```

## 什么是逃逸分析？它会带来什么好处？

### 逃逸分析的定义

逃逸分析（Escape Analysis）是一种编译期优化技术，由JIT编译器执行。它分析对象的动态作用域，判断对象是否会"逃逸"出当前方法或线程。

### 逃逸的三种程度

1. **全局逃逸（Global Escape）**
   - 对象可能被其他线程访问
   - 例如：赋值给静态变量、作为返回值

2. **参数逃逸（Arg Escape）**
   - 对象作为参数传递给其他方法
   - 在其他方法中可能被使用

3. **无逃逸（No Escape）**
   - 对象只在当前方法中使用
   - 不会被其他方法或线程访问
   - **可以进行优化**

### 逃逸分析的好处

#### 1. 减少GC压力
- 通过标量替换，对象不分配在堆上
- 减少堆内存使用
- 降低GC频率

#### 2. 提高性能
- 栈上分配比堆上分配快
- 同步消除减少锁开销
- 缓存友好（栈内存局部性好）

#### 3. 简化编程
- 开发者无需手动优化
- JIT编译器自动处理
- 代码更清晰

### 实际案例对比

**未优化的情况**：
```java
public class BeforeOptimization {
    public void process() {
        // 每次都创建新对象
        Point point = new Point(10, 20);
        int distance = calculate(point);
    }
    
    private int calculate(Point p) {
        return p.x + p.y;
    }
}
```

**逃逸分析优化后**：
```java
public class AfterOptimization {
    public void process() {
        // 对象被拆解，直接在栈上分配
        int x = 10;
        int y = 20;
        int distance = x + y;
    }
}
```

### 注意事项

1. **不是所有对象都能优化**
   - 只有确认无逃逸的对象才能优化
   - 复杂场景下可能无法判断

2. **HotSpot的实现限制**
   - 目前没有完全的栈上分配
   - 主要通过标量替换实现

3. **调试困难**
   - 优化后的代码与原代码不同
   - 调试时可能看不到预期对象

4. **建议**
   - 保持代码简洁，便于JIT优化
   - 避免不必要的对象逃逸
   - 信任JVM的优化能力

## 内存溢出和内存泄漏了解吗？ (区别与原因)

### 概念区别

| 对比项 | 内存溢出（OOM） | 内存泄漏（Memory Leak） |
|--------|---------------|----------------------|
| **定义** | 内存不够用了 | 不再使用的对象没有被回收 |
| **结果** | 程序崩溃 | 最终导致OOM |
| **原因** | 内存需求 > 可用内存 | 对象被意外引用，无法回收 |
| **关系** | 症状 | 病因之一 |

**形象比喻**：
- **内存泄漏**：水龙头没关紧，水慢慢流失
- **内存溢出**：水池满了，水溢出来了
- 内存泄漏久了会导致内存溢出

### 内存溢出的常见原因

#### 1. 堆内存溢出（Java heap space）

**原因**：
- 创建了太多对象
- 对象太大
- 内存泄漏

**错误信息**：
```
java.lang.OutOfMemoryError: Java heap space
```

**示例**：
```java
List<byte[]> list = new ArrayList<>();
while (true) {
    list.add(new byte[1024 * 1024]);  // 每次添加1MB
}
```

#### 2. 栈溢出（StackOverflowError）

**原因**：
- 递归太深
- 局部变量太多

**错误信息**：
```
java.lang.StackOverflowError
```

**示例**：
```java
public void infiniteRecursion() {
    infiniteRecursion();  // 无限递归
}
```

#### 3. 元空间溢出（Metaspace）

**原因**：
- 加载了太多类
- 动态生成类（如CGLIB代理）

**错误信息**：
```
java.lang.OutOfMemoryError: Metaspace
```

#### 4. GC开销过大

**原因**：
- GC时间占比过高
- 回收效果差

**错误信息**：
```
java.lang.OutOfMemoryError: GC overhead limit exceeded
```

### 内存泄漏的常见原因

#### 1. 静态集合类

```java
public class StaticCollectionLeak {
    private static List<Object> cache = new ArrayList<>();
    
    public void add(Object obj) {
        cache.add(obj);  // 只添加，不清理
    }
}
```

#### 2. 未关闭的资源

```java
public class ResourceLeak {
    public void readFile() throws IOException {
        FileInputStream fis = new FileInputStream("test.txt");
        // 忘记关闭，导致泄漏
        // fis.close();
    }
    
    // 正确做法
    public void readFileCorrect() throws IOException {
        try (FileInputStream fis = new FileInputStream("test.txt")) {
            // 自动关闭
        }
    }
}
```

#### 3. 监听器和回调

```java
public class ListenerLeak {
    public void registerListener() {
        EventSource source = new EventSource();
        source.addListener(new EventListener() {
            @Override
            public void onEvent(Event e) {
                // 处理事件
            }
        });
        // 忘记移除监听器，导致泄漏
    }
}
```

#### 4. ThreadLocal未清理

```java
public class ThreadLocalLeak {
    private static ThreadLocal<List<Object>> threadLocal = new ThreadLocal<>();
    
    public void useThreadLocal() {
        threadLocal.set(new ArrayList<>());
        // 使用...
        // 忘记remove，线程复用时导致泄漏
        // threadLocal.remove();
    }
}
```

#### 5. 内部类持有外部类引用

```java
public class InnerClassLeak {
    private byte[] data = new byte[1024 * 1024];  // 1MB
    
    class Inner {
        // 隐式持有外部类引用
        public void doSomething() {
            // 即使只用到少量数据，也会阻止整个外部类被回收
        }
    }
}
```

### 如何排查

**工具**：
- jmap：导出堆转储文件
- MAT（Memory Analyzer Tool）：分析堆转储
- VisualVM：实时监控
- JProfiler：性能分析

**步骤**：
1. 重现问题
2. 导出堆转储：`jmap -dump:format=b,file=heap.hprof <pid>`
3. 用MAT分析
4. 查找支配树（Dominator Tree）
5. 定位泄漏对象和引用链

## **什么情况下会发生栈溢出？**

栈溢出（StackOverflowError）发生在以下情况：

### 1. 无限递归

最常见的原因：

```java
public class InfiniteRecursion {
    public void method() {
        method();  // 无限递归，没有退出条件
    }
    
    public static void main(String[] args) {
        new InfiniteRecursion().method();
    }
}
```

### 2. 递归深度过大

即使有退出条件，但递归太深：

```java
public class DeepRecursion {
    public int factorial(int n) {
        if (n <= 1) return 1;
        return n * factorial(n - 1);  // 如果n很大，会栈溢出
    }
    
    public static void main(String[] args) {
        DeepRecursion dr = new DeepRecursion();
        // 100000层递归，远超栈深度
        System.out.println(dr.factorial(100000));
    }
}
```

### 3. 局部变量过多

虽然少见，但理论上可能：

```java
public class ManyLocals {
    public void method() {
        // 声明大量局部变量
        int a1, a2, a3, /* ... */ a10000;
        // 占用大量栈空间
    }
}
```

### 4. 方法调用链过长

```java
public class LongCallChain {
    public void method1() { method2(); }
    public void method2() { method3(); }
    public void method3() { method4(); }
    // ... 很多层
    public void method1000() { /* ... */ }
}
```

### 栈溢出的原理

**栈帧结构**：
```
栈底
┌──────────────┐
│ main方法栈帧  │
├──────────────┤
│ method1栈帧  │
├──────────────┤
│ method2栈帧  │
├──────────────┤
│ ...          │
├──────────────┤
│ methodN栈帧  │ ← 栈顶
└──────────────┘
```

每次方法调用都会创建一个新的栈帧，如果调用链太长，栈空间耗尽，就会抛出StackOverflowError。

### 配置栈大小

```bash
# 设置线程栈大小
-Xss256k    # 256KB
-Xss1m      # 1MB（默认）
-Xss2m      # 2MB
```

**注意**：
- 栈太小容易溢出
- 栈太大会限制可创建的线程数
- 需要根据应用调整

### 如何避免

1. **检查递归终止条件**
   ```java
   public int factorial(int n) {
       if (n <= 1) return 1;  // 确保有终止条件
       return n * factorial(n - 1);
   }
   ```

2. **改用迭代**
   ```java
   public int factorial(int n) {
       int result = 1;
       for (int i = 2; i <= n; i++) {
           result *= i;
       }
       return result;
   }
   ```

3. **使用尾递归优化**
   ```java
   // 尾递归（某些语言支持优化，Java不支持）
   public int factorialTail(int n, int acc) {
       if (n <= 1) return acc;
       return factorialTail(n - 1, n * acc);
   }
   ```

4. **增加栈大小**
   ```bash
   java -Xss2m MyApp
   ```

### 排查栈溢出

**查看异常堆栈**：

```
创建 → 使用 → 无用 → 回收
  ↓      ↓      ↓      ↓
```

### 3. 编写内存友好的代码

- 及时释放不需要的引用
- 使用合适的数据结构
- 避免不必要的对象创建
- 注意资源关闭

### 4. 调优实践

```bash
# 推荐的JVM参数
-Xms2g -Xmx2g                  # 堆大小

-XX:SurvivorRatio=8            # Survivor比例
```

### 5. 常见问题速查
