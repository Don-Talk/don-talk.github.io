# 基础概念与结构

## 什么是 JVM？

JVM（Java Virtual Machine，Java虚拟机）是一种用于计算设备的规范，它是一个虚构出来的计算机，是通过在实际的计算机上仿真模拟各种计算机功能来实现的。

**核心特点：**
- **跨平台性**：Write Once, Run Anywhere（一次编写，到处运行）
- **自动内存管理**：垃圾回收机制（GC）
- **安全性**：字节码验证、沙箱机制
- **解释执行与编译执行结合**：通过即时编译器（JIT）提升性能

**工作原理：**
```
Java源代码(.java) → 编译器(javac) → 字节码(.class) → JVM加载 → 解释/编译执行 → 机器码
```

---

## 说说 JVM 的其他特性？

1. **平台无关性**：JVM屏蔽了底层操作系统的差异，使得Java程序可以在任何支持JVM的平台上运行
2. **内存管理自动化**：自动进行内存分配和垃圾回收，减少内存泄漏风险
3. **安全性高**：提供字节码验证器、类加载器安全检查、安全管理器等机制
4. **多线程支持**：内置线程模型，支持并发编程
5. **动态链接**：在运行时才进行类的链接，支持动态加载类
6. **异常处理机制**：统一的异常处理框架
7. **性能优化**：通过JIT编译器将热点代码编译为本地代码，提升执行效率

## 能说一下 JVM 的内存区域吗？

JVM内存区域分为**线程私有**和**线程共享**两大部分：

### 线程私有区域

1. **程序计数器（Program Counter Register）**
   - 当前线程执行的字节码行号指示器
   - 唯一不会发生OutOfMemoryError的区域
   - 如果执行的是Native方法，则计数器值为Undefined

2. **Java虚拟机栈（Java Virtual Machine Stack）**
   - 描述Java方法执行的内存模型
   - 每个方法执行时创建一个栈帧（Stack Frame），用于存储局部变量表、操作数栈、动态链接、方法出口等
   - 可能抛出两种异常：
     - StackOverflowError：线程请求的栈深度大于虚拟机所允许的深度
     - OutOfMemoryError：虚拟机栈容量无法扩展时

3. **本地方法栈（Native Method Stack）**
   - 为Native方法服务
   - 与虚拟机栈类似，区别在于服务对象不同

### 线程共享区域

1. **Java堆（Java Heap）**
   - JVM中最大的一块内存区域
   - 所有对象实例和数组都在堆上分配
   - 是垃圾收集器管理的主要区域
   - 可细分为：新生代（Eden、Survivor）、老年代
   - 可能抛出OutOfMemoryError

2. **方法区（Method Area）**
   - 存储已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据
   - JDK 1.8之前称为永久代（Permanent Generation）
   - JDK 1.8之后改为元空间（Metaspace），使用本地内存
   - 可能抛出OutOfMemoryError

3. **运行时常量池（Runtime Constant Pool）**
   - 方法区的一部分
   - 存放编译期生成的各种字面量和符号引用
   - 具有动态性，运行期间也可能将新的常量放入池中

## 堆和栈的区别是什么？

| 对比项 | 堆（Heap） | 栈（Stack） |
|--------|-----------|------------|
| **用途** | 存储对象实例和数组 | 存储局部变量、方法调用信息 |
| **线程共享** | 线程共享 | 线程私有 |
| **内存大小** | 较大（默认约为物理内存的1/4） | 较小（默认1MB左右） |
| **垃圾回收** | 需要GC | 不需要GC，方法执行完自动释放 |
| **生命周期** | 对象的生命周期不确定 | 与方法调用周期一致 |
| **访问速度** | 相对较慢 | 相对较快 |
| **异常类型** | OutOfMemoryError | StackOverflowError |
| **存储内容** | 对象实例、数组 | 基本数据类型、对象引用 |

**示例：**
```java
public class Example {
    public void method() {
        int a = 10;           // a存储在栈中
        String str = "hello"; // str引用在栈中，"hello"对象在堆中
        Object obj = new Object(); // obj引用在栈中，Object实例在堆中
    }
}
```

## 变量存在堆栈的什么位置？

1. **基本数据类型变量**
   - 局部变量：存储在栈中
   - 成员变量：跟随对象存储在堆中
   - 静态变量：存储在方法区

2. **引用类型变量**
   - 引用本身：存储在栈中
   - 对象实例：存储在堆中

3. **常量**
   - 编译期常量：存储在运行时常量池（方法区）
   - 字符串常量：JDK 1.7之前存储在方法区，1.7及之后移动到堆中

**示例分析：**
```java
public class VariableLocation {
    private int instanceVar = 10;      // 实例变量，存储在堆中
    private static int staticVar = 20; // 静态变量，存储在方法区
    
    public void method() {
        int localVar = 30;             // 局部变量，存储在栈中
        String str = "test";           // str引用在栈，"test"在堆(1.7+)或方法区(1.6-)
        Object obj = new Object();     // obj引用在栈，Object实例在堆
    }
}
```

## 说一下 JDK 1.6、1.7、1.8 内存区域的变化？

### JDK 1.6
- **永久代（Permanent Generation）**：存储类信息、常量、静态变量
- **字符串常量池**：位于永久代中
- **运行时常量池**：位于永久代中

### JDK 1.7
- **字符串常量池**：从永久代移到Java堆中
- **静态变量**：从永久代移到Java堆中
- 永久代仍然存在，但内容逐渐减少

### JDK 1.8
- **移除永久代**，引入**元空间（Metaspace）**
- **元空间**：使用本地内存（非JVM内存），存储类元数据
- **字符串常量池**：仍在Java堆中
- **运行时常量池**：仍在方法区（现在是元空间的一部分）

**变化原因：**
1. 永久代大小难以确定，容易导致OutOfMemoryError
2. 字符串常量池放在堆中有利于垃圾回收
3. 元空间使用本地内存，只受系统内存限制，更灵活

---

## 为什么要使用元空间替代永久代？

1. **内存溢出问题**
   - 永久代大小固定（-XX:MaxPermSize），容易触发OutOfMemoryError: PermGen space
   - 元空间使用本地内存，默认无上限，只受系统内存限制

2. **垃圾回收优化**
   - 永久代的GC效率较低
   - 元空间的类元数据回收更高效

3. **HotSpot与其他JVM融合**
   - JRockit和J9没有永久代概念
   - 便于HotSpot与其他JVM整合

4. **动态类加载场景**
   - 如OSGi、Groovy等动态语言频繁生成类
   - 永久代容易溢出，元空间更适合

5. **简化JVM实现**
   - 统一内存管理模型
   - 减少特殊处理逻辑

**配置参数对比：**
```
# JDK 1.6/1.7
-XX:PermSize=64m      # 永久代初始大小
-XX:MaxPermSize=256m   # 永久代最大大小

# JDK 1.8+
-XX:MetaspaceSize=64m      # 元空间初始大小
-XX:MaxMetaspaceSize=256m  # 元空间最大大小（可选，默认无限制）
```

## 说说 JVM 的组织架构？

JVM架构主要分为以下几个部分：

### 1. 类加载器子系统（Class Loader Subsystem）
负责加载、链接和初始化类：
- **加载（Loading）**：查找并加载类的二进制数据
- **链接（Linking）**：
  - 验证（Verification）：确保类的正确性
  - 准备（Preparation）：为静态变量分配内存并设置默认值
  - 解析（Resolution）：将符号引用转换为直接引用
- **初始化（Initialization）**：执行静态代码块和静态变量赋值

**类加载器类型：**
- Bootstrap ClassLoader：启动类加载器，加载核心类库（rt.jar）
- Extension ClassLoader：扩展类加载器，加载ext目录下的jar
- Application ClassLoader：应用程序类加载器，加载classpath下的类
- 自定义类加载器：用户自定义的类加载器

### 2. 运行时数据区（Runtime Data Area）
即前面提到的内存区域：
- 程序计数器
- Java虚拟机栈
- 本地方法栈
- Java堆
- 方法区

### 3. 执行引擎（Execution Engine）
负责执行字节码：
- **解释器（Interpreter）**：逐行解释执行字节码
- **即时编译器（JIT Compiler）**：将热点代码编译为本地代码
  - C1编译器：快速编译，优化较少
  - C2编译器：慢速编译，优化更多
- **垃圾收集器（Garbage Collector）**：自动管理内存

### 4. 本地方法接口（Native Interface）
- JNI（Java Native Interface）：调用本地方法（C/C++）
- 本地方法库：提供操作系统级别的API

### 5. 本地方法库（Native Libraries）
- 提供操作系统相关的功能
- 如文件IO、网络通信等

**架构图：**
```

│         Java 应用程序               │
├─────────────────────────────────────┤
│       类加载器子系统                │

│       运行时数据区                  │
│  ┌──────┬──────┬──────┬────┬────┐  │
│  │ PC   │ 栈   │本地栈│ 堆 │方法│  │
│  │寄存器│      │      │    │区  │  │
│  └──────┴──────┴──────┴────┴────┘  │
├─────────────────────────────────────┤
│       执行引擎                      │
│  ┌────────┬────────┬──────────┐    │
│  │解释器  │ JIT编译器│ GC      │    │
│  └────────┴────────┴──────────┘    │
├─────────────────────────────────────┤
│       本地方法接口 + 本地方法库     │
└─────────────────────────────────────┘
```
