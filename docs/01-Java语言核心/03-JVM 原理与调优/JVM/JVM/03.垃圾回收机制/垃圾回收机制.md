# 垃圾回收机制

## **讲讲 JVM 的垃圾回收机制？**

垃圾回收（Garbage Collection，简称GC）是JVM自动管理内存的机制，它会自动回收不再使用的对象，释放内存空间。

### 为什么需要GC？

**手动管理内存的问题（如C/C++）：**
- 忘记释放内存 → 内存泄漏
- 重复释放内存 → 程序崩溃
- 释放后继续使用 → 悬空指针

**Java的解决方案：**
- 由JVM自动管理内存
- 程序员专注于业务逻辑
- 减少内存相关bug

### GC的核心问题

GC主要解决三个问题：

1. **哪些内存需要回收？** → 判断对象是否存活
2. **什么时候回收？** → GC触发时机
3. **如何回收？** → GC算法和收集器

### GC的工作流程

```
1. 标记阶段：找出所有存活的对象
         ↓
2. 整理阶段：根据需要移动对象或清理空间
         ↓
3. 回收阶段：释放无用对象的内存
```

### GC的分类

| 分类方式 | 类型 | 说明 |
|---------|------|------|
| 按区域 | Minor GC | 新生代GC |
| | Major GC | 老年代GC |
| | Full GC | 整个堆（包括方法区） |
| 按线程 | Serial GC | 单线程 |
| | Parallel GC | 多线程并行 |
| | Concurrent GC | 与应用线程并发 |
| 按模式 | Stop-The-World | 暂停应用线程 |
| | 并发收集 | 不暂停或少暂停 |

## **如何判断对象仍然存活？** (可达性分析 vs 引用计数法)

JVM提供了两种判断对象是否存活的方法：

### 1. 引用计数法（Reference Counting）

**原理**：
- 每个对象有一个引用计数器
- 被引用时+1，引用失效时-1
- 计数器为0时，对象可以被回收

**示意图**：
```
对象A [计数=2] ← 引用1
              ← 引用2

对象B [计数=0] → 可以回收
```

**优点**：
- 实现简单
- 判断效率高
- 可以实时回收

**缺点**：
- **无法解决循环引用问题**

**循环引用示例**：
```java
public class CircularReference {
    private Object reference;
    
    public static void main(String[] args) {
        Object objA = new CircularReference();
        Object objB = new CircularReference();
        
        // 互相引用
        ((CircularReference) objA).reference = objB;
        ((CircularReference) objB).reference = objA;
        
        // 断开外部引用
        objA = null;
        objB = null;
        
        // 此时objA和objB的计数器都是1（互相引用）
        // 但实际上已经无法访问，应该被回收
        // 引用计数法无法处理这种情况！
    }
}
```

**采用者**：Python、PHP等，但Java没有使用

---

### 2. 可达性分析（Reachability Analysis）✅ Java采用

**原理**：
- 从一系列称为"GC Roots"的对象开始
- 向下搜索，走过的路径称为引用链
- 如果一个对象到GC Roots没有任何引用链，则证明此对象不可达，可以回收

**示意图**：
```
GC Roots
   ↓
  对象A ──→ 对象B
   ↓         ↓
  对象C     对象D
   
  对象E (不可达，可回收)
  对象F (不可达，可回收)
```

**GC Roots包括**：
1. 虚拟机栈中引用的对象（局部变量）
2. 方法区中静态属性引用的对象
3. 方法区中常量引用的对象
4. 本地方法栈中JNI引用的对象
5. 正在执行的线程
6. synchronized锁持有的对象

**优点**：
- 解决了循环引用问题
- 准确性高

**缺点**：
- 需要STW（Stop-The-World）
- 实现复杂

**示例**：
```java
public class ReachabilityExample {
    private static Object staticObj;  // 静态变量，是GC Root
    
    public void method() {
        Object localObj = new Object();  // 局部变量，是GC Root
        
        // localObj可达，不会被回收
        
        localObj = null;  // 断开引用
        // 现在这个Object对象不可达，可以被回收
    }
}
```

### 两种方法对比

| 对比项 | 引用计数法 | 可达性分析 |
|--------|-----------|-----------|
| 循环引用 | ❌ 无法处理 | ✅ 可以处理 |
| 实时性 | 好 | 需要STW |
| 实现复杂度 | 简单 | 复杂 |
| 额外开销 | 每个对象需要计数器 | 需要遍历引用链 |
| Java是否采用 | ❌ | ✅ |

## **Java 中可作为 GC Roots 的引用有哪几种？**

GC Roots是可达性分析的起点，以下对象可以作为GC Roots：

### 1. 虚拟机栈中引用的对象

**来源**：栈帧中的局部变量表

```java
public void method() {
    Object obj = new Object();  // obj是GC Root
    // obj引用的对象不会被回收
}
```

### 2. 方法区中静态属性引用的对象

**来源**：类的静态变量

```java
public class StaticExample {
    private static Object staticObj = new Object();  // staticObj是GC Root
}
```

### 3. 方法区中常量引用的对象

**来源**：运行时常量池中的常量

```java
public class ConstantExample {
    private static final String CONSTANT = "hello";  // "hello"是GC Root
}
```

### 4. 本地方法栈中JNI引用的对象

**来源**：Native方法引用的Java对象

```java
public native void nativeMethod(Object obj);  // obj被Native方法引用
```

### 5. Java虚拟机内部的引用

- 基本数据类型对应的Class对象
- 一些常驻的异常对象（如NullPointerException）
- 系统类加载器

### 6. 正在执行的线程

所有活着的线程都是GC Roots

### 7. synchronized锁持有的对象

被synchronized关键字锁定的对象

### 示意图

```
GC Roots集合
├── 栈中的局部变量
│   └── method()中的obj
├── 静态变量
│   └── MyClass.staticField
├── 常量
│   └── "hello"字符串
├── Native方法引用
│   └── JNI全局引用
├── 运行的线程
│   └── Thread-0, Thread-1...
└── 同步锁
    └── synchronized(obj)中的obj
```

### 实际案例

```java
public class GCRootsDemo {
    private static List<String> staticList = new ArrayList<>();  // GC Root
    
    public void method() {
        String localStr = "local";  // GC Root（局部变量）
        String constantStr = "constant";  // GC Root（常量）
        
        // 这些对象都可达，不会被回收
        staticList.add(localStr);
        System.out.println(constantStr);
    }
}
```

## **finalize()方法了解吗？** (对象"自救"机制)

`finalize()`是Object类的一个方法，在对象被回收前会被调用。

### finalize()的执行流程

```
对象不可达
    ↓
第一次标记
    ↓
是否需要执行finalize()？
├─ 否（未重写或已执行过）→ 直接回收
└─ 是 → 放入F-Queue队列
           ↓
      低优先级线程执行finalize()
           ↓
      第二次标记
      ├─ 对象重新关联 → 复活（移除即将回收集合）
      └─ 对象仍不可达 → 回收
```

### 对象"自救"示例

```java
public class FinalizeRescue {
    public static FinalizeRescue instance = null;
    
    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        System.out.println("finalize()方法被执行");
        // 自救：重新建立引用关系
        instance = this;
    }
    
    public static void main(String[] args) throws Exception {
        instance = new FinalizeRescue();
        
        // 第一次：对象变为不可达
        instance = null;
        System.gc();
        Thread.sleep(500);  // 等待finalize执行
        
        if (instance != null) {
            System.out.println("对象自救成功！");
        } else {
            System.out.println("对象已被回收");
        }
        
        // 第二次：finalize()只会执行一次
        instance = null;
        System.gc();
        Thread.sleep(500);
        
        if (instance != null) {
            System.out.println("对象再次自救成功！");
        } else {
            System.out.println("对象已被回收（finalize只执行一次）");
        }
    }
}
```

**输出**：
```
finalize()方法被执行
对象自救成功！
对象已被回收（finalize只执行一次）
```

### finalize()的问题

1. **执行时间不确定**
   - 由低优先级线程执行
   - 可能很久才执行，甚至不执行

2. **性能开销大**
   - 需要额外的队列管理
   - 增加GC复杂度

3. **只能自救一次**
   - finalize()最多执行一次
   - 第二次回收不会再调用

4. **异常处理困难**
   - finalize()中抛出异常会被忽略
   - 难以调试

### 推荐替代方案

**1. try-with-resources（推荐）**
```java
// 不推荐
public void readFile() {
    FileInputStream fis = null;
    try {
        fis = new FileInputStream("test.txt");
        // 读取文件
    } catch (IOException e) {
        e.printStackTrace();
    } finally {
        if (fis != null) {
            try {
                fis.close();  // 手动关闭
            } catch (IOException e) {
                e.printStackTrace();
            }
        }
    }
}

// 推荐
public void readFileBetter() {
    try (FileInputStream fis = new FileInputStream("test.txt")) {
        // 读取文件
        // 自动关闭，无需finalize
    } catch (IOException e) {
        e.printStackTrace();
    }
}
```

**2. 显式的close()方法**
```java
public class Resource implements AutoCloseable {
    public void close() {
        // 释放资源
    }
}
```

### 官方态度

- JDK 9标记为Deprecated
- 不推荐使用
- 未来版本可能移除

## **垃圾收集算法了解吗？** (标记-清除、复制、标记-整理、分代)

JVM提供了多种垃圾收集算法，各有优缺点：

### 1. 标记-清除算法（Mark-Sweep）

**最基础的算法**，分为两个阶段：

#### 阶段1：标记
- 从GC Roots开始，标记所有可达对象

#### 阶段2：清除
- 回收未被标记的对象

**示意图**：
```
标记前：
[存活][死亡][存活][死亡][存活][死亡]

标记后：
[存活][×][存活][×][存活][×]

清除后：
[存活][空闲][存活][空闲][存活][空闲]
```

**优点**：
- 实现简单
- 不需要移动对象

**缺点**：
- **内存碎片**：产生大量不连续的空闲空间
- **效率低**：标记和清除效率都不高
- **需要STW**：整个过程需要暂停应用

**采用者**：老年代的Serial Old、Parallel Old

---

### 2. 复制算法（Copying）

**原理**：
- 将内存分为两块（From和To）
- 每次只使用一块
- GC时将存活对象复制到另一块
- 然后清空当前块

**示意图**：
```
Before GC:
From区: [存活1][死亡][存活2][死亡][存活3]
To区:   [空闲]

After GC:
From区: [空闲]  ← 清空
To区:   [存活1][存活2][存活3]  ← 复制存活对象

下次GC时角色互换
```

**优点**：
- **无内存碎片**
- 实现简单
- 效率高（只需复制存活对象）

**缺点**：
- **内存利用率低**：只能用一半
- 存活对象多时，复制成本高

**优化**：HotSpot将新生代分为Eden和两个Survivor区（8:1:1）
-  Eden + From = 90%可用
-  To = 10%备用

**采用者**：新生代的Serial、ParNew、Parallel Scavenge

---

### 3. 标记-整理算法（Mark-Compact）

**结合了标记-清除和复制的优点**：

#### 阶段1：标记
- 同标记-清除，标记所有存活对象

#### 阶段2：整理
- 让所有存活对象向一端移动
- 然后清理掉边界以外的内存

**示意图**：
```
标记前：
[存活][死亡][存活][死亡][存活][死亡]

标记后：
[存活][×][存活][×][存活][×]

整理后：
[存活][存活][存活][空闲][空闲][空闲]
```

**优点**：
- **无内存碎片**
- 内存利用率高

**缺点**：
- 需要移动对象，成本较高
- 需要更新引用
- 需要STW

**采用者**：老年代的Serial Old、Parallel Old

---

### 4. 分代收集算法（Generational Collection）

**不是独立的算法，而是一种思想**：

**核心思想**：
- 根据对象的生命周期不同，将堆分为不同区域
- 对不同区域采用不同的收集算法

**分区**：
```
Java Heap
├── 新生代（Young Generation）
│   ├── 特点：对象朝生夕死
│   └── 算法：复制算法（效率高）
│
└── 老年代（Old Generation）
    ├── 特点：对象长期存活
    └── 算法：标记-清除或标记-整理
```

**为什么分代？**
- 新生代：大多数对象很快死亡 → 用复制算法，只复制少量存活对象
- 老年代：对象存活率高 → 用标记-整理，避免频繁复制

**优势**：
- 针对不同特点采用最优算法
- 提高整体GC效率
- 是现代JVM的标准做法

**采用者**：所有现代JVM（HotSpot、J9等）

---

### 算法对比表

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 标记-清除 | 实现简单 | 内存碎片、效率低 | 老年代 |
| 复制 | 无碎片、效率高 | 内存利用率低 | 新生代 |
| 标记-整理 | 无碎片、利用率高 | 需要移动对象 | 老年代 |
| 分代收集 | 综合最优 | 实现复杂 | 整个堆 |

## **Minor GC、Major GC、Mixed GC、Full GC 都是什么意思？**

### 1. Minor GC（Young GC / Young Generation GC）

**定义**：发生在新生代的垃圾收集

**触发条件**：
- Eden区空间不足时

**特点**：
- 频率高（因为新生代对象朝生夕死）
- 速度快（采用复制算法，只复制少量存活对象）
- 使用Stop-The-World（暂停所有应用线程）

**过程**：
```
1. Eden区满 → 触发Minor GC
2. 标记Eden和From中的存活对象
3. 复制到To区
4. 清空Eden和From
5. From和To角色互换
```

**影响**：
- 短暂停顿（通常几毫秒到几十毫秒）
- 对应用影响较小

---

### 2. Major GC（Old GC）

**定义**：发生在老年代的垃圾收集

**注意**：
- 不同收集器的Major GC含义不同
- 对于CMS，Major GC指老年代GC
- 对于G1，没有明确的Major GC概念

**特点**：
- 频率低
- 速度慢（老年代对象多）
- 通常需要STW

---

### 3. Mixed GC

**定义**：同时收集新生代和部分老年代

**采用者**：G1收集器特有

**特点**：
- 不是Full GC
- 选择性收集老年代Region
- 目标是控制停顿时间

**触发条件**：
- 老年代占用达到阈值（-XX:InitiatingHeapOccupancyPercent）

---

### 4. Full GC

**定义**：收集整个堆（新生代+老年代+方法区）

**触发条件**：
1. System.gc()被调用（建议，不保证）
2. 老年代空间不足
3. 方法区空间不足
4. Minor GC后，存活对象无法放入Survivor，也无法放入老年代
5. 分配担保失败

**特点**：
- 频率最低
- 速度最慢（可能需要几秒甚至更久）
- **必须STW**
- 对应用影响最大

**影响**：
- 长时间停顿（几百毫秒到几秒）
- 可能导致服务不可用
- 需要尽量避免

---

### GC类型对比

| GC类型 | 收集区域 | 频率 | 速度 | STW | 影响 |
|--------|---------|------|------|-----|------|
| Minor GC | 新生代 | 高 | 快 | 是 | 小 |
| Major GC | 老年代 | 低 | 慢 | 是 | 中 |
| Mixed GC | 新生代+部分老年代 | 中 | 中 | 是 | 中 |
| Full GC | 整个堆 | 很低 | 很慢 | 是 | 大 |

---

## **Young GC 什么时候触发？**

### 主要触发条件

#### 1. Eden区空间不足（最常见）

```java
// 不断创建对象，Eden区满了就会触发Young GC
List<byte[]> list = new ArrayList<>();
for (int i = 0; i < 100000; i++) {
    list.add(new byte[1024]);  // 每次创建1KB对象
}
```

**流程**：
```
新对象分配 → Eden区 → Eden满 → 触发Young GC
```

---

#### 2. 大对象直接进入老年代

如果对象太大，Eden放不下，可能直接触发Young GC或Full GC

```bash
-XX:PretenureSizeThreshold=1048576  # 1MB
```

```java
// 这个大对象可能直接进老年代
byte[] bigArray = new byte[2 * 1024 * 1024];  // 2MB
```

---

#### 3. 动态年龄判定

如果Survivor区中同龄对象总和超过Survivor的一半，会提前晋升到老年代

---

### Young GC的详细流程

```
1. Eden区满，触发Young GC
         ↓
2. 停止所有应用线程（STW）
         ↓
3. 标记Eden和From中的存活对象
         ↓
4. 将存活对象复制到To区
   ├─ 年龄+1
   └─ 年龄≥15 → 进入老年代
         ↓
5. 清空Eden和From
         ↓
6. From和To角色互换
         ↓
7. 恢复应用线程
```

---

### 配置参数

```bash
# 新生代大小
-Xmn512m                    # 新生代512MB
-XX:NewRatio=2              # 老年代:新生代 = 2:1

# Survivor比例
-XX:SurvivorRatio=8         # Eden:Survivor = 8:1

# 对象年龄阈值
-XX:MaxTenuringThreshold=15 # 默认15次GC后进入老年代
```

---

### 如何监控Young GC

**使用jstat**：
```bash
jstat -gc <pid> 1000  # 每秒打印一次GC统计
```

**输出示例**：
```
 S0C    S1C    S0U    S1U      EC       EU        OC         OU       YGC    YGCT
5120.0 5120.0  0.0   1024.0  40960.0  20480.0  102400.0   51200.0    123   1.234
```
- YGC: Young GC次数
- YGCT: Young GC总耗时

## **什么时候会触发 Full GC？** (空间分配担保)

Full GC是最严重的GC，应该尽量避免。以下是触发条件：

### 1. System.gc()被调用

```java
System.gc();  // 建议JVM进行GC，但不保证是Full GC
```

**注意**：
- 只是建议，JVM可以忽略
- 可以通过 `-XX:+DisableExplicitGC` 禁用

---

### 2. 老年代空间不足

**场景**：
- 大对象直接进入老年代，老年代满了
- 对象从新生代晋升到老年代，老年代满了

```java
// 持续创建大对象，最终导致老年代满
List<byte[]> list = new ArrayList<>();
while (true) {
    list.add(new byte[1024 * 1024]);  // 1MB
}
```

---

### 3. 方法区（元空间）空间不足

**场景**：
- 加载了大量类
- 动态生成类（CGLIB、ASM等）

```bash
# 元空间配置过小
-XX:MaxMetaspaceSize=64m
```

---

### 4. Young GC后的分配担保失败

**这是最常见的Full GC触发原因！**

**空间分配担保机制**：

在Young GC之前，JVM会检查：

```
检查1: 老年代最大可用连续空间 > 新生代所有对象总空间？
├─ 是 → Young GC安全
└─ 否 → 检查2

检查2: 是否允许分配担保失败？
├─ 不允许 → 直接Full GC
└─ 允许 → 检查3

检查3: 老年代最大可用连续空间 > 历次晋升到老年代对象的平均大小？
├─ 是 → 冒险进行Young GC
│        └─ 如果Young GC后老年代还是放不下 → Full GC
└─ 否 → Full GC
```

**为什么要担保？**
- Young GC后，存活对象需要放入老年代
- 如果老年代空间不够，就会导致担保失败
- 为了避免这种情况，提前触发Full GC

---

### 5. CMS收集器的特殊情况

CMS收集器在以下情况会触发Full GC（实际上是Serial Old）：

1. **Concurrent Mode Failure**
   - 并发收集过程中，老年代空间不足
   - 退化为Serial Old收集器，进行Full GC

2. **Promotion Failed**
   - Young GC时，对象无法晋升到老年代
   - 触发Full GC

---

### 6. G1收集器的特殊情况

G1收集器在以下情况会触发Full GC（单线程串行收集）：

1. **Evacuation Failure**
   - Mixed GC时，Survivor区放不下存活对象
   - 退化为Full GC

2. **To-space Exhausted**
   - 没有足够的空间复制存活对象

---

### 如何避免Full GC

#### 1. 调整堆大小
```bash
# 增大堆
-Xms4g -Xmx4g

# 调整新生代比例
-XX:NewRatio=2
```

#### 2. 优化代码
- 减少大对象创建
- 及时释放不需要的引用
- 避免内存泄漏

#### 3. 选择合适的收集器
```bash
# G1收集器（推荐）
-XX:+UseG1GC

# 设置最大停顿时间
-XX:MaxGCPauseMillis=200
```

#### 4. 监控和调优
```bash
# 启用GC日志
-XX:+PrintGCDetails -Xloggc:gc.log

# 分析GC日志，找出问题
```

## **知道哪些垃圾收集器？** (Serial, ParNew, CMS, G1, ZGC等)

JVM提供了多种垃圾收集器，每种都有自己的特点：

### 收集器概览图

```
新生代收集器          老年代收集器
┌─────────┐         ┌──────────┐
│ Serial  │────────→│Serial Old│
└─────────┘         └──────────┘
┌─────────┐         ┌──────────┐
│ ParNew  │────────→│Serial Old│
└─────────┘         │  CMS     │
┌─────────┐         └──────────┘
│Parallel │         ┌──────────┐
│Scavenge │────────→│Parallel  │
└─────────┘         │  Old     │
                    └──────────┘
                    ┌──────────┐
                    │   G1     │ (全堆)
                    └──────────┘
                    ┌──────────┐
                    │   ZGC    │ (全堆，JDK 11+)
                    └──────────┘
                    ┌──────────┐
                    │ Shenandoah│ (全堆，JDK 12+)
                    └──────────┘
```

**连线表示可以搭配使用**

---

### 1. Serial收集器

**特点**：
- 单线程收集器
- 收集时必须STW（Stop-The-World）
- 简单高效（无线程切换开销）

**算法**：
- 新生代：复制算法
- 老年代：标记-整理算法

**适用场景**：
- 客户端应用（桌面应用）
- 单核CPU环境
- 内存较小（几百MB）

**配置**：
```bash
-XX:+UseSerialGC
```

**优点**：
- 简单高效
- 没有线程交互开销

**缺点**：
- 单线程，多核CPU浪费
- STW时间长

---

### 2. ParNew收集器

**特点**：
- Serial的多线程版本
- 其他行为与Serial完全相同

**算法**：
- 新生代：复制算法（多线程）
- 老年代：配合Serial Old或CMS

**适用场景**：
- 多核CPU环境
- 常与CMS配合使用

**配置**：
```bash
-XX:+UseParNewGC  # JDK 9后废弃，默认就是ParNew+CMS
```

**优点**：
- 可以利用多核CPU
- 缩短Young GC时间

**缺点**：
- 线程间有通信开销
- CPU敏感，核数越多效果越好

---

### 3. Parallel Scavenge收集器

**特点**：
- 吞吐量优先的收集器
- 关注点：达到可控制的吞吐量
- **吞吐量 = 运行用户代码时间 / (运行用户代码时间 + GC时间)**

**算法**：
- 新生代：复制算法（多线程）
- 老年代：Parallel Old（标记-整理）

**适用场景**：
- 后台计算型应用
- 对响应时间不敏感
- 追求高吞吐量

**配置**：
```bash
-XX:+UseParallelGC            # 新生代
-XX:+UseParallelOldGC         # 老年代
-XX:MaxGCPauseMillis=100      # 最大GC停顿时间
-XX:GCTimeRatio=19            # 吞吐量目标（1/(1+19)=95%）
```

**优点**：
- 吞吐量高
- 自适应调节策略（-XX:+UseAdaptiveSizePolicy）

**缺点**：
- 停顿时间不可控
- 不适合交互式应用

---

### 4. CMS收集器（Concurrent Mark Sweep）

**特点**：
- 以获取最短回收停顿时间为目标
- 并发收集、低停顿
- JDK 9标记为Deprecated，JDK 14移除

**算法**：标记-清除

**收集过程（4个步骤）**：

```
1. 初始标记（Initial Mark）
   - 标记GC Roots能直接关联的对象
   - 需要STW，但很快
   
2. 并发标记（Concurrent Mark）
   - 从GC Roots的直接关联对象开始遍历
   - 与应用线程并发执行
   - 耗时长，但不需要STW
   
3. 重新标记（Remark）
   - 修正并发标记期间产生的变动
   - 需要STW，但比初始标记长
   
4. 并发清除（Concurrent Sweep）
   - 清除未标记的对象
   - 与应用线程并发执行
```

**示意图**：
```
时间轴 →
├─ 初始标记 [STW]
├─ 并发标记 [并发]
├─ 重新标记 [STW]
└─ 并发清除 [并发]
```

**配置**：
```bash
-XX:+UseConcMarkSweepGC
-XX:CMSInitiatingOccupancyFraction=70  # 老年代使用70%时开始CMS
-XX:+UseCMSCompactAtFullCollection     # Full GC后压缩
```

**优点**：
- 并发收集
- 停顿时间短

**缺点**：
- **内存碎片**（标记-清除算法）
- **浮动垃圾**：并发清除时产生的新垃圾无法处理
- **CPU敏感**：并发线程占用CPU资源
- **Concurrent Mode Failure**：并发失败会退化为Serial Old

---

### 5. G1收集器（Garbage-First）

**特点**：
- JDK 7引入，JDK 9成为默认收集器
- 面向服务端应用
- 可预测的停顿时间模型

**核心思想**：
- 将整个堆划分为多个大小相等的独立区域（Region）
- 跟踪每个Region的垃圾堆积价值
- 优先回收价值最大的Region（Garbage-First）

**Region类型**：
```
┌────┬────┬────┬────┬────┐
│ E  │ E  │ S  │ O  │ O  │
├────┼────┼────┼────┼────┤
│ E  │ H  │ O  │ O  │ CS │
└────┴────┴────┴────┴────┘

E = Eden, S = Survivor, O = Old, H = Humongous, CS = Collection Set
```

**收集过程**：

1. **Young GC**
   - 只收集Eden和Survivor区
   - 类似传统的Young GC

2. **Mixed GC**
   - 收集所有新生代 + 部分老年代
   - 选择垃圾最多的Region
   - 分多次进行，控制停顿时间

3. **Full GC**（应尽量避免）
   - 单线程串行收集
   - G1的弱点

**配置**：
```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200      # 目标停顿时间
-XX:G1HeapRegionSize=16m      # Region大小（1-32MB，2的幂）
-XX:InitiatingHeapOccupancyPercent=45  # 触发Mixed GC的阈值
```

**优点**：
- 可预测停顿时间
- 无内存碎片
- 适合大堆（6GB以上）

**缺点**：
- 实现复杂
- 小堆效果不明显
- Full GC性能较差

---

### 6. ZGC收集器（Z Garbage Collector）

**特点**：
- JDK 11引入（实验性），JDK 15生产就绪
- **超低停顿时间**：< 10ms
- 支持TB级堆内存
- 并发收集

**核心技术**：
- 染色指针（Colored Pointers）
- 读屏障（Load Barriers）
- 并发重定位

**配置**：
```bash
-XX:+UseZGC  # JDK 15+
```

**优点**：
- 停顿时间极短且稳定
- 支持超大堆
- 并发执行

**缺点**：
- CPU开销较大（约15%）
- JDK版本要求高

---

### 7. Shenandoah收集器

**特点**：
- Red Hat开发，JDK 12引入
- 与ZGC类似，低停顿
- 使用读屏障和写屏障

**配置**：
```bash
-XX:+UseShenandoahGC
```

---

### 收集器对比表

| 收集器 | 年代 | 算法 | 线程 | 停顿 | 适用场景 |
|--------|------|------|------|------|----------|
| Serial | 新老 | 复制/整理 | 单线程 | 长 | 客户端、小内存 |
| ParNew | 新 | 复制 | 多线程 | 中 | 配合CMS |
| Parallel | 新老 | 复制/整理 | 多线程 | 中 | 吞吐量优先 |
| CMS | 老 | 标记-清除 | 并发 | 短 | 响应时间优先 |
| G1 | 全堆 | 复制/整理 | 并发 | 可控 | 大堆、服务端 |
| ZGC | 全堆 | 染色指针 | 并发 | <10ms | 超大堆、超低延迟 |
| Shenandoah | 全堆 | 读/写屏障 | 并发 | <10ms | 类似ZGC |

---

### 如何选择收集器？

**选择建议**：

1. **小型应用（堆 < 4GB）**
   - Serial GC（单核）
   - Parallel GC（多核）

2. **中型应用（堆 4-8GB）**
   - G1 GC（推荐）
   - CMS（JDK 8）

3. **大型应用（堆 > 8GB）**
   - G1 GC（推荐）
   - ZGC（JDK 11+，超低延迟）

4. **吞吐量优先**
   - Parallel GC

5. **低延迟优先**
   - G1 GC
   - ZGC（极致低延迟）

## **能详细说一下 CMS 的垃圾收集过程吗？**

CMS（Concurrent Mark Sweep）是JDK 8及之前常用的低停顿收集器。

### CMS的完整收集过程

CMS收集分为**7个步骤**，其中2个需要STW，4个可以并发：

```
1. 初始标记（Initial Mark）     [STW]
         ↓
2. 并发标记（Concurrent Mark）   [并发]
         ↓
3. 并发预清理（Concurrent Preclean） [并发]
         ↓
4. 重新标记（Remark）            [STW]
         ↓
5. 并发清除（Concurrent Sweep）  [并发]
         ↓
6. 并发重置（Concurrent Reset）  [并发]
         ↓
7. 等待下次GC触发
```

---

### 详细讲解每个阶段

#### 1. 初始标记（Initial Mark）【STW】

**任务**：
- 标记GC Roots能直接关联到的对象

**特点**：
- 需要STW
- 速度很快（只标记直接关联的对象）

**示例**：
```java
public class CMSExample {
    private static Object root = new Object();  // GC Root
    
    public void method() {
        Object local = new Object();  // GC Root
        local.someField = root;       // root是直接关联的
    }
}
```

初始标记只会标记 `root` 和 `local` 直接引用的对象。

---

#### 2. 并发标记（Concurrent Mark）【并发】

**任务**：
- 从GC Roots的直接关联对象开始，遍历整个对象图
- 标记所有可达对象

**特点**：
- 与应用线程并发执行
- 耗时长（需要遍历整个对象图）
- 不需要STW

**问题**：
- 并发过程中，对象引用关系可能变化
- 会产生"浮动垃圾"

---

#### 3. 并发预清理（Concurrent Preclean）【并发】

**任务**：
- 处理并发标记期间的引用变化
- 为重新标记做准备

**特点**：
- 可选阶段
- 可以减少重新标记的时间

---

#### 4. 重新标记（Remark）【STW】

**任务**：
- 修正并发标记期间产生的变动
- 确保标记的准确性

**特点**：
- 需要STW
- 比初始标记时间长
- 但比完全重新标记短（因为有预清理）

**为什么需要重新标记？**
```
并发标记期间：
对象A → 对象B （标记了）

应用线程修改：
对象A → 对象C （对象B变为垃圾，但未被标记）

重新标记会发现对象B实际上已经是垃圾
```

---

#### 5. 并发清除（Concurrent Sweep）【并发】

**任务**：
- 清除未被标记的对象
- 回收内存空间

**特点**：
- 与应用线程并发执行
- 不需要STW
- **产生内存碎片**（标记-清除算法）

**问题**：
- 清除过程中，应用线程可能继续创建对象
- 这些新对象成为"浮动垃圾"，本次GC无法处理

---

#### 6. 并发重置（Concurrent Reset）【并发】

**任务**：
- 重置CMS收集器的数据结构
- 为下次GC做准备

---

### CMS的问题

#### 1. 内存碎片

**原因**：使用标记-清除算法

**后果**：
- 大对象无法分配
- 提前触发Full GC

**解决**：
```bash
-XX:+UseCMSCompactAtFullCollection  # Full GC后压缩
-XX:CMSFullGCsBeforeCompaction=0    # 每次Full GC都压缩
```

---

#### 2. 浮动垃圾（Floating Garbage）

**原因**：并发清除时，应用线程继续运行，产生新的垃圾

**后果**：
- 这部分垃圾本次GC无法处理
- 需要等到下次GC

**无法避免**，这是并发收集的代价。

---

#### 3. Concurrent Mode Failure

**场景**：
- 并发清除过程中，老年代空间不足
- 应用线程需要分配对象，但空间不够

**后果**：
- CMS收集失败
- 退化为Serial Old收集器
- 进行Full GC，STW时间很长

**解决**：
```bash
# 降低触发CMS的阈值，提前开始收集
-XX:CMSInitiatingOccupancyFraction=70  # 默认92%，改为70%
```

---

#### 4. CPU敏感

**原因**：并发线程需要占用CPU资源

**后果**：
- CPU核数少时，影响应用性能
- 建议CPU核数 >= 4

---

### CMS配置示例

```bash
# 基本配置
-XX:+UseConcMarkSweepGC
-XX:ParNewGC   # 新生代使用ParNew

# 优化配置
-XX:CMSInitiatingOccupancyFraction=70  # 70%时触发CMS
-XX:+UseCMSCompactAtFullCollection     # Full GC后压缩
-XX:CMSFullGCsBeforeCompaction=0       # 每次都压缩
-XX:+CMSClassUnloadingEnabled          # 回收方法区
-XX:MaxTenuringThreshold=10            # 对象年龄阈值
```

---

### CMS vs G1

| 对比项 | CMS | G1 |
|--------|-----|----|
| 算法 | 标记-清除 | 复制+整理 |
| 内存碎片 | 有 | 无 |
| 停顿时间 | 较短 | 可预测 |
| 并发失败 | 会退化 | 不会 |
| 状态 | 已废弃 | 默认推荐 |

**结论**：JDK 9+建议使用G1替代CMS

## **什么是三色标记法？**

三色标记法是CMS和G1等并发收集器使用的标记算法。

### 三种颜色

1. **白色（White）**
   - 对象尚未被GC访问到
   - 最终未被标记的白色对象会被回收

2. **灰色（Gray）**
   - 对象已被GC访问到
   - 但该对象引用的其他对象尚未全部访问
   - 处于"待处理"状态

3. **黑色（Black）**
   - 对象已被GC访问到
   - 且该对象引用的所有对象也都已访问
   - 标记完成，不会被回收

---

### 标记过程

```
初始状态：所有对象都是白色

步骤1：将GC Roots标记为灰色

步骤2：从灰色对象集合中选择一个对象
       ├─ 将其引用的对象标记为灰色
       └─ 将该对象标记为黑色

步骤3：重复步骤2，直到灰色集合为空

最终状态：
- 黑色对象：存活
- 白色对象：死亡，将被回收
```

---

### 示例演示

假设有以下对象图：
```
GC Root → A → B → C
        → D
```

**标记过程**：

```
初始：
Root(白) → A(白) → B(白) → C(白)
       → D(白)

步骤1：Root标记为灰色
Root(灰) → A(白) → B(白) → C(白)
       → D(白)

步骤2：处理Root，将A和D标记为灰色，Root标记为黑色
Root(黑) → A(灰) → B(白) → C(白)
       → D(灰)

步骤3：处理A，将B标记为灰色，A标记为黑色
Root(黑) → A(黑) → B(灰) → C(白)
       → D(灰)

步骤4：处理B，将C标记为灰色，B标记为黑色
Root(黑) → A(黑) → B(黑) → C(灰)
       → D(灰)

步骤5：处理C，C标记为黑色
Root(黑) → A(黑) → B(黑) → C(黑)
       → D(灰)

步骤6：处理D，D标记为黑色
Root(黑) → A(黑) → B(黑) → C(黑)
       → D(黑)

结束：灰色集合为空，所有黑色对象存活
```

---

### 并发标记的问题

在并发标记过程中，应用线程也在运行，可能导致**漏标**问题。

#### 漏标的两种情况

**情况1：赋值器插入引用（Incremental Update）**

```
初始：
A(黑) → B(白)
C(灰)

应用线程执行：
C.ref = B  // C引用了B
A.ref = null  // A不再引用B

结果：
A(黑) → null
C(灰) → B(白)

问题：
- B应该是存活的（被C引用）
- 但B是白色，会被错误回收！
```

**解决方案**：增量更新（Incremental Update）
- 当黑色对象引用白色对象时，将白色对象重新标记为灰色
- CMS使用这种方式

---

**情况2：赋值器删除引用（SATB）**

```
初始：
A(黑) → B(白) → C(白)

应用线程执行：
A.ref = null  // A不再引用B
B.ref = C     // B引用C

结果：
A(黑) → null
B(白) → C(白)

问题：
- B和C应该是存活的
- 但它们都是白色，会被错误回收！
```

**解决方案**：快照初始（SATB, Snapshot At The Beginning）
- 记录删除引用的信息
- G1使用这种方式

---

### 两种解决方案对比

| 方案 | 原理 | 采用者 | 优点 | 缺点 |
|------|------|--------|------|------|
| 增量更新 | 黑色引用白色时，白色变灰色 | CMS | 简单 | 可能重新扫描 |
| SATB | 记录删除引用，保证一致性 | G1 | 精确 | 需要额外记录 |

---

### 写屏障（Write Barrier）

为了实现上述方案，需要在引用赋值时插入"写屏障"：

```java
// 伪代码
void oop_field_store(oop* field, oop new_value) {
    // 写屏障逻辑
    pre_write_barrier(field);  // 写前屏障（SATB）
    *field = new_value;         // 实际赋值
    post_write_barrier(field);  // 写后屏障（增量更新）
}
```

**性能影响**：
- 每次引用赋值都要执行屏障
- 有一定开销
- 但保证了并发标记的正确性

## **G1 垃圾收集器了解吗？** (Region划分、Mixed GC)

G1是JDK 9之后的默认收集器，专为大堆设计。

### G1的核心特性

1. **Region划分**：将堆划分为多个独立区域
2. **可预测停顿**：用户可以指定期望的停顿时间
3. **整体标记-局部复制**：结合两种算法的优点
4. **空间整合**：无内存碎片

---

### Region划分

**传统堆布局**：
```
┌────────────┬──────────────┐
│  新生代     │   老年代      │
└────────────┴──────────────┘
物理连续
```

**G1堆布局**：
```
┌────┬────┬────┬────┬────┬────┐
│ E  │ E  │ S  │ O  │ O  │ O  │
├────┼────┼────┼────┼────┼────┤
│ O  │ E  │ S  │ O  │ H  │ O  │
└────┴────┴────┴────┴────┴────┘
逻辑分区，物理可以不连续
```

**Region类型**：
- **Eden（E）**：新生代伊甸园区
- **Survivor（S）**：新生代幸存区
- **Old（O）**：老年代区
- **Humongous（H）**：大对象区（存放超过Region 50%的对象）

**Region大小**：
- 固定大小：1MB - 32MB（2的幂）
- 根据堆大小自动计算
- 最多2048个Region

---

### G1的收集过程

#### 1. Young GC

**触发条件**：Eden区满

**过程**：
```
1. 选择所有Eden和Survivor Region作为Collection Set (CSet)
2. STW，复制存活对象到新的Survivor或Old Region
3. 清空CSet中的Region
4. 恢复应用线程
```

**特点**：
- 类似传统的Young GC
- 停顿时间短

---

#### 2. Concurrent Marking（并发标记）

**触发条件**：老年代占用达到阈值（默认45%）

**过程**（类似CMS）：
```
1. 初始标记（STW）
2. 并发标记（并发）
3. 最终标记（STW）
4. 筛选回收（计算回收价值）
```

---

#### 3. Mixed GC

**定义**：收集所有新生代 + 部分老年代

**触发条件**：并发标记完成后

**过程**：
```
1. 选择Collection Set：
   - 所有Eden和Survivor Region
   - 部分Old Region（根据回收价值和停顿目标）

2. STW，复制存活对象

3. 清空CSet中的Region

4. 可能分多次Mixed GC，逐步回收老年代
```

**选择Region的策略**：
- 优先回收垃圾多的Region（Garbage-First）
- 考虑停顿时间目标
- 避免一次性回收太多

---

### G1的配置参数

```bash
# 启用G1
-XX:+UseG1GC

# 目标停顿时间
-XX:MaxGCPauseMillis=200

# Region大小（1-32MB，2的幂）
-XX:G1HeapRegionSize=16m

# 触发并发标记的阈值
-XX:InitiatingHeapOccupancyPercent=45

# 年轻代大小范围
-XX:G1NewSizePercent=5   # 最小5%
-XX:G1MaxNewSizePercent=60  # 最大60%

# Mixed GC相关
-XX:G1MixedGCLiveThresholdPercent=85  # Region存活率阈值
-XX:G1MixedGCCountTarget=8  # Mixed GC次数目标
```

---

### G1的优点

1. **可预测停顿**
   - 用户可以设置期望的停顿时间
   - G1会尽量满足

2. **无内存碎片**
   - 使用复制算法
   - 不需要额外的压缩阶段

3. **适合大堆**
   - 6GB以上的堆效果明显
   - 可以处理TB级堆（配合ZGC更好）

4. **并行和并发**
   - 充分利用多核CPU
   - 减少STW时间

---

### G1的缺点

1. **实现复杂**
   - 比传统收集器复杂得多
   - Bug相对多一些

2. **小堆效果不明显**
   - 堆小于4GB时，优势不明显
   - 可能不如Parallel GC

3. **Full GC性能差**
   - G1的Full GC是单线程的
   - 应尽量避免

---

### G1调优建议

1. **设置合理的停顿目标**
   ```bash
   -XX:MaxGCPauseMillis=200  # 根据业务需求调整
   ```

2. **监控Mixed GC频率**
   - 太频繁：提高IHOP阈值
   - 太少：降低IHOP阈值

3. **避免Humongous分配**
   - 大对象会导致性能下降
   - 调整Region大小

4. **升级到最新JDK**
   - G1在不断优化
   - JDK 11+的G1性能更好

---

## **有了 CMS，为什么还要引入 G1？**

虽然CMS已经很优秀，但仍有一些问题，G1应运而生。

### CMS的主要问题

#### 1. 内存碎片

**问题**：
- CMS使用标记-清除算法
- 长期使用后产生大量碎片
- 大对象无法分配

**后果**：
- 需要提前进行内存压缩
- 或者触发Full GC（Serial Old）

**G1的解决**：
- 使用复制算法
- 天然无碎片

---

#### 2. Concurrent Mode Failure

**问题**：
- 并发清除过程中，老年代空间不足
- 退化为Serial Old，长时间STW

**后果**：
- 停顿时间不可控
- 可能长达数秒

**G1的解决**：
- 可预测的停顿模型
- 不会出现模式失败

---

#### 3. 浮动垃圾

**问题**：
- 并发阶段产生的新垃圾无法处理
- 需要等到下次GC

**G1的解决**：
- 同样有浮动垃圾
- 但通过Mixed GC可以更频繁地清理

---

#### 4. CPU敏感

**问题**：
- 并发线程占用CPU
- CPU核数少时影响性能

**G1的解决**：
- 更好的并行度
- 更高效的线程管理

---

#### 5. 不支持大堆

**问题**：
- CMS在大堆（>8GB）上表现不佳
- GC时间长

**G1的解决**：
- 专为大堆设计
- Region化设计，扩展性好

---

### G1的优势

| 对比项 | CMS | G1 |
|--------|-----|----|
| 内存碎片 | 有 | 无 |
| 停顿时间 | 较短但不可控 | 可预测 |
| 并发失败 | 会退化 | 不会 |
| 大堆支持 | 一般 | 优秀 |
| 维护状态 | 已废弃 | 活跃开发 |

---

### 总结

**CMS的历史使命**：
- 在G1成熟之前，是最好的低停顿收集器
- 为并发收集积累了经验

**G1的未来**：
- JDK 9+的默认收集器
- 持续优化和改进
- 适合大多数场景

**选择建议**：
- JDK 8及以下：CMS或Parallel GC
- JDK 9+：G1（默认）
- 超低延迟：ZGC（JDK 11+）

## **垃圾收集器应该如何选择？**

选择合适的垃圾收集器对应用性能至关重要。

### 选择流程图

```
应用类型？
├─ 客户端应用
│  └─ Serial GC
│
└─ 服务端应用
   ├─ 堆大小？
   │  ├─ < 4GB
   │  │  └─ Parallel GC（吞吐量优先）
   │  │     或 G1 GC（低延迟优先）
   │  │
   │  ├─ 4-8GB
   │  │  └─ G1 GC（推荐）
   │  │
   │  └─ > 8GB
   │     └─ G1 GC 或 ZGC
   │
   └─ 延迟要求？
      ├─ 不敏感
      │  └─ Parallel GC
      │
      ├─ 中等（< 200ms）
      │  └─ G1 GC
      │
      └─ 极低（< 10ms）
         └─ ZGC 或 Shenandoah
```

---

### 具体建议

#### 1. 小型应用（堆 < 4GB）

**推荐**：Parallel GC

```bash
-XX:+UseParallelGC
-XX:+UseParallelOldGC
```

**理由**：
- 简单高效
- 吞吐量高
- 对小堆优化好

---

#### 2. 中型应用（堆 4-8GB）

**推荐**：G1 GC

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
```

**理由**：
- 平衡吞吐量和延迟
- 无内存碎片
- 可预测停顿

---

#### 3. 大型应用（堆 > 8GB）

**推荐**：G1 GC 或 ZGC

```bash
# G1
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200

# 或 ZGC（JDK 11+）
-XX:+UseZGC
```

**理由**：
- G1适合大堆
- ZGC适合超大堆和超低延迟

---

#### 4. 低延迟应用

**推荐**：ZGC 或 G1

```bash
# ZGC（JDK 11+）
-XX:+UseZGC

# 或 G1
-XX:+UseG1GC
-XX:MaxGCPauseMillis=100
```

**理由**：
- ZGC停顿时间<10ms
- G1可配置停顿目标

---

#### 5. 高吞吐量应用

**推荐**：Parallel GC

```bash
-XX:+UseParallelGC
-XX:GCTimeRatio=19  # 95%吞吐量
```

**理由**：
- 吞吐量优先
- 适合批处理、计算密集型应用

---

### 不同JDK版本的默认收集器

| JDK版本 | 默认收集器 |
|---------|-----------|
| JDK 6-7 | Parallel GC |
| JDK 8 | Parallel GC |
| JDK 9-10 | G1 GC |
| JDK 11+ | G1 GC |
| JDK 15+ | G1 GC（ZGC生产就绪） |

   - 记录GC日志

2. **选择收集器**
   - 根据应用特点选择

3. **调整参数**
   - 堆大小
   - 新生代比例
   - 停顿目标

4. **监控和迭代**
   - 持续监控GC指标
   - 根据实际情况调整
