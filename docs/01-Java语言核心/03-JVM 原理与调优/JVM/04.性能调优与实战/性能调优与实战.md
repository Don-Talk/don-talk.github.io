# **性能调优与实战**

## **用过哪些性能监控的命令行工具？** (jps, jstat, jmap, jstack等)

JDK提供了丰富的命令行工具，用于监控和诊断JVM性能问题。

### 1. jps - JVM Process Status Tool

**作用**：列出正在运行的Java进程

**常用命令**：
```bash
# 查看所有Java进程
jps

# 查看详细信息（包括main方法参数）
jps -l

# 查看JVM参数
jps -v
```

**输出示例**：
```
12345 MyApp
12346 Jps
23456 org.apache.catalina.startup.Bootstrap
```

---

### 2. jstat - JVM Statistics Monitoring Tool

**作用**：实时监控JVM统计信息（GC、类加载等）

**常用命令**：
```bash
# 监控GC情况，每1秒刷新一次
jstat -gc <pid> 1000

# 监控GC容量，每1秒刷新
jstat -gccapacity <pid> 1000

# 监控GC原因，每1秒刷新
jstat -gccause <pid> 1000

# 监控类加载
jstat -class <pid> 1000
```

**输出示例**（`jstat -gc`）：
```
 S0C    S1C    S0U    S1U      EC       EU        OC         OU       MC     MU    YGC     YGCT    FGC    FGCT     GCT
1024.0 1024.0  0.0   512.0   8192.0   4096.0   20480.0    10240.0   5120.0 4800.0  123    1.234   5     0.567   1.801
```

**字段说明**：
- S0C/S1C: Survivor区容量
- S0U/S1U: Survivor区已使用
- EC/EU: Eden区容量/已使用
- OC/OU: 老年代容量/已使用
- MC/MU: 元空间容量/已使用
- YGC: Young GC次数
- YGCT: Young GC总耗时
- FGC: Full GC次数
- FGCT: Full GC总耗时
- GCT: GC总耗时

---

### 3. jmap - Memory Map

**作用**：生成堆转储文件、查看内存使用情况

**常用命令**：
```bash
# 查看堆使用情况
jmap -heap <pid>

# 查看对象统计
jmap -histo <pid>

# 生成堆转储文件
jmap -dump:format=b,file=heap.hprof <pid>

# 只dump存活对象
jmap -dump:live,format=b,file=heap.hprof <pid>
```

**输出示例**（`jmap -heap`）：
```
Heap Configuration:
   MinHeapFreeRatio = 40
   MaxHeapFreeRatio = 70
   MaxHeapSize      = 2147483648 (2048MB)
   NewSize          = 715653120 (682MB)
   MaxNewSize       = 715653120 (682MB)
   OldSize          = 1431830528 (1365MB)

Heap Usage:
PS Young Generation
Eden Space:
   capacity = 536870912 (512MB)
   used     = 268435456 (256MB)
   free     = 268435456 (256MB)
   50.0% used
```

---

### 4. jstack - Stack Trace

**作用**：打印线程堆栈信息，用于排查死锁、CPU高等问题

**常用命令**：
```bash
# 打印线程堆栈
jstack <pid>

# 打印详细信息（包括锁信息）
jstack -l <pid>

# 导出到文件
jstack <pid> > thread.dump
```

**输出示例**：
```
"main" #1 prio=5 os_prio=0 tid=0x00007f8a1c008000 nid=0x1234 runnable
   java.lang.Thread.State: RUNNABLE
	at com.example.MyApp.main(MyApp.java:10)

"Thread-1" #2 prio=5 os_prio=0 tid=0x00007f8a1c009000 nid=0x1235 waiting
   java.lang.Thread.State: WAITING (parking)
	at sun.misc.Unsafe.park(Native Method)
	- parking to wait for <0x000000076b8c0000>
```

**线程状态**：
- NEW：新建
- RUNNABLE：运行中
- BLOCKED：阻塞
- WAITING：等待
- TIMED_WAITING：超时等待
- TERMINATED：终止

---

### 5. jinfo - Configuration Info

**作用**：查看和修改JVM参数

**常用命令**：
```bash
# 查看所有参数
jinfo <pid>

# 查看指定参数
jinfo -flag MaxHeapSize <pid>

# 动态修改参数（部分参数支持）
jinfo -flag +PrintGCDetails <pid>
```

---

### 6. jcmd - Diagnostic Command

**作用**：多功能诊断工具（JDK 7+）

**常用命令**：
```bash
# 列出所有Java进程
jcmd

# 查看帮助
jcmd <pid> help

# 触发GC
jcmd <pid> GC.run

# 生成堆转储
jcmd <pid> GC.heap_dump /tmp/heap.hprof

# 打印线程堆栈
jcmd <pid> Thread.print
```

---

### 工具对比表

| 工具 | 主要用途 | 特点 |
|------|---------|------|
| jps | 查看Java进程 | 简单快速 |
| jstat | 监控GC统计 | 实时性强 |
| jmap | 内存分析 | 可生成dump |
| jstack | 线程分析 | 排查死锁 |
| jinfo | 参数配置 | 可查看/修改 |
| jcmd | 综合诊断 | 功能全面 |

## **了解哪些可视化的性能监控工具？** (JConsole, VisualVM, MAT, Arthas)

### 1. JConsole

**简介**：JDK自带的图形化监控工具

**启动**：
```bash
jconsole
```

**功能**：
- 内存监控：堆、非堆、内存池
- 线程监控：线程数、死锁检测
- 类加载监控
- MBean管理
- VM概要信息

**优点**：
- JDK自带，无需安装
- 界面简洁

**缺点**：
- 功能相对简单
- 实时性一般

---

### 2. VisualVM

**简介**：强大的可视化监控工具（JDK 8自带，JDK 9+需单独下载）

**启动**：
```bash
jvisualvm
```

**功能**：
- 实时监控：CPU、内存、线程、类
- 性能分析：CPU Profiling、Memory Profiling
- 堆转储分析
- 线程Dump分析
- 插件扩展

**优点**：
- 功能强大
- 支持插件
- 免费开源

**缺点**：
- JDK 9+不再内置
- 大堆分析较慢

---

### 3. MAT (Memory Analyzer Tool)

**简介**：Eclipse开发的堆转储分析工具

**下载**：https://www.eclipse.org/mat/

**功能**：
- 分析heap dump文件
- 查找内存泄漏
- 支配树（Dominator Tree）
- 直方图（Histogram）
- OQL查询

**使用步骤**：
1. 生成heap dump：`jmap -dump:file=heap.hprof <pid>`
2. 用MAT打开heap.hprof
3. 查看Leak Suspects报告
4. 分析支配树，找到泄漏对象

**优点**：
- 专业的内存分析
- 强大的泄漏检测
- 处理大文件能力强

**缺点**：
- 只能分析dump文件
- 不能实时监控

---

### 4. Arthas

**简介**：阿里巴巴开源的Java诊断工具

**GitHub**：https://github.com/alibaba/arthas

**安装**：
```bash
wget https://arthas.aliyun.com/arthas-boot.jar
java -jar arthas-boot.jar
```

**常用命令**：
```bash
# 查看dashboard
dashboard

# 监控方法调用
monitor -c 5 com.example.MyClass myMethod

# 追踪方法调用路径
trace com.example.MyClass myMethod

# 查看方法参数和返回值
watch com.example.MyClass myMethod "{params, returnObj}"

# 反编译类
jad com.example.MyClass

# 热更新代码
redefine /path/to/MyClass.class
```

**优点**：
- 功能强大
- 支持线上诊断
- 可以热更新代码
- 社区活跃

**缺点**：
- 学习成本较高
- 需要谨慎使用（可能影响生产）

---

## **JVM 的常见参数配置知道哪些？** (-Xms, -Xmx, GC日志参数等)

### 1. 堆内存参数

```bash
# 堆初始大小
-Xms2g

# 堆最大大小
-Xmx2g

# 新生代大小
-Xmn512m

# 新生代比例（老年代:新生代）
-XX:NewRatio=2

# Eden与Survivor比例
-XX:SurvivorRatio=8
```

**建议**：
- `-Xms` 和 `-Xmx` 设置为相同值，避免堆动态调整
- 新生代占堆的1/3到1/4

---

### 2. 栈内存参数

```bash
# 线程栈大小
-Xss256k  # 默认1MB，可根据应用调整
```

**注意**：
- 栈太小容易StackOverflowError
- 栈太大会限制可创建的线程数

---

### 3. 元空间参数

```bash
# 元空间初始大小
-XX:MetaspaceSize=256m

# 元空间最大大小
-XX:MaxMetaspaceSize=512m
```

**建议**：
- 根据应用加载的类数量调整
- 动态类多的应用需要更大

---

### 4. GC收集器选择

```bash
# Serial GC
-XX:+UseSerialGC

# Parallel GC
-XX:+UseParallelGC
-XX:+UseParallelOldGC

# CMS GC (JDK 8)
-XX:+UseConcMarkSweepGC

# G1 GC (JDK 9+默认)
-XX:+UseG1GC

# ZGC (JDK 11+)
-XX:+UseZGC
```

---

### 5. GC日志参数

```bash
# JDK 8及之前
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/var/log/gc.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=5
-XX:GCLogFileSize=10M

# JDK 9+
-Xlog:gc*:file=/var/log/gc.log:time,uptime:filecount=5,filesize=10M
```

**日志内容**：
- GC类型（Young/Full）
- GC前后内存变化
- GC耗时
- GC原因

---

### 6. GC行为参数

```bash
# 最大GC停顿时间目标（G1）
-XX:MaxGCPauseMillis=200

# 触发并发标记的阈值（G1）
-XX:InitiatingHeapOccupancyPercent=45

# 对象年龄阈值
-XX:MaxTenuringThreshold=15

# 大对象直接进入老年代的阈值
-XX:PretenureSizeThreshold=1048576  # 1MB

# 禁用System.gc()
-XX:+DisableExplicitGC
```

---

### 7. 其他常用参数

```bash
# 开启OOM时自动生成heap dump
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/heapdump.hprof

# OOM时执行脚本
-XX:OnOutOfMemoryError="kill -9 %p"

# 打印JVM标志
-XX:+PrintCommandLineFlags

# 开启TLAB
-XX:+UseTLAB

# 指针压缩（64位JVM默认开启）
-XX:+UseCompressedOops
```

---

### 生产环境推荐配置

```bash
# 基础配置
-Xms4g -Xmx4g
-Xmn1g
-Xss256k

# G1 GC
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200

# 元空间
```

## **做过 JVM 调优吗？** (调优思路与流程)

JVM调优是一个系统工程，需要遵循科学的方法论。

### 调优目标

1. **降低延迟**：减少GC停顿时间
2. **提高吞吐量**：增加有效工作时间占比
3. **减少内存占用**：优化内存使用效率
4. **提高稳定性**：避免OOM等问题

---

### 调优流程

#### 第1步：建立基准

**收集当前性能数据**：
```bash
# 1. 记录当前JVM参数
java -XX:+PrintCommandLineFlags -version

# 2. 监控GC情况
jstat -gc <pid> 1000 > gc_stats.txt

# 3. 记录响应时间
# 通过应用日志或监控系统

# 4. 记录吞吐量
# QPS、TPS等指标
```

---

#### 第2步：发现问题

**常见问题**：
- GC频率过高
- GC停顿时间过长
- 内存使用率持续高
- 频繁Full GC
- OOM错误

**监控指标**：
```
重点关注：
- Young GC频率：应该几分钟一次
- Full GC频率：应该几小时甚至几天一次
- GC停顿时间：Young GC < 50ms, Full GC < 1s
- 堆使用率：稳定在60-70%
```

---

#### 第3步：定位问题

**使用工具诊断**：

1. **GC问题分析**
   ```bash
   # 分析GC日志
   jstat -gcutil <pid> 1000
   
   # 查看GC原因
   jstat -gccause <pid>
   ```

2. **内存泄漏分析**
   ```bash
   # 生成heap dump
   jmap -dump:live,format=b,file=heap.hprof <pid>
   
   # 用MAT分析
   # 查看Dominator Tree
   # 查找泄漏嫌疑对象
   ```

3. **CPU高问题分析**
   ```bash
   # 找到CPU高的线程
   top -Hp <pid>
   
   # 转换线程ID为16进制
   printf "%x\n" <tid>
   
   # 查看线程堆栈
   jstack <pid> | grep <hex_tid> -A 20
   ```

---

#### 第4步：制定方案

**根据问题类型选择优化方向**：

| 问题 | 可能原因 | 优化方案 |
|------|---------|----------|
| 频繁Young GC | 新生代太小 | 增大新生代 |
| 频繁Full GC | 内存泄漏/老年代小 | 排查泄漏/增大堆 |
| GC停顿时间长 | 收集器不合适 | 换G1/ZGC |
| 内存使用率高 | 对象过多 | 优化代码/缓存策略 |
| CPU高 | 死循环/频繁GC | 排查代码/优化GC |

---

#### 第5步：实施优化

**调整JVM参数**：

```bash
# 示例：优化频繁Full GC

# 原配置
-Xms2g -Xmx2g
-XX:+UseParallelGC

# 优化后
-Xms4g -Xmx4g              # 增大堆
-XX:+UseG1GC               # 换G1收集器
-XX:MaxGCPauseMillis=200   # 设置停顿目标
-XX:InitiatingHeapOccupancyPercent=45  # 提前触发GC
```

**优化代码**：
```java
// 优化前：创建大量临时对象
for (int i = 0; i < 1000000; i++) {
    String str = new String("hello");
}

// 优化后：复用对象
String str = "hello";
for (int i = 0; i < 1000000; i++) {
    // 使用str
}
```

---

#### 第6步：验证效果

**对比优化前后的指标**：

```bash
# 再次监控GC
jstat -gc <pid> 1000

# 对比：
# - GC频率是否降低？
# - GC停顿时间是否缩短？
# - 吞吐量是否提升？
# - 响应时间是否改善？
```

---

#### 第7步：持续监控

**建立监控体系**：
- Prometheus + Grafana监控JVM指标
- ELK分析GC日志
- 设置告警阈值

---

### 调优案例

#### 案例1：频繁Full GC

**现象**：
- 每10分钟一次Full GC
- 每次停顿3-5秒
- 用户感知明显

**排查**：
```bash
# 1. 查看GC日志
cat gc.log | grep "Full GC"

# 2. 生成heap dump
jmap -dump:live,format=b,file=heap.hprof <pid>

# 3. MAT分析发现：大量未关闭的数据库连接
```

**解决**：
1. 修复代码：使用try-with-resources关闭连接
2. 增大堆：-Xms4g -Xmx4g
3. 切换G1：-XX:+UseG1GC

**效果**：
- Full GC从10分钟一次降到几小时一次
- 停顿时间从3-5秒降到200ms以内

---

#### 案例2：内存泄漏

**现象**：
- 内存使用率持续上升
- 几天后OOM

**排查**：
```bash
# 定时生成heap dump
jmap -dump:live,format=b,file=heap1.hprof <pid>
sleep 3600
jmap -dump:live,format=b,file=heap2.hprof <pid>

# MAT对比两个dump，找出增长的对象
```

**发现**：
- HashMap持续增长
- 原因：缓存没有过期策略

**解决**：
- 使用LRU缓存（如LinkedHashMap或Caffeine）
- 设置缓存大小和过期时间

---

### 调优原则

1. **不要过早优化**：先确保功能正确
2. **基于数据决策**：用监控数据说话
3. **小步快跑**：每次只改一个参数
4. **充分测试**：生产变更前必须测试
5. **持续监控**：调优不是一次性的

## **CPU 占用过高怎么排查？**

CPU高是常见的性能问题，需要系统性地排查。

### 排查步骤

#### 第1步：确认问题

```bash
# 查看整体CPU使用率
top

# 查看Java进程的CPU使用
top -p <pid>
```

---

#### 第2步：找到CPU高的线程

```bash
# 查看进程内所有线程的CPU使用
top -Hp <pid>

# 输出示例：
# PID USER PR NI VIRT RES SHR S %CPU %MEM TIME+ COMMAND
# 12345 user 20 0 ... R 90.0 1.0 10:00.00 java
# 12346 user 20 0 ... S 5.0 1.0 0:10.00 java

# 记录CPU最高的线程ID（如12345）
```

---

#### 第3步：转换线程ID

```bash
# 将十进制线程ID转换为十六进制
printf "%x\n" 12345
# 输出：3039
```

---

#### 第4步：查看线程堆栈

```bash
# 查看该线程的堆栈信息
jstack <pid> | grep 0x3039 -A 20

# 输出示例：
# "Thread-1" #2 prio=5 os_prio=0 tid=0x00007f8a1c009000 nid=0x3039 runnable
#    java.lang.Thread.State: RUNNABLE
#     at com.example.HeavyCalculation.calculate(HeavyCalculation.java:15)
#     at com.example.HeavyCalculation.process(HeavyCalculation.java:10)
```

---

#### 第5步：分析代码

**常见原因**：

1. **死循环**
   ```java
   while (true) {
       // 没有退出条件
   }
   ```

2. **复杂计算**
   ```java
   // 大量数学运算
   for (int i = 0; i < Integer.MAX_VALUE; i++) {
       Math.sqrt(i);
   }
   ```

3. **频繁GC**
   - GC线程会占用CPU
   - 通过 `jstat -gc` 确认

4. **序列化/反序列化**
   - JSON、XML解析
   - 大对象序列化

5. **正则表达式**
   - 复杂的正则匹配
   - 回溯过多

---

#### 第6步：解决问题

**针对不同原因的解决方案**：

1. **死循环**：添加退出条件
2. **复杂计算**：优化算法、缓存结果
3. **频繁GC**：优化内存使用、调整GC参数
4. **序列化**：使用更高效的库（如Protobuf）
5. **正则**：优化正则表达式、使用预编译

---

### 自动化脚本

```bash
#!/bin/bash
# cpu_high.sh - CPU高问题排查脚本

PID=$1
if [ -z "$PID" ]; then
    echo "Usage: $0 <pid>"
    exit 1
fi

echo "=== Top 10 CPU Threads ==="
top -Hp $PID -bn1 | head -20

echo ""
echo "=== Converting TIDs to hex ==="
top -Hp $PID -bn1 | awk 'NR>7 {print $1}' | head -10 | while read tid; do
    printf "TID: %d -> Hex: 0x%x\n" $tid $tid
done

echo ""
echo "=== Thread Dumps ==="
jstack $PID | grep -A 20 "nid=0x"
```

**使用**：
```bash
chmod +x cpu_high.sh
./cpu_high.sh <pid>
```

## **内存飙高问题怎么排查？**

内存持续升高可能导致OOM，需要及时排查。

### 排查步骤

#### 第1步：确认内存趋势

```bash
# 实时监控内存使用
jstat -gc <pid> 1000

# 关注OU（老年代使用）是否持续增长
```

---

#### 第2步：生成Heap Dump

```bash
# 生成当前堆转储
jmap -dump:live,format=b,file=heap_$(date +%Y%m%d_%H%M%S).hprof <pid>

# 如果应用已挂掉，检查是否生成了OOM dump
ls -lh /var/log/oom*.hprof
```

---

#### 第3步：分析Heap Dump

**使用MAT分析**：

1. 打开MAT，加载heap dump
2. 查看"Leak Suspects"报告
3. 查看"Dominator Tree"
4. 找出占用内存最大的对象

**关键指标**：
- Shallow Heap：对象自身占用的内存
- Retained Heap：对象被回收后能释放的总内存

---

#### 第4步：定位泄漏点

**常见泄漏原因**：

1. **静态集合**
   ```java
   private static List<Object> cache = new ArrayList<>();
   // 只添加，不清理
   ```

2. **未关闭的资源**
   ```java
   Connection conn = DriverManager.getConnection(...);
   // 忘记close
   ```

3. **监听器未移除**
   ```java
   eventSource.addListener(listener);
   // 忘记removeListener
   ```

4. **ThreadLocal未清理**
   ```java
   threadLocal.set(value);
   // 忘记remove
   ```

5. **缓存无限制**
   ```java
   Map<String, Object> cache = new HashMap<>();
   // 没有大小限制和过期策略
   ```

---

#### 第5步：验证修复

```bash
# 修复代码后，重新部署
# 持续监控内存使用
jstat -gc <pid> 1000 > gc_monitor.log

# 观察内存是否稳定
```

### 实时监控脚本

```bash
#!/bin/bash
# memory_monitor.sh - 内存监控脚本

PID=$1
INTERVAL=${2:-5}  # 默认5秒

echo "Monitoring memory usage for PID: $PID"
echo "Interval: ${INTERVAL}s"
echo ""

while true; do
    TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')
    
    # 获取内存使用情况
    HEAP_INFO=$(jstat -gc $PID 2>/dev/null | tail -1)
    
    if [ -z "$HEAP_INFO" ]; then
        echo "Process $PID not found!"
        exit 1
    fi
    
    # 提取老年代使用率
    OU=$(echo $HEAP_INFO | awk '{print $8}')
    OC=$(echo $HEAP_INFO | awk '{print $7}')
    
    if [ "$OC" != "0" ] && [ -n "$OC" ]; then
        USAGE=$(awk "BEGIN {printf \"%.2f\", ($OU/$OC)*100}")
        echo "[$TIMESTAMP] Old Gen: ${OU}K / ${OC}K = ${USAGE}%"
    fi
    
    sleep $INTERVAL
done
```

**使用**：
```bash
chmod +x memory_monitor.sh
./memory_monitor.sh <pid> 5
```

## **频繁 minor gc 怎么办？**

频繁的Minor GC会影响应用性能。

### 判断是否"频繁"

**正常情况**：
- Minor GC间隔：几分钟到几十分钟
- GC耗时：< 50ms

**异常情况**：
- Minor GC间隔：几秒到几十秒
- GC耗时：> 100ms

---

### 原因分析

#### 1. 新生代太小

**现象**：
- Eden区很快填满
- GC日志显示频繁的Young GC

**解决**：
```bash
# 增大新生代
-Xmn1g  # 原来可能是512m

# 或调整比例
-XX:NewRatio=2  # 新生代占堆的1/3
```

---

#### 2. 对象分配速率过快

**现象**：
- 代码中创建大量短生命周期对象
- 如：循环中创建临时对象

**排查**：
```bash
# 监控对象分配速率
jstat -gcnew <pid> 1000
```

**解决**：
- 优化代码，减少对象创建
- 使用对象池
- 复用对象

```java
// 优化前
for (int i = 0; i < 1000000; i++) {
    StringBuilder sb = new StringBuilder();
    sb.append("item").append(i);
}

// 优化后
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000000; i++) {
    sb.setLength(0);  // 清空
    sb.append("item").append(i);
}
```

---

#### 3. 大对象进入新生代

**现象**：
- 大对象占用Eden空间
- 很快触发GC

**解决**：
```bash
# 让大对象直接进入老年代
-XX:PretenureSizeThreshold=1048576  # 1MB
```

---

#### 4. Survivor区太小

**现象**：
- Survivor区放不下存活对象
- 对象提前进入老年代
- 导致老年代也很快满

**解决**：
```bash
# 调整Survivor比例
-XX:SurvivorRatio=6  # 原来是8，减小比值增大Survivor
```

---

### 调优步骤

1. **监控GC频率**
   ```bash
   jstat -gc <pid> 1000
   ```

2. **分析GC日志**
   ```bash
   # 查看Young GC的频率和耗时
   cat gc.log | grep "Young GC"
   ```

3. **调整新生代大小**
   ```bash
   # 逐步增大，观察效果
   -Xmn512m → -Xmn768m → -Xmn1g
   ```

4. **优化代码**
   - 减少不必要的对象创建
   - 使用基本类型代替包装类型
   - 避免在循环中创建对象

5. **验证效果**
   - GC频率是否降低
   - 应用响应时间是否改善

## **频繁 Full GC 怎么办？**

Full GC是最严重的问题，必须立即处理。

### 判断标准

**正常情况**：
- Full GC间隔：几小时到几天
- GC耗时：< 1秒

**异常情况**：
- Full GC间隔：几分钟到几十分钟
- GC耗时：> 1秒，甚至几秒

---

### 原因分析

#### 1. 内存泄漏

**最常见的原因！**

**排查**：
```bash
# 生成多个heap dump，对比分析
jmap -dump:live,format=b,file=heap1.hprof <pid>
sleep 300
jmap -dump:live,format=b,file=heap2.hprof <pid>

# 用MAT对比，找出增长的对象
```

**解决**：修复泄漏代码

---

#### 2. 堆太小

**现象**：
- 内存使用率持续接近100%
- GC日志显示"Allocation Failure"

**解决**：
```bash
# 增大堆
-Xms4g -Xmx4g  # 原来是2g
```

---

#### 3. 元空间不足

**现象**：
- GC日志显示"Metaspace"
- 加载了大量类

**解决**：
```bash
# 增大元空间
-XX:MaxMetaspaceSize=512m  # 原来是256m
```

---

#### 4. System.gc()调用

**现象**：
- GC日志显示"System.gc()"
- 代码中显式调用

**解决**：
```bash
# 禁用System.gc()
-XX:+DisableExplicitGC
```

或者修改代码，移除`System.gc()`调用。

---

#### 5. CMS并发模式失败

**现象**：
- GC日志显示"Concurrent Mode Failure"
- 退化为Serial Old

**解决**：
```bash
# 降低触发CMS的阈值
-XX:CMSInitiatingOccupancyFraction=70  # 原来是92%
```

---

#### 6. 大对象分配

**现象**：
- 频繁分配大对象
- 老年代很快满

**解决**：
- 优化代码，减少大对象
- 增大堆
- 使用对象池

   - 修复内存泄漏
   - 调整JVM参数
   - 优化代码

---

### 预防措施

1. **代码审查**
   - 检查资源是否正确关闭
   - 检查缓存是否有过期策略
   - 检查静态集合的使用

## **有没有处理过内存泄漏问题？** (排查步骤与工具使用)

内存泄漏是Java应用中最棘手的问题之一。

### 什么是内存泄漏？

**定义**：不再使用的对象由于被意外引用，无法被GC回收，导致内存持续增长。

**特点**：
- 内存使用率持续上升
- GC无法回收
- 最终导致OOM

---

### 排查步骤

#### 第1步：确认泄漏

```bash
# 监控内存趋势
jstat -gc <pid> 1000 > memory.log

# 观察OU（老年代使用）是否持续增长
# 如果持续增长且不下降，可能存在泄漏
```

**判断标准**：
- Full GC后，内存没有明显下降
- 内存使用率呈上升趋势
- 应用运行时间越长，内存越高

---

#### 第2步：定位泄漏对象

**方法1：对比Heap Dump**

```bash
# 时间点1：生成dump
jmap -dump:live,format=b,file=heap_t1.hprof <pid>

# 等待一段时间（如30分钟）
sleep 1800

# 时间点2：生成dump
jmap -dump:live,format=b,file=heap_t2.hprof <pid>

# 用MAT对比两个dump
# Histogram → Compare Result
# 找出数量增长最多的类
```

**方法2：使用jmap histo**

```bash
# 定期执行，观察对象数量变化
jmap -histo <pid> | head -20

# 每隔5分钟执行一次
for i in {1..10}; do
    echo "=== Sample $i at $(date) ==="
    jmap -histo:live <pid> | head -20
    sleep 300
done
```

---

#### 第3步：分析引用链

**使用MAT**：

1. 打开heap dump
2. 找到可疑的类
3. 右键 → Path to GC Roots → exclude weak/soft references
4. 查看谁在引用这些对象

**常见泄漏模式**：

**模式1：静态集合**
```
GC Root (static field)
  ↓
HashMap
  ↓
泄漏的对象
```

**模式2：监听器**
```
GC Root (event source)
  ↓
Listener List
  ↓
泄漏的Listener
```

**模式3：ThreadLocal**
```
GC Root (Thread)
  ↓
ThreadLocalMap
  ↓
泄漏的对象
```

---

#### 第4步：定位代码

**根据引用链，找到对应的代码**：

```java
// 示例：发现是某个Cache类导致的泄漏
public class Cache {
    private static Map<String, Object> cache = new HashMap<>();
    
    public void put(String key, Object value) {
        cache.put(key, value);  // 只put，没有remove
    }
}
```

---

#### 第5步：修复泄漏

**修复方案**：

1. **添加清理逻辑**
   ```java
   public void remove(String key) {
       cache.remove(key);
   }
   ```

2. **使用弱引用**
   ```java
   private static Map<String, WeakReference<Object>> cache = new HashMap<>();
   ```

3. **使用LRU缓存**
   ```java
   // 使用LinkedHashMap实现LRU
   private static Map<String, Object> cache = new LinkedHashMap<>(100, 0.75f, true) {
       @Override
       protected boolean removeEldestEntry(Map.Entry eldest) {
           return size() > MAX_SIZE;
       }
   };
   ```

4. **使用专业缓存库**
   ```java
   // 使用Caffeine
   Cache<String, Object> cache = Caffeine.newBuilder()
       .maximumSize(1000)
       .expireAfterWrite(10, TimeUnit.MINUTES)
       .build();
   ```

---

### 常见泄漏场景

#### 1. 集合类泄漏

```java
// 错误示例
public class LeakExample {
    private static List<Object> list = new ArrayList<>();
    
    public void add(Object obj) {
        list.add(obj);  // 从未清理
    }
}

// 正确示例
public void addAndClean(Object obj) {
    list.add(obj);
    if (list.size() > MAX_SIZE) {
        list.remove(0);  // 移除最旧的
    }
}
```

---

#### 2. 资源未关闭

```java
// 错误示例
public void readFile() throws IOException {
    FileInputStream fis = new FileInputStream("test.txt");
    // 读取文件
    // 忘记close
}

// 正确示例
public void readFileCorrect() throws IOException {
    try (FileInputStream fis = new FileInputStream("test.txt")) {
        // 读取文件
        // 自动关闭
    }
}
```

---

#### 3. ThreadLocal泄漏

```java
// 错误示例
public class ThreadLocalLeak {
    private static ThreadLocal<List<Object>> tl = new ThreadLocal<>();
    
    public void use() {
        tl.set(new ArrayList<>());
        // 使用后未清理
    }
}

// 正确示例
public void useCorrect() {
    try {
        tl.set(new ArrayList<>());
        // 使用
    } finally {
        tl.remove();  // 务必清理
    }
}
```

---

#### 4. 内部类持有外部类引用

```java
// 错误示例
public class Outer {
    private byte[] data = new byte[1024 * 1024];  // 1MB
    
    class Inner {
        // 隐式持有Outer的引用
        public void doSomething() {
            // 即使只用少量数据，也会阻止Outer被回收
        }
    }
}

// 正确示例：使用静态内部类
static class StaticInner {
    // 不持有外部类引用
}
```

---

### 预防内存泄漏

1. **代码规范**
   - 及时关闭资源
   - 及时清理集合
   - 使用try-with-resources
   - ThreadLocal用完要remove

2. **代码审查**
   - 重点检查静态集合
   - 检查监听器注册/注销
   - 检查缓存策略

3. **自动化检测**
   - 使用静态分析工具（如SonarQube）
   - 单元测试中加入内存检测
   - CI/CD集成泄漏检测

4. **监控告警**
   - 监控内存趋势
   - 设置告警阈值
   - 定期分析heap dump

## **有没有处理过内存溢出问题？**

OOM（Out Of Memory）是Java应用最严重的故障之一。

### OOM的类型

#### 1. Java heap space

**原因**：堆内存不足

**错误信息**：
```
java.lang.OutOfMemoryError: Java heap space
```

**排查**：
```bash
# 检查堆配置
jmap -heap <pid>

# 生成heap dump
jmap -dump:live,format=b,file=heap.hprof <pid>

# 分析dump，找出大对象
```

**解决**：
- 增大堆：-Xmx4g
- 修复内存泄漏
- 优化代码，减少对象创建

---

#### 2. GC overhead limit exceeded

**原因**：GC时间占比过高（>98%），但回收效果差（<2%）

**错误信息**：
```
java.lang.OutOfMemoryError: GC overhead limit exceeded
```

**排查**：
```bash
# 查看GC日志
cat gc.log | grep "GC"

# 监控GC频率
jstat -gc <pid> 1000
```

**解决**：
- 增大堆
- 优化GC参数
- 修复内存泄漏

---

#### 3. Metaspace

**原因**：元空间不足

**错误信息**：
```
java.lang.OutOfMemoryError: Metaspace
```

**排查**：
```bash
# 查看元空间使用
jstat -gc <pid>

# 查看加载的类
jmap -clstats <pid>
```

**解决**：
- 增大元空间：-XX:MaxMetaspaceSize=512m
- 检查是否有大量动态类生成
- 检查类加载器泄漏

---

#### 4. StackOverflowError

**原因**：栈深度超过限制

**错误信息**：
```
java.lang.StackOverflowError
```

**排查**：
```bash
# 查看线程堆栈
jstack <pid>

# 检查是否有无限递归
```

**解决**：
- 修复递归逻辑
- 改用迭代
- 增大栈：-Xss2m

---

#### 5. Direct buffer memory

**原因**：直接内存不足

**错误信息**：
```
java.lang.OutOfMemoryError: Direct buffer memory
```

**排查**：
```bash
# 查看直接内存使用
jmap -heap <pid>
```

**解决**：
- 增大直接内存：-XX:MaxDirectMemorySize=2g
- 检查ByteBuffer是否正确释放
- 修复直接内存泄漏

---

### OOM应急处理

**当生产环境发生OOM时**：

1. **保护现场**
   ```bash
   # 如果配置了-XX:+HeapDumpOnOutOfMemoryError
   # 会自动生成heap dump
   
   # 否则手动生成
   jmap -dump:live,format=b,file=oom.hprof <pid>
   ```

2. **收集日志**
   ```bash
   # GC日志
   cp gc.log gc_before_restart.log
   
   # 应用日志
   cp app.log app_before_restart.log
   
   # 线程堆栈
   jstack <pid> > thread_before_restart.dump
   ```

3. **重启服务**
   ```bash
   # 重启应用
   systemctl restart app
   
   # 或滚动重启
   ```

4. **分析原因**
   - 分析heap dump
   - 查看GC日志
   - 检查最近的代码变更

5. **修复并验证**
   - 修复问题
   - 测试验证
   - 重新部署

---

### 预防OOM

1. **合理配置**
   ```bash
   # 根据服务器内存合理配置
   -Xms4g -Xmx4g
   -XX:MaxMetaspaceSize=512m
   -XX:MaxDirectMemorySize=1g
   ```

2. **启用OOM保护**
   ```bash
   -XX:+HeapDumpOnOutOfMemoryError
   -XX:HeapDumpPath=/var/log/oom.hprof
   -XX:OnOutOfMemoryError="/scripts/oom_handler.sh %p"
   ```

3. **监控告警**
   - 内存使用率 > 80% 告警
   - Full GC频率 > 1次/小时 告警
   - GC停顿时间 > 1s 告警

4. **压力测试**
   - 上线前进行压力测试
   - 长时间稳定性测试
   - 找出内存瓶颈

5. **代码质量**
   - 避免内存泄漏
   - 合理使用缓存
   - 及时释放资源
