### 什么是Redis？

Redis是一种基于内存的、键值对类型的NoSQL数据库。

#### 补充说明

**详细介绍下Bitmap(位图)**

Redis 的 Bitmap 并非独立的数据类型，而是基于 String 类型实现的一组位操作命令。它将字符串视为一个由 0 和 1 组成的位数组，其中每个 bit 代表一个状态的开关

- 底层结构：Bitmap 的底层是二进制安全的 String，最大长度为 512MB。因此，一个 Bitmap 最多可表示 `512 * 1024 * 1024 * 8 = 2^32` 位（约 43 亿），偏移量（offset）范围为 `[0, 2^32-1]`。
- 自动扩展：当使用 `SETBIT` 设置一个超出当前字符串长度的偏移量时，Redis 会自动用 0 填充中间的空白位，以扩展字符串。
- 空间效率：相比使用 `Set` 或 `Hash` 存储大量布尔值，Bitmap 的空间占用极小。例如，存储 1 亿用户的登录状态，Bitmap 仅需约 12.5MB，而 `Set` 方案则需要数百MB。

常用命令

| 命令           | 语法                                                         | 说明                                                         |
| :------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **`SETBIT`**   | `SETBIT key offset value`                                    | 设置指定偏移量 `offset` 的位为 `value` (0或1)。若 `key` 不存在则自动创建。时间复杂度 **O(1)**。 |
| **`GETBIT`**   | `GETBIT key offset`                                          | 获取指定偏移量 `offset` 的位值。若 `offset` 超出范围或 `key` 不存在，返回 0。时间复杂度 **O(1)**。 |
| **`BITCOUNT`** | `BITCOUNT key [start end]`                                   | 统计指定字节范围 `[start, end]` 内值为 1 的位数。`start` 和 `end` 以**字节**为单位。时间复杂度 **O(N)**。 |
| **`BITPOS`**   | `BITPOS key bit [start end]`                                 | 查找指定字节范围 `[start, end]` 内第一个值为 `bit` (0或1) 的**位偏移量**。时间复杂度 **O(N)**。 |
| **`BITOP`**    | `BITOP op destkey key [key ...]`                             | 对一个或多个 Bitmap 执行位运算（`AND`/`OR`/`XOR`/`NOT`），结果存入 `destkey`。时间复杂度 **O(N)**。 |
| **`BITFIELD`** | `BITFIELD key [GET type offset] [SET type offset value] ...` | 在单个命令中对多个位域执行 `get`/`set`/`incrby` 操作，支持指定数据类型（如 `u8`, `i32`）和溢出策略。每个子命令时间复杂度 **O(1)**。 |



详细介绍下HyperLogLog

HyperLogLog是一种概率数据结构，用于高效估算一个集合中不重复元素的数量（基数）。

特点：

- 空间占用极小：无论数据量多大，内存占用基本固定（Redis中约为12KB）
- 结果近似：估算结果存在一定误差（标准误差约为0.81%），但足以满足绝大多数业务需求

非常适合统计网站UV、App日活、广告独立曝光等大规模去重计数场景。

| 特性         | HyperLogLog                    | Set                      | Bitmap                   |
| :----------- | :----------------------------- | :----------------------- | :----------------------- |
| **内存占用** | **固定 (~12KB)**，与数据量无关 | 随数据量线性增长，巨大   | 取决于 ID 范围，可能很大 |
| **结果精度** | **近似值** (误差 ~0.81%)       | **精确值**               | **精确值**               |
| **核心功能** | 估算基数                       | 精确去重、查询、删除     | 精确去重、按位操作       |
| **适用场景** | 大规模、允许误差的计数         | 小规模、需精确操作的场景 | ID 范围不大的精确去重    |



详细介绍下GEO

GEO是3.2版本引入的功能，专门用于存储和查询地理位置（经纬度）信息。它基于有序集合ZSet实现，通过GeoHash编码将二维坐标映射为一维分数，从而高效支持“附近的人/店”、“两点距离”等LBS场景。 

**核心概念与原理**

- 本质：GEO并非独立的数据结构，其底层是Sorted Set
  - member：成员，即地点名称（如“Beijing”）
  - score：分数，是该地点经纬度GeoHash编码后得到的52位整数
- 坐标系统：使用WGS84坐标系
  - 经度（longitude）：-180°到180°
  - 纬度（latitude）：-85.05112878° 到 85.05112878°
- GeoHash编码：一种将二维经纬度编码为一堆字符串的算法
  - 原理：通过二分法交替划分经度和纬度区间，生成二进制串，再转为Base32字符串
  - 特性：字符串前缀相同，地理位置越接近。Redis利用此特性，通过查询目标区域及周围8个区域的ZSet分数范围，实现高效的范围查询

**常用命令速查**

| 命令                | 作用                                       | 示例                                                   |
| :------------------ | :----------------------------------------- | :----------------------------------------------------- |
| `GEOADD`            | 添加一个或多个地点的经纬度。               | `GEOADD cities 116.404269 39.91582 "Beijing"`          |
| `GEOPOS`            | 获取一个或多个地点的经纬度。               | `GEOPOS cities "Beijing"`                              |
| `GEODIST`           | 计算两个地点间的直线距离。                 | `GEODIST cities "Beijing" "Shanghai" km`               |
| `GEOHASH`           | 获取一个或多个地点的 GeoHash 字符串。      | `GEOHASH cities "Beijing"`                             |
| `GEORADIUS`         | 以给定经纬度为中心，查询指定半径内的地点。 | `GEORADIUS cities 116.404269 39.91582 500 km WITHDIST` |
| `GEORADIUSBYMEMBER` | 以指定地点为中心，查询其指定半径内的地点。 | `GEORADIUSBYMEMBER cities "Beijing" 500 km`            |



Redis为什么使⽤ hash 类型⽽不使⽤ string 类型序列化存储？

**核心思想：结构化 vs. 扁平化**

- String + 序列化：

  将一个对象的所有字段“拍平”，用一个序列化工具（如 JSON, MessagePack, Java Serializable）打包成一个大的字符串，然后整体存到一个 Key 下。

  **例如**：`user:1000` -> `"{"id":1000,"name":"Alice","age":30,"city":"Beijing"}"`

- Hash：

  将一个对象的每个字段作为 Hash 中的一个独立的 Field-Value 对，存储在一个 Key 下。

  **例如**：`user:1000` -> `{ "id": "1000", "name": "Alice", "age": "30", "city": "Beijing" }`

**详细对比分析**

| 特性                | String + 序列化 (JSON/XML等)                                 | Hash                                                         | 胜出方 & 原因                   |
| :------------------ | :----------------------------------------------------------- | :----------------------------------------------------------- | :------------------------------ |
| **1. 读写单个字段** | **差**。必须读取整个字符串，反序列化，修改字段，再序列化整个对象，最后写回。网络开销和CPU开销巨大。 | **优**。可直接使用 `HGET user:1000 name` 或 `HMGET user:1000 name age` 只读写需要的字段。 | **Hash**                        |
| **2. 内存占用**     | **通常更高**。序列化后的字符串包含键名、格式符号（如引号、冒号、花括号），有冗余。且不同大小的字符串无法压缩。 | **通常更低**。Hash 内部结构紧凑，特别是小 Hash 会使用 **ziplist（压缩列表）** 编码，极大地节省内存。Redis 会对 field 和 value 进行优化存储。 | **Hash** (尤其当字段多或值小时) |
| **3. 灵活性**       | **差**。要增加一个字段，必须读取整个对象，修改，再写回。无法单独操作。 | **优**。可以动态地增加、删除、修改任意字段，无需影响其他字段。`HSET`， `HDEL`。 | **Hash**                        |
| **4. 部分更新**     | **不支持**。任何字段的修改都导致整个值被重写。               | **原生支持**。只更新变化的字段。                             | **Hash**                        |
| **5. 原子性操作**   | **弱**。需要客户端自己保证多个操作的原子性（如使用事务或Lua脚本）。 | **强**。所有 Hash 命令都是原子的，天然支持对多个字段的原子读写。 | **Hash**                        |
| **6. 大数据量遍历** | **可行但低效**。需要一次性加载所有数据到客户端，内存压力大。 | **高效**。可以使用 `HSCAN` 命令渐进式地、分批地遍历字段，避免阻塞服务器。 | **Hash**                        |
| **String 的优势**   | **简单直观**。对于结构简单、不常变动、或者需要作为一个整体传输的对象（如配置项、整个会话），序列化方案更直接。 | **无**                                                       | **String** (特定场景)           |
| **String 的优势**   | **兼容性**。序列化格式（如 JSON）是跨语言的标准，易于与其他系统集成。 | **无**                                                       | **String** (特定场景)           |

为什么Hash比String更省内存？

这是Hash最大的优势之一，其核心在于ziplist（压缩列表）编码。

当一个Hash满足以下条件时，Redis会使用ziplist来存储，而不是标准的哈希表（HashTable）：

1. 哈希对象保存的所有字段名和值的字符串长度都小于`hash-max-ziplist-value`（默认64字节）
2. 哈希对象保存的字段数量小于`hash-max-ziplist-entries`（默认512个）

**ziplist的特点：**

- 连续内存：他将所有的field-value对紧挨着存放在一块连续的内存中，像数组一样
- 无冗余指针：不像哈希表那样为每个entry分配额外的指针空间
- 编码优化：对小整数和短字符串使用特殊的编码方式，进一步压缩存储空间

**举例说明**：
假设有一个用户对象 `user:1000`，包含 `name`, `age`, `city` 三个字段。

- **String 序列化 (JSON)**：
  - 存储内容：`"{\"name\":\"Alice\",\"age\":30,\"city\":\"Beijing\"}"`
  - 包含了大量的语法字符（`{ } " , :`）和重复的键名。
- **Hash (使用 ziplist)**：
  - 存储内容可以看作是：`[name, Alice, age, 30, city, Beijing]`
  - 只有纯粹的数据，没有任何格式符号和重复键名，并且紧密排列。

因此，对于存储大量小型对象（如数百万用户资料），使用 Hash 可以显著减少 Redis 的内存占用，降低成本。

### Redis为什么快？

> 总结：
> Redis快的原因：内存存储奠定基础，单线程+I/O多路复用解决并发瓶颈，定制化数据结构最小化操作耗时，再加上一系列工程优化，最终实现了“高性能、低延迟”的特性，适合作为缓存、计数器、消息队列等场景的核心组件。

1. 基于内存存储（核心原因）

   Redis的数据默认存储在内存中，而内存的读写速度比磁盘快几个数量级。

   虽然Redis也支持持久化（RDB/AOF），但持久化是异步或定期执行的，不影响正常的内存读写性能。

2. 单线程模型（避免上下文切换和锁竞争）

   Redis6.0之前采用单线程处理网络I/O和命令执行（6.0之后引入多线程仅用于网络读写，命令执行仍单线程），这种设计的好处：

   - 无上下文切换开销：多线程需要频繁切换CPU上下文，单线程避免了这部分损耗
   - 无锁竞争：多线程操作共享数据时需要加锁，锁会导致等待和性能下降；
   - 简化设计：单线程逻辑更简单，容易保证命令执行的原子性

3. 高效的数据结构

   Redis 针对不同的数据类型设计了定制化的高效数据结构，而非直接使用语言内置结构：

   - String：动态字符串，相比C字符串，预分配空间减少内存重分配次数，且二进制安全（可存储任意数据，如图片、序列化对象） 
   - Hash：当字段少是用压缩列表（ziplist）（连续内存，节省空间且遍历快）；字段多时转哈希表，兼顾查询和更新效率
   - List：双向链表（quicklist，结合ziplist和linkedlist的优点，平衡内存和性能）或压缩列表，支持O(1)头尾插入、删除
   - Set：整数集合（intset，元素全为整数且数量少时，紧凑存储）或哈希表
   - ZSet：跳表+哈希表，跳表支持O(logN)范围查询和排序，哈希表实现O(1)分数/成员查询

4. I/O多路复用（高效处理并发连接）

   Redis使用I/O多路复用技术监听多个客户端连接的事件，用一个线程即可高效处理数千级别的并发请求：

   - 传统阻塞I/O模型中，一个连接对应一个线程，线程数过多会导致资源耗尽
   - I/O多路复用让单线程能同时监控多个连接，仅在连接有数据可读、可写时才处理，避免了无效等待，极大提升了并发处理能力

5. 优化的命令执行流程

   - 命令队列：客户端发送的命令先进入队列，按顺序执行（单线程保证原子性），避免乱序执行导致的数据不一致
   - O(1)/O(logN)时间复杂度：大部分常用命令都是常数时间或对数时间复杂度，极少有O(N)的高成本命令（除非主动使用KEYS * 这类全量扫描命令，生产环境应避免）
   - 管道（Pipeline）：支持将多个命令打包一次性发送，减少网络往返次数(RTT)，大幅提升批量操作的吞吐量

6. 其他优化细节

   - 内存分配器：使用jemalloc替代系统默认的malloc，jemalloc按size class分配内存，减少内存碎片，提升内存利用率。
   - 惰性删除+定期删除：过期键的删除策略结合了惰性删除（访问时检查是否过期）和定期删除（周期性抽样删除过期键），避免集中删除导致的性能波动
   - 持久化优化：RDB是快照式持久化（fork子进程执行，不阻塞主线程），AOF重写也是fork子进程，且支持appendfsync everysec(每秒刷盘，平衡安全性和性能)

### Redis早期为什么选择单线程？

> 总结：
>
> 1. 避免CPU切换上下文还有多线程锁竞争这些开销
> 2. Redis的瓶颈不在CPU，在内存、网络带宽这些
> 3. 单线程天然的保证了执行命令的原子性

Redis在6.0版本之前，其核心网络模型采用单线程，主要基于以下几点考量：

- **性能足够：瓶颈不在CPU**

  Redis将数据存储于内存，执行命令是纯内存操作，速度极快。官方指出，Redis的性能瓶颈通常在内存容量或网络带宽，而非CPU计算能力。

  在普通Linux机器上，单个Redis实例的QPS可达8万+，使用pipeline批量请求时甚至能接近100万。

  这意味着，即便单线程已经能够轻松支撑极高并发，引入复杂的多线程并不能带来同比例的收益。

- **避免开销：省去线程切换和锁竞争**

  多线程虽然能利用多核，但也带来了显著的额外开销：

  - 上下文切换：CPU在多个线程之间切换，需要保存和恢复寄存器、栈等状态，消耗客观
  - 同步开销：为保证数据一致性，多线程访问共享数据需引入锁（如互斥锁）。这不仅有加锁、解锁的性能损耗，还可能导致死锁或线程阻塞，使问题排查变得极为复杂。

  Redis通过单线程+I/O多路复用，用少量线程即可高效管理海量并发连接，完美避开了上述开销。



### Redis6.0使用多线程是怎么回事？

Redis 6.0的“多线程”不是指命令的执行是多线程，而是引入了多线程I/O，旨在解决网络瓶颈，同时保留核心的单线程命令执行模型。

**🤔 为什么引入多线程？**

Redis的性能瓶颈通常不在CPU，而在于内存大小和网络带宽。在6.0版本之前，其单线程模型（一个主线程处理所有请求）存在以下局限：

- 网络I/O瓶颈：随着网络带宽的提升和高并发场景的出现，单个线程处理网络读写的CPU时间占比越来越高，容易达到瓶颈
- 多核资源浪费：主线程只能利用一个CPU核心，无法充分发挥多核服务器的性能

因此Redis 6.0选择将耗时的网络I/O操作进行并行化，而命令执行依旧保持单线程，以最小的改动换区最大的性能收益。

**⚙️ 多线程 I/O 的工作原理**

Redis 6.0的多线程 I/O模型可以概括为：主线程+多个I/O线程。

- 主线程：负责接收新连接、事件循环调度，以及所有命令的串行执行
- I/O线程：仅负责并行处理网络数据的读写和协议解析，不执行任何命令，也不直接访问数据

典型处理流程：

1. 接收连接：主线程accept新连接，并将socket放入待处理队列
2. 分发读任务：当socket可读时，主线程将连接通过轮询分配给I/O线程，自己也参与读取
3. 并行读&解析：所有I/O线程并行从socket读取数据并解析成命令对象。主线程会阻塞等待所有I/O完成
4. 串行执行：主线程从队列中取出已解析的命令，单线程顺序执行。此过程无锁，保证了原子性
5. 分发写任务：主线程将执行结果放入回复队列，并分配给I/O线程进行回写。主线程同样会阻塞等待
6. 并行写：I/O线程并行将数据写回客户端，完成后清空队列，等待下一轮事件

设计特点：

- 读写分离：I/O线程在同一时刻要么全部读，要么全部写，避免复杂的锁竞争
- 主线程参与：主线程也分担一部分I/O工作，以充分利用多核
- 无数据竞争：因为数据访问仍在主线程，所以无需为Redis的核心数据结构加锁

**🛠️ 如何配置与使用？**

1. 默认状态

多线程 I/O 在 Redis 6.0 及以后版本中**默认是关闭的**。即使您的服务器是多核CPU，也需要手动配置才能启用该功能。

2. 配置方法

必须在 `redis.conf` 配置文件中**静态配置**，重启 Redis 服务后生效。**不支持**在运行时通过 `CONFIG SET` 命令动态修改。

```java
# 在 redis.conf 中添加或修改以下行

# 开启 IO 线程数量。建议设置为 CPU 核心数 - 1，且最大值不要超过 8。
# 例如，在 8 核 CPU 上，可以设置为 6。
io-threads 4

# 是否让 IO 线程也负责读取客户端数据和解析协议。默认为 no。
# 设置为 yes 才能发挥多线程 I/O 的最大效能。
io-threads-do-reads yes
```

3. 配置建议

- **CPU 核数 < 4**：通常无需开启，多线程带来的调度开销可能大于性能收益。
- **4 核 CPU**：建议 `io-threads` 设为 **2 或 3**。
- **8 核 CPU**：建议设为 **6**。
- **> 8 核 CPU**：收益会递减，不建议超过 **8**，过多的线程反而会成为负担。

4. 注意事项

- **SSL/TLS 限制**：当 Redis 启用了 SSL/TLS 加密时，**不支持**多线程 I/O 功能。
- **适用场景**：如果您的服务器 CPU 使用率不高，但吞吐量已达上限，这通常意味着瓶颈在于网络或客户端。此时开启多线程 I/O 可能会有帮助。但如果瓶颈在内存或命令执行本身，则提升不明显。

**十、事务与管道**

1. 事务（保证原子性，但不支持回滚）

Redis 事务通过 `MULTI`、`EXEC` 包裹命令，**按顺序执行，不被其他命令打断**，但某条命令错误不会导致整个事务回滚（仅跳过错误命令）。

| 命令        | 作用                                 | 示例                                                         |
| ----------- | ------------------------------------ | ------------------------------------------------------------ |
| `MULTI`     | 开启事务                             | `MULTI`                                                      |
| `EXEC`      | 执行事务队列中的命令                 | `EXEC`                                                       |
| `DISCARD`   | 取消事务（清空队列）                 | `DISCARD`                                                    |
| `WATCH key` | 监控键，若被修改则事务取消（乐观锁） | `WATCH balance` → `MULTI` → `DECR balance 100` → `EXEC`（若balance被改则EXEC返回nil） |

2. 管道（Pipeline）

将多个命令打包发送，**减少网络往返时间（RTT）**，提升性能（非原子性，命令按顺序执行但可能被其他客户端命令穿插）。

示例（Redis CLI）：

```java
# 管道命令（Linux shell）
echo -e "SET a 1\nSET b 2\nGET a\nGET b" | redis-cli --pipe
```

**十一、持久化相关命令**

Redis 支持 RDB（快照）和 AOF（日志）两种持久化方式，相关命令：

| 命令           | 作用                                | 示例                    |
| -------------- | ----------------------------------- | ----------------------- |
| `SAVE`         | 同步生成RDB快照（阻塞主线程，慎用） | `SAVE`                  |
| `BGSAVE`       | 异步生成RDB快照（后台执行，不阻塞） | `BGSAVE`                |
| `LASTSAVE`     | 获取最后一次成功生成RDB的时间戳     | `LASTSAVE` → 1644567890 |
| `BGREWRITEAOF` | 异步重写AOF文件（压缩冗余命令）     | `BGREWRITEAOF`          |

**十二、键（Key）管理命令**

用于查找、删除、过期等操作：

| 命令                                        | 作用                                         | 示例                                  |
| ------------------------------------------- | -------------------------------------------- | ------------------------------------- |
| `KEYS pattern`                              | 查找匹配模式的键（生产环境慎用，可能阻塞）   | `KEYS user:*` → 查找所有user:开头的键 |
| `SCAN cursor [MATCH pattern] [COUNT count]` | 渐进式遍历键（非阻塞，替代KEYS）             | `SCAN 0 MATCH user:* COUNT 10`        |
| `DEL key1 key2 ...`                         | 删除键（返回成功删除的数量）                 | `DEL name age` → 2                    |
| `EXISTS key`                                | 判断键是否存在（存在返回1，否则0）           | `EXISTS name` → 1                     |
| `EXPIRE key secs`                           | 设置键的过期时间（秒）                       | `EXPIRE name 3600` → 1                |
| `PEXPIRE key ms`                            | 设置键的过期时间（毫秒）                     | `PEXPIRE name 3600000`                |
| `TTL key`                                   | 查看键的剩余过期时间（-1永不过期，-2已过期） | `TTL name` → 3599                     |
| `PTTL key`                                  | 查看键的剩余过期时间（毫秒）                 | `PTTL name` → 3599000                 |

以上是 Redis 最常用的命令，覆盖了日常开发的大部分场景。实际使用时需注意**命令的时间复杂度**（如`KEYS`是O(N)，大集群下会阻塞），优先选择高效命令（如`SCAN`替代`KEYS`、`HGETALL`慎用大哈希）。





### 单线程的Redis QPS能到多少？

> 总结：
>
> 理论峰值10万到12万，实际生产中3万到6万

首先需要明确，我们通常说的“Redis是单线程的”，指的是其核心命令执行流程由单个主线程完成。从Redis 4.0和6.0版本开始，后台线程和I/O多线程已被引入以处理特定耗时任务，但这并不改变命令执行的单线程模型。

在讨论QPS时，通常区分两种场景：

- 理论峰值

  ​	在理想的基准测试环境下，使用redis-benchmark对简单命令（如GET/SET）进行压测，并配合pipeline和多核CPU，单个Redis实例的QPS通常能达到10万至12万左右。

- 生产环境（线上实际）

  ​	线上业务的复杂度远超基准测试，实际QPS会受命令类型、数据大小、网络状况等多种因素影响，通常表现如下：

  - 常见范围：多数线上服务的QPS稳定在3万至6万之间
  - 复杂场景：若业务涉及大key、复杂命令（如SORT、ZUNIONSTORE）、跨机房高延迟网络，或AOF配置为always刷盘等，QPS可能降至数万甚至更低

核心瓶颈：

Redis单实例的QPS天花板主要受限于单个CPU核心的性能，而非多线程并发能力。一旦命令执行本身成为瓶颈（如处理大key、执行慢命令），及时增加客户端并发也无法提升QPS。。





## 持久化

### Redis持久化方式有哪些？

主要有两种，RDB和AOF。

RDB通过创建时间点快照来实现持久化，AOF通过记录每个写操作命令来实现持久化。

这两种方式可以单独使用，也可以同时使用。这样就可以波爱护增Redis服务器在重启后不丢失数据，通过RDB和AOF文件来恢复内存中原有的数据。



### RDB

RDB（Redis Database Backup）是一种常见的快照式持久化方式。它将Redis在内存中的数据以二进制文件的形式定期保存到磁盘上，形成一个快照文件（通常名为**dump.rdb**）。

#### **工作原理：**

1. 触发机制：RDB可以手动触发或自动触发
   - 手动触发：使用命令SAVE或BGSAVE。SAVE会阻塞主进程，直到快照完成；BGSAVE是异步的，使用fork子进程生成快照，不会阻塞主进程
   - 自动触发：根据配置文件中的规则，当满足一定条件时（如在指定时间内有指定数量的键发生变化），Redis会自动执行BGSAVE  
2. 快照过程：
   - Redis fork出子进程
   - 子进程将内存数据序列化为RDB文件
   - 完成后，替换旧的RDB文件
   - 如果配置了压缩，文件会使用LZF算法压缩以节省空间
3. 加载过程：Redis启动时，如果存在RDB文件，会自动加载它来恢复数据。加载速度较快，因为是直接从二进制文件读取。

#### **RDB的配置：**

在Redis的配置人间redis.conf中，可以设置RDB的相关参数：

- 保存策略：例如 save 900 1表示如果在900秒内至少有1个键被修改，则出发快照。多个save规则可以并存
- 文件路径：dbfilename "dump.rdb"指定文件名，dir "/path/to/dir"指定目录
- 压缩：让rdbcompression yes启用压缩
- 校验： rdbchecksum yes启用CRC64校验以确保文件完整性
- 停止写入：stop-writes-on-bgsave-error yes 如果快照失败，停止写入以避免数据不一致 

#### 示例配置片段:

```java
save 3600 1
save 300 100
save 60 10000
dbfilename dump.rdb
dir ./
rdbcompression yes
```

#### **RDB的优缺点**

| 方面           | 优点                                                       | 缺点                                                         |
| -------------- | ---------------------------------------------------------- | ------------------------------------------------------------ |
| **性能**       | 生成快照时主进程不阻塞（使用 BGSAVE），适合高负载场景。    | fork 子进程需要消耗内存（几乎双倍于当前数据集），可能导致短暂延迟。 |
| **数据完整性** | 恢复速度快，文件紧凑，便于备份和传输。                     | 只能恢复到最后一次快照点，中间的修改可能丢失（RPO 较高）。   |
| **适用场景**   | 适合对数据丢失有一定容忍度的应用，如缓存系统或非实时数据。 | 不适合对数据零丢失要求高的场景（如金融系统）。               |
| **文件大小**   | 压缩后文件较小，易于存储。                                 | 如果数据频繁变化，快照文件可能较大。                         |



### AOF

AOF(Append Only File)是一种日志式的持久化机制，它将每条写命令追加到AOF文件（通常名为appendonly.aof）中，形成一个操作日志。这种方式更加注重数据完整性，适合对数据丢失零容忍的场景。

#### AOF的工作原理

- 记录方式：Redis将每条写命令（如SET、DEL）以Redis协议格式追加到AOF文件。
- 同步策略（fsync）：
  - always：每条命令都同步到磁盘（最安全，但性能低）
  - everysec：每秒同步一次（默认，平衡安全与性能）
  - no：有操作系统决定（最快，但风险高）
- 重写过程（Rewrite）
  - AOF文件会随着操作增长变大，Redis可自动或手动触发BGREWRITEAOF，重写为最小命令集的等效文件，减少大小
  - 重写时fork子进程，不阻塞主进程
- 恢复过程：启动时，Redis重放AOF文件中的所有命令，重建数据集。恢复时间随文件大小而定。

#### AOF的配置

在 redis.conf 中配置：

- appendonly yes：启用 AOF。
- appendfilename "appendonly.aof"：文件名。
- appendfsync everysec：同步策略。
- auto-aof-rewrite-percentage 100：当文件增长 100% 时自动重写。
- auto-aof-rewrite-min-size 64mb：最小重写阈值。
- aof-use-rdb-preamble yes：混合模式，使用 RDB 格式作为 AOF 前缀，提高效率。

#### AOF的优缺点

- 优点：数据丢失最小（可达秒级），文件可读（纯文本命令），异域审计
- 缺点：文件较大、恢复慢（需重放名命令）、性能开销高（）



### RDB和AOF对比

| 方面           | RDB (快照式)                           | AOF (日志式)                         |
| -------------- | -------------------------------------- | ------------------------------------ |
| **持久化机制** | 定期快照整个数据集                     | 追加每条写命令到文件                 |
| **数据完整性** | 可能丢失最后快照后的数据（RPO 分钟级） | 丢失最小（可配置到秒级，甚至零丢失） |
| **文件大小**   | 较小（支持压缩）                       | 较大（随操作增加，可通过重写优化）   |
| **恢复速度**   | 快（直接加载二进制）                   | 慢（需重放所有命令）                 |
| **性能开销**   | 低（间歇 fork）                        | 高（每操作追加，取决于 fsync）       |
| **适用场景**   | 缓存、非实时数据，对丢失有容忍         | 高可靠性需求，如数据库、金融系统     |
| **文件格式**   | 二进制，不易读                         | 文本（Redis 命令），易读易修改       |
| **潜在风险**   | fork 时内存峰值高，崩溃中途数据不一致  | 文件损坏可能需手动修复               |

总体上，RDB更搞笑但安全性低；AOF更安全但资源消耗大。Redis支持混合模式（AOF以RDB格式开头），结合两者优点。

### RDB和AOF如何选择？

- 根据业务需求：
  - 高性能、低丢失容忍：优先RDB，如缓存服务，可接受分钟级数据丢失
  - 高可靠性、零丢失：优先AOF，如订单系统、金融交易，需要每秒持久化
  - 推荐组合：同时启用RDB和AOF，Redis优先加载AOF，RDB作为备份。混合模式（aof-use-rdb-preamble yes）可优化AOF文件大小和恢复速度。
- 考虑资源：
  - 内存大（>10GB）：RDB frok可能卡顿，偏向AOF
  - 磁盘I/O瓶颈：RDB更好，避免AOF的频繁写入
  - 数据规模：小数据集用RDB；大数据集用AOF+重写
- 风险评估：如果服务器崩溃频繁，用AOF；否则RDB够用。

### 应用建议

- 主从复制：主节点用AOF，确保从节点同步完整数据；从节点可选RDB节省资源。
- 备份策略：定期复制RDB/AOF文件到远程（如S3），使用BGSAVE或BGREWRITEAOF手动备份。结合Sentinel或Cluster实现高可用
- 监控与优化：
  - 用INFO Persistence监控最后保存时间、AOF大小。
  - 自动重写AOF防止文件膨胀
  - 测试恢复：定期模拟崩溃，验证恢复时间
- 常见pitfalls：
  - 避免SAVE在生产中使用（阻塞）
  - AOF用SSD磁盘提升性能
  - 大流量下，调用AOF重写阈值避免频繁fork

### 示例生产配置

假设一个中等负载的电商缓存系统（优先RDB，辅助AOF）：

```java
# RDB 配置

save 300 100
save 60 10000

dir /var/redis/data

# AOF 配置
```

### Redis如何恢复数据？

Redis作为内存数据库，默认数据在重启后丢失，但通过持久化机制可以实现数据恢复。

恢复过程发生在Redis服务启动时，自动从持久化文件中加载数据。如果没有持久化文件或文件损坏，数据将从0开始。 

#### 1.RDB方式恢复数据

RDB是快照式持久化，恢复时直接加载二进制文件，速度快。

- 自动恢复：
  - Redis启动时，会检查配置目录（dir参数指定，通常是当前目录）下的RDB文件（默认dump.rdb，由dbfilename指定）
  - 如果文件存在且完整（校验和通过，如果启用rdbchecksum yes），Redis会自动加载它
  - 加载过程：直接将文件中的数据反序列化到内存中，通常只需要几秒到几分钟，取决于数据集大小。
- 手动恢复：
  - 如果需要从备份文件恢复，先停止Redis。
  - 将备份文件复制到配置目录，替换原有文件。
  - 启动Redis，它会自动加载
  - 命令行测试：使用redis-server redis.conf启动，观察日志中是否有“DB loaded from disk”消息。
- 注意：
  - RDB只能恢复到最后一次快照时刻的数据，中间的修改会丢失
  - 如果文件损坏，可尝试 redis-check-dump工具检查和修复（Redis工具集提供）

#### 2.AOF 方式恢复数据

AOF是日志持久化，恢复时重放文件中的写命令，数据更完整但速度较慢。

- 自动恢复
  - Redis启动时，如果启用appendonly yes，会优先检查AOF文件（默认appendonly.aof，由appendfilename指定）。
  - 如果AOF存在且RDB也存在，Redis默认优先加载AOF（因为更完整）
  - 加载过程：逐条执行AOF中的命令，重建内存数据集。支持混合模式（aof-use-rdb-preamble yes），前部分用RDB格式加速
- 手动恢复
  - 停止Redis，将备份AOF文件复制到目录
  - 如果AOF文件太大或损坏，可用redis-check-aof --fix修复（会尝试移除无效命令）
  - 启动Redis，观察日志中“AOF loaded”消息
  - 手动重写AOF：用BGREWRITEAOF命令在运行时优化文件，然后重启
- 注意：
  - 恢复时间随文件大小增加，可能需要几分钟到几小时
  - 如果配置appendfsync always，数据丢失几乎为零；everysec可能丢失1秒数据

#### 3.RDB和AOF混合恢复

-  如果两者都启用，Redis优先AOF（配置中无须额外设置）
- 恢复逻辑：
  1. 检查AOF是否存在且有效-->加载AOF
  2. 如果AOF无效或不存在-->加载RDB
- 推荐生产中使用混合：RDB提供快速备份，AOF确保完整性。恢复后，课手动触发BGREWRITEAOF，更新AOF。

#### 4.其他恢复场景

- 主从复制：在集群中，从节点可从主节点同步数据（使用REPLICAOF命令）。如果主节点崩溃，通过Sentinel或Cluster自动failover，从节点提升Wie主，数据从复制中恢复。
- 从备份恢复：定期备份RDB/AOF到远程（如AWS S3），奔溃时下载替换
- 无持久化恢复：如果未启用持久化，只能通过外部工具（如Redis导出命令）或应用层重构数据，但通常不可行。

#### 5.配置建议与注意事项

- 配置示例（redis.conf）：

```java
dir /var/redis/data  # 存储目录
dbfilename dump.rdb
appendonly yes
appendfilename appendonly.aof
appendfsync everysec  # AOF 同步策略
aof-use-rdb-preamble yes  # 混合模式
```

- **测试恢复**：在开发环境模拟崩溃（kill 进程），重启检查数据。

- **潜在问题**：
  - 内存不足：加载大文件可能 OOM，需确保服务器资源充足。
  - 版本兼容：不同 Redis 版本的 RDB/AOF 可能不兼容，备份时记录版本。
  - 安全：文件权限设为 600，避免泄露。
  - 监控：用 INFO Persistence 查看加载状态，日志文件记录错误。

如果恢复失败，检查日志（默认 redis.log）中的错误，如 "Short read or OOM loading DB"。有具体错误或场景，可以提供更多细节帮忙排查！

### Redis 4.0的混合持久化了解吗？ 

Redis4.0引入了混合持久化（Hybrid Persistence）。这是对传统RDB和AOF持久化的优化，旨在结合两者的优点，解决AOF文件体积过大和恢复速度慢的问题，同事保持RDB的高效性。混合持久化不是一种全新的模式，而是通过在AOF中嵌入RDB格式来实现的。

它默认不启用，需要手动配置，但**从Redis7.0开始成为默认选项**。

#### 为什么引入混合持久化

- 传统RDB的问题：RDB定期生成快照，文件小、恢复快，但可能丢失最后快照后的数据
- 传统AOF的问题：AOF记录每条写命令，数据丢失最小（可达妙秒级），但文件会随操作积累变大，导致恢复时需要重放所有命令，时间长且I/O开销高。
- 混合的益处：通过在AOF文件开头嵌入RDB快照，后续只追加增量命令，文件大小显著减小（可压缩百分之五十以上），恢复速度提升（先加载RDB部分，再重放少量AOF）。这在大数据集场景下特别有用，如电商缓存或实时数据库。

#### 混合持久化的工作原理

混合持久化主要发生在AOF的重写过程中：

1. 触发重写：当AOF文件增长到一定阈值（配置如auto-aof-rewrite-percentage 100），Redis会自动或手动（BGREWRITEAOF命令）出发重写。不阻塞主进程，使用fork子进程。
2. 生成过程：
   - 子进程现将当前内存数据集以RDB二进制格式写入新的AOF文件开头（称为RDB前缀）
   - 然后，将重写期间的增量写命令以AOF格式追加到文件末尾
   - 完成后，原子替换旧AOF文件
3. 恢复过程：
   - Redis启动时，如果启用混合模式，会先加载AOF文件的RDB前缀（快速恢复大部分数据）
   - 然后重放文件末尾的AOF命令（只处理增量变化）
   - 这比纯AOF快得多，因为避免了重放整个历史日志。

注意：混合模式依赖AOF启用（appendonly yes），RDB可以作为辅助备份，但恢复优先AOF

#### 配置混合持久化

在 redis.conf 文件中配置（Redis 4.0 及以上版本）：

- 核心参数：aof-use-rdb-preamble yes（启用混合模式，从 Redis 7.0 默认 yes）。
- 相关 AOF 配置示例：

```java
appendonly yes                # 启用 AOF
appendfilename "appendonly.aof"  # AOF 文件名
appendfsync everysec          # 同步策略（每秒 fsync）
auto-aof-rewrite-percentage 100  # 当 AOF 增长 100% 时重写
auto-aof-rewrite-min-size 64mb   # 最小重写阈值 64MB
aof-use-rdb-preamble yes      # 启用 RDB 前缀（混合关键）
aof-rewrite-incremental-fsync yes  # 重写时增量 fsync，提升性能
```

- RDB 配置可选结合（如 save 900 1），用于额外快照备份。
- 应用配置后，重启 Redis 或用 CONFIG REWRITE 动态更新。

#### 混合持久化优缺点

| 方面             | 优点                                                   | 缺点                                                     |
| ---------------- | ------------------------------------------------------ | -------------------------------------------------------- |
| **性能**         | 恢复速度快（RDB 部分主导），文件大小小，适合大数据集。 | 重写时仍需 fork 子进程，短暂消耗内存（几乎双倍数据集）。 |
| **数据安全**     | 继承 AOF 的低丢失（秒级），比纯 RDB 更可靠。           | 依赖 AOF 重写频率，如果重写失败可能回退到纯 AOF。        |
| **适用场景**     | 生产环境，如高负载数据库，需要平衡耐用性和速度。       | 不适合极小数据集（开销无明显优势）。                     |
| **与纯模式比较** | 比纯 AOF 恢复快 2-10 倍，文件小；比纯 RDB 丢失少。     | 配置稍复杂，需要监控重写日志。                           |

总体优于单一模式，尤其在Redis 4.0后。

#### 实际应用建议

- 生产选择：推荐启用混合持久化+RDB作为备份。监控INFO Persistence命令，查看最后重写时间和AOF大小。
- 升级注意：从旧版本升级到4.0时，确保兼容性；测试恢复流程。
- 潜在问题：如果磁盘空间不足，重写可能失败，导致AOF膨胀。定期备份文件到远程存储。
- 与更高版本：Redis 7.0引入多部分AOF，进一步优化，但4.0的混合仍是基础

#### 生产配置实例

```java
# 启⽤AOF
appendonly yes
# 使⽤混合持久化
aof-use-rdb-preamble yes
# 每秒同步⼀次AOF，平衡性能和安全性
appendfsync everysec
# AOF重写触发条件：⽂件增⻓100%且⾄少达到64MB
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
# RDB备份策略
save 900 1 # 15分钟内有1个修改
save 300 10 # 5分钟内有10个修改
save 60 10000 # 1分钟内有10000个修改
```
