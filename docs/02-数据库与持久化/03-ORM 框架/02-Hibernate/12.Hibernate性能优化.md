## 1. N+1 查询问题

### 1.1 问题描述

```java
// 查询所有用户
List<User> users = session.createQuery("FROM User", User.class).list();  // 1次查询

// 遍历用户，访问订单
for (User user : users) {
    System.out.println(user.getOrders().size());  // 每个用户执行1次查询 → N次
}

// 总共执行：1 + N 次查询
```

<!-- TODO: 插入图片：N+1 查询问题示意图 -->

### 1.2 解决方案一：JOIN FETCH

```java
// 使用 JOIN FETCH 一次性加载
List<User> users = session.createQuery(
    "FROM User u LEFT JOIN FETCH u.orders", User.class)
    .list();

// 只执行 1 次查询
```

**生成 SQL**：
```sql
SELECT u.*, o.* 
FROM user u 
LEFT JOIN orders o ON u.id = o.user_id;
```

### 1.3 解决方案二：@BatchSize

```java
@Entity
public class User {
    
    @OneToMany(mappedBy = "user")
    @BatchSize(size = 10)  // 批量抓取，每次加载10个用户的订单
    private List<Order> orders;
}
```

**效果**：
* 如果有 100 个用户，会执行 10 次查询（每次加载 10 个用户的订单）
* 而非 100 次查询

### 1.4 解决方案三：子查询

```java
@Entity
public class User {
    
    @OneToMany(mappedBy = "user")
    @org.hibernate.annotations.BatchLoader(
        strategy = BatchLoaderType.SUBSELECT
    )
    private List<Order> orders;
}
```

**生成 SQL**：
```sql
-- 第一次：查询所有用户
SELECT * FROM user;

-- 第二次：使用子查询加载所有相关订单
SELECT * FROM orders WHERE user_id IN (SELECT id FROM user);
```

### 1.5 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| JOIN FETCH | 一次查询，性能最好 | 可能产生笛卡尔积 |
| @BatchSize | 灵活控制批量大小 | 仍需多次查询 |
| SUBSELECT | 适合复杂场景 | SQL 较复杂 |

**推荐**：优先使用 JOIN FETCH

## 2. 懒加载优化

### 2.1 合理设置 Fetch 策略

```java
@Entity
public class User {
    
    // 一对多默认 LAZY，无需特别设置
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private List<Order> orders;
    
    // 多对一默认 EAGER，建议改为 LAZY
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "department_id")
    private Department department;
}
```

### 2.2 解决 LazyInitializationException

#### 方式一：JOIN FETCH

```java
List<User> users = session.createQuery(
    "FROM User u LEFT JOIN FETCH u.department", User.class)
    .list();
```

#### 方式二：Open Session In View（不推荐）

```yaml
spring:
  jpa:
    open-in-view: true  # 开发环境可用，生产环境禁用
```

#### 方式三：在 Service 层加载

```java
@Service
@Transactional
public class UserService {
    
    public User getUserWithOrders(Long id) {
        User user = userRepository.findById(id).orElse(null);
        
        // 在事务内访问，触发懒加载
        Hibernate.initialize(user.getOrders());
        
        return user;
    }
}
```

## 3. 批量操作优化

### 3.1 配置批量大小

```yaml
spring:
  jpa:
    properties:
      hibernate:
        jdbc:
          batch_size: 50
        order_inserts: true
        order_updates: true
```

### 3.2 批量插入

```java
@Test
public void testBatchInsert() {
    Session session = sessionFactory.openSession();
    Transaction tx = session.beginTransaction();
    
    for (int i = 0; i < 1000; i++) {
        User user = new User();
        user.setUsername("用户" + i);
        session.save(user);
        
        if (i % 50 == 0) {
            session.flush();
            session.clear();
        }
    }
    
    tx.commit();
    session.close();
}
```

### 3.3 批量更新

```java
@Test
public void testBatchUpdate() {
    Session session = sessionFactory.openSession();
    Transaction tx = session.beginTransaction();
    
    // HQL 批量更新（推荐）
    int affectedRows = session.createQuery(
            "UPDATE User SET status = :status WHERE age < :age")
            .setParameter("status", 0)
            .setParameter("age", 18)
            .executeUpdate();
    
    tx.commit();
    session.close();
}
```

## 4. 缓存优化

### 4.1 启用二级缓存

```yaml
spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_second_level_cache: true
          use_query_cache: true
          region:
            factory_class: org.hibernate.cache.ehcache.EhCacheRegionFactory
```

### 4.2 实体类标注

```java
@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Product {
    // ...
}
```

### 4.3 监控缓存命中率

```java
Statistics stats = sessionFactory.getStatistics();

long hitCount = stats.getSecondLevelCacheHitCount();
long missCount = stats.getSecondLevelCacheMissCount();

double hitRate = (double) hitCount / (hitCount + missCount);
System.out.println("缓存命中率：" + String.format("%.2f%%", hitRate * 100));
```

## 5. SQL 优化

### 5.1 避免 SELECT *

```java
// ❌ 不推荐
@Query("FROM User")
List<User> findAll();

// ✅ 推荐：只查询需要的字段
@Query("SELECT new com.example.dto.UserDTO(u.id, u.username) FROM User u")
List<UserDTO> findUserDTOs();
```

### 5.2 合理使用索引

```sql
-- 为常用查询条件创建索引
CREATE INDEX idx_username ON user(username);
CREATE INDEX idx_age_email ON user(age, email);
```

### 5.3 分析执行计划

```sql
EXPLAIN SELECT * FROM user WHERE age > 20 AND username LIKE '%张%';
```

**关注指标**：
* `type`：访问类型（ALL、index、range、ref、const）
* `key`：实际使用的索引
* `rows`：扫描行数
* `Extra`：额外信息

## 6. 连接池优化

### 6.1 HikariCP 配置

```yaml
spring:
  datasource:
    hikari:
      minimum-idle: 5
      maximum-pool-size: 20
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
      pool-name: MyHikariCP
```

### 6.2 参数调优

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| minimum-idle | 最小空闲连接 | CPU 核心数 |
| maximum-pool-size | 最大连接数 | CPU核心数 × 2 + 磁盘数 |
| connection-timeout | 连接超时 | 30秒 |
| idle-timeout | 空闲超时 | 10分钟 |
| max-lifetime | 最大生命周期 | 30分钟 |

## 7. 分页优化

### 7.1 传统分页问题

```java
// 深分页性能差
Page<User> page = userRepository.findAll(PageRequest.of(1000, 10));

// SQL: SELECT * FROM user LIMIT 10000, 10
-- 需要扫描前 10010 条记录
```

### 7.2 游标分页（推荐）

```java
@Query("FROM User u WHERE u.id > :lastId ORDER BY u.id")
List<User> findByCursor(@Param("lastId") Long lastId, Pageable pageable);

// 使用
List<User> users = repository.findByCursor(0L, PageRequest.of(0, 10));
Long lastId = users.get(users.size() - 1).getId();
List<User> nextUsers = repository.findByCursor(lastId, PageRequest.of(0, 10));
```

**优势**：利用主键索引，性能稳定

## 9. 常见问题排查

### 9.1 内存溢出

**症状**：`OutOfMemoryError`

**原因**：
* 一次性加载大量数据
* Session 缓存未清空

**解决**：

```java
// 分批处理
int batchSize = 1000;
for (int i = 0; i < totalCount; i += batchSize) {
    List<User> users = repository.findAll(
        PageRequest.of(i / batchSize, batchSize));
    
    // 处理数据
    
    session.clear();  // 清空缓存
}
```

### 9.2 慢查询

**排查步骤**：

1. 开启 SQL 日志
2. 分析执行计划
3. 检查索引
4. 优化查询语句

```yaml
logging:
  level:
    org.hibernate.SQL: DEBUG
    org.hibernate.type.descriptor.sql.BasicBinder: TRACE
```

### 9.3 连接泄露

**症状**：连接池耗尽

**原因**：
* Session 未关闭
* 事务未提交

**解决**：

```java
// 确保关闭 Session
try (Session session = sessionFactory.openSession()) {
    // 使用资源
} finally {
    session.close();
}
```
