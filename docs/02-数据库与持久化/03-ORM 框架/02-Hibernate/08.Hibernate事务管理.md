## 1. 事务基础

### 1.1 什么是事务？

事务是数据库操作的最小工作单元，要么全部成功，要么全部失败。

### 1.2 ACID 特性

| 特性 | 说明 | 示例 |
|------|------|------|
| **Atomicity（原子性）** | 事务中的操作要么全部完成，要么全部不完成 | 转账：扣款和存款必须同时成功 |
| **Consistency（一致性）** | 事务执行前后，数据库保持一致状态 | 转账前后总金额不变 |
| **Isolation（隔离性）** | 多个事务互不干扰 | 事务 A 看不到事务 B 的中间状态 |
| **Durability（持久性）** | 事务提交后，数据永久保存 | 断电后数据不丢失 |

## 2. Hibernate 事务管理

### 2.1 JDBC 事务

```java
@Test
public void testJDBCTransaction() {
    Session session = sessionFactory.openSession();
    Transaction tx = null;
    
    try {
        tx = session.beginTransaction();
        
        // 执行数据库操作
        User user = new User();
        user.setUsername("张三");
        session.save(user);
        
        Order order = new Order();
        order.setOrderNo("ORD001");
        order.setUser(user);
        session.save(order);
        
        // 提交事务
        tx.commit();
        
    } catch (Exception e) {
        // 回滚事务
        if (tx != null && tx.isActive()) {
            tx.rollback();
        }
        e.printStackTrace();
    } finally {
        session.close();
    }
}
```

### 2.2 Spring 集成事务

```java
@Service
public class UserService {
    
    @Autowired
    private UserMapper userMapper;
    
    @Transactional
    public void createUser(User user) {
        userMapper.save(user);
        // 如果抛出异常，自动回滚
    }
}
```

## 3. 事务隔离级别

### 3.1 隔离级别分类

| 级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|------|------|-----------|------|------|
| READ_UNCOMMITTED | ✅ | ✅ | ✅ | 最高 |
| READ_COMMITTED | ❌ | ✅ | ✅ | 高 |
| REPEATABLE_READ | ❌ | ❌ | ✅ | 中 |
| SERIALIZABLE | ❌ | ❌ | ❌ | 最低 |

### 3.2 配置隔离级别

```xml
<!-- hibernate.cfg.xml -->
<property name="hibernate.connection.isolation">2</property>
```

**值对应关系**：
* 1 - READ_UNCOMMITTED
* 2 - READ_COMMITTED
* 4 - REPEATABLE_READ
* 8 - SERIALIZABLE

### 3.3 编程式设置

```java
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();

// 设置隔离级别
session.doWork(connection -> {
    connection.setTransactionIsolation(Connection.TRANSACTION_READ_COMMITTED);
});

tx.commit();
session.close();
```

## 4. 锁机制

### 4.1 悲观锁

**原理**：假设会发生冲突，先加锁再操作。

#### （1）SELECT FOR UPDATE

```java
@Test
public void testPessimisticLock() {
    Session session = sessionFactory.openSession();
    Transaction tx = session.beginTransaction();
    
    // 锁定记录
    User user = session.get(User.class, 1L, LockMode.PESSIMISTIC_WRITE);
    
    // 修改
    user.setUsername("新名字");
    
    tx.commit();
    session.close();
}
```

**生成 SQL**：
```sql
SELECT * FROM user WHERE id = 1 FOR UPDATE;
```

#### （2）锁模式

| 模式 | 说明 |
|------|------|
| PESSIMISTIC_READ | 共享锁，允许其他事务读 |
| PESSIMISTIC_WRITE | 排他锁，不允许其他事务读写 |
| PESSIMISTIC_FORCE_INCREMENT | 排他锁 + 版本号递增 |

### 4.2 乐观锁

**原理**：假设不会发生冲突，提交时检查版本。

#### （1）使用 @Version

```java
@Entity
public class User {
    
    @Id
    @GeneratedValue
    private Long id;
    
    private String username;
    
    @Version
    private Integer version;  // 版本号
}
```

#### （2）工作流程

```
事务 A: 读取 (version=1) → 修改 → 提交 (version=2) ✓
事务 B: 读取 (version=1) → 修改 → 提交 (version!=1) ✗ 失败
```

#### （3）测试

```java
@Test
public void testOptimisticLock() {
    Session session1 = sessionFactory.openSession();
    Session session2 = sessionFactory.openSession();
    
    Transaction tx1 = session1.beginTransaction();
    Transaction tx2 = session2.beginTransaction();
    
    try {
        // 两个事务同时读取
        User user1 = session1.get(User.class, 1L);
        User user2 = session2.get(User.class, 1L);
        
        System.out.println("user1 version: " + user1.getVersion());
        System.out.println("user2 version: " + user2.getVersion());
        
        // 事务 1 先提交
        user1.setUsername("新名字1");
        tx1.commit();
        
        // 事务 2 后提交（会失败）
        user2.setUsername("新名字2");
        tx2.commit();  // 抛出 OptimisticLockException
        
    } catch (OptimisticLockException e) {
        tx2.rollback();
        System.out.println("乐观锁冲突：" + e.getMessage());
    } finally {
        session1.close();
        session2.close();
    }
}
```

### 4.3 悲观锁 vs 乐观锁

| 特性 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| **适用场景** | 写多读少 | 读多写少 |
| **性能** | 较低（阻塞） | 较高（无阻塞） |
| **并发度** | 低 | 高 |
| **实现复杂度** | 简单 | 需处理冲突 |
| **死锁风险** | 有 | 无 |

**选择建议**：
* **高并发、读多写少**：乐观锁
* **写操作频繁、冲突概率高**：悲观锁

## 5. 并发问题

### 5.1 脏读（Dirty Read）

**场景**：事务 A 读取了事务 B 未提交的数据。

```
事务 A: 读取余额 (1000)
事务 B: 扣款 200 → 余额 800（未提交）
事务 A: 读取余额 (800) ← 脏读
事务 B: 回滚 → 余额 1000
事务 A: 基于错误数据操作
```

**解决**：使用 READ_COMMITTED 或更高级别

### 5.2 不可重复读（Non-Repeatable Read）

**场景**：同一事务中多次读取同一数据，结果不一致。

```
事务 A: 读取余额 (1000)
事务 B: 扣款 200 → 提交
事务 A: 再次读取余额 (800) ← 不可重复读
```

**解决**：使用 REPEATABLE_READ 或更高级别

### 5.3 幻读（Phantom Read）

**场景**：同一查询条件，两次查询返回的记录数不同。

```
事务 A: 查询年龄 > 20 的用户 (10条)
事务 B: 插入一条年龄 25 的用户 → 提交
事务 A: 再次查询年龄 > 20 的用户 (11条) ← 幻读
```

**解决**：使用 SERIALIZABLE

## 6. 死锁处理

### 6.1 死锁场景

```
事务 A: 锁定记录 1 → 请求记录 2（等待）
事务 B: 锁定记录 2 → 请求记录 1（等待）
→ 死锁
```

### 6.2 预防死锁

1. **统一访问顺序**：所有事务按相同顺序访问资源
2. **设置超时时间**：避免无限等待
3. **减少事务持有锁的时间**：快速提交

### 6.3 配置超时

```xml
<property name="hibernate.timeout">10</property>
```

```java
tx.setTimeout(10);  // 10秒超时
```

## 7. 传播行为（Spring）

### 7.1 传播类型

| 类型 | 说明 |
|------|------|
| REQUIRED | 如果存在事务则加入，否则新建（默认） |
| REQUIRES_NEW | 总是新建事务，挂起当前事务 |
| NESTED | 嵌套事务 |
| SUPPORTS | 支持当前事务，没有则以非事务执行 |
| NOT_SUPPORTED | 以非事务执行，挂起当前事务 |
| MANDATORY | 必须在事务中执行 |
| NEVER | 必须在非事务中执行 |

### 7.2 使用示例

```java
@Service
public class OrderService {
    
    @Transactional(propagation = Propagation.REQUIRED)
    public void createOrder(Order order) {
        // 主事务
        orderMapper.save(order);
        
        // 调用其他服务
        userService.updateStock(order);
    }
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logOperation(String operation) {
        // 独立事务，不受主事务影响
        operationLogMapper.save(operation);
    }
}
```

## 8. 完整示例

### 8.1 转账业务

```java
@Service
public class AccountService {
    
    @Autowired
    private AccountMapper accountMapper;
    
    @Transactional
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        // 1. 检查余额
        Account fromAccount = accountMapper.findById(fromId);
        if (fromAccount.getBalance().compareTo(amount) < 0) {
            throw new RuntimeException("余额不足");
        }
        
        // 2. 扣款
        fromAccount.setBalance(fromAccount.getBalance().subtract(amount));
        accountMapper.update(fromAccount);
        
        // 3. 收款
        Account toAccount = accountMapper.findById(toId);
        toAccount.setBalance(toAccount.getBalance().add(amount));
        accountMapper.update(toAccount);
        
        // 4. 记录日志
        TransactionLog log = new TransactionLog();
        log.setFromId(fromId);
        log.setToId(toId);
        log.setAmount(amount);
        transactionLogMapper.save(log);
        
        // 如果任何步骤失败，整个事务回滚
    }
}
```

### 8.2 测试代码

```java
@Test
public void testTransfer() {
    Session session = sessionFactory.openSession();
    Transaction tx = session.beginTransaction();
    
    try {
        AccountService service = new AccountService();
        service.transfer(1L, 2L, new BigDecimal("100"));
        
        tx.commit();
        System.out.println("转账成功");
        
    } catch (Exception e) {
        tx.rollback();
        System.out.println("转账失败：" + e.getMessage());
    } finally {
        session.close();
    }
}
```

## 9. 最佳实践

### 9.1 事务管理原则

📌 **事务尽量小**：减少锁持有时间  
📌 **避免在事务中调用外部接口**：如 HTTP 请求、邮件发送  
📌 **合理选择隔离级别**：根据业务需求  
📌 **使用乐观锁处理并发**：适合读多写少场景  
📌 **设置超时时间**：防止长时间占用资源  

### 9.2 常见陷阱

❌ **长事务**：事务中包含耗时操作  
❌ **嵌套事务滥用**：导致复杂性问题  
❌ **忽略异常处理**：未正确回滚  
❌ **跨会话操作**：在不同 Session 中操作同一对象
