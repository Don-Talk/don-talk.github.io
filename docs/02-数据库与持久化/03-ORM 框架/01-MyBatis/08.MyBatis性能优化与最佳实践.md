> * 学会使用连接池优化数据库访问

## 1. 性能问题分析

### 1.1 常见性能瓶颈

1. **N+1 查询问题**：循环中执行查询
2. **全表扫描**：缺少索引或索引失效
3. **大结果集**：一次性加载过多数据
4. **连接池配置不当**：连接数过少或过多
5. **缓存未命中**：频繁查询数据库
6. **SQL 语句复杂**：多表 JOIN、子查询嵌套

### 1.2 性能监控工具

* **MyBatis Log**：查看执行的 SQL
* **P6Spy**：SQL 性能分析
* **Druid Monitor**：连接池监控
* **Arthas**：Java 诊断工具

## 2. N+1 查询问题

### 2.1 问题描述

```java
// 错误示例：N+1 查询
List<User> users = userMapper.findAll();  // 1次查询
for (User user : users) {
    List<Order> orders = orderMapper.findByUserId(user.getId());  // N次查询
    user.setOrders(orders);
}
```

如果有 100 个用户，将执行 1 + 100 = 101 次查询！

### 2.2 解决方案一：JOIN 查询

```xml
<resultMap id="UserWithOrdersResultMap" type="User">
    <id property="id" column="user_id"/>
    <result property="username" column="username"/>
    
    <collection property="orders" ofType="Order">
        <id property="id" column="order_id"/>
        <result property="orderNo" column="order_no"/>
        <result property="amount" column="amount"/>
    </collection>
</resultMap>

<select id="findAllWithOrders" resultMap="UserWithOrdersResultMap">
    SELECT u.id AS user_id, u.username,
           o.id AS order_id, o.order_no, o.amount
    FROM user u
    LEFT JOIN orders o ON u.id = o.user_id
</select>
```

**优点**：只执行一次 SQL  
**缺点**：数据量大时会产生冗余

### 2.3 解决方案二：批量查询

```java
// 先查询所有用户
List<User> users = userMapper.findAll();

// 提取所有用户 ID
List<Integer> userIds = users.stream()
    .map(User::getId)
    .collect(Collectors.toList());

// 批量查询订单
List<Order> orders = orderMapper.findByUserIds(userIds);

// 组装数据
Map<Integer, List<Order>> orderMap = orders.stream()
    .collect(Collectors.groupingBy(Order::getUserId));

users.forEach(user -> 
    user.setOrders(orderMap.getOrDefault(user.getId(), new ArrayList<>()))
);
```

**XML**：
```xml
<select id="findByUserIds" resultType="Order">
    SELECT * FROM orders WHERE user_id IN
    <foreach collection="list" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</select>
```

**优点**：只执行 2 次 SQL  
**推荐**：数据量大的场景

### 2.4 解决方案三：延迟加载

```xml
<resultMap id="UserWithOrdersResultMap" type="User">
    <id property="id" column="id"/>
    <result property="username" column="username"/>
    
    <collection property="orders" 
                column="id" 
                select="findOrdersByUserId"
                fetchType="lazy"/>
</resultMap>
```

**配置**：
```yaml
mybatis:
  configuration:
    lazy-loading-enabled: true
    aggressive-lazy-loading: false
```

**优点**：按需加载  
**注意**：必须在 SqlSession 关闭前访问

## 3. SQL 优化

### 3.1 避免 SELECT *

```xml
<!-- 错误 -->
<select id="findAll" resultType="User">
    SELECT * FROM user
</select>

<!-- 正确 -->
<select id="findAll" resultType="User">
    SELECT id, username, email, age FROM user
</select>
```

**原因**：
* 减少网络传输
* 提高索引覆盖率
* 避免字段变更影响

### 3.2 合理使用索引

#### 创建索引

```sql
-- 单列索引
CREATE INDEX idx_username ON user(username);

-- 联合索引
CREATE INDEX idx_age_email ON user(age, email);

-- 唯一索引
CREATE UNIQUE INDEX idx_email ON user(email);
```

#### 索引失效场景

```xml
<!-- 失效：对索引列使用函数 -->
<select id="findByYear" resultType="User">
    SELECT * FROM user WHERE YEAR(create_time) = 2024
</select>

<!-- 正确 -->
<select id="findByYear" resultType="User">
    SELECT * FROM user 
    WHERE create_time &gt;= '2024-01-01' 
      AND create_time &lt; '2025-01-01'
</select>
```

```xml
<!-- 失效：模糊查询前缀通配符 -->
<select id="search" resultType="User">
    SELECT * FROM user WHERE username LIKE '%张%'
</select>

<!-- 正确：使用前缀匹配 -->
<select id="search" resultType="User">
    SELECT * FROM user WHERE username LIKE '张%'
</select>
```

```xml
<!-- 失效：类型转换 -->
<select id="findById" resultType="User">
    SELECT * FROM user WHERE id = #{id}
</select>
<!-- 如果 id 是字符串类型，会导致索引失效 -->
```

### 3.3 优化分页查询

#### 传统分页（性能差）

```xml
<select id="findByPage" resultType="User">
    SELECT * FROM user ORDER BY id LIMIT 10000, 10
</select>
```

**问题**：需要扫描前 10010 条记录

#### 优化分页（推荐）

```xml
<select id="findByPageOptimized" resultType="User">
    SELECT * FROM user 
    WHERE id &gt; #{lastId} 
    ORDER BY id 
    LIMIT #{pageSize}
</select>
```

**使用**：
```java
// 第一页
List<User> page1 = mapper.findByPageOptimized(0, 10);
Long lastId = page1.get(page1.size() - 1).getId();

// 第二页
List<User> page2 = mapper.findByPageOptimized(lastId, 10);
```

**优点**：利用索引，性能稳定

### 3.4 避免大事务

```java
// 错误：事务中包含耗时操作
@Transactional
public void createUser(User user) {
    userMapper.insert(user);
    
    // 耗时操作（发送邮件、调用外部接口等）
    emailService.sendEmail(user.getEmail());  // 不要放在事务中
    smsService.sendSms(user.getPhone());      // 不要放在事务中
}

// 正确：缩小事务范围
@Transactional
public void createUser(User user) {
    userMapper.insert(user);
}

public void createUserAsync(User user) {
    createUser(user);
    
    // 异步执行耗时操作
    CompletableFuture.runAsync(() -> {
        emailService.sendEmail(user.getEmail());
        smsService.sendSms(user.getPhone());
    });
}
```

## 4. 连接池优化

### 4.1 HikariCP 配置（Spring Boot 默认）

```yaml
spring:
  datasource:
    hikari:
      # 最小空闲连接数
      minimum-idle: 5
      # 最大连接数
      maximum-pool-size: 20
      # 连接超时时间（毫秒）
      connection-timeout: 30000
      # 空闲连接存活时间（毫秒）
      idle-timeout: 600000
      # 连接最大生命周期（毫秒）
      max-lifetime: 1800000
      # 连接测试查询
      connection-test-query: SELECT 1
```

### 4.2 参数调优建议

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| minimum-idle | 最小空闲连接 | CPU 核心数 |
| maximum-pool-size | 最大连接数 | CPU核心数 * 2 + 磁盘数 |
| connection-timeout | 连接超时 | 30秒 |
| idle-timeout | 空闲超时 | 10分钟 |
| max-lifetime | 最大生命周期 | 30分钟 |

### 4.3 监控连接池

```java
@Component
public class DataSourceMonitor {
    
    @Autowired
    private HikariDataSource dataSource;
    
    @Scheduled(fixedRate = 60000)  // 每分钟监控
    public void monitor() {
        HikariPoolMXBean poolMXBean = dataSource.getHikariPoolMXBean();
        
        System.out.println("活跃连接数：" + poolMXBean.getActiveConnections());
        System.out.println("空闲连接数：" + poolMXBean.getIdleConnections());
        System.out.println("总连接数：" + poolMXBean.getTotalConnections());
        System.out.println("等待线程数：" + poolMXBean.getThreadsAwaitingConnection());
    }
}
```

## 5. 缓存优化

### 5.1 一级缓存

* **默认开启**，无需配置
* **会话级别**，SqlSession 关闭后失效
* **注意**：增删改操作会自动清空缓存

### 5.2 二级缓存

#### 启用二级缓存

```yaml
mybatis:
  configuration:
    cache-enabled: true
```

```xml
<mapper namespace="com.example.mapper.UserMapper">
    <cache eviction="LRU" flushInterval="60000" size="512" readOnly="true"/>
</mapper>
```

#### 实体类实现 Serializable

```java
@Data
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    // ...
}
```

### 5.3 Redis 缓存（推荐）

#### 添加依赖

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
```

#### 配置

```yaml
spring:
  redis:
    host: localhost
    port: 6379
    database: 0
```

#### 使用

```java
@Service
public class UserService {
    
    @Autowired
    private UserMapper userMapper;
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    public User findById(Integer id) {
        String key = "user:" + id;
        
        // 从缓存获取
        User user = (User) redisTemplate.opsForValue().get(key);
        if (user != null) {
            return user;
        }
        
        // 从数据库获取
        user = userMapper.findById(id);
        
        // 写入缓存（过期时间 1 小时）
        redisTemplate.opsForValue().set(key, user, 1, TimeUnit.HOURS);
        
        return user;
    }
    
    @Transactional
    public void update(User user) {
        userMapper.update(user);
        
        // 删除缓存
        String key = "user:" + user.getId();
        redisTemplate.delete(key);
    }
}
```

## 6. 批量操作优化

### 6.1 批量插入

```xml
<insert id="batchInsert">
    INSERT INTO user (username, email, age) VALUES
    <foreach collection="list" item="user" separator=",">
        (#{user.username}, #{user.email}, #{user.age})
    </foreach>
</insert>
```

**分批处理**：
```java
public void batchInsertUsers(List<User> users) {
    int batchSize = 1000;
    for (int i = 0; i < users.size(); i += batchSize) {
        List<User> batch = users.subList(
            i, Math.min(i + batchSize, users.size()));
        userMapper.batchInsert(batch);
    }
}
```

### 6.2 批量更新

```xml
<update id="batchUpdate">
    UPDATE user
    <set>
        username = CASE id
        <foreach collection="list" item="user">
            WHEN #{user.id} THEN #{user.username}
        </foreach>
        END,
        email = CASE id
        <foreach collection="list" item="user">
            WHEN #{user.id} THEN #{user.email}
        </foreach>
        END
    </set>
    WHERE id IN
    <foreach collection="list" item="user" open="(" separator="," close=")">
        #{user.id}
    </foreach>
</update>
```

### 6.3 使用 ExecutorType.BATCH

```java
@Autowired
private SqlSessionTemplate sqlSessionTemplate;

public void batchInsertWithExecutor(List<User> users) {
    SqlSession session = sqlSessionTemplate.getSqlSessionFactory()
        .openSession(ExecutorType.BATCH);
    
    try {
        UserMapper mapper = session.getMapper(UserMapper.class);
        for (User user : users) {
            mapper.insert(user);
        }
        session.commit();
    } finally {
        session.close();
    }
}
```

### 7.3 异常处理

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(DuplicateKeyException.class)
    public ResponseEntity<String> handleDuplicateKey(DuplicateKeyException e) {
        return ResponseEntity.badRequest().body("数据已存在");
    }
    
    @ExceptionHandler(BadSqlGrammarException.class)
    public ResponseEntity<String> handleBadSql(BadSqlGrammarException e) {
        log.error("SQL 语法错误", e);

    }

        log.error("系统异常", e);

    }
}
```

### 7.4 慢查询监控

```java
@Aspect
@Component
@Slf4j
public class SlowQueryAspect {
    
    @Around("@annotation(com.example.annotation.SlowQueryMonitor)")
    public Object monitor(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        
        Object result = joinPoint.proceed();
        
        long duration = System.currentTimeMillis() - start;
        if (duration > 1000) {  // 超过 1 秒
            log.warn("慢查询 detected: {} ms, method: {}", 
                     duration, joinPoint.getSignature());
        }
        
        return result;
    }
}
```

## 8. 常见问题排查

### 8.1 内存溢出

**症状**：`OutOfMemoryError`

**原因**：
* 一次性查询大量数据
* 缓存数据过多

**解决**：
```xml
<!-- 使用流式查询 -->
<select id="findAllStream" resultType="User" resultSetType="FORWARD_ONLY" fetchSize="100">
    SELECT * FROM user
</select>
```

```java
@Select("SELECT * FROM user")
@Options(resultSetType = ResultSetType.FORWARD_ONLY, fetchSize = 100)
void findAllStream(ResultHandler<User> handler);
```

### 8.2 连接泄露

**症状**：连接池耗尽

**原因**：
* SqlSession 未关闭
* 事务未提交/回滚

**解决**：
```java
// 确保关闭 SqlSession
try (SqlSession session = sqlSessionFactory.openSession()) {
    // 使用资源
} finally {
    session.close();
}
```

### 8.3 死锁

**症状**：事务长时间等待

**原因**：
* 多个事务互相等待锁
* 事务持有锁时间过长

**解决**：
* 统一访问顺序
* 缩短事务时间
* 设置超时时间

```yaml
spring:
  datasource:
    hikari:
      connection-timeout: 30000
```

## 10. 总结

### 10.1 核心优化点

✅ **避免 N+1 查询**：使用 JOIN 或批量查询  
✅ **SQL 优化**：合理使用索引，避免全表扫描  
✅ **连接池调优**：根据负载调整连接数  
✅ **缓存策略**：合理使用二级缓存或 Redis  
✅ **批量操作**：减少数据库交互次数  

### 10.2 最佳实践清单

📌 避免 SELECT *，只查询需要的字段  
📌 为常用查询条件创建索引  
📌 大事务拆分为小事务  
📌 使用连接池监控工具  
📌 生产环境关闭 SQL 日志  
📌 定期清理无用缓存  
📌 分批处理大数据量  
📌 设置合理的超时时间
