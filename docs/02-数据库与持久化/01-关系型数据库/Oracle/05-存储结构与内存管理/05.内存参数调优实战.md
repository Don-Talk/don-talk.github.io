# 05内存参数调优实战

**学习目标：**
> * 掌握 Oracle 内存参数的配置方法
> * 学会根据负载类型调整内存分配
> * 能够使用 AWR 报告分析内存性能
> * 了解不同场景下的最佳配置
> * 掌握内存问题的诊断和解决

---

## 1. 内存参数概览

### 1.1 核心内存参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| **memory_target** | AMM 总内存目标 | 0（禁用） |
| **memory_max_target** | AMM 最大内存 | 0 |
| **sga_target** | SGA 目标大小 | 0 |
| **sga_max_size** | SGA 最大大小 | - |
| **pga_aggregate_target** | PGA 目标大小 | 自动 |
| **db_cache_size** | Buffer Cache 大小 | 自动 |
| **shared_pool_size** | Shared Pool 大小 | 自动 |
| **large_pool_size** | Large Pool 大小 | 自动 |
| **java_pool_size** | Java Pool 大小 | 自动 |

### 1.2 参数关系

```
AMM 模式：
memory_target >= sga_target + pga_aggregate_target

手动管理模式：
sga_target = db_cache_size + shared_pool_size + ...
pga_aggregate_target = 各会话 PGA 总和
```

## 2. 内存管理模式选择

### 2.1 AMM（自动内存管理）

**适用**：中小规模数据库，简化管理。

```sql
-- 启用 AMM
ALTER SYSTEM SET memory_target = 4G SCOPE=SPFILE;
ALTER SYSTEM SET memory_max_target = 6G SCOPE=SPFILE;
ALTER SYSTEM SET sga_target = 0 SCOPE=SPFILE;
ALTER SYSTEM SET pga_aggregate_target = 0 SCOPE=SPFILE;

SHUTDOWN IMMEDIATE;
STARTUP;
```

**优点**：
- ✅ 自动调整 SGA 和 PGA
- ✅ 简化管理
- ✅ 适合动态负载

**缺点**：
- ❌ 不能使用 HugePages
- ❌ 大规模系统可能不稳定
- ❌ 精细控制能力弱

### 2.2 ASMM（自动共享内存管理）

**适用**：生产环境，需要 HugePages。

```sql
-- 启用 ASMM
ALTER SYSTEM SET memory_target = 0 SCOPE=SPFILE;
ALTER SYSTEM SET sga_target = 3G SCOPE=SPFILE;
ALTER SYSTEM SET sga_max_size = 4G SCOPE=SPFILE;
ALTER SYSTEM SET pga_aggregate_target = 1G SCOPE=SPFILE;

SHUTDOWN IMMEDIATE;
STARTUP;
```

**优点**：
- ✅ 可以使用 HugePages
- ✅ 自动调整 SGA 内部组件
- ✅ 稳定性好

**缺点**：
- ❌ PGA 需要单独管理
- ❌ 配置稍复杂

### 2.3 手动管理

**适用**：特殊需求，精细控制。

```sql
-- 手动设置各组件
ALTER SYSTEM SET db_cache_size = 2G;
ALTER SYSTEM SET shared_pool_size = 1G;
ALTER SYSTEM SET large_pool_size = 256M;
ALTER SYSTEM SET java_pool_size = 128M;
ALTER SYSTEM SET pga_aggregate_target = 1G;
```

**优点**：
- ✅ 完全控制
- ✅ 可预测性高

**缺点**：
- ❌ 管理复杂
- ❌ 需要专业知识
- ❌ 不适应负载变化

## 3. OLTP 系统内存配置

### 3.1 特点

- 大量小事务
- 高并发
- 随机 I/O
- 需要快速响应

### 3.2 推荐配置

```sql
-- 假设总内存 16GB

-- SGA: 60-70% (9-11GB)
ALTER SYSTEM SET sga_target = 10G SCOPE=SPFILE;
ALTER SYSTEM SET sga_max_size = 12G SCOPE=SPFILE;

-- PGA: 30-40% (5-6GB)
ALTER SYSTEM SET pga_aggregate_target = 4G SCOPE=SPFILE;

-- Buffer Cache: SGA 的 50-60%
ALTER SYSTEM SET db_cache_size = 6G;

-- Shared Pool: SGA 的 20-30%
ALTER SYSTEM SET shared_pool_size = 2G;

-- Redo Log Buffer: 默认即可
-- 通常 1-3MB，由 Oracle 自动管理

-- 其他组件：自动管理
```

### 3.3 调优重点

✅ **Buffer Cache 命中率**：目标 > 95%  
✅ **Library Cache 命中率**：目标 > 90%  
✅ **减少硬解析**：使用绑定变量  
✅ **优化索引**：减少全表扫描  

### 3.4 监控脚本

```sql
-- OLTP 关键指标
SELECT 
    'Buffer Cache Hit Ratio' as metric,
    ROUND((1 - phy.value / NULLIF(log.value + con.value, 0)) * 100, 2) || '%' as value
FROM v$sysstat phy, v$sysstat log, v$sysstat con
WHERE phy.name = 'physical reads'
AND log.name = 'db block gets'
AND con.name = 'consistent gets';

-- Library Cache Hit Ratio
SELECT 
    'Library Cache Hit Ratio' as metric,
    ROUND(SUM(gethitratio) / COUNT(*) * 100, 2) || '%' as value
FROM v$librarycache;
```

## 4. 数据仓库系统内存配置

### 4.1 特点

- 大查询、大事务
- 批量加载
- 顺序 I/O
- 大量排序和哈希操作

### 4.2 推荐配置

```sql
-- 假设总内存 32GB

-- SGA: 40-50% (13-16GB)
ALTER SYSTEM SET sga_target = 14G SCOPE=SPFILE;
ALTER SYSTEM SET sga_max_size = 16G SCOPE=SPFILE;

-- PGA: 50-60% (16-19GB)
ALTER SYSTEM SET pga_aggregate_target = 16G SCOPE=SPFILE;

-- Buffer Cache: 可以较小
ALTER SYSTEM SET db_cache_size = 8G;

-- Shared Pool: 适中
ALTER SYSTEM SET shared_pool_size = 2G;

-- Large Pool: 较大（并行查询）
ALTER SYSTEM SET large_pool_size = 1G;
```

### 4.3 调优重点

✅ **PGA 大小**：支持大规模排序和哈希  
✅ **并行查询**：合理设置并行度  
✅ **直接路径操作**：减少 Buffer Cache 污染  
✅ **物化视图**：预计算聚合  

### 4.4 监控脚本

```sql
-- 数据仓库关键指标
SELECT 
    name,
    value/1024/1024 as mb
FROM v$pgastat
WHERE name IN ('aggregate PGA target parameter',
               'total PGA inuse',
               'maximum PGA allocated');

-- 检查磁盘排序
SELECT 
    name,
    value
FROM v$sysstat
WHERE name LIKE '%sort%';
```

## 5. 混合负载系统配置

### 5.1 特点

- OLTP + 报表查询
- 白天 OLTP，夜间批处理
- 需要平衡两种负载

### 5.2 推荐配置

```sql
-- 假设总内存 24GB

-- 使用 ASMM，让 Oracle 自动调整
ALTER SYSTEM SET sga_target = 14G SCOPE=SPFILE;
ALTER SYSTEM SET pga_aggregate_target = 8G SCOPE=SPFILE;

-- 或使用 Resource Manager 区分负载
```

### 5.3 动态调整策略

```sql
-- 白天（OLTP 优先）
ALTER SYSTEM SET pga_aggregate_target = 6G;

-- 夜间（批处理优先）
ALTER SYSTEM SET pga_aggregate_target = 12G;
```

## 6. 内存问题诊断

### 6.1 Buffer Cache 命中率低

**症状**：Physical I/O 高，响应慢

**诊断**：
```sql
-- 检查命中率
SELECT 
    ROUND((1 - physical_reads / NULLIF(db_block_gets + consistent_gets, 0)) * 100, 2) as hit_ratio
FROM v$sysstat
WHERE name IN ('db block gets', 'consistent gets', 'physical reads');

-- 如果 < 90%，需要优化
```

**解决**：
1. 增加 `db_cache_size`
2. 优化 SQL，减少全表扫描
3. 添加合适的索引
4. 使用 KEEP 池缓存热点表

### 6.2 Shared Pool 不足

**症状**：ORA-04031，硬解析多

**诊断**：
```sql
-- 检查 Library Cache 命中率
SELECT namespace, gethitratio FROM v$librarycache;

-- 检查硬解析比例
SELECT 
    ROUND(hard_parses / NULLIF(total_parses, 0) * 100, 2) as hard_parse_pct
FROM (
    SELECT 
        SUM(CASE WHEN name = 'parse count (hard)' THEN value END) as hard_parses,
        SUM(CASE WHEN name = 'parse count (total)' THEN value END) as total_parses
    FROM v$sysstat
    WHERE name LIKE 'parse count%'
);
```

**解决**：
1. 增加 `shared_pool_size`
2. 使用绑定变量
3. 刷新 Shared Pool（临时方案）
4. 优化 SQL，减少不同语句

### 6.3 PGA 不足

**症状**：磁盘排序多，ORA-04030

**诊断**：
```sql
-- 检查 PGA 使用
SELECT * FROM v$pgastat;

-- 检查磁盘排序
SELECT name, value FROM v$sysstat WHERE name LIKE '%sort%disk%';
```

**解决**：
1. 增加 `pga_aggregate_target`
2. 优化 SQL，减少排序
3. 增加临时表空间
4. 杀死异常会话

### 6.4 内存泄漏

**症状**：内存使用持续增长

**诊断**：
```sql
-- 查找 PGA 使用最多的会话
SELECT 
    s.sid,
    s.username,
    p.program,
    pga_used_mem/1024/1024 as used_mb
FROM v$session s, v$process p
WHERE s.paddr = p.addr
ORDER BY pga_used_mem DESC
FETCH FIRST 10 ROWS ONLY;
```

**解决**：
1. 杀死泄漏会话
2. 查找应用代码问题
3. 重启实例（最后手段）

## 8. 最佳实践

### 8.1 配置原则

✅ **选择合适的管理模式**
- 小规模：AMM
- 中大规模：ASMM + HugePages
- 特殊需求：手动

✅ **根据负载类型调整**
- OLTP：大 Buffer Cache
- DW：大 PGA
- 混合：平衡配置

✅ **预留余量**
- 不要将所有内存分配给 Oracle
- 预留 20-30% 给操作系统

✅ **监控和调整**
- 定期检查内存使用
- 根据实际负载调整
- 记录配置变更

### 8.2 常见问题避免

❌ **不要过度分配内存**  
❌ **不要忘记禁用 THP**  
❌ **不要忽略统计信息**  
❌ **不要频繁调整参数**  
❌ **不要在高峰期进行重大变更**  

### 8.3 变更流程

1. **评估当前状态**
2. **制定变更计划**
3. **在测试环境验证**
4. **选择维护窗口**
5. **执行变更**
6. **监控效果**
7. **回滚预案**
