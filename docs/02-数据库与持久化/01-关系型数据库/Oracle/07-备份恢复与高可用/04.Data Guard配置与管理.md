## 1. Data Guard 概述

### 1.1 什么是 Data Guard？

Data Guard 是 Oracle 提供的高可用性和灾难恢复解决方案。

**核心功能**：
- ✅ 数据保护（灾难恢复）
- ✅ 高可用性（自动故障转移）
- ✅ 读写分离（报表卸载）
- ✅ 数据完整性保证

<!-- TODO: 插入图片：Data Guard 架构图 -->

### 1.2 Data Guard 组件

| 组件 | 说明 |
|------|------|
| **Primary Database** | 主数据库，处理生产负载 |
| **Standby Database** | 备数据库，接收 redo 数据 |
| **Redo Transport** | 传输 redo 到备库 |
| **Redo Apply** | 在备库应用 redo |
| **Role Management** | 主备角色管理 |

### 1.3 备库类型

#### （1）物理备库（Physical Standby）

- 与主库完全相同的物理结构
- 通过 Redo Apply 同步
- 性能最好，最常用

#### （2）逻辑备库（Logical Standby）

- 逻辑上相同，物理结构可以不同
- 通过 SQL Apply 同步
- 可以打开进行只读操作

#### （3）快照备库（Snapshot Standby）

- 可更新的测试环境
- 可以暂时脱离同步
- 适合测试和开发

## 2. 保护模式

### 2.1 三种保护模式

| 模式 | 数据保护 | 性能影响 | 可用性 |
|------|---------|---------|--------|
| **Maximum Protection** | 零数据丢失 | 高 | 低 |
| **Maximum Availability** | 零数据丢失（正常时） | 中 | 中 |
| **Maximum Performance** | 可能少量丢失 | 低 | 高 |

### 2.2 Maximum Protection

**特点**：
- 事务提交前必须写入备库
- 如果备库不可用，主库停止
- 零数据丢失保证

```sql
-- 设置保护模式
ALTER DATABASE SET STANDBY DATABASE TO MAXIMIZE PROTECTION;
```

### 2.3 Maximum Availability

**特点**：
- 正常时零数据丢失
- 备库不可用时降级为 Maximum Performance
- 平衡保护和可用性

```sql
ALTER DATABASE SET STANDBY DATABASE TO MAXIMIZE AVAILABILITY;
```

### 2.4 Maximum Performance（默认）

**特点**：
- 异步传输 redo
- 对主库性能影响最小
- 可能丢失少量数据

```sql
ALTER DATABASE SET STANDBY DATABASE TO MAXIMIZE PERFORMANCE;
```

## 3. 配置物理备库

### 3.1 前提条件

✅ **主库 ARCHIVELOG 模式**  
✅ **FORCE LOGGING 启用**  
✅ **相同的 Oracle 版本**  
✅ **网络连通性**  
✅ **足够的存储空间**  

### 3.2 配置步骤

#### 步骤 1：主库准备

```sql
-- 启用 FORCE LOGGING
ALTER DATABASE FORCE LOGGING;

-- 检查参数
SHOW PARAMETER db_unique_name;
SHOW PARAMETER log_archive_config;
SHOW PARAMETER log_archive_dest_2;
SHOW PARAMETER fal_server;

-- 创建 standby redo logs
ALTER DATABASE ADD STANDBY LOGFILE GROUP 4 
    ('/u01/oradata/stdby_redo04.log') SIZE 50M;
ALTER DATABASE ADD STANDBY LOGFILE GROUP 5 
    ('/u01/oradata/stdby_redo05.log') SIZE 50M;
ALTER DATABASE ADD STANDBY LOGFILE GROUP 6 
    ('/u01/oradata/stdby_redo06.log') SIZE 50M;
```

#### 步骤 2：配置监听和网络

```ini
# tnsnames.ora
PRIMARY_DB =
  (DESCRIPTION =
    (ADDRESS = (PROTOCOL = TCP)(HOST = primary_host)(PORT = 1521))
    (CONNECT_DATA = (SERVICE_NAME = primary_db))
  )

STANDBY_DB =
  (DESCRIPTION =
    (ADDRESS = (PROTOCOL = TCP)(HOST = standby_host)(PORT = 1521))
    (CONNECT_DATA = (SERVICE_NAME = standby_db))
  )
```

#### 步骤 3：创建备库

**方法 1：使用 RMAN 复制**

```bash
# 在备库服务器执行
rman TARGET sys/password@primary_db AUXILIARY sys/password@standby_db

RUN {
    DUPLICATE TARGET DATABASE FOR STANDBY FROM ACTIVE DATABASE
    DORECOVER
    SPFILE
      SET db_unique_name='standby_db' COMMENT 'Is Standby'
      SET log_archive_dest_2='service=primary_db ASYNC VALID_FOR=(ONLINE_LOGFILES,PRIMARY_ROLE) DB_UNIQUE_NAME=primary_db'
      SET fal_server='primary_db'
    NOFILENAMECHECK;
}
```

**方法 2：手动创建**

```bash
# 1. 备份主库
RMAN> BACKUP DATABASE FORMAT '/backup/%U';

# 2. 复制备份到备库

# 3. 在备库恢复
RMAN> STARTUP NOMOUNT;
RMAN> RESTORE CONTROLFILE FROM '/backup/controlfile.bak';
RMAN> ALTER DATABASE MOUNT;
RMAN> RESTORE DATABASE;
RMAN> RECOVER DATABASE;
```

#### 步骤 4：启动备库同步

```sql
-- 在备库
ALTER DATABASE RECOVER MANAGED STANDBY DATABASE DISCONNECT FROM SESSION;

-- 或使用实时应用（需要 standby redo logs）
ALTER DATABASE RECOVER MANAGED STANDBY DATABASE USING CURRENT LOGFILE DISCONNECT;
```

## 4. 监控 Data Guard

### 4.1 检查同步状态

```sql
-- 在主库
SELECT 
    dest_id,
    status,
    error,
    transmitted_seq,
    applied_seq
FROM v$archive_dest_status
WHERE target = 'STANDBY';

-- 在备库
SELECT 
    sequence#,
    first_time,
    next_time,
    applied
FROM v$archived_log
ORDER BY sequence# DESC;
```

### 4.2 查看延迟

```sql
-- 主备库延迟（秒）
SELECT 
    name,
    value,
    datum_time,
    time_computed
FROM v$dataguard_stats
WHERE name IN ('transport lag', 'apply lag');
```

### 4.3 监控进程

```sql
-- 查看 MRP 进程（Managed Recovery Process）
SELECT 
    process,
    status,
    thread#,
    sequence#,
    block#,
    blocks
FROM v$managed_standby;

-- 关键进程：
-- MRP0：Redo Apply
-- RFS：Remote File Server（接收 redo）
-- LGWR：写入本地 redo
```

### 4.4 告警脚本

```sql
-- 检查是否有 GAP
SELECT * FROM v$archive_gap;

-- 检查错误
SELECT 
    message,
    timestamp
FROM v$dataguard_status
WHERE severity IN ('Error', 'Fatal')
ORDER BY timestamp DESC;

-- 检查备库状态
SELECT 
    database_role,
    protection_mode,
    protection_level,
    open_mode
FROM v$database;
```

## 5. 角色转换

### 5.1 Switchover（计划内切换）

主备角色互换，无数据丢失。

```sql
-- 步骤 1：检查是否可以进行 switchover
SELECT switchover_status FROM v$database;
-- 应该返回：TO STANDBY 或 SESSIONS ACTIVE

-- 步骤 2：主库切换到备库
ALTER DATABASE COMMIT TO SWITCHOVER TO STANDBY WITH SESSION SHUTDOWN;

-- 步骤 3：备库切换到主库
ALTER DATABASE COMMIT TO SWITCHOVER TO PRIMARY WITH SESSION SHUTDOWN;

-- 步骤 4：原主库（现在是备库）启动同步
STARTUP MOUNT;
ALTER DATABASE RECOVER MANAGED STANDBY DATABASE DISCONNECT;

-- 步骤 5：原备库（现在是主库）打开
ALTER DATABASE OPEN;
```

### 5.2 Failover（故障转移）

主库故障，备库接管。

```sql
-- 步骤 1：检查是否有 GAP
SELECT * FROM v$archive_gap;

-- 如果有 GAP，尝试解决
-- 否则继续

-- 步骤 2：激活备库为主库
ALTER DATABASE RECOVER MANAGED STANDBY DATABASE FINISH FORCE;
ALTER DATABASE ACTIVATE STANDBY DATABASE;

-- 步骤 3：打开新主库
ALTER DATABASE OPEN;

-- 注意：这是破坏性操作，原主库不能再加入 DG
```

### 5.3 Fast-Start Failover（自动故障转移）

需要 Data Guard Broker 配置。

```bash
# 配置 Broker
dgmgrl /

DGMGRL> CREATE CONFIGURATION dg_config AS PRIMARY DATABASE IS primary_db CONNECT IDENTIFIER IS primary_db;
DGMGRL> ADD DATABASE standby_db AS CONNECT IDENTIFIER IS standby_db MAINTAINED AS PHYSICAL;
DGMGRL> ENABLE CONFIGURATION;

# 启用 Fast-Start Failover
DGMGRL> EDIT DATABASE primary_db SET PROPERTY FastStartFailoverTarget='standby_db';
DGMGRL> ENABLE FAST_START FAILOVER;
```

## 6. Active Data Guard

### 6.1 什么是 Active Data Guard？

允许备库在应用 redo 的同时打开进行只读查询。

**优势**：
- ✅ 报表查询卸载到备库
- ✅ 实时数据（几乎无延迟）
- ✅ 不影响主库性能

### 6.2 启用 Active Data Guard

```sql
-- 在备库
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE OPEN READ ONLY;
ALTER DATABASE RECOVER MANAGED STANDBY DATABASE USING CURRENT LOGFILE DISCONNECT;

-- 验证
SELECT open_mode FROM v$database;
-- 应该返回：READ ONLY WITH APPLY
```

### 6.3 使用场景

📊 **报表查询**
```sql
-- 连接到备库执行报表
SELECT /*+ PARALLEL */ COUNT(*) FROM large_table;
```

📊 **备份卸载**
```bash
# 在备库执行备份，减轻主库负担
RMAN> BACKUP DATABASE;
```

📊 **数据验证**
```sql
-- 验证数据一致性
SELECT COUNT(*) FROM employees@standby_db;
SELECT COUNT(*) FROM employees@primary_db;
```

## 7. 最佳实践

### 7.1 配置建议

✅ **使用 Maximum Availability 模式**（平衡保护和性能）  
✅ **配置 Standby Redo Logs**（支持实时应用）  
✅ **定期测试 Switchover**  
✅ **监控延迟和 GAP**  
✅ **使用 Data Guard Broker 简化管理**  

### 7.2 性能优化

⚡ **异步传输**（Maximum Performance）  
⚡ **专用网络**（redo 传输）  
⚡ **压缩传输**（带宽有限时）  
⚡ **并行应用**（大负载时）  

### 7.3 运维建议

📌 **每日检查同步状态**  
📌 **每周测试查询备库**  
📌 **每月执行 Switchover 测试**  
📌 **每季度进行 Failover 演练**  
📌 **记录所有变更**

## 8. 常见问题

### 8.1 GAP 处理

```sql
-- 检查 GAP
SELECT * FROM v$archive_gap;

-- 手动注册缺失的归档日志
ALTER DATABASE REGISTER LOGICAL LOGFILE '/path/to/archivelog';

-- 或从主库复制
scp primary_host:/archive/seq_100.arc standby_host:/archive/
```

### 8.2 延迟过大

**原因**：
- 网络带宽不足
- 备库 I/O 瓶颈
- 主库负载过高

**解决**：
```sql
-- 增加网络带宽
-- 优化备库 I/O
-- 调整 redo 传输参数
ALTER SYSTEM SET log_archive_max_processes = 10;
```

### 8.3 备库无法同步

**检查**：
- 网络连接
- 监听器状态
- 参数配置
- 磁盘空间
