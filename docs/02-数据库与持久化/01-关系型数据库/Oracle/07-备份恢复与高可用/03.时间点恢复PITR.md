## 1. PITR 概述

### 1.1 什么是 PITR？

PITR（Point-In-Time Recovery，时间点恢复）是将数据库或表空间恢复到过去某个特定时间点的技术。

**应用场景**：
- 误删除表或数据
- 逻辑错误（错误更新）
- 应用 bug 导致数据损坏
- 需要回滚到已知良好状态

### 1.2 PITR 类型

| 类型 | 范围 | 复杂度 |
|------|------|--------|
| **数据库级 PITR** | 整个数据库 | 高 |
| **表空间级 PITR（TSPITR）** | 特定表空间 | 中 |
| **Flashback** | 表/行级 | 低 |

## 2. 表空间级时间点恢复（TSPITR）

### 2.1 TSPITR 原理

TSPITR 通过创建辅助实例来恢复特定表空间，不影响其他表空间。

**工作流程**：
1. 创建辅助实例
2. 恢复辅助实例到目标时间点
3. 导出目标表空间的数据
4. 导入到主数据库
5. 清理辅助实例

### 2.2 TSPITR 前提条件

✅ **ARCHIVELOG 模式**  
✅ **目标表空间的备份在目标时间点之前**  
✅ **足够的磁盘空间（辅助实例）**  
✅ **归档日志从备份到目标时间点完整**

## 3. 自动 TSPITR

### 3.1 基本语法

```sql
RUN {
    RECOVER TABLESPACE users, app_data
    UNTIL TIME "TO_DATE('2026-05-19 10:00:00', 'YYYY-MM-DD HH24:MI:SS')"
    AUXILIARY DESTINATION '/u01/auxiliary';
}
```

### 3.2 完整示例

```sql
-- 场景：users 表空间中的数据被误删除
-- 需要恢复到 2 小时前

RUN {
    -- 设置目标时间
    RECOVER TABLESPACE users
    UNTIL TIME "SYSDATE - 2/24"
    
    -- 辅助实例位置
    AUXILIARY DESTINATION '/u01/app/oracle/auxiliary'
    
    -- 可选：指定辅助实例参数
    AUXILIARY PARAMETER FILE='/tmp/aux_init.ora';
}
```

### 3.3 监控 TSPITR 进度

```sql
-- 查看恢复进度
SELECT 
    sid,
    serial#,
    sofar,
    totalwork,
    ROUND(sofar/totalwork*100, 2) as pct_complete
FROM v$session_longops
WHERE opname LIKE 'RMAN%';

-- 查看辅助实例状态
ps -ef | grep aux
```

## 4. 手动 TSPITR

### 4.1 步骤详解

#### 步骤 1：确定恢复时间点

```sql
-- 查找误操作的时间
SELECT 
    operation,
    timestamp,
    table_name
FROM dba_tab_modifications
WHERE table_name = 'EMPLOYEES'
ORDER BY timestamp DESC;
```

#### 步骤 2：检查依赖对象

```sql
-- 查找跨表空间依赖
SELECT 
    owner,
    name,
    type,
    referenced_owner,
    referenced_name,
    referenced_type
FROM dba_dependencies
WHERE owner IN (
    SELECT DISTINCT owner 
    FROM dba_segments 
    WHERE tablespace_name = 'USERS'
);
```

#### 步骤 3：准备辅助实例

```bash
# 创建目录
mkdir -p /u01/app/oracle/auxiliary
mkdir -p /u01/app/oracle/auxiliary/adump

# 创建初始化参数文件
cat > /tmp/aux_init.ora << EOF
db_name=ORCL
compatible=19.0.0
db_block_size=8192
sga_target=2G
pga_aggregate_target=500M
control_files='/u01/app/oracle/auxiliary/control01.ctl'
EOF
```

#### 步骤 4：执行恢复

```sql
RUN {
    SET UNTIL TIME "TO_DATE('2026-05-19 10:00:00', 'YYYY-MM-DD HH24:MI:SS')";
    
    # 恢复控制文件
    RESTORE CONTROLFILE TO '/u01/app/oracle/auxiliary/control01.ctl';
    
    # 启动辅助实例
    STARTUP FORCE NOMOUNT AUXILIARY;
    
    # 挂载辅助数据库
    ALTER DATABASE MOUNT AUXILIARY;
    
    # 恢复数据文件
    RESTORE CLONE DATAFILE 4, 5;  -- users 表空间的文件
    
    # 恢复数据库
    RECOVER CLONE DATABASE;
    
    # 打开辅助数据库
    ALTER DATABASE OPEN RESETLOGS AUXILIARY;
}
```

#### 步骤 5：导出数据

```bash
# 使用 Data Pump 导出
expdp system/password@aux_db \
    tablespaces=users \
    directory=dp_dir \
    dumpfile=users_tspitr.dmp \
    logfile=users_tspitr.log
```

#### 步骤 6：导入到主数据库

```bash
# 先删除当前表空间中的对象
DROP TABLESPACE users INCLUDING CONTENTS AND DATAFILES;

# 重新创建表空间
CREATE TABLESPACE users DATAFILE '/u01/oradata/users01.dbf' SIZE 1G;

# 导入数据
impdp system/password@main_db \
    tablespaces=users \
    directory=dp_dir \
    dumpfile=users_tspitr.dmp \
    logfile=users_import.log
```

#### 步骤 7：清理辅助实例

```sql
-- 关闭辅助实例
SHUTDOWN IMMEDIATE;

-- 删除辅助文件
rm -rf /u01/app/oracle/auxiliary/*
```

## 5. TSPITR 限制

### 5.1 不支持的对象

❌ **SYS 用户的对象**  
❌ **回滚段**  
❌ **包含物化视图日志的表空间**  
❌ **含有 VARRAY 或嵌套表的对象**  

### 5.2 依赖关系问题

⚠️ **跨表空间外键**
```sql
-- 如果表 A（USERS）引用表 B（SYSTEM）
-- TSPITR 后可能导致引用完整性问题

-- 解决：同时恢复依赖的表空间
RECOVER TABLESPACE users, system UNTIL TIME ...;
```

⚠️ **索引在不同表空间**
```sql
-- 表和索引在不同表空间
-- 需要同时恢复
RECOVER TABLESPACE users, idx_users UNTIL TIME ...;
```

### 5.3 其他限制

⚠️ **不能恢复已删除的表空间**  
⚠️ **目标时间必须在备份之后**  
⚠️ **需要足够的磁盘空间**  
⚠️ **恢复期间表空间离线**

## 6. Flashback vs TSPITR

### 6.1 对比

| 特性 | Flashback | TSPITR |
|------|-----------|--------|
| **恢复粒度** | 表/行级 | 表空间级 |
| **速度** | 快 | 慢 |
| **复杂度** | 简单 | 复杂 |
| **空间需求** | UNDO/FRA | 辅助实例 |
| **适用场景** | 小范围错误 | 大范围错误 |

### 6.2 选择指南

**使用 Flashback**：
- 少量表或行被误修改
- UNDO 数据可用
- 需要快速恢复

**使用 TSPITR**：
- 整个表空间需要恢复
- Flashback 不可用
- 可以接受较长停机时间

## 7. 实战案例

### 案例 1：误删除表

```sql
-- 场景：开发人员误删除了重要表
DROP TABLE employees PURGE;

-- 解决方案 1：Flashback Drop（如果启用回收站）
FLASHBACK TABLE employees TO BEFORE DROP;

-- 解决方案 2：TSPITR（如果 Purge 或回收站禁用）
RUN {
    RECOVER TABLESPACE users
    UNTIL TIME "SYSDATE - 1/24"  -- 1 小时前
    AUXILIARY DESTINATION '/u01/auxiliary';
}
```

### 案例 2：批量错误更新

```sql
-- 场景：错误更新了所有员工的工资
UPDATE employees SET salary = salary * 10;
COMMIT;

-- 解决方案：TSPITR
RUN {
    RECOVER TABLESPACE users
    UNTIL SCN 1234567  -- 更新前的 SCN
    AUXILIARY DESTINATION '/u01/auxiliary';
}
```

### 案例 3：应用升级失败

```sql
-- 场景：应用升级后数据混乱，需要回滚

-- 记录升级前的时间点
SELECT SYSTIMESTAMP FROM dual;
-- 2026-05-19 08:00:00

-- 升级失败后，恢复到升级前
RUN {
    RECOVER TABLESPACE app_data, app_index
    UNTIL TIME "TO_DATE('2026-05-19 07:55:00', 'YYYY-MM-DD HH24:MI:SS')"
    AUXILIARY DESTINATION '/u01/auxiliary';
}
```

## 9. 故障排查

### 9.1 常见问题

#### 问题 1：空间不足

**错误**：ORA-01653: unable to extend table

**解决**：
```sql
-- 增加辅助实例空间
ALTER SYSTEM SET db_recovery_file_dest_size = 200G;

-- 或清理 FRA
DELETE OBSOLETE;
```

#### 问题 2：归档日志缺失

**错误**：ORA-00308: cannot open archived log

**解决**：
- 从备份恢复归档日志
- 或选择更早的恢复时间点

#### 问题 3：依赖关系冲突

**错误**：表空间包含依赖对象

**解决**：
```sql
-- 同时恢复依赖的表空间
RECOVER TABLESPACE users, system, idx_users UNTIL TIME ...;
```

### 9.2 诊断脚本

```sql
-- 检查表空间自包含
EXEC DBMS_TTS.TRANSPORT_SET_CHECK('users', TRUE);
SELECT * FROM TRANSPORT_SET_VIOLATIONS;

-- 查看需要的归档日志
SELECT 
    thread#,
    sequence#,
    first_time,
    next_time
FROM v$archived_log
WHERE first_time BETWEEN <backup_time> AND <target_time>
ORDER BY sequence#;
```
