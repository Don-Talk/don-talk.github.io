## 1. SQL Trace 概述

### 1.1 什么是 SQL Trace？

SQL Trace 是 Oracle 提供的 SQL 执行跟踪工具，记录 SQL 语句的详细执行过程。

**功能**：
- ✅ 记录 SQL 执行计划
- ✅ 统计执行次数和时间
- ✅ 记录等待事件
- ✅ 分析解析、执行、提取各阶段
- ✅ 生成详细的性能报告

## 2. 启用 SQL Trace

### 2.1 当前会话跟踪

```sql
-- 方法 1：ALTER SESSION（推荐）
ALTER SESSION SET sql_trace = TRUE;

-- 执行需要跟踪的 SQL
SELECT * FROM employees WHERE department_id = 10;

-- 停止跟踪
ALTER SESSION SET sql_trace = FALSE;

-- 方法 2：DBMS_SESSION
EXEC DBMS_SESSION.SET_SQL_TRACE(TRUE);
EXEC DBMS_SESSION.SET_SQL_TRACE(FALSE);

-- 方法 3：10046 事件（更详细）
ALTER SESSION SET EVENTS '10046 trace name context forever, level 12';
-- 执行 SQL
ALTER SESSION SET EVENTS '10046 trace name context off';
```

### 2.2 跟踪其他会话

```sql
-- 查找会话信息
SELECT sid, serial#, username, program
FROM v$session
WHERE username = 'HR';

-- 启用跟踪
EXEC DBMS_SYSTEM.SET_SQL_TRACE_IN_SESSION(sid => 123, serial# => 456, sql_trace => TRUE);

-- 停止跟踪
EXEC DBMS_SYSTEM.SET_SQL_TRACE_IN_SESSION(sid => 123, serial# => 456, sql_trace => FALSE);

-- 使用 10046 事件
EXEC DBMS_SYSTEM.SET_EV(si => 123, se => 456, ev => 10046, le => 12, nm => '');
```

### 2.3 跟踪级别

| 级别 | 说明 |
|------|------|
| **Level 1** | 基本 SQL Trace |
| **Level 4** | Level 1 + 绑定变量值 |
| **Level 8** | Level 1 + 等待事件 |
| **Level 12** | Level 1 + 绑定变量 + 等待事件（最常用） |

## 3. TKPROF 工具

### 3.1 TKPROF 基本用法

```bash
# 基本语法
tkprof tracefile.trc outputfile.prf [options]

# 示例
tkprof /u01/diag/orcl_ora_12345.trc /tmp/report.prf \
    sort=fchela \
    print=10 \
    sys=no \
    explain=hr/password
```

### 3.2 常用选项

| 选项 | 说明 |
|------|------|
| **sort=option** | 排序方式 |
| **print=n** | 只输出前 n 条 SQL |
| **sys=no** | 排除 SYS 用户的 SQL |
| **explain=user/pass** | 显示执行计划 |
| **aggregate=yes/no** | 是否合并相同 SQL |
| **insert=filename** | 生成 INSERT 语句存储到表 |
| **record=filename** | 生成 SQL*Plus 脚本 |

### 3.3 排序选项

| sort 值 | 说明 |
|---------|------|
| **prscnt** | 解析次数 |
| **prscpu** | 解析 CPU 时间 |
| **prsela** | 解析耗时 |
| **execnt** | 执行次数 |
| **exeela** | 执行耗时 |
| **fchcnt** | 提取次数 |
| **fchela** | 提取耗时 |
| **fchrow** | 提取行数 |

## 4. TKPROF 报告解读

### 4.1 报告结构

```
TKPROF: Release 19.0.0.0.0 - Development on Mon May 19 10:30:00 2026

Trace file: /u01/diag/orcl_ora_12345.trc
Sort options: fchela

********************************************************************************

count    = number of times OCI procedure was executed
cpu      = cpu time in seconds executing
elapsed  = elapsed time in seconds executing
disk     = number of physical reads of buffers from disk
query    = number of buffers gotten for consistent read
current  = number of buffers gotten in current mode (usually for update)
rows     = number of rows processed by the fetch or execute call

********************************************************************************
```

### 4.2 SQL 统计信息

```
SELECT * FROM employees WHERE department_id = 10

call     count       cpu    elapsed       disk      query    current        rows
------- ------  -------- ---------- ---------- ---------- ----------  ----------
Parse        1      0.01       0.02          0          0          0           0
Execute      1      0.00       0.00          0          0          0           0
Fetch       10      0.05       0.08         15        150          0         100
------- ------  -------- ---------- ---------- ---------- ----------  ----------
total       12      0.06       0.10         15        150          0         100

Misses in library cache during parse: 1
Optimizer mode: ALL_ROWS
Parsing user id: 85

Execution Plan:
----------------------------------------------------------
Plan hash value: 1234567890

--------------------------------------------------------------------------------
| Id  | Operation                   | Name            | Rows  | Bytes | Cost  |
--------------------------------------------------------------------------------
|   0 | SELECT STATEMENT            |                 |   100 |  5000 |    15 |
|   1 |  TABLE ACCESS BY INDEX ROWID| EMPLOYEES       |   100 |  5000 |    15 |
|*  2 |   INDEX RANGE SCAN          | IDX_EMP_DEPT    |   100 |       |     2 |
--------------------------------------------------------------------------------
```

### 4.3 关键指标解读

#### （1）Call 类型
- **Parse**：解析 SQL，检查语法和权限
- **Execute**：执行 SQL
- **Fetch**：提取结果集

#### （2）重要指标
- **cpu**：CPU 时间，越低越好
- **elapsed**：实际耗时，包括等待时间
- **disk**：物理读次数，应尽量减少
- **query**：逻辑读（一致性读），主要性能指标
- **current**：当前模式读（通常是 DML）
- **rows**：处理的行数

#### （3）Library Cache Misses
- **Misses = 0**：软解析（好）
- **Misses > 0**：硬解析（需要优化）

### 4.4 性能判断标准

| 指标 | 优秀 | 良好 | 需要优化 |
|------|------|------|---------|
| **CPU/Elapsed** | > 80% | 50-80% | < 50% |
| **Disk/Query** | < 1% | 1-5% | > 5% |
| **Rows/Fetch** | > 50 | 10-50 | < 10 |
| **Library Cache Miss** | 0 | < 10% | > 10% |

## 5. 10046 事件详解

### 5.1 启用 10046 跟踪

```sql
-- 当前会话
ALTER SESSION SET EVENTS '10046 trace name context forever, level 12';

-- 其他会话
EXEC DBMS_SYSTEM.SET_EV(si => 123, se => 456, ev => 10046, le => 12, nm => '');

-- 使用 DBMS_MONITOR（推荐）
EXEC DBMS_MONITOR.SESSION_TRACE_ENABLE(
    session_id => 123,
    serial_num => 456,
    waits => TRUE,
    binds => TRUE
);

-- 停止跟踪
EXEC DBMS_MONITOR.SESSION_TRACE_DISABLE(session_id => 123, serial_num => 456);
```

### 5.2 跟踪文件内容示例

```

SELECT employee_id, first_name, last_name FROM employees WHERE department_id = :b1
END OF STMT

BINDS #140234567890:
 Bind#0
  oacdty=02 mxl=22(22) mxlc=00 mal=00 scl=00 pre=00
  oacflg=03 fl2=1000000 frm=00 csi=00 siz=24 off=0
  kxsbbbfp=7f1234567890  bln=22  avl=02  flg=05
  value=10
```

### 5.3 字段说明

| 字段 | 说明 |
|------|------|
| **c** | CPU 时间（微秒） |
| **e** | 经过时间（微秒） |

| **mis** | Library Cache Miss |
| **r** | 返回行数 |

## 6. 实战案例

### 6.1 诊断慢查询

```sql
-- 步骤 1：启用跟踪
ALTER SESSION SET EVENTS '10046 trace name context forever, level 12';

-- 步骤 2：执行慢查询
SELECT e.*, d.department_name
FROM employees e
JOIN departments d ON e.department_id = d.department_id
WHERE e.salary > 5000
ORDER BY e.hire_date;

-- 步骤 3：停止跟踪
ALTER SESSION SET EVENTS '10046 trace name context off';

-- 步骤 4：找到 trace 文件
SELECT value FROM v$diag_info WHERE name = 'Default Trace File';

-- 步骤 5：使用 TKPROF 分析
-- tkprof tracefile.trc output.prf sort=fchela explain=hr/password
```

### 6.2 分析绑定变量问题

```sql
-- 启用带绑定变量的跟踪
ALTER SESSION SET EVENTS '10046 trace name context forever, level 12';

-- 执行多次不同值的查询
SELECT * FROM employees WHERE employee_id = 100;
SELECT * FROM employees WHERE employee_id = 200;
SELECT * FROM employees WHERE employee_id = 300;

ALTER SESSION SET EVENTS '10046 trace name context off';

-- TKPROF 分析
-- tkprof trace.trc report.prf sort=execnt aggregate=yes
```

### 6.3 监控特定用户

```sql
-- 查找用户会话
SELECT sid, serial# FROM v$session WHERE username = 'APP_USER';

-- 启用跟踪
EXEC DBMS_MONITOR.SESSION_TRACE_ENABLE(
    session_id => 123,
    serial_num => 456,
    waits => TRUE,
    binds => TRUE
);

-- 等待一段时间后停止
EXEC DBMS_MONITOR.SESSION_TRACE_DISABLE(
    session_id => 123,
    serial_num => 456
);
```

## 7. 自动化跟踪

### 7.1 基于条件的跟踪

```sql
-- 跟踪特定模块
BEGIN
    DBMS_MONITOR.SERV_MOD_ACT_TRACE_ENABLE(
        service_name => 'orcl',
        module_name => 'HR_APP',
        action_name => NULL,
        waits => TRUE,
        binds => TRUE
    );
END;
/

-- 停止跟踪
BEGIN
    DBMS_MONITOR.SERV_MOD_ACT_TRACE_DISABLE(
        service_name => 'orcl',
        module_name => 'HR_APP'
    );
END;
/
```

### 7.2 客户端标识符跟踪

```sql
-- 设置客户端标识符
EXEC DBMS_SESSION.SET_IDENTIFIER('user123');

-- 启用跟踪
BEGIN
    DBMS_MONITOR.CLIENT_ID_TRACE_ENABLE(
        client_id => 'user123',
        waits => TRUE,
        binds => TRUE
    );
END;
/

-- 在应用中设置
-- Java: connection.setClientInfo("OCSID.CLIENTID", "user123");
```

## 8. 最佳实践

### 8.1 使用建议

✅ **选择合适的跟踪级别**
- 一般问题：Level 1
- 绑定变量问题：Level 4
- 等待事件：Level 8
- 综合分析：Level 12

✅ **控制跟踪范围**
- 避免在生产环境全局跟踪
- 限制跟踪时间
- 定期清理 trace 文件

✅ **结合其他工具**
- AWR：宏观性能趋势
- ASH：实时会话分析
- SQL Trace：详细 SQL 分析

### 8.2 性能影响

⚠️ **SQL Trace 有性能开销**
- 生产环境谨慎使用
- 避免长时间跟踪
- 监控系统负载

### 8.3 文件管理

```bash
# 定期清理旧的 trace 文件
find /u01/diag -name "*.trc" -mtime +7 -delete
find /u01/diag -name "*.trm" -mtime +7 -delete

# 压缩归档
find /u01/diag -name "*.trc" -mtime +1 -exec gzip {} \;
```

## 9. 常见问题

### 9.1 找不到 Trace 文件

**解决**：
```sql
-- 确认 trace 目录
SHOW PARAMETER diagnostic_dest;

-- 获取当前 trace 文件
SELECT value FROM v$diag_info WHERE name = 'Default Trace File';

-- 列出所有 trace 文件
ls -lt $ORACLE_BASE/diag/rdbms/*/*/trace/*.trc | head
```

### 9.2 Trace 文件太大

**解决**：
```sql
-- 限制跟踪时间
-- 尽快执行 STOP 命令

-- 使用筛选条件
-- 只跟踪特定 SQL 或模块

-- 分段跟踪
-- 多次短时间跟踪代替长时间跟踪
```

### 9.3 TKPROF 输出为空

- 格式不正确

**解决**：

```bash
head -50 tracefile.trc

# 尝试不使用 explain 选项
tkprof trace.trc output.prf sys=no
```
