## 1. AWR 概述

### 1.1 什么是 AWR？

AWR（Automatic Workload Repository，自动工作负载仓库）是 Oracle 内置的性能数据收集和分析工具。

**特点**：
- ✅ 自动收集性能数据
- ✅ 存储在 SYSAUX 表空间
- ✅ 提供详细的性能分析
- ✅ 支持历史对比
- ✅ 生成 HTML/TXT 报告

## 2. AWR 架构

### 2.1 数据收集机制

```
MMON 进程（每 60 秒）
    ↓
采集内存中的统计数据
    ↓
写入 AWR（SYSAUX 表空间）
    ↓
生成快照（Snapshot）
```

### 2.2 关键组件

| 组件 | 说明 |
|------|------|
| **MMON** | Monitor 进程，负责收集 |
| **MMNL** | Monitor Light 进程，辅助收集 |
| **Snapshots** | 性能数据快照 |
| **Baselines** | 性能基线 |
| **WRH$ 表** | AWR 历史数据表 |

### 2.3 默认配置

```sql
-- 查看 AWR 配置
SELECT 
    snap_interval,
    retention,
    topnsql
FROM dba_hist_wr_control;

-- 默认值：
-- snap_interval: +00000 01:00:00.0 (1 小时)
-- retention: +00008 00:00:00.0 (8 天)
-- topnsql: 30 (Top SQL 数量)
```

## 3. 生成 AWR 报告

### 3.1 命令行生成

```bash
# 进入 SQL*Plus
sqlplus / as sysdba

# 运行 AWR 报告脚本
@?/rdbms/admin/awrrpt.sql
```

**交互式步骤**：
```
1. 选择报告类型：html 或 txt
2. 选择天数范围
3. 选择开始快照 ID
4. 选择结束快照 ID
5. 输入报告文件名
```

### 3.2 指定快照 ID

```sql
-- 先查看可用的快照
SELECT 
    snap_id,
    TO_CHAR(begin_interval_time, 'YYYY-MM-DD HH24:MI') as begin_time,
    TO_CHAR(end_interval_time, 'YYYY-MM-DD HH24:MI') as end_time
FROM dba_hist_snapshot
ORDER BY snap_id DESC
FETCH FIRST 20 ROWS ONLY;

-- 然后生成报告
@?/rdbms/admin/awrrpt.sql
-- 输入开始和结束 snap_id
```

### 3.3 SQL 生成

```sql
-- 使用 DBMS_WORKLOAD_REPOSITORY
BEGIN
    DBMS_WORKLOAD_REPOSITORY.AWR_REPORT_HTML(
        l_dbid => 1234567890,
        l_inst_num => 1,
        l_bid => 100,
        l_eid => 101,
        l_options => 0
    );
END;
```

### 3.4 RAC 环境

```bash
# RAC 全局报告
@?/rdbms/admin/awrgrpt.sql

# 特定实例报告
@?/rdbms/admin/awrrpti.sql
```

## 4. AWR 报告解读

### 4.1 报告结构

```
1. Report Summary（报告摘要）
2. Load Profile（负载概况）
3. Instance Efficiency Percentages（实例效率）
4. Top 5 Timed Events（前 5 等待事件）
5. SQL Statistics（SQL 统计）
6. I/O Statistics（I/O 统计）
7. Memory Statistics（内存统计）
8. Advisory Statistics（顾问统计）
```

### 4.2 Load Profile（负载概况）

**关键指标**：

| 指标 | 说明 | 正常范围 |
|------|------|---------|
| **DB Time** | 数据库总耗时 | - |
| **DB CPU** | CPU 时间 | < DB Time |
| **Redo size** | 重做日志大小 | 根据业务 |
| **Logical reads** | 逻辑读 | - |
| **Block changes** | 块变更 | - |
| **Physical reads** | 物理读 | 越少越好 |
| **Physical writes** | 物理写 | - |
| **User calls** | 用户调用 | - |
| **Parses** | 解析次数 | - |
| **Hard parses** | 硬解析 | < 10% of Parses |
| **Transactions** | 事务数 | 根据业务 |

**示例分析**：
```
Load Profile                    Per Second    Per Transaction
~~~~~~~~~~~~                    ~~~~~~~~~~    ~~~~~~~~~~~~~~~
DB Time(s):                         150.5           15.05
DB CPU(s):                          45.2             4.52
Redo size:                      1,234,567        123,456
Logical reads:                   50,000          5,000
Block changes:                   10,000          1,000
Physical reads:                     500             50
Physical writes:                    200             20
User calls:                       1,000            100
Parses:                             800             80
Hard parses:                         50              5
Transactions:                        10              1
```

**分析要点**：
- Hard Parse % = 50/800 = 6.25% ✅ 良好
- Physical Reads / Logical Reads = 500/50000 = 1% ✅ Buffer Cache 命中率高
- DB CPU / DB Time = 45.2/150.5 = 30% ⚠️ 可能有等待事件

### 4.3 Instance Efficiency Percentages

**理想值**：

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **Buffer Nowait %** | > 99% | Buffer Cache 无等待 |
| **Redo NoWait %** | 100% | Redo Log 无等待 |
| **Buffer Hit %** | > 95% | Buffer Cache 命中率 |
| **Library Hit %** | > 90% | Library Cache 命中率 |
| **Soft Parse %** | > 90% | 软解析比例 |
| **Execute to Parse %** | > 90% | 执行/解析比 |
| **Latch Hit %** | > 99% | Latch 命中率 |

**问题诊断**：
- Buffer Hit % 低 → 增加 db_cache_size
- Library Hit % 低 → 增加 shared_pool_size
- Soft Parse % 低 → 使用绑定变量

### 4.4 Top 5 Timed Events

**最重要的部分！**

```
Top 5 Timed Events
~~~~~~~~~~~~~~~~~~
Event                              Waits    Time(s)  Avg wait(ms)  %DB time
---------------------------------- -------- --------- ------------ ----------
db file scattered read             50,000   45,000    900           30%
CPU time                           -        30,000    -             20%
log file sync                      100,000  25,000    250           17%
db file sequential read            30,000   20,000    667           13%
latch: library cache               500,000  10,000    20            7%
```

**常见等待事件及解决**：

| 等待事件 | 原因 | 解决方案 |
|---------|------|---------|
| **db file scattered read** | 全表扫描 | 添加索引、优化 SQL |
| **db file sequential read** | 索引范围扫描 | 优化索引、检查聚簇因子 |
| **log file sync** | Commit 频繁 | 批量提交、减少 commit |
| **latch: library cache** | 硬解析多 | 使用绑定变量 |
| **enq: TX - row lock contention** | 锁竞争 | 优化事务、减少锁持有时间 |
| **direct path read** | 直接路径读 | 增加 Buffer Cache |
| **gc cr request** (RAC) | 缓存融合 | 优化 RAC 网络 |

### 4.5 SQL Statistics

#### （1）SQL ordered by Elapsed Time

## 5. AWR 管理

### 5.1 修改配置

```sql
-- 修改快照间隔和保留时间
BEGIN
    DBMS_WORKLOAD_REPOSITORY.MODIFY_SNAPSHOT_SETTINGS(
        interval => 30,      -- 30 分钟
        retention => 14400   -- 10 天（分钟）
    );
END;

-- 或使用 ALTER SYSTEM
ALTER SYSTEM SET statistics_level = TYPICAL;  -- BASIC/TYPICAL/ALL
```

### 5.2 手动创建快照

```sql
-- 创建快照
EXEC DBMS_WORKLOAD_REPOSITORY.CREATE_SNAPSHOT();

-- 删除快照
EXEC DBMS_WORKLOAD_REPOSITORY.DROP_SNAPSHOT_RANGE(
    low_snap_id => 100,
    high_snap_id => 110
);
```

### 5.3 创建基线（Baseline）

```sql
-- 创建固定基线
BEGIN
    DBMS_WORKLOAD_REPOSITORY.CREATE_BASELINE(
        start_snap_id => 100,
        end_snap_id => 110,
        baseline_name => 'baseline_normal_hours',
        expiration => 30  -- 30 天后过期
    );
END;

-- 创建移动窗口基线
BEGIN
    DBMS_WORKLOAD_REPOSITORY.CREATE_BASELINE_TEMPLATE(
        start_time => SYSDATE,
        end_time => SYSDATE + 7,
        baseline_name_prefix => 'baseline_weekly_',
        template_name => 'template_weekly',
        expiration => NULL  -- 永久
    );
END;

-- 查看基线
SELECT * FROM dba_hist_baseline;
```

**用途**：
- 性能对比
- 异常检测
- 容量规划

### 5.4 清理 AWR 数据

```sql
-- 删除过期快照（自动清理）
-- 由 MMON 进程根据 retention 设置自动清理

-- 手动清理
EXEC DBMS_WORKLOAD_REPOSITORY.DROP_SNAPSHOT_RANGE(
    low_snap_id => 1,
    high_snap_id => 50
);

-- 清空整个 AWR
TRUNCATE TABLE wrh$_active_session_history;
```

## 6. AWR 最佳实践

### 6.1 配置建议

✅ **快照间隔**：生产环境 15-30 分钟  
✅ **保留时间**：至少 7-14 天  
✅ **statistics_level**：TYPICAL（默认）  
✅ **定期创建基线**：正常负载时期  
✅ **监控 SYSAUX 空间**  

### 6.2 分析流程

📌 **查看 Top 5 Timed Events**（定位主要问题）  
📌 **检查 Load Profile**（了解负载特征）  
📌 **分析 Instance Efficiency**（评估效率）  
📌 **审查 Top SQL**（找出问题 SQL）  
📌 **检查 I/O 和内存**（资源使用情况）  
📌 **对比基线**（识别异常）  

### 6.3 监控 SYSAUX

```sql
-- 查看 SYSAUX 使用情况
SELECT 
    occupant_name,
    space_usage_kbytes/1024 as usage_mb,
    schema_name
FROM v$sysaux_occupants
ORDER BY space_usage_kbytes DESC;

-- AWR 通常占用 1-5 GB
-- 如果过大，考虑缩短 retention
```

## 7. 常见问题

### 7.1 AWR 未收集数据

**原因**：
- statistics_level = BASIC
- 许可证问题

**解决**：
```sql
ALTER SYSTEM SET statistics_level = TYPICAL;
```

### 7.2 SYSAUX 空间不足

**解决**：
```sql
-- 缩短 retention
BEGIN
    DBMS_WORKLOAD_REPOSITORY.MODIFY_SNAPSHOT_SETTINGS(
        retention => 4320  -- 3 天
    );
END;

-- 或删除旧快照
EXEC DBMS_WORKLOAD_REPOSITORY.DROP_SNAPSHOT_RANGE(1, 100);
```

### 7.3 快照缺失

**原因**：
- 数据库关闭
- MMON 进程异常

**解决**：
```sql
-- 检查 MMON 状态
SELECT program, status FROM v$process WHERE program LIKE '%MMON%';

-- 重启数据库或联系 Oracle Support
```
