### 01. AWR报告深度解读

**核心内容**：
- AWR 架构和快照机制
- 生成 AWR 报告
- Report Header 解读
- Load Profile 分析
- Top 5 Timed Events
- SQL Statistics 分析
- Instance Activity Stats
- Wait Events 分析
- Latch 统计

**关键知识点**：
```sql
-- 生成 AWR 报告
-- 方法 1：命令行
@$ORACLE_HOME/rdbms/admin/awrrpt.sql

-- 方法 2：PL/SQL
DECLARE
    l_snap_id_start NUMBER;
    l_snap_id_end NUMBER;
    l_report CLOB;
BEGIN
    -- 获取快照 ID
    SELECT MIN(snap_id), MAX(snap_id)
    INTO l_snap_id_start, l_snap_id_end
    FROM dba_hist_snapshot
    WHERE begin_interval_time >= SYSDATE - 1;
    
    -- 生成 HTML 报告
    l_report := DBMS_WORKLOAD_REPOSITORY.AWR_REPORT_HTML(
        l_dbid => (SELECT dbid FROM v$database),
        l_inst_num => 1,
        l_bid => l_snap_id_start,
        l_eid => l_snap_id_end
    );
    
    -- 保存到文件
    DBMS_XSLPROCESSOR.CLOB2FILE(l_report, 'DP_DIR', 'awr_report.html');
END;

-- 查看快照信息
SELECT 
    snap_id,
    begin_interval_time,
    end_interval_time,
    elapsed_seconds
FROM dba_hist_snapshot
ORDER BY snap_id DESC
FETCH FIRST 20 ROWS ONLY;

-- 修改快照设置
BEGIN
    DBMS_WORKLOAD_REPOSITORY.MODIFY_SNAPSHOT_SETTINGS(
        retention => 43200,  -- 保留 30 天（分钟）
        interval => 30       -- 每 30 分钟采集一次
    );
END;

-- 手动创建快照
EXEC DBMS_WORKLOAD_REPOSITORY.CREATE_SNAPSHOT;

-- 删除快照范围
EXEC DBMS_WORKLOAD_REPOSITORY.DROP_SNAPSHOT_RANGE(
    low_snap_id => 100,
    high_snap_id => 200
);
```

**AWR 报告关键指标**：
- **DB Time**：数据库总耗时
- **DB CPU**：CPU 使用时间
- **Redo size**：重做日志生成量
- **Executions**：执行次数
- **Parse calls**：解析调用次数
- **Buffer gets**：逻辑读
- **Disk reads**：物理读

### 02. ASH活动会话历史

**核心内容**：
- ASH 数据采集机制
- v$active_session_history 视图
- dba_hist_active_sess_history 视图
- ASH 报告生成
- 实时性能诊断
- 历史性能回溯

**关键知识点**：
```sql
-- 查看当前活动会话
SELECT 
    sample_time,
    session_id,
    session_serial#,
    sql_id,
    event,
    wait_class,
    blocking_session,
    current_obj#,
    current_file#,
    current_block#
FROM v$active_session_history
WHERE sample_time > SYSDATE - 1/24  -- 最近 1 小时
ORDER BY sample_time DESC;

-- 查找最耗时的 SQL
SELECT 
    sql_id,
    COUNT(*) as ash_samples,
    ROUND(COUNT(*) * 10 / 60, 2) as estimated_minutes
FROM v$active_session_history
WHERE sample_time > SYSDATE - 1
GROUP BY sql_id
ORDER BY ash_samples DESC
FETCH FIRST 10 ROWS ONLY;

-- 查找主要等待事件
SELECT 
    event,
    wait_class,
    COUNT(*) as wait_count,
    ROUND(COUNT(*) * 100 / SUM(COUNT(*)) OVER (), 2) as pct
FROM v$active_session_history
WHERE sample_time > SYSDATE - 1
AND event IS NOT NULL
GROUP BY event, wait_class
ORDER BY wait_count DESC
FETCH FIRST 10 ROWS ONLY;

-- 查找阻塞链
SELECT 
    sample_time,
    session_id,
    blocking_session,
    sql_id,
    event
FROM v$active_session_history
WHERE blocking_session IS NOT NULL
AND sample_time > SYSDATE - 1/12  -- 最近 5 分钟
ORDER BY sample_time DESC;

-- 生成 ASH 报告
@$ORACLE_HOME/rdbms/admin/ashrpt.sql
```

**实时监控脚本**：
```sql
-- 实时活动会话监控
SELECT 
    s.sid,
    s.serial#,
    s.username,
    s.program,
    s.status,
    w.event,
    w.wait_time,
    w.seconds_in_wait,
    q.sql_text
FROM v$session s
LEFT JOIN v$session_wait w ON s.sid = w.sid
LEFT JOIN v$sql q ON s.sql_id = q.sql_id
WHERE s.type = 'USER'
AND s.username IS NOT NULL
ORDER BY w.seconds_in_wait DESC;
```

### 03. ADDM自动诊断

**核心内容**：
- ADDM 工作原理
- 自动诊断流程
- 查看 ADDM 建议
- 实施优化建议
- 自定义 ADDM 分析

**关键知识点**：
```sql
-- 查看最新的 ADDM 报告
SELECT 
    task_name,
    status,
    creation_time,
    description
FROM dba_advisor_tasks
WHERE advisor_name = 'ADDM'
ORDER BY creation_time DESC
FETCH FIRST 10 ROWS ONLY;

-- 获取 ADDM 报告文本
SET LONG 1000000
SET PAGESIZE 0
SELECT DBMS_ADVISOR.GET_TASK_REPORT('ADDM_TASK_123') FROM dual;

-- 运行手动 ADDM 分析
DECLARE
    l_task_name VARCHAR2(100);
    l_task_desc VARCHAR2(1000);
BEGIN
    l_task_name := 'MY_ADDM_ANALYSIS';
    l_task_desc := 'Manual ADDM Analysis';
    
    DBMS_ADVISOR.CREATE_TASK(
        advisor_name => 'ADDM',
        task_name => l_task_name,
        task_desc => l_task_desc
    );
    
    DBMS_ADVISOR.SET_TASK_PARAMETER(
        task_name => l_task_name,
        parameter => 'START_SNAPSHOT',
        value => 100
    );
    
    DBMS_ADVISOR.SET_TASK_PARAMETER(
        task_name => l_task_name,
        parameter => 'END_SNAPSHOT',
        value => 110
    );
    
    DBMS_ADVISOR.EXECUTE_TASK(task_name => l_task_name);
END;

-- 查看建议
SELECT 
    finding_name,
    impact_type,
    impact,
    recommendation
FROM dba_advisor_findings
WHERE task_name = 'MY_ADDM_ANALYSIS';
```

**常见 ADDM 发现**：
- 硬解析过多
- I/O 瓶颈
- 内存不足
- 锁竞争
- 低效 SQL

### 04. SQL Trace与10046事件

**核心内容**：
- SQL Trace 基础
- 10046 事件级别
- 跟踪会话 SQL
- 跟踪整个实例
- 跟踪文件管理

**关键知识点**：
```sql
-- 方法 1：跟踪当前会话
ALTER SESSION SET SQL_TRACE = TRUE;
-- 执行 SQL...
ALTER SESSION SET SQL_TRACE = FALSE;

-- 方法 2：使用 DBMS_SESSION
EXEC DBMS_SESSION.SESSION_TRACE_ENABLE;
-- 执行 SQL...
EXEC DBMS_SESSION.SESSION_TRACE_DISABLE;

-- 方法 3：跟踪其他会话
-- 先找到 SID 和 SERIAL#
SELECT sid, serial#, username FROM v$session WHERE username = 'HR';

-- 启用跟踪
EXEC DBMS_SYSTEM.SET_SQL_TRACE_IN_SESSION(sid => 123, serial# => 456, sql_trace => TRUE);

-- 禁用跟踪
EXEC DBMS_SYSTEM.SET_SQL_TRACE_IN_SESSION(sid => 123, serial# => 456, sql_trace => FALSE);

-- 方法 4：使用 10046 事件（更详细）
-- Level 1: 基本 SQL Trace
ALTER SESSION SET EVENTS '10046 trace name context forever, level 1';

-- Level 4: 包括绑定变量
ALTER SESSION SET EVENTS '10046 trace name context forever, level 4';

-- Level 8: 包括等待事件
ALTER SESSION SET EVENTS '10046 trace name context forever, level 8';

-- Level 12: 包括绑定变量和等待事件
ALTER SESSION SET EVENTS '10046 trace name context forever, level 12';

-- 禁用
ALTER SESSION SET EVENTS '10046 trace name context off';

-- 查找跟踪文件
SELECT 
    s.sid,
    s.serial#,
    p.tracefile
FROM v$session s
JOIN v$process p ON s.paddr = p.addr
WHERE s.sid = 123;

-- 或使用
SHOW PARAMETER user_dump_dest;
```

### 05. TKPROF工具使用

**核心内容**：
- TKPROF 基本用法
- 输出报告解读
- 排序选项
- 过滤选项
- 性能分析技巧

**关键知识点**：
```bash
# 基本用法
tkprof trace_file.trc output_file.txt

# 带排序和解释
tkprof trace_file.trc output_file.txt \
    sort=exeela,fchela \
    explain=user/password \
    sys=no \
    aggregate=yes \
    print=10

# 常用排序选项
# exeela - 执行时间
# fchela - 获取时间
# cpucpu - CPU 时间
# diskrw - 磁盘读取
# buffer - 缓冲区获取
# rows   - 处理行数

# 示例
tkprof /u01/diag/rdbms/orcl/ORCL/trace/orcl_ora_12345.trc \
    /tmp/tkprof_output.txt \
    sort=fchela \
    explain=hr/hr \
    sys=no \
    print=20
```

**TKPROF 输出解读**：
```
call     count       cpu    elapsed       disk      query    current        rows
------- ------  -------- ---------- ---------- ---------- ----------  ----------
Parse       1      0.01       0.01          0          0          0           0
Execute     1      0.02       0.02          0          0          0           0
Fetch      10      0.15       0.20         50        500          0         100
------- ------  -------- ---------- ---------- ---------- ----------  ----------
total      12      0.18       0.23         50        500          0         100
```

**关键指标**：
- **cpu**：CPU 时间（秒）
- **elapsed**：经过时间（秒）
- **disk**：物理读次数
- **query**：一致性读（逻辑读）
- **current**：当前模式读
- **rows**：处理的行数

### 06. 等待事件分析

**核心内容**：
- 等待事件分类
- 常见等待事件详解
- I/O 相关等待
- 并发相关等待
- 网络相关等待
- 等待事件优化策略

**关键知识点**：
```sql
-- 查看当前等待事件
SELECT 
    sid,
    event,
    wait_class,
    wait_time,
    seconds_in_wait,
    state
FROM v$session_wait
WHERE wait_class != 'Idle'
ORDER BY seconds_in_wait DESC;

-- 系统级等待事件统计
SELECT 
    event,
    wait_class,
    total_waits,
    time_waited,
    average_wait,
    max_wait
FROM v$system_event
WHERE wait_class != 'Idle'
ORDER BY time_waited DESC
FETCH FIRST 20 ROWS ONLY;

-- 按等待类别汇总
SELECT 
    wait_class,
    SUM(total_waits) as total_waits,
    SUM(time_waited) as total_time_waited
FROM v$system_event
GROUP BY wait_class
ORDER BY total_time_waited DESC;
```

**常见等待事件及解决方案**：

| 等待事件 | 原因 | 解决方案 |
|---------|------|---------|
| `db file scattered read` | 全表扫描 | 添加索引、优化 SQL |
| `db file sequential read` | 索引扫描过多 | 检查索引效率、考虑 FTS |
| `log file sync` | 频繁提交 | 批量提交、减少提交频率 |
| `enq: TX - row lock contention` | 行锁竞争 | 优化事务、减少锁持有时间 |
| `latch: cache buffers chains` | Buffer Cache 热点块 | 优化 SQL、分散访问 |
| `cursor: pin S wait on X` | 硬解析竞争 | 使用绑定变量 |
| `enq: TM - contention` | 表级锁竞争 | 避免长时间持锁 |
| `direct path read/write` | 直接路径 I/O | 检查 PGA、临时表空间 |

```sql
-- 查找热点对象
SELECT 
    o.owner,
    o.object_name,
    o.object_type,
    SUM(h.time_waited) as total_wait_time
FROM v$active_session_history h
JOIN dba_objects o ON h.current_obj# = o.object_id
WHERE h.sample_time > SYSDATE - 1
GROUP BY o.owner, o.object_name, o.object_type
ORDER BY total_wait_time DESC
FETCH FIRST 10 ROWS ONLY;
```

### 07. 绑定变量与游标共享

**核心内容**：
- 硬解析 vs 软解析
- 绑定变量优势
- 游标共享机制
- 绑定变量窥探
- 自适应游标共享（ACS）
- 游标失效原因

**关键知识点**：
```sql
-- 查看库缓存命中率
SELECT 
    namespace,
    gethitratio,
    pinhitratio,
    reloads,
    invalidations
FROM v$librarycache;

-- 查看硬解析和软解析
SELECT 
    name,
    value
FROM v$sysstat
WHERE name IN ('parse count (total)', 'parse count (hard)', 'parse count (soft)');

-- 计算硬解析比例
SELECT 
    ROUND(
        (SELECT value FROM v$sysstat WHERE name = 'parse count (hard)') /
        NULLIF((SELECT value FROM v$sysstat WHERE name = 'parse count (total)'), 0) * 100,
        2
    ) as hard_parse_pct
FROM dual;

-- 查看未使用绑定变量的 SQL
SELECT 
    sql_id,
    executions,
    parse_calls,
    ROUND(parse_calls / NULLIF(executions, 0) * 100, 2) as parse_pct,
    SUBSTR(sql_text, 1, 100) as sql_text
FROM v$sqlarea
WHERE executions > 10
AND parse_calls / NULLIF(executions, 0) > 0.8
ORDER BY parse_calls DESC
FETCH FIRST 20 ROWS ONLY;

-- 查看子游标数量（判断是否过度共享）
SELECT 
    sql_id,
    COUNT(child_number) as child_count,
    executions
FROM v$sql
GROUP BY sql_id, executions
HAVING COUNT(child_number) > 10
ORDER BY child_count DESC
FETCH FIRST 20 ROWS ONLY;

-- 查看游标失效原因
SELECT 
    reason,
    COUNT(*) as count
FROM v$shared_cursor
GROUP BY reason
ORDER BY count DESC;
```

**使用绑定变量**：
```sql
-- ❌ 不使用绑定变量（每次都是硬解析）
SELECT * FROM employees WHERE employee_id = 100;
SELECT * FROM employees WHERE employee_id = 101;
SELECT * FROM employees WHERE employee_id = 102;

-- ✅ 使用绑定变量（软解析）
VARIABLE emp_id NUMBER;
:emp_id := 100;
SELECT * FROM employees WHERE employee_id = :emp_id;

:emp_id := 101;
SELECT * FROM employees WHERE employee_id = :emp_id;

-- PL/SQL 中自动使用绑定变量
BEGIN
    FOR i IN 1..100 LOOP
        UPDATE employees SET salary = salary * 1.1
        WHERE employee_id = i;  -- 自动绑定
    END LOOP;
END;
```

**自适应游标共享（ACS）**：
```sql
-- 查看 ACS 信息
SELECT 
    sql_id,
    child_number,
    is_bind_sensitive,
    is_bind_aware,
    is_shareable
FROM v$sql
WHERE sql_id = 'your_sql_id';

-- 绑定变量窥探
ALTER SYSTEM SET "_optim_peek_user_binds" = TRUE;  -- 默认 TRUE
```

### 08. 并行查询优化

**核心内容**：
- 并行执行架构
- 并行度设置
- 并行提示（Hint）
- 并行执行计划
- 并行负载平衡
- 并行执行监控

**关键知识点**：
```sql
-- 表级并行度设置
ALTER TABLE large_table PARALLEL 4;
ALTER TABLE large_table NOPARALLEL;

-- 会话级并行度
ALTER SESSION FORCE PARALLEL QUERY PARALLEL 8;
ALTER SESSION FORCE PARALLEL DML PARALLEL 4;

-- 使用 Hint
SELECT /*+ PARALLEL(e 4) */ *
FROM employees e
WHERE department_id = 10;

SELECT /*+ PARALLEL(4) */ *
FROM large_table;

-- 并行 DDL
CREATE INDEX idx_large PARALLEL 8 AS SELECT ...;
ALTER TABLE large_table MOVE PARALLEL 8;

-- 查看并行执行
SELECT 
    qcsid,
    qcserial#,
    degree,
    req_degree,
    COUNT(*) as slave_count
FROM v$px_session
GROUP BY qcsid, qcserial#, degree, req_degree;

-- 监控并行查询
SELECT 
    s.sid,
    s.username,
    px.qcsid,
    px.degree,
    px.req_degree,
    s.sql_id
FROM v$session s
JOIN v$px_session px ON s.sid = px.sid
WHERE px.qcsid IS NOT NULL;

-- 并行执行统计
SELECT 
    name,
    value
FROM v$sysstat
WHERE name LIKE 'Parallel operations%';
```

**并行度计算**：
```sql
-- Oracle 推荐公式
-- PARALLEL_DEGREE_LIMIT = CPU_COUNT * PARALLEL_THREADS_PER_CPU

SHOW PARAMETER cpu_count;
SHOW PARAMETER parallel_threads_per_cpu;

-- 自动并行度
ALTER SYSTEM SET parallel_degree_policy = AUTO;
ALTER TABLE large_table PARALLEL;  -- Oracle 自动决定并行度
```

**注意事项**：
- ⚠️ 并行查询消耗更多资源
- ⚠️ 适合大数据量操作（通常 > 1GB）
- ⚠️ OLTP 系统谨慎使用
- ⚠️ 监控系统负载，避免过载
