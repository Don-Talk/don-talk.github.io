## 1. 等待事件概述

### 1.1 什么是等待事件？

等待事件是 Oracle 会话在等待某种资源或条件时记录的事件。

**DB Time 组成**：
```
DB Time = CPU Time + Wait Time
```

**等待事件分类**：
- **Idle Events**：空闲等待（正常）
- **User I/O**：用户 I/O 等待
- **System I/O**：系统 I/O 等待
- **Concurrency**：并发等待
- **Application**：应用层等待
- **Configuration**：配置问题
- **Administrative**：管理操作
- **Network**：网络等待
- **Commit**：提交等待
- **Scheduler**：调度器等待

## 2. I/O 等待事件

### 2.1 db file scattered read

**含义**：多块读，通常是全表扫描或快速全索引扫描。

**常见原因**：
- 缺少索引
- 统计信息过时
- SQL 写法导致无法使用索引
- 小表全表扫描（正常）

**诊断**：
```sql
-- 查看等待该事件的会话
SELECT 
    sid,
    serial#,
    username,
    event,
    p1 as file_id,
    p2 as block_id,
    p3 as blocks,
    seconds_in_wait
FROM v$session_wait
WHERE event = 'db file scattered read';

-- 查找对应的对象
SELECT owner, segment_name, segment_type
FROM dba_extents
WHERE file_id = &file_id
AND &block_id BETWEEN block_id AND block_id + blocks - 1;

-- 查找相关 SQL
SELECT sql_id, sql_text
FROM v$sql
WHERE sql_id IN (
    SELECT sql_id FROM v$session WHERE event = 'db file scattered read'
);
```

**优化方案**：
1. 添加合适的索引
2. 重写 SQL 避免全表扫描
3. 考虑分区表
4. 增加 Buffer Cache
5. 使用 IOT（索引组织表）

### 2.2 db file sequential read

**含义**：单块读，通常是索引范围扫描或回表操作。

**常见原因**：
- 索引范围扫描
- 通过 ROWID 访问表
- 聚簇因子差导致大量回表

**诊断**：
```sql
-- 查看等待会话
SELECT 
    sid,
    serial#,
    username,
    event,
    p1 as file_id,
    p2 as block_id,
    p3 as blocks,
    seconds_in_wait,
    sql_id
FROM v$session_wait
WHERE event = 'db file sequential read';

-- 检查索引聚簇因子
SELECT 
    i.index_name,
    i.table_name,
    i.clustering_factor,
    t.num_rows,
    t.blocks
FROM user_indexes i, user_tables t
WHERE i.table_name = t.table_name
AND i.clustering_factor > t.num_rows * 0.9;
```

**优化方案**：
1. 优化索引聚簇因子（重建表）
2. 使用覆盖索引避免回表
3. 调整 SQL 减少索引扫描范围
4. 考虑索引压缩

### 2.3 direct path read / direct path write

**含义**：直接路径读写，绕过 Buffer Cache。

**常见场景**：
- 并行查询
- 临时段操作（排序、哈希）
- 批量加载
- 全表扫描大表

**诊断**：
```sql
-- 查看直接路径读写
SELECT 
    event,
    COUNT(*) as wait_count,
    SUM(time_waited) as total_wait_time
FROM v$active_session_history
WHERE event LIKE 'direct path%'
AND sample_time > SYSDATE - 1/24
GROUP BY event;

-- 检查 PGA 使用
SELECT 
    name,
    value/1024/1024 as mb
FROM v$pgastat
WHERE name IN ('aggregate PGA target parameter',
               'total PGA inuse',
               'maximum PGA allocated');
```

**优化方案**：
1. 增加 PGA_AGGREGATE_TARGET
2. 优化排序操作（添加索引）
3. 减少 DISTINCT、GROUP BY、ORDER BY
4. 使用物化视图预聚合

## 3. 锁和闩锁等待

### 3.1 enq: TX - row lock contention

**含义**：行级锁竞争，事务等待其他事务释放行锁。

**常见原因**：
- 多个会话更新同一行
- 外键未加锁
- 唯一索引冲突
- 位图索引更新

**诊断**：
```sql
-- 查找锁等待
SELECT 
    w.sid as waiting_sid,
    w.event,
    w.seconds_in_wait,
    b.sid as blocking_sid,
    b.username as blocking_user,
    b.program as blocking_program
FROM v$session w, v$session b
WHERE w.blocking_session = b.sid
AND w.event = 'enq: TX - row lock contention';

-- 查找被锁定的对象
SELECT 
    l.session_id,
    o.owner,
    o.object_name,
    o.object_type,
    l.locked_mode
FROM v$locked_object l, dba_objects o
WHERE l.object_id = o.object_id;

-- 查看锁详细信息
SELECT * FROM v$lock WHERE type = 'TX';
```

**解决**：
1. 提交或回滚阻塞事务
2. 杀死阻塞会话（最后手段）
3. 优化应用逻辑，减少锁持有时间
4. 使用乐观锁机制

```sql
-- 紧急处理：杀死阻塞会话
ALTER SYSTEM KILL SESSION 'sid,serial#' IMMEDIATE;
```

### 3.2 latch: library cache

**含义**：库缓存闩锁竞争，通常是硬解析过多。

**常见原因**：
- 大量硬解析
- 未使用绑定变量
- Shared Pool 太小
- 高并发解析

**诊断**：
```sql
-- 查看 Library Cache 命中率
SELECT 
    namespace,
    gets,
    gethits,
    ROUND(gethitratio * 100, 2) as hit_ratio,
    pins,
    pinhits,
    ROUND(pinhitratio * 100, 2) as pin_hit_ratio
FROM v$librarycache;

-- 检查硬解析比例
SELECT 
    name,
    value
FROM v$sysstat
WHERE name IN ('parse count (total)', 'parse count (hard)');

-- 查找未使用绑定变量的 SQL
SELECT 
    substr(sql_text, 1, 100) as sql_preview,
    COUNT(*) as duplicate_count
FROM v$sql
GROUP BY substr(sql_text, 1, 100)
HAVING COUNT(*) > 10
ORDER BY duplicate_count DESC;
```

**优化方案**：
1. 使用绑定变量
2. 增加 SHARED_POOL_SIZE
3. 设置 CURSOR_SHARING = FORCE（谨慎）
4. 使用会话池减少解析

```sql
-- 调整 Shared Pool
ALTER SYSTEM SET shared_pool_size = 2G SCOPE=BOTH;

-- 设置游标共享（谨慎使用）
ALTER SYSTEM SET cursor_sharing = FORCE SCOPE=BOTH;
```

### 3.3 latch: cache buffers chains

**含义**：缓冲区链闩锁竞争，热点块访问。

**常见原因**：
- 热点块（频繁访问的数据块）
- 嵌套循环连接导致重复访问
- Buffer Cache 不足

**诊断**：
```sql
-- 查找热点块
SELECT 
    file#,
    dbablk,
    tch as touch_count,
    obj
FROM x$bh
WHERE tch > 100
ORDER BY tch DESC
FETCH FIRST 20 ROWS ONLY;

-- 查找热点块对应的对象
SELECT 
    owner,
    object_name,
    subobject_name,
    object_type
FROM dba_objects
WHERE data_object_id = &obj_id;

-- 检查闩锁统计
SELECT 
    name,
    gets,
    misses,
    ROUND(misses/NULLIF(gets,0)*100, 2) as miss_ratio,
    sleeps
FROM v$latch
WHERE name = 'cache buffers chains';
```

**优化方案**：
1. 优化 SQL 减少逻辑读
2. 使用哈希连接代替嵌套循环
3. 分区表分散热点
4. 增加 Buffer Cache
5. 使用反向键索引打散数据

## 4. 网络和提交等待

### 4.1 SQL*Net message from client

**含义**：等待客户端发送请求（空闲等待）。

**说明**：
- 这是正常的空闲等待
- 如果占比过高，说明应用响应慢
- 不是数据库问题

### 4.2 log file sync

**含义**：等待 redo 日志写入磁盘（commit 时发生）。

**常见原因**：
- 频繁的 commit
-  redo 日志磁盘 I/O 慢
- 日志文件太小导致频繁切换

**诊断**：
```sql
-- 查看 log file sync 等待
SELECT 
    event,
    total_waits,
    time_waited,
    average_wait
FROM v$system_event
WHERE event = 'log file sync';

-- 检查 redo 日志切换频率
SELECT 
    TO_CHAR(first_time, 'YYYY-MM-DD HH24:MI') as time,
    sequence#
FROM v$log_history
ORDER BY first_time DESC
FETCH FIRST 20 ROWS ONLY;

-- 检查 redo 大小
SELECT 
    name,
    value/1024/1024 as mb
FROM v$sysstat
WHERE name = 'redo size';
```

**优化方案**：
1. 减少 commit 频率（批量提交）
2. 使用nologging操作（谨慎）
3. 优化 redo 日志文件大小和数量
4. 使用更快的磁盘存储 redo 日志
5. 使用 COMMIT WRITE BATCH/NOWAIT

```sql
-- 批量提交
FOR i IN 1..10000 LOOP
    INSERT INTO table_name VALUES (...);
    IF MOD(i, 1000) = 0 THEN
        COMMIT;
    END IF;
END LOOP;

-- 异步提交（有风险）
COMMIT WRITE NOWAIT;
```

## 5. 等待事件分析方法

### 5.1 实时等待分析

```sql
-- 当前最严重的等待事件
SELECT 
    event,
    wait_class,
    COUNT(*) as sessions_waiting,
    SUM(seconds_in_wait) as total_wait_seconds
FROM v$session
WHERE wait_class != 'Idle'
GROUP BY event, wait_class
ORDER BY sessions_waiting DESC
FETCH FIRST 10 ROWS ONLY;

-- 等待会话详情
SELECT 
    s.sid,
    s.serial#,
    s.username,
    s.event,
    s.seconds_in_wait,
    s.sql_id,
    q.sql_text
FROM v$session s
LEFT JOIN v$sql q ON s.sql_id = q.sql_id
WHERE s.wait_class != 'Idle'
ORDER BY s.seconds_in_wait DESC;
```

### 5.2 历史等待分析

```sql
-- AWR 中的 Top 等待事件
SELECT 
    event,
    wait_class,
    total_waits,
    ROUND(time_waited/100, 2) as time_waited_sec,
    ROUND(average_wait/100, 2) as avg_wait_sec
FROM dba_hist_system_event
WHERE snap_id BETWEEN &start_snap AND &end_snap
AND wait_class != 'Idle'
ORDER BY time_waited DESC
FETCH FIRST 15 ROWS ONLY;

-- 等待事件趋势
SELECT 
    TO_CHAR(begin_interval_time, 'YYYY-MM-DD HH24:MI') as time,
    event,
    total_waits,
    time_waited
FROM dba_hist_system_event hse, dba_hist_snapshot hs
WHERE hse.snap_id = hs.snap_id
AND event = 'db file scattered read'
ORDER BY begin_interval_time;
```

### 5.3 ASH 等待分析

```sql
-- 过去 1 小时的等待事件分布
SELECT 
    event,
    wait_class,
    COUNT(*) as samples,
    ROUND(COUNT(*) * 100 / SUM(COUNT(*)) OVER (), 2) as pct
FROM v$active_session_history
WHERE sample_time > SYSDATE - 1/24
AND event IS NOT NULL
GROUP BY event, wait_class
ORDER BY samples DESC;

-- 特定等待事件的会话
SELECT 
    session_id,
    sql_id,
    current_obj#,
    COUNT(*) as samples
FROM v$active_session_history
WHERE event = 'enq: TX - row lock contention'
AND sample_time > SYSDATE - 1/24
GROUP BY session_id, sql_id, current_obj#
ORDER BY samples DESC;
```

## 6. 实战案例

### 案例 1：全表扫描导致 I/O 等待

**现象**：系统响应慢，大量 `db file scattered read` 等待。

**诊断步骤**：
```sql
-- 1. 确认等待事件
SELECT event, COUNT(*) FROM v$session 
WHERE wait_class != 'Idle' 
GROUP BY event;

-- 2. 查找相关 SQL
SELECT sql_id, COUNT(*) FROM v$active_session_history
WHERE event = 'db file scattered read'
AND sample_time > SYSDATE - 15/1440
GROUP BY sql_id
ORDER BY COUNT(*) DESC;

-- 3. 获取 SQL 文本
SELECT sql_text FROM v$sql WHERE sql_id = '&sql_id';

-- 4. 检查执行计划
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR('&sql_id'));

-- 5. 检查是否缺少索引
SELECT index_name, column_name 
FROM user_ind_columns 
WHERE table_name = 'LARGE_TABLE';
```

**解决**：
```sql
-- 添加索引
CREATE INDEX idx_large_table_col ON large_table(column_name);

-- 收集统计信息
EXEC DBMS_STATS.GATHER_TABLE_STATS('HR', 'LARGE_TABLE');
```

### 案例 2：死锁导致的应用挂起

**现象**：应用多个会话hang住。

**诊断**：
```sql
-- 查找阻塞链
SELECT 
    LPAD(' ', LEVEL*2) || sid as sid_tree,
    serial#,
    username,
    event,
    blocking_session
FROM v$session
START WITH blocking_session IS NULL
CONNECT BY PRIOR sid = blocking_session;

-- 查找根本阻塞者
SELECT 
    sid,
    serial#,
    username,
    program,
    sql_id
FROM v$session
WHERE sid IN (
    SELECT blocking_session 
    FROM v$session 
    WHERE blocking_session IS NOT NULL
)
AND sid NOT IN (
    SELECT sid 
    FROM v$session 
    WHERE blocking_session IS NOT NULL
);
```

**解决**：
```sql
-- 联系应用方确认后，杀死阻塞会话
ALTER SYSTEM KILL SESSION '123,4567' IMMEDIATE;

-- 或者定位到具体 SQL 后修复应用逻辑
```

## 7. 最佳实践

### 7.1 监控策略

✅ **实时监控**
- 设置等待事件告警阈值
- 监控活跃会话数
- 跟踪 Top 等待事件

✅ **定期分析**
- 每日审查 AWR 报告
- 分析等待事件趋势
- 识别新出现的瓶颈

✅ **基线对比**
- 建立性能基线
- 对比异常时段
- 发现性能退化

### 7.2 优化原则

📌 **优先解决影响最大的等待**  
📌 **关注 DB Time 占比高的事件**  
📌 **结合业务场景分析**  
📌 **避免过度优化**  
📌 **持续监控验证效果**  

### 7.3 常用脚本

创建监控视图：
```sql
-- 创建等待事件监控视图
CREATE OR REPLACE VIEW vw_current_waits AS
SELECT 
    s.sid,
    s.serial#,
    s.username,
    s.program,
    s.event,
    s.wait_class,
    s.seconds_in_wait,
    s.sql_id,
    q.sql_text
FROM v$session s
LEFT JOIN v$sql q ON s.sql_id = q.sql_id
WHERE s.wait_class != 'Idle';

-- 使用
SELECT * FROM vw_current_waits ORDER BY seconds_in_wait DESC;
```
