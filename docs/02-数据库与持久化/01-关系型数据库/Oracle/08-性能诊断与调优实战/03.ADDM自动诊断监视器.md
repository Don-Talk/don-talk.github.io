## 1. ADDM 概述

### 1.1 什么是 ADDM？

ADDM（Automatic Database Diagnostic Monitor，自动数据库诊断监视器）是 Oracle 的智能性能诊断工具。

**特点**：
- ✅ 自动分析 AWR 数据
- ✅ 识别性能瓶颈的根本原因
- ✅ 提供具体的优化建议
- ✅ 量化性能问题的影响
- ✅ 每小时自动运行

<!-- TODO: 插入图片：ADDM 工作流程图 -->

### 1.2 ADDM vs AWR vs ASH

| 特性 | ADDM | AWR | ASH |
|------|------|-----|-----|
| **类型** | 诊断引擎 | 数据仓库 | 采样数据 |
| **数据来源** | AWR | 内存统计 | 会话采样 |
| **输出** | 诊断报告 | 统计报告 | 原始数据 |
| **自动化** | 完全自动 | 手动生成报告 | 手动查询 |
| **建议** | ✅ 有 | ❌ 无 | ❌ 无 |

### 1.3 ADDM 工作流程

```
AWR 快照（每小时）
    ↓
ADDM 自动分析
    ↓
识别性能问题
    ↓
计算影响（DB Time）
    ↓
生成诊断和建议
    ↓
存储到 DBA_ADVISOR_* 视图
```

## 2. 查看 ADDM 报告

### 2.1 通过 SQL*Plus 生成

```bash
# 运行 ADDM 报告脚本
sqlplus / as sysdba
@?/rdbms/admin/addmrpt.sql
```

**交互式步骤**：
1. 选择开始快照 ID
2. 选择结束快照 ID
3. 输入报告文件名
4. 查看报告

### 2.2 通过 PL/SQL 获取

```sql
-- 获取最新的 ADDM 报告
DECLARE
    l_task_name VARCHAR2(100);
    l_report CLOB;
BEGIN
    -- 获取最新的 ADDM 任务名
    SELECT task_name INTO l_task_name
    FROM (
        SELECT task_name
        FROM dba_advisor_tasks
        WHERE advisor_name = 'ADDM'
        AND status = 'COMPLETED'
        ORDER BY created DESC
    )
    WHERE ROWNUM = 1;
    
    -- 获取报告
    l_report := DBMS_ADVISOR.GET_TASK_REPORT(
        task_name => l_task_name,
        type => 'TEXT',
        level => 'TYPICAL'
    );
    
    -- 输出报告
    DBMS_OUTPUT.PUT_LINE(SUBSTR(l_report, 1, 32767));
END;
/
```

### 2.3 通过 Enterprise Manager

1. 登录 Oracle Enterprise Manager
2. 导航到 Performance → ADDM
3. 选择时间范围
4. 查看分析报告

## 3. ADDM 报告结构

### 3.1 报告概览

```
ADDM Report for Task 'TASK_12345'
---------------------------------

Analysis Period
---------------
AWR snapshot range from 100 to 101.
Time period starts at 19-MAY-26 10.00.00 AM
Time period ends at 19-MAY-26 11.00.00 AM

Analysis Target
---------------
Database 'ORCL' with DB ID 1234567890.
Database version 19.0.0.0.0.
Analysis was requested for all instances, but ADDM analyzed instance orcl.

Activity During the Analysis Period
-----------------------------------
Total database time was 12345 seconds.
The average number of active sessions was 3.43.
```

### 3.2 Findings（发现）

ADDM 按影响程度排序列出发现的问题：

```
Finding 1: Hard Parse Due to Literals
Impact is 2.3 active sessions, 18.67% of total activity.
----------------------------------------------------------
SQL statements consuming significant database time were found 
to be spending significant time on hard parses due to literals.

Recommendation 1: Application Analysis
Estimated benefit is 2.3 active sessions, 18.67% of total activity.
--------------------------------------------------------------------
Action
  Investigate application logic to use bind variables instead of 
  literals.

Rationale
  The SQL statements shown below were found to be causing excessive 
  hard parses. Use bind variables to reduce parsing overhead.

Affected SQL Statements
-----------------------
  UPDATE employees SET salary = 5000 WHERE employee_id = 100
  UPDATE employees SET salary = 6000 WHERE employee_id = 101
```

### 3.3 常见 Finding 类型

| Finding 类型 | 说明 | 典型影响 |
|------------|------|---------|
| **Hard Parse** | 硬解析过多 | 高 CPU 使用 |
| **Soft Parse** | 软解析过多 | 中等 CPU 使用 |
| **Buffer Busy** | 缓冲区忙等待 | I/O 瓶颈 |
| **Lock Contention** | 锁竞争 | 会话阻塞 |
| **I/O Bottleneck** | I/O 瓶颈 | 磁盘等待 |
| **Memory Configuration** | 内存配置不当 | 性能下降 |
| **Top Segments** | 热点段 | I/O 集中 |

## 4. 常见诊断场景

### 4.1 CPU 使用率高

**ADDM 发现**：
```
Finding: High CPU Usage
Impact is 5.2 active sessions, 42% of total activity.

Symptom:
  Wait class "CPU" was consuming significant database time.

Recommendation:
  Investigate SQL statements consuming high CPU.
```

**解决步骤**：
```sql
-- 1. 查找高 CPU 的 SQL
SELECT sql_id, cpu_time, executions, elapsed_time
FROM v$sql
ORDER BY cpu_time DESC
FETCH FIRST 10 ROWS ONLY;

-- 2. 分析执行计划
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR('sql_id'));

-- 3. 优化 SQL 或添加索引
```

### 4.2 I/O 瓶颈

**ADDM 发现**：
```
Finding: I/O Bottleneck
Impact is 3.8 active sessions, 30.8% of total activity.

Symptom:
  Wait class "User I/O" was consuming significant database time.

Recommendation:
  Consider using Automatic Segment Advisor to identify hot segments.
```

**解决步骤**：
```sql
-- 1. 查找 I/O 密集的表
SELECT 
    owner,
    segment_name,
    segment_type,
    physical_reads,
    buffer_busy_waits
FROM v$segment_statistics
WHERE statistic_name = 'physical reads'
ORDER BY physical_reads DESC
FETCH FIRST 10 ROWS ONLY;

-- 2. 检查索引使用情况
SELECT index_name, table_name, status
FROM user_indexes
WHERE table_name = 'HOT_TABLE';

-- 3. 考虑分区或索引优化
```

### 4.3 锁竞争

**ADDM 发现**：
```
Finding: Lock Contention
Impact is 2.1 active sessions, 17% of total activity.

Symptom:
  Wait event "enq: TX - row lock contention" was significant.

Recommendation:
  Investigate application logic causing lock contention.
```

**解决步骤**：
```sql
-- 1. 查找锁等待
SELECT 
    blocking_session,
    sid,
    serial#,
    wait_class,
    seconds_in_wait
FROM v$session
WHERE blocking_session IS NOT NULL;

-- 2. 查找热点表
SELECT 
    object_name,
    sum(waits) as total_waits
FROM v$rowcache
GROUP BY object_name
ORDER BY total_waits DESC;

-- 3. 优化事务逻辑
```

### 4.4 内存不足

**ADDM 发现**：
```
Finding: Memory Configuration
Impact is 1.5 active sessions, 12.2% of total activity.

Symptom:
  Buffer cache hit ratio is low.

Recommendation:
  Increase SGA target size or optimize SQL to reduce logical reads.
```

**解决步骤**：
```sql
-- 1. 检查当前内存配置
SHOW PARAMETER sga_target;
SHOW PARAMETER pga_aggregate_target;

-- 2. 查看内存建议
SELECT * FROM v$sga_target_advice;
SELECT * FROM v$pga_target_advice;

-- 3. 调整内存
ALTER SYSTEM SET sga_target = 8G SCOPE=BOTH;
```

## 5. DBMS_ADVISOR 包

### 5.1 手动运行 ADDM

```sql
-- 创建 ADDM 任务
DECLARE
    l_task_name VARCHAR2(100);
    l_task_desc VARCHAR2(200);
BEGIN
    l_task_name := 'MY_ADDM_TASK';
    l_task_desc := 'Manual ADDM Analysis';
    
    -- 创建任务
    DBMS_ADVISOR.CREATE_TASK(
        advisor_name => 'ADDM',
        task_name => l_task_name,
        task_desc => l_task_desc
    );
    
    -- 设置参数
    DBMS_ADVISOR.SET_TASK_PARAMETER(
        task_name => l_task_name,
        parameter => 'START_SNAPSHOT',
        value => 100
    );
    
    DBMS_ADVISOR.SET_TASK_PARAMETER(
        task_name => l_task_name,
        parameter => 'END_SNAPSHOT',
        value => 101
    );
    
    -- 执行任务
    DBMS_ADVISOR.EXECUTE_TASK(l_task_name);
END;
/

-- 获取报告
SELECT DBMS_ADVISOR.GET_TASK_REPORT('MY_ADDM_TASK') 
FROM dual;

-- 删除任务
EXEC DBMS_ADVISOR.DELETE_TASK('MY_ADDM_TASK');
```

### 5.2 其他顾问工具

```sql
-- SQL Access Advisor
DBMS_ADVISOR.QUICK_TUNE(
    advisor_name => DBMS_ADVISOR.SQLACCESS_ADVISOR,
    task_name => 'my_sql_access_task',
    attr1 => 'SELECT * FROM employees WHERE dept_id = 10'
);

-- Segment Advisor
DECLARE
    l_object_id NUMBER;
BEGIN
    DBMS_ADVISOR.CREATE_OBJECT(
        task_name => 'my_segment_task',
        object_type => 'TABLE',
        attr1 => 'HR',
        attr2 => 'EMPLOYEES',
        attr3 => NULL,
        attr4 => NULL,
        object_id => l_object_id
    );
    
    DBMS_ADVISOR.EXECUTE_TASK('my_segment_task');
END;
/
```

## 6. ADDM 最佳实践

### 6.1 使用建议

✅ **定期查看 ADDM 报告**（每天或每周）  
✅ **优先处理高影响的 Finding**  
✅ **结合 AWR 和 ASH 一起分析**  
✅ **记录优化前后的对比**  
✅ **建立性能基线**  

### 6.2 解读技巧

📌 **关注 Impact 值**：表示对 DB Time 的影响  
📌 **查看 Recommendation**：具体的优化建议  
📌 **检查 Rationale**：问题的根本原因  
📌 **分析 Affected SQL**：相关的 SQL 语句  
📌 **考虑业务场景**：某些"问题"可能是正常的  

### 6.3 自动化监控

```sql
-- 创建定期 ADDM 分析作业
BEGIN
    DBMS_SCHEDULER.CREATE_JOB(
        job_name => 'DAILY_ADDM_ANALYSIS',
        job_type => 'PLSQL_BLOCK',
        job_action => q'[
            DECLARE
                l_task_name VARCHAR2(100);
            BEGIN
                l_task_name := 'ADDM_' || TO_CHAR(SYSDATE, 'YYYYMMDD');
                
                DBMS_ADVISOR.CREATE_TASK(
                    advisor_name => 'ADDM',
                    task_name => l_task_name
                );
                
                DBMS_ADVISOR.EXECUTE_TASK(l_task_name);
            END;
        ]',
        start_date => SYSTIMESTAMP,
        repeat_interval => 'FREQ=DAILY; BYHOUR=2',
        enabled => TRUE
    );
END;
/
```

## 7. 常见问题

### 7.1 ADDM 未运行

**原因**：
- statistics_level = BASIC
- 许可证问题

**解决**：
```sql
ALTER SYSTEM SET statistics_level = TYPICAL;
```

### 7.2 报告为空

**原因**：
- 数据库空闲
- AWR 数据缺失

**解决**：
```sql
-- 检查 AWR 数据
SELECT COUNT(*) FROM dba_hist_snapshot;

-- 手动创建快照
EXEC DBMS_WORKLOAD_REPOSITORY.CREATE_SNAPSHOT();
```

### 7.3 建议不可行

**原因**：
- ADDM 不了解业务约束
- 环境限制

**解决**：
- 结合实际情况评估建议
- 咨询 DBA 团队
- 在测试环境验证

## 8. 总结

### 8.1 核心知识点

✅ **ADDM 原理**：基于 AWR 的智能诊断  
✅ **报告结构**：Findings、Recommendations、Rationale  
✅ **常见问题**：CPU、I/O、锁、内存  
✅ **DBMS_ADVISOR**：手动运行诊断  
✅ **最佳实践**：定期查看、优先处理高影响问题  

### 8.2 关键视图

📌 `dba_advisor_tasks` - 顾问任务  
📌 `dba_advisor_findings` - 诊断发现  
📌 `dba_advisor_recommendations` - 优化建议  
📌 `dba_hist_snapshot` - AWR 快照
