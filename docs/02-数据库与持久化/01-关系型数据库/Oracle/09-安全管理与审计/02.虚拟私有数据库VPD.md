## 1. VPD 概述

### 1.1 什么是 VPD？

VPD（Virtual Private Database，虚拟私有数据库），也称为 FGAC（Fine-Grained Access Control，细粒度访问控制）。

**功能**：
- ✅ 行级访问控制
- ✅ 列级访问控制
- ✅ 动态添加 WHERE 条件
- ✅ 对用户透明
- ✅ 防止数据泄露

### 1.2 VPD vs 传统权限

| 特性 | 传统权限 | VPD |
|------|---------|-----|
| **粒度** | 表级 | 行级/列级 |
| **控制方式** | GRANT/REVOKE | 动态谓词 |
| **灵活性** | 低 | 高 |
| **复杂性** | 简单 | 复杂 |
| **性能影响** | 无 | 有（较小） |

### 1.3 应用场景

📊 **多租户系统**：不同客户只能看到自己的数据  
📊 **部门隔离**：员工只能看到本部门数据  
📊 **敏感数据保护**：隐藏薪资等敏感信息  
📊 **审计要求**：满足合规性要求

## 2. VPD 工作原理

### 2.1 工作流程

```
用户查询：SELECT * FROM employees

    ↓

VPD 策略函数执行

    ↓

生成动态谓词：WHERE department_id = 10

    ↓

最终执行：SELECT * FROM employees WHERE department_id = 10
```

### 2.2 关键组件

| 组件 | 说明 |
|------|------|
| **策略函数** | PL/SQL 函数，返回谓词条件 |
| **策略** | 将函数绑定到表或视图 |
| **上下文** | 存储会话信息（如用户角色） |
| **谓词** | 动态生成的 WHERE 条件 |

## 3. 创建 VPD 策略

### 3.1 基本语法

```sql
DBMS_RLS.ADD_POLICY(
    object_schema     => 'HR',
    object_name       => 'EMPLOYEES',
    policy_name       => 'emp_dept_policy',
    function_schema   => 'SECURITY',
    policy_function   => 'get_dept_predicate',
    statement_types   => 'SELECT, INSERT, UPDATE, DELETE',
    update_check      => TRUE,
    enable            => TRUE,
    static_policy     => FALSE,
    policy_type       => DBMS_RLS.DYNAMIC,
    long_predicate    => FALSE,
    sec_relevant_cols => NULL,
    sec_relevant_cols_opt => NULL
);
```

### 3.2 创建策略函数

#### 示例 1：基于部门的行级控制

```sql
-- 创建策略函数
CREATE OR REPLACE FUNCTION security.get_dept_predicate(
    p_schema VARCHAR2,
    p_object VARCHAR2
) RETURN VARCHAR2 IS
    v_dept_id NUMBER;
BEGIN
    -- 获取当前用户的部门 ID
    SELECT dept_id INTO v_dept_id
    FROM hr.user_departments
    WHERE username = SYS_CONTEXT('USERENV', 'SESSION_USER');
    
    -- 返回谓词
    RETURN 'department_id = ' || v_dept_id;
EXCEPTION
    WHEN NO_DATA_FOUND THEN
        -- 如果没有部门信息，返回 1=0（无数据）
        RETURN '1=0';
END;
/
```

#### 示例 2：基于角色的控制

```sql
CREATE OR REPLACE FUNCTION security.get_role_predicate(
    p_schema VARCHAR2,
    p_object VARCHAR2
) RETURN VARCHAR2 IS
    v_role VARCHAR2(50);
BEGIN
    -- 获取用户角色
    v_role := SYS_CONTEXT('USERENV', 'SESSION_USER');
    
    -- DBA 可以看到所有数据
    IF v_role = 'DBA_USER' THEN
        RETURN NULL;  -- 无限制
    END IF;
    
    -- 经理可以看到本部门数据
    IF v_role LIKE 'MANAGER_%' THEN
        RETURN 'department_id = ' || SUBSTR(v_role, 9);
    END IF;
    
    -- 普通员工只能看到自己的数据
    RETURN 'employee_id = (SELECT employee_id FROM hr.users WHERE username = ''' || 
           v_role || ''')';
END;
/
```

#### 示例 3：使用应用上下文

```sql
-- 设置上下文
CREATE CONTEXT app_context USING security.set_context;

-- 上下文设置过程
CREATE OR REPLACE PROCEDURE security.set_context IS
    v_dept_id NUMBER;
BEGIN
    SELECT dept_id INTO v_dept_id
    FROM hr.user_departments
    WHERE username = SYS_CONTEXT('USERENV', 'SESSION_USER');
    
    DBMS_SESSION.SET_CONTEXT('app_context', 'dept_id', v_dept_id);
END;
/

-- 在登录触发器中调用
CREATE OR REPLACE TRIGGER set_app_context
AFTER LOGON ON DATABASE
BEGIN
    security.set_context;
END;
/

-- 策略函数使用上下文
CREATE OR REPLACE FUNCTION security.get_ctx_predicate(
    p_schema VARCHAR2,
    p_object VARCHAR2
) RETURN VARCHAR2 IS
BEGIN
    RETURN 'department_id = SYS_CONTEXT(''app_context'', ''dept_id'')';
END;
/
```

### 3.3 添加策略

```sql
-- 添加行级策略
BEGIN
    DBMS_RLS.ADD_POLICY(
        object_schema   => 'HR',
        object_name     => 'EMPLOYEES',
        policy_name     => 'emp_dept_policy',
        function_schema => 'SECURITY',
        policy_function => 'get_dept_predicate',
        statement_types => 'SELECT, INSERT, UPDATE, DELETE',
        update_check    => TRUE
    );
END;
/

-- 验证策略
SELECT * FROM dba_policies WHERE object_name = 'EMPLOYEES';
```

### 3.4 测试 VPD

```sql
-- 以不同用户登录测试
CONNECT manager_10/password@orcl

-- 结果：所有记录的 department_id = 20
```

## 4. 列级 VPD

### 4.1 隐藏敏感列

```sql
-- 创建列级策略函数
CREATE OR REPLACE FUNCTION security.hide_salary(
    p_schema VARCHAR2,
    p_object VARCHAR2
) RETURN VARCHAR2 IS
BEGIN
    -- 非 HR 用户看不到薪资
    IF SYS_CONTEXT('USERENV', 'SESSION_USER') NOT LIKE 'HR_%' THEN
        RETURN '1=0';  -- 隐藏列
    END IF;
    
    RETURN NULL;  -- HR 用户可以看
END;
/

-- 添加列级策略
BEGIN
    DBMS_RLS.ADD_POLICY(
        object_schema         => 'HR',
        object_name           => 'EMPLOYEES',
        policy_name           => 'salary_policy',
        function_schema       => 'SECURITY',
        policy_function       => 'hide_salary',
        sec_relevant_cols     => 'SALARY, COMMISSION_PCT',
        sec_relevant_cols_opt => DBMS_RLS.ALL_ROWS
    );
END;
/
```

**效果**：
- HR 用户：可以看到 salary 和 commission_pct
- 其他用户：这些列显示为 NULL

### 4.2 部分隐藏列

```sql
-- 脱敏显示：只显示前几位
CREATE OR REPLACE FUNCTION security.mask_phone(
    p_schema VARCHAR2,
    p_object VARCHAR2
) RETURN VARCHAR2 IS
BEGIN
    RETURN NULL;  -- 不阻止查询，但在视图中脱敏
END;
/

-- 或使用视图实现脱敏
CREATE VIEW hr.employees_safe AS
SELECT 
    employee_id,
    first_name,
    last_name,
    department_id,
    SUBSTR(phone_number, 1, 3) || '-***-' || SUBSTR(phone_number, -4) as phone_number,
    email
FROM hr.employees;

GRANT SELECT ON hr.employees_safe TO PUBLIC;
```

## 5. 策略类型

### 5.1 动态策略（默认）

每次查询都执行策略函数。

```sql
policy_type => DBMS_RLS.DYNAMIC
```

**适用**：谓词依赖会话状态

### 5.2 静态策略

策略函数只执行一次，结果缓存。

```sql
policy_type => DBMS_RLS.STATIC
static_policy => TRUE
```

**适用**：谓词对所有用户相同

### 5.3 共享静态策略

多个对象共享同一个策略。

```sql
policy_type => DBMS_RLS.SHARED_STATIC
```

### 5.4 上下文敏感策略

根据上下文变化。

```sql
policy_type => DBMS_RLS.CONTEXT_SENSITIVE
```

## 6. VPD 管理

### 6.1 查看策略

```sql
-- 查看所有 VPD 策略

    object_owner,
    object_name,
    policy_name,
    policy_function,
    sel, ins, upd, del,
    enable,
    static_policy

ORDER BY object_owner, object_name;

-- 查看特定表的策略
SELECT * FROM dba_policies 
WHERE object_name = 'EMPLOYEES';
```

### 6.2 启用/禁用策略

```sql
-- 禁用策略
BEGIN
    DBMS_RLS.ENABLE_POLICY(
        object_schema => 'HR',
        object_name   => 'EMPLOYEES',
        policy_name   => 'emp_dept_policy',
        enable        => FALSE
    );
END;
/

-- 启用策略
BEGIN
    DBMS_RLS.ENABLE_POLICY(
        object_schema => 'HR',
        object_name   => 'EMPLOYEES',
        policy_name   => 'emp_dept_policy',
        enable        => TRUE
    );
END;
/
```

### 6.3 删除策略

```sql
BEGIN
    DBMS_RLS.DROP_POLICY(
        object_schema => 'HR',
        object_name   => 'EMPLOYEES',
        policy_name   => 'emp_dept_policy'
    );
END;
/
```

### 6.4 刷新策略

```sql
-- 刷新共享策略缓存
BEGIN
    DBMS_RLS.REFRESH_POLICY(
        object_schema => 'HR',
        object_name   => 'EMPLOYEES',
        policy_name   => 'emp_dept_policy'
    );
END;
/
```

## 7. 性能优化

### 7.1 性能影响

⚠️ **每次查询都执行策略函数**  
⚠️ **复杂函数可能影响性能**  
⚠️ **多策略叠加会增加开销**  

### 7.2 优化建议

✅ **使用静态策略（如果可能）**
```sql
-- 静态策略只执行一次
static_policy => TRUE
```

✅ **简化策略函数**
```sql
-- ❌ 差：复杂查询
CREATE OR REPLACE FUNCTION complex_predicate(...) 
RETURN VARCHAR2 IS
BEGIN
    -- 多次查询、复杂逻辑
    RETURN predicate;
END;

-- ✅ 好：使用上下文
CREATE OR REPLACE FUNCTION simple_predicate(...) 
RETURN VARCHAR2 IS
BEGIN
    RETURN 'dept_id = SYS_CONTEXT(''app_ctx'', ''dept_id'')';
END;
```

✅ **使用绑定变量**
```sql
-- 避免硬解析
RETURN 'department_id = :dept_id';
```

✅ **缓存上下文值**
```sql
-- 在登录时设置，后续直接使用
DBMS_SESSION.SET_CONTEXT('app_ctx', 'dept_id', v_dept_id);
```

✅ **监控性能**
```sql
-- 查看策略执行统计
SELECT 
    policy_name,
    executions,
    cpu_time,
    elapsed_time
FROM v$sql
WHERE sql_text LIKE '%POLICY_FUNCTION%';
```

## 8. 实战案例

### 案例 1：多租户 SaaS 系统

**需求**：不同公司的用户只能看到自己公司的数据。

**实现**：

```sql
-- 1. 创建上下文
CREATE CONTEXT tenant_context USING security.set_tenant;

-- 2. 设置租户 ID
CREATE OR REPLACE PROCEDURE security.set_tenant IS
    v_tenant_id NUMBER;
BEGIN
    SELECT tenant_id INTO v_tenant_id
    FROM tenants
    WHERE username = SYS_CONTEXT('USERENV', 'SESSION_USER');
    
    DBMS_SESSION.SET_CONTEXT('tenant_context', 'tenant_id', v_tenant_id);
END;
/

-- 3. 登录触发器
CREATE OR REPLACE TRIGGER set_tenant_ctx
AFTER LOGON ON DATABASE
BEGIN
    security.set_tenant;
END;
/

-- 4. 策略函数
CREATE OR REPLACE FUNCTION security.tenant_predicate(
    p_schema VARCHAR2,
    p_object VARCHAR2
) RETURN VARCHAR2 IS
BEGIN
    RETURN 'tenant_id = SYS_CONTEXT(''tenant_context'', ''tenant_id'')';
END;
/

-- 5. 添加策略到所有业务表
BEGIN
    FOR t IN (SELECT table_name FROM user_tables WHERE table_name LIKE 'APP_%') LOOP
        DBMS_RLS.ADD_POLICY(
            object_schema   => USER,
            object_name     => t.table_name,
            policy_name     => 'tenant_policy',
            function_schema => 'SECURITY',
            policy_function => 'tenant_predicate'
        );
    END LOOP;
END;
/
```

### 案例 2：医疗系统隐私保护

**需求**：医生只能看到自己科室的患者，且敏感信息脱敏。

```sql
-- 行级控制：只看本科室患者
CREATE OR REPLACE FUNCTION security.dept_patient_policy(
    p_schema VARCHAR2,
    p_object VARCHAR2
) RETURN VARCHAR2 IS
BEGIN
    RETURN 'department_id = SYS_CONTEXT(''med_ctx'', ''dept_id'')';
END;
/

-- 列级控制：隐藏敏感信息
BEGIN
    DBMS_RLS.ADD_POLICY(
        object_schema         => 'MEDICAL',
        object_name           => 'PATIENTS',
        policy_name           => 'sensitive_data_policy',
        function_schema       => 'SECURITY',
        policy_function       => 'hide_sensitive',
        sec_relevant_cols     => 'SSN, DIAGNOSIS, TREATMENT',
        sec_relevant_cols_opt => DBMS_RLS.ALL_ROWS
    );
END;
/
```

## 9. 最佳实践

### 9.1 设计原则

✅ **最小权限原则**：只暴露必要的数据  
✅ **透明性**：对应用层透明，无需修改代码  
✅ **集中管理**：统一的安全策略管理  
✅ **审计跟踪**：记录访问日志  

### 9.2 实施建议

📌 **先设计后实施**：明确安全需求  
📌 **测试充分**：验证各种场景  
📌 **性能测试**：评估性能影响  
📌 **文档化**：记录策略逻辑  
📌 **定期审查**：更新策略以适应变化  

### 9.3 常见问题

⚠️ **递归查询问题**：策略函数中的查询可能再次触发策略  
⚠️ **权限问题**：确保策略函数有足够权限  
⚠️ **NULL 处理**：注意 NULL 值的谓词生成  
⚠️ **性能监控**：定期检查策略执行效率

## 10. 总结

### 10.1 核心知识点

✅ **VPD 原理**：动态添加 WHERE 条件  
✅ **策略函数**：返回谓词的 PL/SQL 函数  
✅ **行级控制**：过滤行数据  
✅ **列级控制**：隐藏敏感列  
✅ **上下文**：存储会话信息  

### 10.2 关键 API

📌 `DBMS_RLS.ADD_POLICY` - 添加策略  
📌 `DBMS_RLS.DROP_POLICY` - 删除策略  
📌 `DBMS_RLS.ENABLE_POLICY` - 启用/禁用  
📌 `SYS_CONTEXT` - 获取上下文  
📌 `DBMS_SESSION.SET_CONTEXT` - 设置上下文
