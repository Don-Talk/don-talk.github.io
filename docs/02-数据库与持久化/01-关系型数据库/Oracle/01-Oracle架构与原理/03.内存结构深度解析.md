## 1. Oracle 内存架构总览

<!-- TODO: 插入图片：Oracle 内存架构图 -->

Oracle 内存分为两大区域：

**SGA（System Global Area）**：
- 所有进程共享
- 实例启动时分配
- 包含多个子组件

**PGA（Program Global Area）**：
- 每个进程私有
- 进程创建时分配
- 用于排序、哈希等操作

```sql
-- 查看整体内存使用
SELECT 
    component,
    current_size/1024/1024 as current_mb,
    min_size/1024/1024 as min_mb,
    max_size/1024/1024 as max_mb
FROM v$sga_dynamic_components
ORDER BY current_size DESC;
```

## 2. SGA 自动管理

### 2.1 ASMM（Automatic Shared Memory Management）

从 Oracle 10g 引入，自动调整 SGA 各组件大小。

**启用 ASMM**：
```sql
-- 设置 SGA 目标大小
ALTER SYSTEM SET sga_target=4G SCOPE=BOTH;

-- SGA 最大大小（可选）
ALTER SYSTEM SET sga_max_size=6G SCOPE=SPFILE;

-- 确保以下参数为 0（让 Oracle 自动管理）
ALTER SYSTEM SET db_cache_size=0 SCOPE=BOTH;
ALTER SYSTEM SET shared_pool_size=0 SCOPE=BOTH;
ALTER SYSTEM SET large_pool_size=0 SCOPE=BOTH;
ALTER SYSTEM SET java_pool_size=0 SCOPE=BOTH;
ALTER SYSTEM SET streams_pool_size=0 SCOPE=BOTH;
```

**工作原理**：
- Oracle 监控各组件使用情况
- 根据负载动态调整大小
- 每 1 秒检查一次
- 在最小值和最大值之间调整

### 2.2 AMM（Automatic Memory Management）

从 Oracle 11g 引入，同时管理 SGA 和 PGA。

**启用 AMM**：
```sql
-- 设置总内存目标
ALTER SYSTEM SET memory_target=6G SCOPE=BOTH;

-- 设置最大内存（可选）
ALTER SYSTEM SET memory_max_target=8G SCOPE=SPFILE;

-- 确保以下参数为 0
ALTER SYSTEM SET sga_target=0 SCOPE=BOTH;
ALTER SYSTEM SET pga_aggregate_target=0 SCOPE=BOTH;
```

**注意**：
- Linux 系统需要配置 `/dev/shm` 足够大
- `memory_target` 不能超过 `/dev/shm` 大小

```bash
# 检查 /dev/shm 大小
df -h /dev/shm

# 如果需要增大
mount -o remount,size=8G /dev/shm
```

### 2.3 手动管理

传统方式，手动设置每个组件大小。

```sql
ALTER SYSTEM SET db_cache_size=2G SCOPE=BOTH;
ALTER SYSTEM SET shared_pool_size=1G SCOPE=BOTH;
ALTER SYSTEM SET large_pool_size=256M SCOPE=BOTH;
ALTER SYSTEM SET java_pool_size=128M SCOPE=BOTH;
```

**适用场景**：
- 需要精确控制内存分配
- 特殊应用需求
- 调试性能问题

### 2.4 内存管理方式对比

| 特性 | 手动管理 | ASMM | AMM |
|------|---------|------|-----|
| **复杂度** | 高 | 中 | 低 |
| **灵活性** | 低 | 中 | 高 |
| **适用版本** | 所有 | 10g+ | 11g+ |
| **推荐场景** | 特殊需求 | 生产环境 | 通用环境 |

## 3. Database Buffer Cache

### 3.1 作用和结构

Buffer Cache 是 SGA 中最大的组件，缓存从数据文件读取的数据块。

<!-- TODO: 插入图片：Buffer Cache 结构图 -->

**主要功能**：
- 减少物理 I/O
- 提高查询性能
- 支持多版本读一致性

**组织结构**：
- **Write List**：脏块链表
- **LRU List**：最近最少使用链表
- **LRUW List**：最近最少使用写链表

### 3.2 缓冲池类型

#### （1）Default Pool（默认池）
- 大多数对象使用
- 标准 LRU 算法

#### （2）Keep Pool（保持池）
- 缓存热点小表
- 避免被置换出去

```sql
-- 将表放入 Keep Pool
ALTER TABLE employees STORAGE (BUFFER_POOL KEEP);

-- 配置 Keep Pool 大小
ALTER SYSTEM SET db_keep_cache_size=256M SCOPE=BOTH;
```

#### （3）Recycle Pool（回收池）
- 缓存大表的一次性扫描
- 快速置换

```sql
-- 将表放入 Recycle Pool
ALTER TABLE large_table STORAGE (BUFFER_POOL RECYCLE);

-- 配置 Recycle Pool 大小
ALTER SYSTEM SET db_recycle_cache_size=128M SCOPE=BOTH;
```

### 3.3 缓冲区命中率

```sql
-- 计算 Buffer Cache 命中率
SELECT 
    name,
    value
FROM v$sysstat
WHERE name IN ('db block gets', 'consistent gets', 'physical reads');

-- 命中率公式
-- Hit Ratio = 1 - (physical reads / (db block gets + consistent gets))
```

**理想值**：> 90%

**如果命中率低**：
1. 增加 `db_cache_size`
2. 优化 SQL，减少全表扫描
3. 使用 Keep Pool 缓存热点表
4. 检查是否有大表污染缓存

### 3.4 监控 Buffer Cache

```sql
-- 查看 Buffer Cache 使用率
SELECT 
    component,
    current_size/1024/1024 as size_mb,
    oper_count,
    last_oper_type
FROM v$sga_dynamic_components
WHERE component = 'DEFAULT buffer cache';

-- 查看缓冲池统计
SELECT 
    name,
    block_size,
    buffers,
    bytes/1024/1024 as size_mb
FROM v$buffer_pool_statistics;

-- 查看等待事件
SELECT 
    event,
    total_waits,
    time_waited,
    average_wait
FROM v$system_event
WHERE event LIKE 'buffer busy waits%'
   OR event LIKE 'free buffer waits%';
```

### 3.5 常见问题

#### 问题：Buffer Busy Waits

**原因**：
- 多个会话同时访问同一块
- 热块竞争

**解决**：
```sql
-- 1. 查找热块
SELECT 
    p1 as file_id,
    p2 as block_id,
    COUNT(*) as waits
FROM v$active_session_history
WHERE event = 'buffer busy waits'
GROUP BY p1, p2
ORDER BY waits DESC
FETCH FIRST 10 ROWS ONLY;

-- 2. 找到对应的对象
SELECT 
    owner,
    segment_name,
    segment_type
FROM dba_extents
WHERE file_id = :file_id
AND :block_id BETWEEN block_id AND block_id + blocks - 1;

-- 3. 解决方案
-- - 使用 Hash Partitioning 分散热点
-- - 增加 PCTFREE 减少行迁移
-- - 优化 SQL 减少重复访问
```

## 4. Shared Pool

### 4.1 组成结构

Shared Pool 是 SGA 中最复杂的组件，包含：

<!-- TODO: 插入图片：Shared Pool 结构图 -->

1. **Library Cache**：SQL 和执行计划
2. **Dictionary Cache**：数据字典信息
3. **Result Cache**：查询结果缓存（可选）
4. **Server Result Cache**：PL/SQL 函数结果

### 4.2 Library Cache

缓存 SQL 语句、执行计划、PL/SQL 代码。

**关键概念**：
- **Parent Cursor**：SQL 文本和哈希值
- **Child Cursor**：具体的执行计划
- **Hard Parse**：硬解析，创建新游标
- **Soft Parse**：软解析，重用现有游标

```sql
-- 查看 Library Cache 命中率
SELECT 
    namespace,
    gets,
    gethits,
    ROUND(gethits/NULLIF(gets,0)*100, 2) as hit_ratio,
    pins,
    pinhits,
    ROUND(pinhits/NULLIF(pins,0)*100, 2) as pin_hit_ratio
FROM v$librarycache;
```

**理想值**：
- SQL Area 命中率 > 90%
- Pin Hit Ratio > 95%

### 4.3 Dictionary Cache

缓存数据字典信息（表定义、列信息、权限等）。

```sql
-- 查看 Dictionary Cache 命中率
SELECT 
    parameter,
    gets,
    getmisses,
    ROUND((1 - getmisses/NULLIF(gets,0))*100, 2) as hit_ratio
FROM v$rowcache
WHERE gets > 0
ORDER BY getmisses DESC;
```

**理想值**：> 90%

### 4.4 共享池碎片

**问题**：频繁硬解析导致碎片化

**诊断**：
```sql
-- 查看共享池空闲空间
SELECT 
    pool,
    name,
    bytes/1024/1024 as mb
FROM v$sgastat
WHERE pool = 'shared pool'
AND name = 'free memory';

-- 查看大型空闲 chunk
SELECT 
    ksmchcls as class,
    ksmchcom as comment,
    ksmchsiz as size_bytes
FROM x$ksmsp
WHERE ksmchcls = 'freeable'
ORDER BY ksmchsiz DESC
FETCH FIRST 20 ROWS ONLY;
```

**解决**：
```sql
-- 1. 刷新共享池（谨慎使用，生产环境慎用）
ALTER SYSTEM FLUSH SHARED_POOL;

-- 2. 使用绑定变量减少硬解析
-- 3. 保持 CURSOR_SHARING = EXACT（默认）
-- 4. 增加 shared_pool_size
```

### 4.5 监控 Shared Pool

```sql
-- Shared Pool 使用统计
SELECT 
    component,
    current_size/1024/1024 as size_mb,
    min_size/1024/1024 as min_mb,
    max_size/1024/1024 as max_mb
FROM v$sga_dynamic_components
WHERE component = 'shared pool';

-- 查看硬解析次数
SELECT 
    name,
    value
FROM v$sysstat
WHERE name IN ('parse count (total)', 'parse count (hard)');

-- 计算硬解析比例
SELECT 
    ROUND(
        (SELECT value FROM v$sysstat WHERE name = 'parse count (hard)') /
        NULLIF((SELECT value FROM v$sysstat WHERE name = 'parse count (total)'), 0) * 100,
        2
    ) as hard_parse_pct
FROM dual;
```

**理想值**：硬解析比例 < 10%

## 5. Redo Log Buffer

### 5.1 作用和特点

Redo Log Buffer 缓存重做日志条目，由 LGWR 定期写入磁盘。

**特点**：
- 固定大小，不能动态调整
- 循环使用
- 写入速度快

### 5.2 大小配置

```sql
-- 查看当前大小
SHOW PARAMETER log_buffer;

-- 修改大小（需要重启）
ALTER SYSTEM SET log_buffer=16777216 SCOPE=SPFILE;  -- 16MB
```

**建议大小**：
- 小型系统：4-8 MB
- 中型系统：8-16 MB
- 大型系统：16-32 MB

**注意**：过大的 Redo Log Buffer 不会显著提升性能

### 5.3 私有 Redo Strands

Oracle 12c+ 引入，减少 latch 竞争。

```sql
-- 查看是否启用
SHOW PARAMETER use_private_strands;

-- 在高并发系统中自动启用
```

## 9. PGA 管理

### 9.1 PGA 组成

<!-- TODO: 插入图片：PGA 结构图 -->

**Private SQL Area**：
- 绑定变量信息
- 游标状态

**Session Information**：
- 会话变量
- 权限信息

**Sort Area**：
- ORDER BY 排序
- GROUP BY 分组
- DISTINCT 去重
- 索引创建

**Hash Area**：
- Hash Join
- Hash Group By

**Bitmap Merge Area**：
- Bitmap 索引合并

### 9.2 PGA 自动管理

```sql
-- 启用 PGA 自动管理
ALTER SYSTEM SET pga_aggregate_target=2G SCOPE=BOTH;

-- 查看当前设置
SHOW PARAMETER pga_aggregate_target;
```

**工作原理**：
- Oracle 根据工作负载自动调整每个会话的工作区大小
- 总 PGA 使用量不超过目标值
- 单个会话最多使用 5% 的 PGA_TARGET

### 9.3 工作区大小

```sql
-- 查看工作区统计
SELECT 
    name,
    value
FROM v$pgastat;

-- 重要指标：
-- aggregate PGA target parameter
-- aggregate PGA auto target
-- total PGA inuse
-- total PGA freeable
-- maximum PGA allocated
-- cache hit percentage
```

**理想值**：Cache Hit Percentage > 90%

### 9.4 PGA 溢出

当 PGA 不足时，操作会溢出到临时表空间。

```sql
-- 查看溢出统计
SELECT 
    name,
    value
FROM v$pgastat
WHERE name LIKE '%extra%';

-- 查看临时表空间使用
SELECT 
    tablespace_name,
    bytes_used/1024/1024 as used_mb,
    bytes_free/1024/1024 as free_mb
FROM v$temp_space_header;
```

**如果溢出频繁**：
1. 增加 `pga_aggregate_target`
2. 优化 SQL，减少排序和哈希操作
3. 增加索引避免排序

### 9.5 监控 PGA

```sql
-- 查看 PGA 使用最多的会话
SELECT 
    s.sid,
    s.username,
    s.program,
    p.pga_used_mem/1024/1024 as used_mb,
    p.pga_alloc_mem/1024/1024 as alloc_mb,
    p.pga_max_mem/1024/1024 as max_mb
FROM v$session s
JOIN v$process p ON s.paddr = p.addr
ORDER BY p.pga_used_mem DESC
FETCH FIRST 20 ROWS ONLY;

-- 查看 PGA 顾问建议
SELECT 
    pga_target_for_estimate/1024/1024 as estimate_mb,
    estd_pga_cache_hit_percentage as hit_pct,
    estd_overalloc_count as overalloc
FROM v$pga_target_advice
ORDER BY pga_target_for_estimate;
```

## 10. 内存诊断视图

### 10.1 常用视图

| 视图 | 用途 |
|------|------|
| `v$sga` | SGA 总体信息 |
| `v$sgastat` | SGA 详细统计 |
| `v$sga_dynamic_components` | 动态组件大小 |
| `v$pgastat` | PGA 统计 |
| `v$memory_dynamic_components` | 内存组件（AMM） |
| `v$memory_target_advice` | AMM 顾问 |
| `v$sga_target_advice` | ASMM 顾问 |
| `v$pga_target_advice` | PGA 顾问 |

### 10.2 综合监控脚本

```sql
-- 内存使用概览
SELECT 
    'SGA Target' as component,
    TO_NUMBER(value)/1024/1024 as size_mb
FROM v$parameter
WHERE name = 'sga_target'
UNION ALL
SELECT 
    'PGA Target',
    TO_NUMBER(value)/1024/1024
FROM v$parameter
WHERE name = 'pga_aggregate_target'
UNION ALL
SELECT 
    'Memory Target',
    TO_NUMBER(value)/1024/1024
FROM v$parameter
WHERE name = 'memory_target';

-- SGA 各组件使用情况
SELECT 
    component,
    current_size/1024/1024 as current_mb,
    user_specified_size/1024/1024 as specified_mb,
    min_size/1024/1024 as min_mb,
    max_size/1024/1024 as max_mb
FROM v$sga_dynamic_components
ORDER BY current_size DESC;

-- 内存命中率汇总
SELECT 
    'Buffer Cache' as component,
    ROUND(
        (1 - phy.value / NULLIF((cur.value + con.value), 0)) * 100,
        2
    ) as hit_ratio
FROM v$sysstat cur, v$sysstat con, v$sysstat phy
WHERE cur.name = 'db block gets'
AND con.name = 'consistent gets'
AND phy.name = 'physical reads'
UNION ALL
SELECT 
    'Library Cache',
    ROUND(AVG(gethits/NULLIF(gets,0)*100), 2)
FROM v$librarycache
UNION ALL
SELECT 
    'Dictionary Cache',
    ROUND((1 - SUM(getmisses)/NULLIF(SUM(gets),0))*100, 2)
FROM v$rowcache;
```

## 11. 内存调优最佳实践

### 11.1 SGA 调优

✅ **Buffer Cache**：
- 命中率 > 90%
- 关注 `db file scattered read` 和 `db file sequential read`
- 使用 Keep Pool 缓存热点小表

✅ **Shared Pool**：
- 硬解析比例 < 10%
- 使用绑定变量
- 定期监控碎片

✅ **Redo Log Buffer**：
- 通常不需要调整
- 关注 `log file sync` 等待

### 11.2 PGA 调优

✅ **PGA Target**：
- Cache Hit Percentage > 90%
- 避免过多溢出到临时表空间
- 监控单个会话的 PGA 使用

✅ **排序优化**：
- 为 ORDER BY 列创建索引
- 减少 DISTINCT 和 GROUP BY
- 使用物化视图预聚合

### 11.3 内存参数建议

```sql
-- OLTP 系统
ALTER SYSTEM SET sga_target=4G;
ALTER SYSTEM SET pga_aggregate_target=1G;

-- 数据仓库系统
ALTER SYSTEM SET sga_target=8G;
ALTER SYSTEM SET pga_aggregate_target=4G;

-- 混合负载
ALTER SYSTEM SET memory_target=10G;
```

## 12. 常见问题排查

### 问题 1：ORA-04031（共享池内存不足）

**症状**：
- 无法分配共享池内存
- 硬解析失败

**解决**：
```sql
-- 1. 刷新共享池（临时方案）
ALTER SYSTEM FLUSH SHARED_POOL;

-- 2. 增加共享池大小
ALTER SYSTEM SET shared_pool_size=2G SCOPE=BOTH;

-- 3. 查找并优化大量硬解析的 SQL
SELECT 
    sql_id,
    executions,
    parse_calls,
    ROUND(parse_calls/NULLIF(executions,0)*100, 2) as parse_pct
FROM v$sqlarea
WHERE executions > 100
ORDER BY parse_pct DESC
FETCH FIRST 20 ROWS ONLY;
```

### 问题 2：ORA-04030（PGA 内存不足）

**症状**：
- 进程无法分配 PGA 内存
- 会话异常终止

**解决**：
```sql
-- 1. 增加 PGA 目标
ALTER SYSTEM SET pga_aggregate_target=4G SCOPE=BOTH;

-- 2. 限制单个进程的 PGA 使用
ALTER SYSTEM SET pga_aggregate_limit=8G SCOPE=BOTH;

-- 3. 查找 PGA 使用过多的会话
SELECT 
    s.sid,
    s.username,
    p.pga_used_mem/1024/1024 as used_mb
FROM v$session s
JOIN v$process p ON s.paddr = p.addr
ORDER BY p.pga_used_mem DESC
FETCH FIRST 10 ROWS ONLY;
```

### 问题 3：内存泄漏

**症状**：
- 内存使用持续增长
- 重启后恢复正常

**排查**：
```sql
-- 监控内存增长趋势
SELECT 
    snap_id,
    BEGIN_INTERVAL_TIME,
    END_INTERVAL_TIME,
    sgasize/1024/1024 as sga_mb,
    pgasize/1024/1024 as pga_mb
FROM dba_hist_sgastat, dba_hist_snapshot
WHERE snap_id = dba_hist_snapshot.snap_id
ORDER BY snap_id;
```

## 13. 总结

### 13.1 核心知识点

✅ **SGA 组件**：Buffer Cache、Shared Pool、Redo Log Buffer、Large Pool  
✅ **PGA 管理**：自动管理、工作区、溢出处理  
✅ **内存管理模式**：手动、ASMM、AMM  
✅ **缓冲池**：Default、Keep、Recycle  
✅ **监控视图**：v$sgastat、v$pgastat、v$sga_dynamic_components  

### 13.2 最佳实践

📌 优先使用 ASMM 或 AMM  
📌 监控关键命中率指标  
📌 使用绑定变量减少硬解析  
📌 合理配置 Keep/Recycle Pool  
📌 定期分析内存顾问建议  

### 13.3 性能调优要点

- **Buffer Cache**：关注命中率，优化全表扫描
- **Shared Pool**：减少硬解析，避免碎片
- **PGA**：避免溢出，优化排序操作
- **整体**：使用 AWR 报告识别瓶颈
