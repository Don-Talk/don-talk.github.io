## 1. 后台进程总览

Oracle 实例由内存结构和后台进程组成。后台进程是 Oracle 数据库的"工人"，负责各种维护和管理工作。

### 1.1 进程分类

**必需进程**（Instance 启动时必须存在）：
- PMON - Process Monitor
- SMON - System Monitor
- DBWn - Database Writer
- LGWR - Log Writer
- CKPT - Checkpoint

**可选进程**（根据配置和功能需求）：
- ARCn - Archiver
- RECO - Recoverer
- MMON/MMNL - Manageability Monitor
- CJQ0 - Job Queue Coordinator
- QMNC - AQ Coordinator
- LREG - Listener Registration

## 2. DBWR（Database Writer）- 数据库写入进程

### 2.1 职责和作用

DBWR 是 Oracle 最重要的后台进程之一，负责将修改过的数据块（脏块）从 Database Buffer Cache 写入到数据文件。

**主要职责**：
1. 管理 Buffer Cache 中的脏数据块
2. 执行检查点时的批量写入
3. 释放 Buffer Cache 空间供新数据使用
4. 确保数据的一致性和持久性

<!-- TODO: 插入图片：DBWR 工作流程图 -->

### 2.2 触发写入的条件

DBWR 在以下情况下会被触发：

1. **Checkpoint 发生**
   - 完全检查点
   - 增量检查点
   - 日志切换触发的检查点

2. **Buffer Cache 压力**
   - 空闲缓冲区低于阈值（通常 25%）
   - Server Process 找不到干净缓冲区

3. **超时**
   - 每 3 秒唤醒一次检查

4. **表空间 Offline**
   - 表空间设置为 OFFLINE 时

5. **Drop/Truncate 表**
   - 删除或截断表时

6. **Begin Backup**
   - 开始热备份时

### 2.3 多 DBWR 进程

从 Oracle 8i 开始，可以配置多个 DBWR 进程（DBW0 ~ DBWJ，最多 36 个）。

```sql
-- 查看当前 DBWR 进程数量
SHOW PARAMETER db_writer_processes;

-- 查看 IO  slaves（如果启用异步 IO）
SELECT name, value 
FROM v$parameter 
WHERE name LIKE '%io_slaves%';
```

**配置建议**：

| CPU 核心数 | 推荐 DBWR 数量 |
|-----------|--------------|
| 1-8       | 1            |
| 9-16      | 2            |
| 17-32     | 4            |
| 33-64     | 8            |
| > 64      | 根据负载测试调整 |

```sql
-- 修改 DBWR 进程数（需要重启）
ALTER SYSTEM SET db_writer_processes=4 SCOPE=SPFILE;
```

### 2.4 DBWR 写入算法

DBWR 使用多种算法优化写入效率：

#### （1）LRU 链扫描
- 扫描 LRU（Least Recently Used）链
- 找到脏块并批量写入

#### （2）增量检查点队列
- 维护一个低 RBA（Redo Byte Address）队列
- 优先写入最旧的脏块

#### （3）批量写入
- 使用 scatter I/O 一次性写入多个块
- 减少 I/O 次数，提高吞吐量

#### （4）延迟写入
- 不立即写入，积累一批后批量写入
- 减少随机 I/O，提高性能

### 2.6 常见问题和调优

#### 问题 1：Free Buffer Waits

**症状**：
- 会话等待 `free buffer waits` 事件
- 应用响应变慢

**原因**：
- DBWR 写入速度跟不上
- Buffer Cache 太小
- I/O 子系统瓶颈

**解决方案**：
```sql
-- 1. 增加 DBWR 进程
ALTER SYSTEM SET db_writer_processes=4 SCOPE=SPFILE;

-- 2. 增大 Buffer Cache
ALTER SYSTEM SET db_cache_size=4G;

-- 3. 检查 I/O 性能
SELECT * FROM v$iostat_function;

-- 4. 使用异步 I/O
SHOW PARAMETER disk_asynch_io;
```

#### 问题 2：Checkpoint 不完整

**症状**：
- 实例恢复时间过长
- alert.log 中出现警告

**解决方案**：
```sql
-- 调整快速启动 MTTR 目标
ALTER SYSTEM SET fast_start_mttr_target=300;  -- 5分钟

-- 强制检查点
ALTER SYSTEM CHECKPOINT GLOBAL;
```

## 3. LGWR（Log Writer）- 日志写入进程

### 3.1 职责和作用

LGWR 负责将 Redo Log Buffer 中的重做日志条目写入到联机重做日志文件。这是保证事务持久性的关键进程。

**主要职责**：
1. 将 redo entries 从内存写入磁盘
2. 管理日志文件的切换
3. 协调归档进程（ARCn）
4. 通知用户进程提交完成

<!-- TODO: 插入图片：LGWR 工作流程图 -->

### 3.2 触发写入的条件

LGWR 在以下情况下写入日志：

1. **用户提交事务（COMMIT）**
   - 最重要的触发条件
   - 保证事务持久性

2. **Redo Log Buffer 达到 1/3 满**
   - 防止缓冲区溢出

3. **每秒定时写入**
   - 即使没有提交，也每秒写入一次

4. **DBWR 写入前**
   - 保证先写日志再写数据（Write-Ahead Logging）

5. **Redo Log Buffer 有 1MB 未写入数据**

### 3.3 LGWR 工作模式

#### 传统模式（单 LGWR）
- 单个 LGWR 进程处理所有写入
- 适用于低并发环境

#### 现代模式（LGWR + Worker Threads）
- Oracle 12c+ 引入
- 主 LGWR 协调多个 worker 线程
- 并行写入多个日志文件
- 提高高并发环境性能

```sql
-- 查看日志写入器配置
SELECT 
    name,
    value
FROM v$parameter
WHERE name LIKE 'use_large_pages%'
   OR name LIKE 'max_log%';
```

### 3.4 日志写入流程

```
用户进程 COMMIT
    ↓
生成 redo entry 放入 Redo Log Buffer
    ↓
通知 LGWR
    ↓
LGWR 获取 redo copy latch
    ↓
将 redo entries 写入所有日志组成员
    ↓
等待所有写入完成
    ↓
释放 latch，返回成功给用户
```

**关键点**：
- LGWR 必须等待所有日志组成员写入完成
- 这就是为什么建议日志组成员放在不同磁盘
- 写入是最慢的成员决定整体性能

### 3.5 监控 LGWR

```sql
-- LGWR 统计信息
SELECT 
    name,
    value
FROM v$sysstat
WHERE name LIKE 'redo%'
ORDER BY name;

-- 重要指标：
-- redo size                    : 生成的 redo 总量
-- redo wastage                 : 浪费的 redo 空间
-- redo writes                  : 写入次数
-- redo blocks written          : 写入的块数
-- redo write time              : 写入总时间（厘秒）
```

```sql
-- 计算平均写入时间
SELECT 
    ROUND(
        (SELECT value FROM v$sysstat WHERE name = 'redo write time') /
        NULLIF((SELECT value FROM v$sysstat WHERE name = 'redo writes'), 0),
        2
    ) as avg_write_time_ms
FROM dual;
```

```sql
-- 日志切换频率
SELECT 
    TO_CHAR(first_time, 'YYYY-MM-DD HH24:MI') as time,
    COUNT(*) as switches
FROM v$log_history
WHERE first_time > SYSDATE - 1
GROUP BY TO_CHAR(first_time, 'YYYY-MM-DD HH24:MI')
ORDER BY time;
```

```sql
-- 等待事件
SELECT 
    event,
    total_waits,
    time_waited,
    average_wait
FROM v$system_event
WHERE event LIKE 'log file%';

-- 常见等待事件：
-- log file sync          : 等待 LGWR 确认写入
-- log file parallel write: LGWR 等待 I/O 完成
```

### 3.6 日志切换优化

**理想切换频率**：每 15-20 分钟切换一次

```sql
-- 查看当前日志文件大小和切换频率
SELECT 
    l.group#,
    l.bytes/1024/1024 as size_mb,
    l.members,
    l.status,
    lf.member
FROM v$log l
JOIN v$logfile lf ON l.group# = lf.group#
ORDER BY l.group#;
```

**调整日志文件大小**：

```sql
-- 添加新的更大的日志组
ALTER DATABASE ADD LOGFILE GROUP 4 
('/u01/app/oracle/oradata/ORCL/redo04.log') 
SIZE 500M;

ALTER DATABASE ADD LOGFILE GROUP 5 
('/u01/app/oracle/oradata/ORCL/redo05.log') 
SIZE 500M;

-- 切换日志直到要删除的组变为 INACTIVE
ALTER SYSTEM SWITCH LOGFILE;

-- 删除旧的小日志组
ALTER DATABASE DROP LOGFILE GROUP 1;
ALTER DATABASE DROP LOGFILE GROUP 2;
ALTER DATABASE DROP LOGFILE GROUP 3;
```

### 3.7 常见问题和调优

#### 问题 1：Log File Sync 等待过高

**症状**：
- 应用响应慢，特别是 COMMIT 操作
- `log file sync` 等待事件占比高

**原因**：
- I/O 子系统慢
- 日志文件布局不合理
- 频繁的 COMMIT

**解决方案**：
```sql
-- 1. 将日志文件放到快速存储（SSD）
-- 2. 确保日志组成员在不同磁盘
-- 3. 减少不必要的 COMMIT（批量提交）
-- 4. 使用 COMMIT_WRITE 参数
ALTER SESSION SET COMMIT_WRITE = BATCH,NOWAIT;

-- 5. 检查 I/O 延迟
SELECT 
    name,
    singleblkrdtim / NULLIF(singleblkrds, 0) as avg_read_time
FROM v$filestat;
```

#### 问题 2：日志切换过于频繁

**症状**：
- 每小时切换超过 10 次
- Checkpoint 频繁发生
- 性能波动

**解决方案**：
- 增大日志文件大小
- 增加日志组数量

## 4. CKPT（Checkpoint）- 检查点进程

### 4.1 职责和作用

CKPT 负责更新控制文件和数据文件头，记录检查点位置。它不直接写入数据块，而是协调 DBWR 的工作。

**主要职责**：
1. 更新控制文件中的检查点信息
2. 更新数据文件头的检查点 SCN
3. 通知 DBWR 执行写入
4. 维护检查点队列

<!-- TODO: 插入图片：Checkpoint 流程图 -->

### 4.2 检查点类型

#### （1）完全检查点（Full Checkpoint）
- 将所有脏块写入磁盘
- 更新所有文件头的 SCN
- 通常在关闭数据库时发生

```sql
-- 手动触发完全检查点
ALTER SYSTEM CHECKPOINT GLOBAL;
```

#### （2）增量检查点（Incremental Checkpoint）
- 只记录低 RBA 位置
- 不立即写入所有脏块
- 持续进行，减少恢复时间

#### （3）线程检查点（Thread Checkpoint）
- 针对特定实例（RAC 环境）
- 日志切换时触发

#### （4）对象检查点（Object Checkpoint）
- 针对特定对象（表、索引）
- DROP/TRUNCATE 时触发

#### （5）并行查询检查点
- 并行操作完成后触发

### 4.3 FAST_START_MTTR_TARGET

这个参数控制实例恢复的目标时间。

```sql
-- 查看当前设置
SHOW PARAMETER fast_start_mttr_target;

-- 设置目标恢复时间为 5 分钟（300 秒）
ALTER SYSTEM SET fast_start_mttr_target=300;

-- 禁用自动检查点调优
ALTER SYSTEM SET fast_start_mttr_target=0;
```

**权衡**：
- 值越小 → 检查点越频繁 → I/O 开销大，但恢复快
- 值越大 → 检查点越少 → I/O 开销小，但恢复慢

### 4.4 监控检查点


## 5. SMON（System Monitor）- 系统监控进程

### 5.1 职责和作用

SMON 是 Oracle 的"清洁工"和"医生"，负责实例恢复和各种清理工作。

**主要职责**：
1. **实例恢复**（Instance Recovery）
   - 前滚（Roll Forward）：应用 redo
   - 回滚（Roll Back）：撤销未提交事务
   
2. **清理临时段**
   - 释放排序操作使用的临时空间
   
3. **合并空闲空间**
   - 合并字典管理表空间中的相邻空闲区
   
4. **恢复失败的事务**
   - 清理标记为失败的事务
   
5. **监控其他进程**
   - 检测并报告异常

<!-- TODO: 插入图片：SMON 恢复流程图 -->

### 5.2 实例恢复流程

当实例崩溃后重新启动时，SMON 自动执行恢复：

#### 阶段 1：前滚（Cache Recovery）
```
读取 redo log
    ↓
应用所有已提交的更改
    ↓
包括已写入和未写入数据文件的更改
    ↓
恢复到崩溃前的状态
```

#### 阶段 2：回滚（Transaction Recovery）
```
识别未提交的事务
    ↓
使用 undo 段回滚这些事务
    ↓
保证数据一致性
```

```sql
-- 查看恢复进度
SELECT 
    undotsn,
    usn,
    slt,
    seq,
    state,
    cputime,
    elapsed_seconds
FROM v$fast_start_transactions;

-- 查看恢复统计
SELECT 
    recovery_estimated_ios,
    actual_redo_blks,
    estimated_mttr
FROM v$instance_recovery;
```

### 5.3 监控 SMON

```sql

    name,
    value

WHERE name LIKE 'SMON%';

-- 常见统计项：
```

### 5.4 SMON 相关问题

#### 问题：SMON 清理临时段导致性能下降

**症状**：
- 定期出现性能抖动
- SMON 进程 CPU 使用率高

**解决方案**：
```sql
-- 1. 使用本地管理表空间（避免空间合并）
CREATE TABLESPACE temp_ts
TEMPFILE '/u01/temp01.dbf' SIZE 1G
EXTENT MANAGEMENT LOCAL UNIFORM SIZE 1M;

-- 2. 调整临时表空间大小，避免频繁扩展
ALTER DATABASE TEMPFILE '/u01/temp01.dbf' RESIZE 2G;

-- 3. 监控临时表空间使用
SELECT 
    tablespace_name,
    bytes_used/1024/1024 as used_mb,
    bytes_free/1024/1024 as free_mb
FROM v$temp_space_header;
```

## 6. PMON（Process Monitor）- 进程监控进程

### 6.1 职责和作用

PMON 是 Oracle 的"警察"，负责监控和清理失败的进程。

**主要职责**：
1. **检测失败的进程**
   - 定期轮询所有服务器进程
   
2. **清理资源**
   - 释放锁
   - 释放内存
   - 清理游标
   
3. **服务注册**
   - 向监听器注册实例和服务
   - 每 60 秒发送心跳
   
4. **重启 Dispatcher**（共享服务器模式）

<!-- TODO: 插入图片：PMON 清理流程图 -->

### 6.2 进程清理流程

```
检测到进程失败
    ↓
回滚未提交的事务
    ↓
释放行锁和表锁
    ↓
释放 PGA 内存
    ↓
清理游标状态
    ↓
更新数据字典
```

### 6.3 服务注册

#### 动态注册（默认）
- PMON 自动向监听器注册
- 每 60 秒发送心跳
- 监听器配置文件不需要静态配置

```sql
-- 查看服务注册状态
SELECT 
    name,
    network_name,
    status
FROM v$active_services;

-- 手动注册
ALTER SYSTEM REGISTER;
```

#### 静态注册
- 在 listener.ora 中配置
- 用于远程启动数据库
- 用于 Data Guard 环境

```
# listener.ora
SID_LIST_LISTENER =
  (SID_LIST =
    (SID_DESC =
      (GLOBAL_DBNAME = ORCL)
      (ORACLE_HOME = /u01/app/oracle/product/19.0.0/dbhome_1)
      (SID_NAME = ORCL)
    )
  )
```

### 6.4 监控 PMON

```sql
-- PMON 统计

    name,
    value
FROM v$sysstat
WHERE name LIKE 'PMON%';

-- 查看服务注册
```

### 6.5 PMON 相关问题

#### 问题：PMON 无法注册到监听器

**症状**：
- TNS-12514: listener does not currently know of service
- 客户端无法连接

**排查步骤**：
```bash
# 1. 检查监听器状态
lsnrctl status

# 2. 检查 local_listener 参数
SHOW PARAMETER local_listener;

# 3. 手动注册
ALTER SYSTEM REGISTER;

# 4. 检查监听器日志
tail -f $ORACLE_BASE/diag/tnslsnr/hostname/listener/alert/log.xml
```

**解决方案**：
```sql
-- 如果非标准端口，需要配置 local_listener
ALTER SYSTEM SET local_listener='(ADDRESS=(PROTOCOL=TCP)(HOST=localhost)(PORT=1522))';
ALTER SYSTEM REGISTER;
```

## 7. 可选进程详解

### 7.1 ARCn（Archiver）- 归档进程

**职责**：
- 在日志切换后复制联机日志到归档位置
- 只在 ARCHIVELOG 模式下运行

```sql
-- 查看归档模式
ARCHIVE LOG LIST;

-- 查看归档进程
SELECT * FROM v$archive_processes;

-- 查看归档目标
SHOW PARAMETER log_archive_dest;
```

**配置多个归档进程**：
```sql
ALTER SYSTEM SET log_archive_max_processes=4;
```

### 7.2 RECO（Recoverer）- 分布式事务恢复

**职责**：
- 自动解决失败的分布式事务
- 在分布式数据库中同步事务状态

```sql
-- 查看待处理的分布式事务
SELECT * FROM DBA_2PC_PENDING;
```

### 7.3 MMON/MMNL（Manageability Monitor）

**职责**：
- MMON：采集 AWR 快照（每小时）
- MMNL：协助 MMON，处理轻量级任务

```sql
-- 查看 AWR 快照
SELECT 
    snap_id,
    begin_interval_time,
    end_interval_time
FROM dba_hist_snapshot
ORDER BY snap_id DESC
FETCH FIRST 10 ROWS ONLY;

-- 手动创建快照
EXEC DBMS_WORKLOAD_REPOSITORY.CREATE_SNAPSHOT;
```

### 7.4 CJQ0（Job Queue Coordinator）

**职责**：
- 管理作业队列
- 启动 Jnnn 进程执行 DBMS_JOB 和 DBMS_SCHEDULER

```sql
-- 查看作业队列进程
SHOW PARAMETER job_queue_processes;

-- 查看调度作业
SELECT 
    job_name,
    state,
    next_run_date
FROM dba_scheduler_jobs;
```

## 8. 进程监控实战

### 8.1 综合监控脚本

```sql
-- 后台进程健康检查
SELECT 
    p.name,
    p.description,
    CASE WHEN p.paddr != '00' THEN 'RUNNING' ELSE 'STOPPED' END as status,
    s.sid,
    s.serial#,
    sp.spid as os_pid
FROM v$bgprocess p
LEFT JOIN v$session s ON p.paddr = s.paddr
LEFT JOIN v$process sp ON p.paddr = sp.addr
WHERE p.paddr != '00'
ORDER BY p.name;
```

### 8.2 等待事件分析

```sql
-- 后台进程的等待事件
SELECT 
    p.name,
    sw.event,
    sw.wait_time,
    sw.seconds_in_wait,
    sw.state
FROM v$bgprocess p
JOIN v$session s ON p.paddr = s.paddr
JOIN v$session_wait sw ON s.sid = sw.sid
WHERE p.paddr != '00'
AND sw.wait_time > 0;
```

### 8.3 进程资源使用

```sql
-- 后台进程的 CPU 和内存使用
SELECT 
    p.name,
    sp.spid,
    sp.pga_used_mem/1024/1024 as pga_used_mb,
    sp.pga_alloc_mem/1024/1024 as pga_alloc_mb
FROM v$bgprocess p
JOIN v$process sp ON p.paddr = sp.addr
WHERE p.paddr != '00'
ORDER BY sp.pga_used_mem DESC;
```

## 9. 故障排查案例

### 案例 1：DBWR 写入缓慢

**现象**：
- 系统整体变慢
- 大量 `free buffer waits`

**排查**：
```sql
-- 1. 检查 DBWR 写入速度
SELECT name, value 
FROM v$sysstat 
WHERE name LIKE 'DBWR%write%';

-- 2. 检查 I/O 延迟
SELECT 
    file#,
    avg_read_time,
    avg_write_time
FROM v$filestat;

-- 3. 检查 Buffer Cache 命中率
SELECT 
    1 - (phy.value / (cur.value + con.value)) as hit_ratio
FROM v$sysstat cur, v$sysstat con, v$sysstat phy
WHERE cur.name = 'db block gets'
AND con.name = 'consistent gets'
AND phy.name = 'physical reads';
```

**解决**：
- 增加 DBWR 进程数
- 优化 I/O 子系统
- 调整 Buffer Cache 大小

### 案例 2：LGWR 成为瓶颈

**现象**：
- COMMIT 响应慢
- `log file sync` 等待高

**排查**：
```sql
-- 1. 检查日志写入时间
SELECT 
    ROUND(
        (SELECT value FROM v$sysstat WHERE name = 'redo write time') /
        NULLIF((SELECT value FROM v$sysstat WHERE name = 'redo writes'), 0),
        2
    ) as avg_write_ms
FROM dual;

-- 2. 检查日志切换频率
SELECT 
    TO_CHAR(first_time, 'YYYY-MM-DD HH24') as hour,
    COUNT(*) as switches
FROM v$log_history
WHERE first_time > SYSDATE - 7
GROUP BY TO_CHAR(first_time, 'YYYY-MM-DD HH24')
ORDER BY hour;
```

**解决**：
- 日志文件迁移到 SSD
- 增大日志文件
- 减少不必要的 COMMIT

## 10. 总结

### 10.1 核心知识点

✅ **DBWR**：脏块写入，检查点触发，批量 I/O  
✅ **LGWR**：日志写入，COMMIT 触发，先写日志  
✅ **CKPT**：更新检查点信息，协调 DBWR  
✅ **SMON**：实例恢复，清理临时段  
✅ **PMON**：进程清理，服务注册  
✅ **可选进程**：ARCn、RECO、MMON、CJQ0  

### 10.2 最佳实践

📌 合理配置 DBWR 进程数（根据 CPU）  
📌 日志文件放在快速存储，多路复用  
📌 设置合理的 FAST_START_MTTR_TARGET  
📌 监控关键等待事件  
📌 定期检查后台进程健康状态  

### 10.3 性能调优要点

- **DBWR 优化**：关注 `free buffer waits`
- **LGWR 优化**：关注 `log file sync`
- **CKPT 优化**：平衡检查点频率和恢复时间
- **整体优化**：使用 AWR 报告识别瓶颈
