### 02. Bitmap 索引适用场景

**核心内容**：
- Bitmap 索引结构
- 适用场景（低基数列）
- 位图连接索引
- Bitmap 索引的优缺点
- 并发问题

**关键知识点**：
```sql
-- 创建 Bitmap 索引
CREATE BITMAP INDEX idx_gender ON employees(gender);

-- 适用场景
-- ✅ 性别、状态、类型等低基数列
-- ❌ 高基数列（如 ID、时间戳）

-- 位图连接索引
CREATE BITMAP INDEX idx_emp_dept 
ON employees(departments.dept_name)
FROM employees, departments
WHERE employees.dept_id = departments.dept_id;
```

### 03. 函数索引与反向索引

**核心内容**：
- 函数索引原理
- 常用函数索引场景
- 反向索引（Reverse Key Index）
- RAC 环境中的反向索引

**关键知识点**：
```sql
-- 函数索引
CREATE INDEX idx_emp_upper_name 
ON employees(UPPER(name));

-- 使用函数索引
SELECT * FROM employees 
WHERE UPPER(name) = 'JOHN';

-- 反向索引（适合序列主键）
CREATE INDEX idx_reverse ON orders(order_id) REVERSE;

-- 日期函数索引
CREATE INDEX idx_order_date_trunc 
ON orders(TRUNC(order_date));
```

### 04. 复合索引与覆盖索引

**核心内容**：
- 复合索引设计原则
- 最左前缀原则
- 覆盖索引（Covering Index）
- 索引跳跃扫描（Index Skip Scan）
- 索引压缩

**关键知识点**：
```sql
-- 复合索引（注意列顺序）
CREATE INDEX idx_emp_dept_sal 
ON employees(dept_id, salary DESC, hire_date);

-- 最左前缀原则
-- ✅ 可以使用索引
WHERE dept_id = 10
WHERE dept_id = 10 AND salary > 5000
WHERE dept_id = 10 AND salary > 5000 AND hire_date > SYSDATE-30

-- ❌ 不能使用索引
WHERE salary > 5000
WHERE hire_date > SYSDATE-30

-- 覆盖索引
CREATE INDEX idx_emp_covering 
ON employees(emp_id, name, email);

-- 查询只从索引获取数据（无需访问表）
SELECT emp_id, name, email FROM employees WHERE emp_id = 100;
```

### 05. 索引失效场景全解析

**核心内容**：
- 隐式类型转换
- 函数导致索引失效
- LIKE 左模糊
- OR 条件陷阱
- NOT 条件
- 计算表达式
- NULL 值处理

**关键知识点**：
```sql
-- ❌ 隐式类型转换导致索引失效
SELECT * FROM employees WHERE employee_id = '100';

-- ✅ 显式转换
SELECT * FROM employees WHERE employee_id = TO_NUMBER('100');

-- ❌ 函数导致索引失效
SELECT * FROM employees WHERE UPPER(name) = 'JOHN';

-- ✅ 使用函数索引或改写
SELECT * FROM employees WHERE name LIKE 'John%';

-- ❌ LIKE 左模糊
SELECT * FROM employees WHERE name LIKE '%ohn';

-- ✅ LIKE 右模糊
SELECT * FROM employees WHERE name LIKE 'Joh%';

-- ❌ 计算表达式
SELECT * FROM orders WHERE amount * 1.1 > 1000;

-- ✅ 改写计算
SELECT * FROM orders WHERE amount > 1000 / 1.1;

-- ❌ OR 条件（部分列无索引）
SELECT * FROM employees WHERE emp_id = 100 OR email = 'test@test.com';

-- ✅ 使用 UNION ALL
SELECT * FROM employees WHERE emp_id = 100
UNION ALL
SELECT * FROM employees WHERE email = 'test@test.com' AND emp_id != 100;
```

### 06. 执行计划解读

**核心内容**：
- EXPLAIN PLAN 使用
- DBMS_XPLAN 包
- AUTOTRACE
- 访问路径（Access Path）
- 连接方法（Join Method）
- 成本估算

**关键知识点**：
```sql
-- 方法 1：EXPLAIN PLAN
EXPLAIN PLAN FOR
SELECT e.name, d.department_name
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id
WHERE e.salary > 5000;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 方法 2：AUTOTRACE
SET AUTOTRACE ON EXPLAIN;
SELECT * FROM employees WHERE dept_id = 10;
SET AUTOTRACE OFF;

-- 方法 3：实时执行计划
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR(NULL, NULL, 'ALL'));

-- 方法 4：AWR 中的执行计划
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_AWR('sql_id'));
```

**执行计划关键元素**：
- **Operation**：操作类型（TABLE ACCESS、INDEX SCAN 等）
- **Cost**：成本估算
- **Cardinality**：预计返回行数
- **Bytes**：预计数据量

**常见访问路径**：
- `TABLE ACCESS FULL`：全表扫描
- `INDEX UNIQUE SCAN`：唯一索引扫描
- `INDEX RANGE SCAN`：索引范围扫描
- `INDEX FULL SCAN`：索引全扫描
- `INDEX FAST FULL SCAN`：索引快速全扫描

**常见连接方法**：
- `NESTED LOOPS`：嵌套循环（小结果集）
- `HASH JOIN`：哈希连接（大结果集）
- `MERGE JOIN`：排序合并（已排序数据）

### 07. Hint 提示完全指南

**核心内容**：
- Hint 语法和作用域
- 访问路径 Hint
- 连接方法 Hint
- 连接顺序 Hint
- 并行执行 Hint
- 优化器模式 Hint

**关键知识点**：
```sql
-- 强制使用索引
SELECT /*+ INDEX(e idx_emp_salary) */ *
FROM employees e
WHERE salary > 5000;

-- 强制全表扫描
SELECT /*+ FULL(e) */ *
FROM employees e
WHERE dept_id = 10;

-- 强制连接方法
SELECT /*+ USE_NL(e d) */ *  -- Nested Loops
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id;

SELECT /*+ USE_HASH(e d) */ *  -- Hash Join
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id;

SELECT /*+ USE_MERGE(e d) */ *  -- Merge Join
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id;

-- 强制连接顺序
SELECT /*+ ORDERED */ *
FROM employees e, departments d, locations l
WHERE e.dept_id = d.dept_id
AND d.location_id = l.location_id;

-- 指定驱动表
SELECT /*+ LEADING(d e) */ *
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id;

-- 并行查询
SELECT /*+ PARALLEL(4) */ *
FROM large_table;

-- 优化器模式
SELECT /*+ ALL_ROWS */ * FROM employees;  -- 吞吐量优先
SELECT /*+ FIRST_ROWS(10) */ * FROM employees;  -- 响应时间优先
```

**常用 Hint 分类**：

| 类别 | Hint | 作用 |
|------|------|------|
| **访问路径** | `INDEX(table index)` | 使用指定索引 |
| | `FULL(table)` | 全表扫描 |
| | `INDEX_FFS(table index)` | 索引快速全扫描 |
| **连接方法** | `USE_NL(t1 t2)` | Nested Loops |
| | `USE_HASH(t1 t2)` | Hash Join |
| | `USE_MERGE(t1 t2)` | Merge Join |
| **连接顺序** | `ORDERED` | 按 FROM 顺序 |
| | `LEADING(t1 t2)` | 指定驱动表 |
| **并行** | `PARALLEL(n)` | 并行度 |
| **优化器** | `ALL_ROWS` | 吞吐量优先 |
| | `FIRST_ROWS(n)` | 响应时间优先 |
| **其他** | `NO_MERGE` | 禁止视图合并 |
| | `PUSH_PRED` | 下推谓词 |

## 🔍 实战案例预告

### 案例 1：慢查询优化
```sql
-- 原始查询（慢）
SELECT * FROM orders 
WHERE TO_CHAR(order_date, 'YYYY-MM') = '2026-01';

-- 优化后（快）
SELECT * FROM orders 
WHERE order_date >= DATE '2026-01-01'
AND order_date < DATE '2026-02-01';
```

### 案例 2：索引设计
```sql
-- 针对高频查询创建复合索引
CREATE INDEX idx_orders_optimized 
ON orders(customer_id, order_date DESC, status)
COMPRESS 1;
```
