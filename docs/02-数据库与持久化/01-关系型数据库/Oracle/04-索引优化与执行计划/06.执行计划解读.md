## 1. 执行计划概述

### 1.1 什么是执行计划？

执行计划是 Oracle 优化器生成的 SQL 执行步骤说明。

**作用**：
- 了解 SQL 如何执行
- 识别性能瓶颈
- 指导优化方向

### 1.2 执行计划关键元素

| 元素 | 说明 |
|------|------|
| **Id** | 步骤编号 |
| **Operation** | 操作类型 |
| **Name** | 对象名称 |
| **Rows** | 预计返回行数 |
| **Bytes** | 预计数据量 |
| **Cost** | 成本估算 |
| **Time** | 预计时间 |

## 2. 查看执行计划的方法

### 2.1 EXPLAIN PLAN（最常用）

```sql
-- 生成执行计划
EXPLAIN PLAN FOR
SELECT e.name, d.department_name
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id
WHERE e.salary > 5000;

-- 查看执行计划
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**输出示例**：
```
Plan hash value: 1234567890

----------------------------------------------------------------------------------------
| Id  | Operation                    | Name     | Rows  | Bytes | Cost (%CPU)| Time     |
----------------------------------------------------------------------------------------
|   0 | SELECT STATEMENT             |          |    10 |   500 |     5  (20)| 00:00:01 |
|*  1 |  HASH JOIN                   |          |    10 |   500 |     5  (20)| 00:00:01 |
|*  2 |   TABLE ACCESS FULL          | EMPLOYEES|    10 |   300 |     3  (34)| 00:00:01 |
|   3 |   TABLE ACCESS FULL          | DEPARTMENTS|  27 |   459 |     2   (0)| 00:00:01 |
----------------------------------------------------------------------------------------

Predicate Information (identified by operation id):
---------------------------------------------------
   1 - access("E"."DEPT_ID"="D"."DEPT_ID")
   2 - filter("E"."SALARY">5000)
```

### 2.2 AUTOTRACE

```sql
-- 启用 AUTOTRACE
SET AUTOTRACE ON EXPLAIN;

-- 执行查询
SELECT * FROM employees WHERE dept_id = 10;

-- 关闭
SET AUTOTRACE OFF;

-- 显示统计信息
SET AUTOTRACE ON STATISTICS;
```

### 2.3 实时执行计划

```sql
-- 查看最近执行的 SQL 的实际计划
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR(NULL, NULL, 'ALL'));

-- 查看特定 SQL_ID 的计划
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR('sql_id_here', NULL, 'ALL'));
```

### 2.4 AWR 中的执行计划

```sql
-- 查看历史执行计划
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_AWR('sql_id_here'));
```

### 2.5 SQL Developer / Toad

图形化工具可以直接显示执行计划（可视化）。

## 3. 访问路径（Access Path）

### 3.1 TABLE ACCESS FULL（全表扫描）

```
TABLE ACCESS FULL | EMPLOYEES
```

**含义**：扫描整个表  
**适用**：小表、返回大部分行、无合适索引  

### 3.2 TABLE ACCESS BY INDEX ROWID（通过 ROWID 访问表）

```
TABLE ACCESS BY INDEX ROWID | EMPLOYEES
  INDEX RANGE SCAN | IDX_EMP_DEPT
```

**含义**：先扫描索引获取 ROWID，再访问表  
**适用**：返回少量行  

### 3.3 INDEX UNIQUE SCAN（唯一索引扫描）

```
INDEX UNIQUE SCAN | PK_EMPLOYEES
```

**含义**：通过唯一索引查找单行  
**适用**：主键或唯一约束的等值查询  
**效率**：⭐⭐⭐⭐⭐（最快）

### 3.4 INDEX RANGE SCAN（索引范围扫描）

```
INDEX RANGE SCAN | IDX_EMP_SALARY
```

**含义**：扫描索引的一部分  
**适用**：范围查询、非唯一索引的等值查询  
**效率**：⭐⭐⭐⭐

### 3.5 INDEX FULL SCAN（索引全扫描）

```
INDEX FULL SCAN | IDX_EMP_ID
```

**含义**：扫描整个索引（有序）  
**适用**：需要有序结果、覆盖索引  

### 3.6 INDEX FAST FULL SCAN（索引快速全扫描）

```
INDEX FAST FULL SCAN | IDX_EMP_ID
```

**含义**：类似全表扫描，但扫描索引  
**适用**：统计查询、不需要有序  
**特点**：多块读，速度快  

### 3.7 BITMAP 相关

```
BITMAP CONVERSION TO ROWIDS
  BITMAP AND
    BITMAP INDEX SINGLE VALUE | IDX_GENDER
    BITMAP INDEX SINGLE VALUE | IDX_STATUS
```

**含义**：位图运算后转换为 ROWID  
**适用**：Bitmap 索引

## 4. 连接方法（Join Method）

### 4.1 NESTED LOOPS（嵌套循环）

```
NESTED LOOPS
  TABLE ACCESS FULL | EMPLOYEES
  INDEX UNIQUE SCAN | PK_DEPARTMENTS
```

**原理**：
- 外层表每行，内层表查找匹配
- 类似双重循环

**适用**：
- 小表驱动大表
- 外层返回行数少
- 需要快速返回第一行

**优点**：响应时间快  
**缺点**：总吞吐量可能低  

### 4.2 HASH JOIN（哈希连接）

```
HASH JOIN
  TABLE ACCESS FULL | EMPLOYEES
  TABLE ACCESS FULL | DEPARTMENTS
```

**原理**：
- 将小表构建哈希表
- 扫描大表，在哈希表中查找

**适用**：
- 大表连接大表
- 等值连接
- 内存充足

**优点**：吞吐量大  
**缺点**：需要内存，响应时间慢  

### 4.3 MERGE JOIN（排序合并连接）

```
MERGE JOIN
  SORT JOIN
    TABLE ACCESS FULL | EMPLOYEES
  SORT JOIN
    TABLE ACCESS FULL | DEPARTMENTS
```

**原理**：
- 两表分别排序
- 合并排序后的结果

**适用**：
- 数据已排序
- 非等值连接
- 大范围连接

**优点**：适合已排序数据  
**缺点**：需要排序开销

## 5. 执行计划解读技巧

### 5.1 从下往上读

```
SELECT STATEMENT              ← 最后执行
  HASH JOIN                   ← 然后
    TABLE ACCESS FULL         ← 先执行
    INDEX RANGE SCAN          ← 先执行
```

### 5.2 关注 Cost

```
Cost (%CPU)
5  (20)     ← Cost=5, CPU占20%
```

**Cost 越低越好**（但不绝对）

### 5.3 检查 Rows 估算

```
Rows
10          ← 预计返回 10 行
```

**如果与实际相差很大**：
- 统计信息可能过时
- 需要收集统计信息

### 5.4 查看 Predicate Information

```
Predicate Information:
   1 - access("E"."DEPT_ID"="D"."DEPT_ID")  ← 连接条件
   2 - filter("E"."SALARY">5000)            ← 过滤条件
```

**access**：用于访问数据（可以使用索引）  
**filter**：用于过滤数据（可能需要额外处理）

## 6. 执行计划分析案例

### 案例 1：全表扫描优化

```sql
-- 原始查询
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE dept_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 执行计划显示：TABLE ACCESS FULL

-- 问题：没有使用索引

-- 解决：创建索引
CREATE INDEX idx_emp_dept ON employees(dept_id);

-- 再次查看执行计划
-- 现在显示：INDEX RANGE SCAN
```

### 案例 2：连接方法优化

```sql
-- 原始查询
EXPLAIN PLAN FOR
SELECT e.name, d.department_name
FROM employees e, departments d
WHERE e.dept_id = d.dept_id;

-- 执行计划显示：HASH JOIN

-- 如果 employees 很小，可以强制使用 NESTED LOOPS
SELECT /*+ USE_NL(e d) */ e.name, d.department_name
FROM employees e, departments d
WHERE e.dept_id = d.dept_id;

-- 执行计划变为：NESTED LOOPS
```

### 案例 3：统计信息问题

```sql
-- 执行计划显示 Rows 估算严重不准
-- 实际返回 1000 行，估算只有 10 行

-- 解决：收集统计信息
EXEC DBMS_STATS.GATHER_TABLE_STATS('HR', 'EMPLOYEES');

-- 重新生成执行计划
-- Rows 估算应该更准确
```

## 7. 常见问题

### 7.1 为什么优化器选择了全表扫描？

**可能原因**：
1. 表很小
2. 返回行数太多（> 10%-20%）
3. 没有合适的索引
4. 统计信息过时
5. 聚簇因子太高

### 7.2 为什么 Cost 很低但实际很慢？

**可能原因**：
1. 统计信息不准确
2. 系统负载高
3. I/O 瓶颈
4. 锁等待

### 7.3 如何判断执行计划是否最优？

**检查点**：
- ✅ 使用了合适的索引
- ✅ Rows 估算准确
- ✅ Cost 合理
- ✅ 连接方法合适
- ✅ 无不必要的排序
