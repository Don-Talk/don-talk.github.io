# 03函数索引与反向索引

**学习目标：**
> * 掌握函数索引的创建和使用
> * 理解反向索引的原理和适用场景
> * 能够解决常见索引失效问题
> * 学会在 RAC 环境中使用反向索引

---

## 1. 函数索引（Function-Based Index）

### 1.1 什么是函数索引？

函数索引基于表达式或函数的结果创建索引，而不是列本身。

**用途**：
- 解决函数导致索引失效的问题
- 支持复杂表达式的快速查询
- 实现大小写不敏感搜索

### 1.2 创建函数索引

```sql
-- 基本语法
CREATE INDEX idx_emp_upper_name 
ON employees(UPPER(name));

-- 使用函数索引
SELECT * FROM employees 
WHERE UPPER(name) = 'JOHN DOE';
-- 可以使用索引！

-- 没有函数索引时
SELECT * FROM employees 
WHERE UPPER(name) = 'JOHN DOE';
-- 无法使用普通索引，需要全表扫描
```

### 1.3 常用函数索引场景

#### （1）大小写不敏感搜索

```sql
-- 创建索引
CREATE INDEX idx_emp_name_upper ON employees(UPPER(name));

-- 查询
SELECT * FROM employees WHERE UPPER(name) = 'JOHN';

-- 或使用 LOWER
CREATE INDEX idx_emp_name_lower ON employees(LOWER(name));
SELECT * FROM employees WHERE LOWER(name) = 'john';
```

#### （2）日期截断

```sql
-- 按天统计
CREATE INDEX idx_order_date_trunc 
ON orders(TRUNC(order_date));

-- 查询某一天的订单
SELECT * FROM orders 
WHERE TRUNC(order_date) = DATE '2026-05-19';
```

#### （3）字符串函数

```sql
-- SUBSTR 函数索引
CREATE INDEX idx_phone_prefix 
ON employees(SUBSTR(phone, 1, 3));

SELECT * FROM employees 
WHERE SUBSTR(phone, 1, 3) = '138';

-- LENGTH 函数索引
CREATE INDEX idx_name_length 
ON employees(LENGTH(name));

SELECT * FROM employees 
WHERE LENGTH(name) > 10;
```

#### （4）数学运算

```sql
-- 计算字段索引
CREATE INDEX idx_annual_salary 
ON employees(salary * 12);

SELECT * FROM employees 
WHERE salary * 12 > 100000;
```

#### （5）NVL 处理 NULL

```sql
CREATE INDEX idx_comm_nvl 
ON employees(NVL(commission_pct, 0));

SELECT * FROM employees 
WHERE NVL(commission_pct, 0) > 0.1;
```

### 1.4 复合函数索引

```sql
-- 多个函数的组合
CREATE INDEX idx_emp_name_dept 
ON employees(UPPER(name), dept_id);

SELECT * FROM employees 
WHERE UPPER(name) = 'JOHN' AND dept_id = 10;
```

### 1.5 注意事项

⚠️ **必须使用相同的函数**
```sql
-- 创建了 UPPER 索引
CREATE INDEX idx_name ON employees(UPPER(name));

-- ✅ 正确
SELECT * FROM employees WHERE UPPER(name) = 'JOHN';

-- ❌ 错误（无法使用索引）
SELECT * FROM employees WHERE name = 'John';
SELECT * FROM employees WHERE LOWER(name) = 'john';
```

⚠️ **需要权限**
```sql
-- 创建函数索引需要 QUERY REWRITE 权限
GRANT QUERY REWRITE TO hr;
```

⚠️ **统计信息**
```sql
-- 收集统计信息以确保优化器使用函数索引
EXEC DBMS_STATS.GATHER_TABLE_STATS('HR', 'EMPLOYEES');
```

## 2. 反向索引（Reverse Key Index）

### 2.1 什么是反向索引？

反向索引将索引键值的字节顺序反转后存储。

**示例**：
```
原始值：123456
反转后：654321

原始值：123457
反转后：754321
```

### 2.2 为什么需要反向索引？

**问题**：序列生成的主键导致索引热点

```sql
-- 假设使用序列生成 ID
INSERT INTO orders (order_id) VALUES (seq_order.NEXTVAL);
-- 1, 2, 3, 4, 5...

-- 这些值在 B-Tree 索引中相邻
-- 导致所有 INSERT 都竞争同一个叶子块
-- 产生 "buffer busy waits" 等待事件
```

**解决方案**：使用反向索引分散热点

```sql
CREATE INDEX idx_order_id_rev ON orders(order_id) REVERSE;

-- 现在值被分散到不同的叶子块
-- 1 → 100000
-- 2 → 200000
-- 3 → 300000
-- 减少了块竞争
```

### 2.3 创建反向索引

```sql
-- 基本语法
CREATE INDEX idx_order_id_rev 
ON orders(order_id) 
REVERSE;

-- 复合反向索引
CREATE INDEX idx_order_compound_rev 
ON orders(customer_id, order_date) 
REVERSE;
```

### 2.4 查看反向索引

```sql
SELECT 
    index_name,
    table_name,
    index_type
FROM user_indexes
WHERE index_type LIKE '%REVERSE%';
```

### 2.5 反向索引的限制

❌ **不支持范围扫描**
```sql
-- ❌ 这个查询无法使用反向索引
SELECT * FROM orders 
WHERE order_id BETWEEN 1000 AND 2000;

-- ✅ 等值查询可以使用
SELECT * FROM orders 
WHERE order_id = 1234;
```

**原因**：反转后，原本相邻的值不再相邻，无法进行范围扫描。

### 2.6 适用场景

✅ **RAC 环境**
- 多实例并发插入
- 减少全局缓存竞争

✅ **高并发 INSERT**
- 序列生成的主键
- 时间戳列

✅ **索引热点块问题**
- 监控到 "buffer busy waits"
- AWR 报告显示索引竞争

❌ **不适用场景**
- 需要范围查询
- 低并发系统
- ORDER BY 查询

## 3. 实战案例

### 案例 1：大小写不敏感搜索优化

```sql
-- 原始查询（慢，全表扫描）
SELECT * FROM users 
WHERE UPPER(email) = 'TEST@EXAMPLE.COM';

-- 创建函数索引
CREATE INDEX idx_users_email_upper 
ON users(UPPER(email));

-- 现在查询很快
EXPLAIN PLAN FOR
SELECT * FROM users 
WHERE UPPER(email) = 'TEST@EXAMPLE.COM';

-- 执行计划显示使用 INDEX RANGE SCAN
```

### 案例 2：RAC 环境优化

```sql
-- RAC 环境，高并发插入
-- 监控发现索引热点

-- 创建反向索引
CREATE INDEX idx_orders_pk_rev 
ON orders(order_id) 
REVERSE;

-- 观察等待事件减少
SELECT event, total_waits, time_waited
FROM v$system_event
WHERE event LIKE 'buffer busy waits%';
```

### 案例 3：日期查询优化

```sql
-- 频繁按天查询
SELECT * FROM logs 
WHERE TRUNC(log_time) = DATE '2026-05-19';

-- 创建函数索引
CREATE INDEX idx_logs_date_trunc 
ON logs(TRUNC(log_time));

-- 或者更好的方式：使用范围查询（无需函数索引）
SELECT * FROM logs 
WHERE log_time >= DATE '2026-05-19'
AND log_time < DATE '2026-05-20';
```

## 4. 最佳实践

### 4.1 函数索引

✅ **优先改写 SQL 避免函数**
```sql
-- ❌ 使用函数索引
WHERE UPPER(name) = 'JOHN'

-- ✅ 更好：存储时统一格式
-- 或在应用层处理
```

✅ **只创建必要的函数索引**
- 每个函数索引都有维护成本
- 监控使用情况

✅ **注意函数确定性**
- 只能使用确定性函数
- 自定义函数需要 DETERMINISTIC 关键字

### 4.2 反向索引

✅ **只在必要时使用**
- RAC 环境
- 高并发 INSERT
- 确认有热点块问题

✅ **测试范围查询影响**
- 确保不需要范围扫描
- 或为范围查询创建额外索引

❌ **不要滥用**
- 普通 B-Tree 索引优先
- 反向索引是特殊场景的解决方案

## 5. 总结

### 5.1 核心知识点

✅ **函数索引**：基于表达式创建  
✅ **反向索引**：反转键值字节顺序  
✅ **函数索引用途**：解决函数导致的索引失效  
✅ **反向索引用途**：分散索引热点  

### 5.2 选择指南

| 场景 | 推荐方案 |
|------|---------|
| 函数查询 | 函数索引 |
| 大小写不敏感 | 函数索引（UPPER/LOWER） |
| RAC 热点块 | 反向索引 |
| 序列主键高并发 | 反向索引 |
| 范围查询 | 普通 B-Tree |
