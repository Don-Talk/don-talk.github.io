# 04复合索引与覆盖索引

**学习目标：**
> * 掌握复合索引的设计原则
> * 理解最左前缀原则
> * 学会使用覆盖索引优化查询
> * 了解索引跳跃扫描机制

---

## 1. 复合索引（Composite Index）

### 1.1 什么是复合索引？

复合索引是基于多个列创建的索引。

```sql
-- 创建复合索引
CREATE INDEX idx_emp_dept_sal 
ON employees(dept_id, salary, hire_date);
```

### 1.2 最左前缀原则

复合索引遵循最左前缀原则：可以使用索引的前导列。

```sql
-- 索引：(dept_id, salary, hire_date)

-- ✅ 可以使用索引
WHERE dept_id = 10
WHERE dept_id = 10 AND salary > 5000
WHERE dept_id = 10 AND salary > 5000 AND hire_date > SYSDATE-30

-- ❌ 不能使用索引（跳过前导列）
WHERE salary > 5000
WHERE hire_date > SYSDATE-30
WHERE salary > 5000 AND hire_date > SYSDATE-30
```

### 1.3 列顺序设计原则

**原则 1：等值查询列在前**
```sql
-- 常见查询
SELECT * FROM employees 
WHERE dept_id = 10      -- 等值
AND salary > 5000       -- 范围
ORDER BY hire_date;     -- 排序

-- 推荐索引
CREATE INDEX idx_optimal 
ON employees(dept_id, salary, hire_date);
```

**原则 2：高选择性列在前**
```sql
-- 如果两个列都是等值查询
-- 选择性高的列放前面
CREATE INDEX idx_selective 
ON employees(email, status);  -- email 选择性更高
```

**原则 3：考虑 ORDER BY**
```sql
-- 如果需要排序，排序列放在最后
CREATE INDEX idx_order_optimized 
ON employees(dept_id, hire_date DESC);

SELECT * FROM employees 
WHERE dept_id = 10 
ORDER BY hire_date DESC;  -- 避免排序操作
```

### 1.4 索引跳跃扫描（Index Skip Scan）

Oracle 9i+ 支持索引跳跃扫描，即使不使用前导列也可能使用索引。

```sql
-- 索引：(gender, employee_id)
CREATE INDEX idx_gender_emp ON employees(gender, employee_id);

-- 这个查询可能使用跳跃扫描
SELECT * FROM employees WHERE employee_id = 100;

-- 执行计划
-- INDEX SKIP SCAN IDX_GENDER_EMP
```

**原理**：
- Oracle 将索引逻辑上分成多个子索引
- 对每个子索引进行扫描
- 适合前导列不同值很少的情况

**注意**：跳跃扫描效率低于直接使用索引，不应依赖。

## 2. 覆盖索引（Covering Index）

### 2.1 什么是覆盖索引？

覆盖索引包含查询所需的所有列，无需访问表（回表）。

```sql
-- 查询
SELECT employee_id, name, email 
FROM employees 
WHERE dept_id = 10;

-- 覆盖索引（包含所有需要的列）
CREATE INDEX idx_emp_covering 
ON employees(dept_id, employee_id, name, email);

-- 执行计划
-- INDEX RANGE SCAN IDX_EMP_COVERING
-- 无需 TABLE ACCESS BY INDEX ROWID
```

### 2.2 覆盖索引的优势

✅ **避免回表**：减少 I/O  
✅ **提高性能**：尤其对大表  
✅ **减少锁竞争**：不访问表  

### 2.3 覆盖索引设计

```sql
-- 原始查询
SELECT order_id, customer_id, amount 
FROM orders 
WHERE status = 'COMPLETED'
AND order_date >= SYSDATE - 30;

-- 覆盖索引
CREATE INDEX idx_orders_covering 
ON orders(status, order_date, order_id, customer_id, amount);

-- 查询直接从索引获取数据
```

### 2.4 权衡考虑

⚠️ **索引大小**：列越多，索引越大  
⚠️ **维护成本**：DML 开销增加  
⚠️ **只用于高频查询**：不是所有查询都需要覆盖

## 3. 索引压缩（Index Compression）

### 3.1 为什么需要压缩？

复合索引中前导列可能有大量重复值，压缩可以节省空间。

### 3.2 创建压缩索引

```sql
-- 压缩前 N 列
CREATE INDEX idx_emp_compressed 
ON employees(dept_id, name, salary)
COMPRESS 1;  -- 压缩 dept_id 列

-- 压缩更多列
CREATE INDEX idx_emp_compressed2 
ON employees(dept_id, location, name)
COMPRESS 2;  -- 压缩 dept_id 和 location
```

### 3.3 压缩效果

```sql
-- 查看压缩前后大小对比
SELECT 
    segment_name,
    bytes/1024/1024 as size_mb
FROM user_segments
WHERE segment_name LIKE 'IDX_EMP%';
```

**典型压缩率**：30%-70%

### 3.4 注意事项

✅ **适合前导列重复度高**  
❌ **过度压缩可能影响性能**  
⚠️ **测试后再应用到生产**

## 4. 实战案例

### 案例 1：优化多条件查询

```sql
-- 常见查询模式
SELECT * FROM orders
WHERE customer_id = 100
AND status = 'PENDING'
AND order_date >= SYSDATE - 7
ORDER BY order_date DESC;

-- 分析：
-- customer_id：等值查询
-- status：等值查询
-- order_date：范围查询 + 排序

-- 创建索引
CREATE INDEX idx_orders_opt 
ON orders(customer_id, status, order_date DESC);

-- 验证执行计划
EXPLAIN PLAN FOR
SELECT * FROM orders
WHERE customer_id = 100
AND status = 'PENDING'
AND order_date >= SYSDATE - 7
ORDER BY order_date DESC;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
-- 应该看到 INDEX RANGE SCAN，无 SORT ORDER BY
```

### 案例 2：覆盖索引优化报表

```sql
-- 报表查询（频繁执行）
SELECT 
    product_id,
    product_name,
    price,
    category
FROM products
WHERE category = 'Electronics'
AND price BETWEEN 100 AND 500;

-- 创建覆盖索引
CREATE INDEX idx_products_report 
ON products(category, price, product_id, product_name);

-- 查询直接从索引获取，无需访问表
```

### 案例 3：索引压缩节省空间

```sql
-- 大表，dept_id 重复度高
SELECT COUNT(DISTINCT dept_id), COUNT(*) 
FROM employees;
-- 结果：20 / 1000000

-- 未压缩索引
CREATE INDEX idx_emp_uncompressed 
ON employees(dept_id, employee_id);
-- 大小：50 MB

-- 压缩索引
CREATE INDEX idx_emp_compressed 
ON employees(dept_id, employee_id)
COMPRESS 1;
-- 大小：20 MB（节省 60%）
```

## 5. 最佳实践

### 5.1 复合索引设计

✅ **列数适中**：通常 2-4 列  
✅ **等值在前，范围在后**  
✅ **考虑 ORDER BY**  
✅ **高选择性优先**  
✅ **定期监控使用情况**  

### 5.2 覆盖索引使用

✅ **只用于高频查询**  
✅ **权衡索引大小和性能**  
✅ **避免过多列**  
✅ **优先考虑 SELECT 列少的查询**  

### 5.3 压缩策略

✅ **前导列重复度高时压缩**  
✅ **测试压缩效果**  
✅ **监控性能影响**

## 6. 总结

### 6.1 核心知识点

✅ **最左前缀原则**：必须使用前导列  
✅ **列顺序设计**：等值→范围→排序  
✅ **覆盖索引**：避免回表  
✅ **索引压缩**：节省空间  
✅ **跳跃扫描**：Oracle 自动优化  

### 6.2 设计要点

📌 分析查询模式  
📌 合理排列列顺序  
📌 权衡索引数量和大小  
📌 监控和优化
