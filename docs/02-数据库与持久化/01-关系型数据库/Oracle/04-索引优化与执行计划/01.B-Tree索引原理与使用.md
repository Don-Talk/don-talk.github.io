## 1. B-Tree 索引概述

### 1.1 什么是 B-Tree 索引？

B-Tree（Balanced Tree，平衡树）是 Oracle 最常用的索引类型，适用于高基数列（唯一值多的列）。

**特点**：
- 自动保持平衡
- 所有叶子节点在同一层
- 支持等值查询和范围查询
- 默认索引类型

### 1.2 B-Tree 索引结构

```
根块（Root Block）
    ↓
分支块（Branch Blocks）
    ↓       ↓
分支块    分支块
    ↓       ↓       ↓       ↓
叶子块（Leaf Blocks）→ 双向链表
```

**三层结构**：
1. **根块**：入口点，只有一个
2. **分支块**：中间层，存储键值和指针
3. **叶子块**：最底层，存储键值和 ROWID

**叶子块特点**：
- 包含索引键值
- 包含对应行的 ROWID
- 通过双向链表连接（支持范围扫描）
- 按键值排序

## 2. 创建和管理 B-Tree 索引

### 2.1 创建索引

```sql
-- 基本语法
CREATE INDEX idx_emp_name ON employees(name);

-- 指定表空间
CREATE INDEX idx_emp_name 
ON employees(name)
TABLESPACE users_idx;

-- 并行创建（大表）
CREATE INDEX idx_emp_name 
ON employees(name)
PARALLEL 4;

-- 在线创建（不阻塞 DML）
CREATE INDEX idx_emp_name 
ON employees(name)
ONLINE;

-- 压缩索引
CREATE INDEX idx_emp_dept 
ON employees(dept_id, name)
COMPRESS 1;  -- 压缩前 1 列
```

### 2.2 查看索引信息

```sql
-- 查看用户索引
SELECT 
    index_name,
    table_name,
    index_type,
    uniqueness,
    status,
    num_rows,
    leaf_blocks,
    clustering_factor
FROM user_indexes
WHERE table_name = 'EMPLOYEES';

-- 查看索引列
SELECT 
    index_name,
    column_name,
    column_position,
    descend
FROM user_ind_columns
WHERE table_name = 'EMPLOYEES'
ORDER BY index_name, column_position;

-- 查看索引大小
SELECT 
    segment_name,
    bytes/1024/1024 as size_mb,
    extents
FROM user_segments
WHERE segment_name = 'IDX_EMP_NAME';
```

### 2.3 修改索引

```sql
-- 重建索引
ALTER INDEX idx_emp_name REBUILD;

-- 重建到不同表空间
ALTER INDEX idx_emp_name REBUILD TABLESPACE new_ts;

-- 在线重建
ALTER INDEX idx_emp_name REBUILD ONLINE;

-- 合并索引（整理碎片）
ALTER INDEX idx_emp_name COALESCE;

-- 重命名索引
ALTER INDEX idx_emp_name RENAME TO idx_employee_name;

-- 禁用索引
ALTER INDEX idx_emp_name UNUSABLE;

-- 启用索引（需要重建）
ALTER INDEX idx_emp_name REBUILD;
```

### 2.4 删除索引

```sql
DROP INDEX idx_emp_name;

-- 注意：删除主键或唯一约束关联的索引需要先删除约束
ALTER TABLE employees DROP CONSTRAINT pk_employees CASCADE;
```

## 3. 索引扫描类型

### 3.1 INDEX UNIQUE SCAN（唯一索引扫描）

用于唯一索引的等值查询，最多返回一行。

```sql
-- 假设 employee_id 有唯一索引
SELECT * FROM employees WHERE employee_id = 100;

-- 执行计划
-- INDEX UNIQUE SCAN IDX_EMP_ID
```

**特点**：
- 最快访问方式
- 最多返回一行
- 成本最低

### 3.2 INDEX RANGE SCAN（索引范围扫描）

用于非唯一索引或范围查询。

```sql
-- 范围查询
SELECT * FROM employees 
WHERE salary BETWEEN 5000 AND 10000;

-- 前缀匹配
SELECT * FROM employees 
WHERE name LIKE 'John%';

-- 复合索引的部分列
SELECT * FROM employees 
WHERE dept_id = 10;  -- 假设索引是 (dept_id, name)

-- 执行计划
-- INDEX RANGE SCAN IDX_EMP_SALARY
```

**特点**：
- 返回多行
- 按索引顺序访问
- 适合小范围查询

### 3.3 INDEX FULL SCAN（索引全扫描）

扫描整个索引，按索引顺序返回数据。

```sql
-- 查询索引覆盖的列，需要排序
SELECT employee_id, name 
FROM employees 
ORDER BY employee_id;

-- 执行计划
-- INDEX FULL SCAN IDX_EMP_ID
```

**特点**：
- 扫描所有叶子块
- 避免排序操作
- 适合小表或需要有序结果

### 3.4 INDEX FAST FULL SCAN（索引快速全扫描）

类似全表扫描，但扫描的是索引。

```sql
-- 统计查询（只需要索引列）
SELECT COUNT(*) FROM employees;

-- 查询索引覆盖的所有列（无序）
SELECT employee_id, name FROM employees;

-- 执行计划
-- INDEX FAST FULL SCAN IDX_EMP_ID
```

**特点**：
- 多块读（效率高）
- 不保证顺序
- 适合统计和覆盖查询

### 3.5 TABLE ACCESS BY INDEX ROWID（通过 ROWID 访问表）

通过索引找到 ROWID，再访问表获取其他列。

```sql
-- 查询非索引列
SELECT name, email 
FROM employees 
WHERE dept_id = 10;

-- 执行计划
-- INDEX RANGE SCAN IDX_EMP_DEPT
-- TABLE ACCESS BY INDEX ROWID EMPLOYEES
```

**特点**：
- 需要回表（额外 I/O）
- 随机 I/O（性能较差）
- 返回行数少时效率高

## 4. 聚簇因子（Clustering Factor）

### 4.1 什么是聚簇因子？

聚簇因子衡量索引键值顺序与表中数据物理存储顺序的一致性。

**取值范围**：
- 最小值 = 表的块数（理想情况）
- 最大值 = 表的行数（最差情况）

### 4.2 查看聚簇因子

```sql
SELECT 
    index_name,
    num_rows,
    leaf_blocks,
    clustering_factor,
    table_name
FROM user_indexes
WHERE table_name = 'EMPLOYEES';
```

### 4.3 聚簇因子的影响

#### 低聚簇因子（好）
```
索引顺序：1, 2, 3, 4, 5, 6...
物理存储：1, 2, 3, 4, 5, 6...
```
- 数据在表中按索引顺序存储
- 范围扫描效率高
- 减少 I/O 次数

#### 高聚簇因子（差）
```
索引顺序：1, 2, 3, 4, 5, 6...
物理存储：3, 1, 5, 2, 6, 4...
```
- 数据在表中随机分布
- 范围扫描需要大量随机 I/O
- 优化器可能选择全表扫描

### 4.4 改善聚簇因子

```sql
-- 方法 1：按索引列顺序重组表
CREATE TABLE employees_new AS
SELECT * FROM employees ORDER BY dept_id;

DROP TABLE employees;
RENAME employees_new TO employees;

-- 重建索引
CREATE INDEX idx_emp_dept ON employees(dept_id);

-- 方法 2：使用索引组织表（IOT）
CREATE TABLE employees_iot (
    employee_id NUMBER PRIMARY KEY,
    name VARCHAR2(50),
    dept_id NUMBER
) ORGANIZATION INDEX;
```

## 5. 索引选择性

### 5.1 什么是选择性？

选择性 = 不同值的数量 / 总行数

**高选择性**（接近 1）：适合建索引
- 主键、唯一列
- 例如：employee_id、email

**低选择性**（接近 0）：不适合建索引
- 性别、状态标志
- 例如：gender（只有 2 个值）

### 5.2 计算选择性

```sql
-- 计算列的选择性
SELECT 
    column_name,
    num_distinct,
    num_rows,
    ROUND(num_distinct / NULLIF(num_rows, 0) * 100, 2) as selectivity_pct
FROM user_tab_columns
WHERE table_name = 'EMPLOYEES';
```

### 5.3 选择性指导原则

| 选择性 | 建议 |
|--------|------|
| > 20% | ✅ 非常适合建索引 |
| 5%-20% | ⚠️ 可以考虑建索引 |
| < 5% | ❌ 通常不适合建索引 |

**例外情况**：
- 低选择性列但数据分布不均
- 复合索引中的前导列
- Bitmap 索引适合低选择性

## 6. 索引维护成本

### 6.1 DML 操作对索引的影响

每次 INSERT、UPDATE、DELETE 都需要维护索引：

```sql
-- INSERT：添加索引条目
INSERT INTO employees (employee_id, name, dept_id)
VALUES (1000, 'New Employee', 10);
-- 需要在所有相关索引中添加条目

-- UPDATE：可能更新索引
UPDATE employees SET dept_id = 20 WHERE employee_id = 100;
-- 如果 dept_id 有索引，需要更新

-- DELETE：删除索引条目
DELETE FROM employees WHERE employee_id = 100;
-- 需要从所有相关索引中删除条目
```

### 6.2 索引过多问题

**问题**：
- 每个索引都会增加 DML 开销
- 占用额外存储空间
- 增加备份时间

**建议**：
- 只创建必要的索引
- 定期清理未使用的索引
- 权衡查询性能和 DML 性能

### 6.3 监控索引使用情况

```sql
-- 启用索引监控
ALTER INDEX idx_emp_name MONITORING USAGE;

-- 查看使用情况
SELECT 
    index_name,
    used,
    start_monitoring,
    end_monitoring
FROM v$object_usage;

-- 停止监控
ALTER INDEX idx_emp_name NOMONITORING USAGE;

-- 查找未使用的索引
SELECT 
    i.index_name,
    i.table_name,
    i.num_rows
FROM user_indexes i
LEFT JOIN v$object_usage u ON i.index_name = u.index_name
WHERE u.used = 'NO' OR u.index_name IS NULL;
```

## 7. 何时使用 B-Tree 索引

### 7.1 适合场景

✅ **高选择性列**
```sql
CREATE INDEX idx_emp_email ON employees(email);
```

✅ **经常用于 WHERE 条件**
```sql
-- 频繁查询
SELECT * FROM employees WHERE dept_id = 10;
CREATE INDEX idx_emp_dept ON employees(dept_id);
```

✅ **经常用于 JOIN 条件**
```sql
SELECT e.name, d.department_name
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id;
CREATE INDEX idx_emp_dept ON employees(dept_id);
```

✅ **经常用于 ORDER BY**
```sql
SELECT * FROM employees ORDER BY hire_date DESC;
CREATE INDEX idx_emp_hire ON employees(hire_date DESC);
```

✅ **需要唯一约束**
```sql
CREATE UNIQUE INDEX idx_emp_email ON employees(email);
```

### 7.2 不适合场景

❌ **小表（< 1000 行）**
- 全表扫描更快
- 索引维护成本高

❌ **低选择性列**
```sql
-- gender 只有 2 个值，不适合 B-Tree 索引
-- 考虑 Bitmap 索引或不建索引
```

❌ **频繁更新的列**
- 索引维护开销大
- 可能导致索引碎片

❌ **LIKE 左模糊查询**
```sql
-- 这个查询无法使用索引
SELECT * FROM employees WHERE name LIKE '%ohn';
```

## 8. 索引设计最佳实践

### 8.1 命名规范

```sql
-- 推荐的命名约定
idx_<table>_<column>          -- 普通索引
uk_<table>_<column>           -- 唯一索引
pk_<table>                    -- 主键索引

-- 示例
CREATE INDEX idx_emp_dept ON employees(dept_id);
CREATE UNIQUE INDEX uk_emp_email ON employees(email);
```

### 8.2 复合索引列顺序

**原则**：
1. 等值查询列在前
2. 范围查询列在后
3. 高选择性列在前

```sql
-- 常见查询
SELECT * FROM employees 
WHERE dept_id = 10 
AND salary > 5000 
ORDER BY hire_date;

-- 推荐索引
CREATE INDEX idx_emp_optimal 
ON employees(dept_id, salary, hire_date);

-- dept_id：等值查询，放第一
-- salary：范围查询，放第二
-- hire_date：排序，放第三
```

### 8.3 覆盖索引

```sql
-- 如果查询只需要某些列，创建覆盖索引
SELECT employee_id, name, email 
FROM employees 
WHERE dept_id = 10;

-- 覆盖索引（无需回表）
CREATE INDEX idx_emp_covering 
ON employees(dept_id, employee_id, name, email);
```

### 8.4 函数索引

```sql
-- 如果查询使用函数，创建函数索引
SELECT * FROM employees WHERE UPPER(name) = 'JOHN';

-- 函数索引
CREATE INDEX idx_emp_name_upper 
ON employees(UPPER(name));
```

## 9. 索引故障排查

### 9.1 索引未被使用

```sql
-- 检查统计信息是否最新
EXEC DBMS_STATS.GATHER_TABLE_STATS('HR', 'EMPLOYEES');

-- 查看执行计划
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE dept_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 可能的原因：
-- 1. 统计信息过时
-- 2. 选择性太低
-- 3. 返回行数太多（优化器选择 FTS）
-- 4. 索引失效
```

### 9.2 索引碎片

```sql
-- 检查索引碎片
SELECT 
    index_name,
    blevel,
    leaf_blocks,
    num_rows,
    ROUND((leaf_blocks * 8192) / NULLIF(num_rows * 100, 0) * 100, 2) as fragmentation_pct
FROM user_indexes
WHERE table_name = 'EMPLOYEES';

-- 重建索引消除碎片
ALTER INDEX idx_emp_name REBUILD;
```

### 9.3 索引膨胀

```sql
-- 比较索引大小和预期大小
SELECT 
    segment_name,
    bytes/1024/1024 as actual_mb,
    (num_rows * avg_row_len) / 1024 / 1024 as expected_mb
FROM user_segments s
JOIN user_tables t ON s.segment_name = t.table_name
WHERE s.segment_name = 'EMPLOYEES';

-- 如果实际远大于预期，考虑重建
ALTER INDEX idx_emp_name REBUILD;
```

## 10. 实战案例

### 案例 1：优化慢查询

```sql
-- 原始查询（慢）
SELECT * FROM orders 
WHERE customer_id = 100 
AND order_date >= SYSDATE - 30;

-- 分析执行计划
EXPLAIN PLAN FOR
SELECT * FROM orders 
WHERE customer_id = 100 
AND order_date >= SYSDATE - 30;

-- 发现是全表扫描

-- 创建复合索引
CREATE INDEX idx_orders_opt ON orders(customer_id, order_date DESC);

-- 再次分析执行计划
-- 现在使用 INDEX RANGE SCAN
```

### 案例 2：索引维护策略

```sql
-- 定期监控索引使用情况
BEGIN
    -- 启用所有索引监控
    FOR idx IN (SELECT index_name FROM user_indexes) LOOP
        EXECUTE IMMEDIATE 'ALTER INDEX ' || idx.index_name || ' MONITORING USAGE';
    END LOOP;
END;

-- 一周后检查
SELECT * FROM v$object_usage WHERE used = 'NO';

-- 删除未使用的索引
DROP INDEX idx_unused;
```

## 11. 总结

### 11.1 核心知识点

✅ **B-Tree 结构**：根块、分支块、叶子块  
✅ **扫描类型**：UNIQUE、RANGE、FULL、FAST FULL  
✅ **聚簇因子**：影响范围扫描效率  
✅ **选择性**：决定索引价值  
✅ **维护成本**：权衡查询和 DML 性能  

### 11.2 最佳实践

📌 为高选择性列创建索引  
📌 复合索引注意列顺序  
📌 定期监控索引使用情况  
📌 及时重建碎片化索引  
📌 避免过度索引  

### 11.3 性能要点

- 唯一查询 → INDEX UNIQUE SCAN
- 小范围查询 → INDEX RANGE SCAN
- 有序结果 → INDEX FULL SCAN
- 统计查询 → INDEX FAST FULL SCAN
- 大范围查询 → 可能全表扫描更好
