## 1. 索引失效概述

### 1.1 什么是索引失效？

索引失效是指查询无法使用索引，导致全表扫描或效率低下。

**影响**：
- 查询性能大幅下降
- 增加 I/O 开销
- 系统负载升高

## 2. 常见索引失效场景

### 2.1 隐式类型转换

```sql
-- ❌ 索引失效：字符串与数字比较
SELECT * FROM employees WHERE employee_id = '100';
-- employee_id 是 NUMBER 类型，'100' 是字符串
-- Oracle 会进行隐式转换：TO_NUMBER(employee_id) = 100

-- ✅ 正确写法
SELECT * FROM employees WHERE employee_id = 100;

-- ❌ 另一个例子
SELECT * FROM employees WHERE hire_date = '2026-05-19';
-- hire_date 是 DATE 类型

-- ✅ 正确写法
SELECT * FROM employees 
WHERE hire_date = TO_DATE('2026-05-19', 'YYYY-MM-DD');
```

**检测方法**：
```sql
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE employee_id = '100';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
-- 如果看到 TABLE ACCESS FULL，说明索引失效
```

### 2.2 函数导致索引失效

```sql
-- ❌ 索引失效
SELECT * FROM employees WHERE UPPER(name) = 'JOHN';
SELECT * FROM employees WHERE TRUNC(hire_date) = SYSDATE;
SELECT * FROM employees WHERE SUBSTR(phone, 1, 3) = '138';

-- ✅ 解决方案 1：创建函数索引
CREATE INDEX idx_emp_name_upper ON employees(UPPER(name));
CREATE INDEX idx_emp_date_trunc ON employees(TRUNC(hire_date));

-- ✅ 解决方案 2：改写 SQL 避免函数
SELECT * FROM employees 
WHERE name >= 'John' AND name < 'Joho';  -- 范围查询
```

### 2.3 LIKE 左模糊

```sql
-- ❌ 索引失效（左模糊）
SELECT * FROM employees WHERE name LIKE '%ohn';
SELECT * FROM employees WHERE email LIKE '%@gmail.com';

-- ✅ 可以使用索引（右模糊）
SELECT * FROM employees WHERE name LIKE 'Joh%';

-- ✅ 解决方案：使用全文索引或反向存储
CREATE INDEX idx_name_reverse ON employees(REVERSE(name));
SELECT * FROM employees WHERE REVERSE(name) LIKE REVERSE('%ohn');
```

### 2.4 NOT 条件

```sql
-- ❌ 索引通常失效
SELECT * FROM employees WHERE dept_id != 10;
SELECT * FROM employees WHERE status <> 'ACTIVE';
SELECT * FROM employees WHERE name NOT LIKE 'A%';

-- ✅ 解决方案：改写为正向条件
SELECT * FROM employees 
WHERE dept_id IN (20, 30, 40, 50);  -- 列出所有其他值

-- 或使用 MINUS
SELECT employee_id FROM employees
MINUS
SELECT employee_id FROM employees WHERE dept_id = 10;
```

### 2.5 OR 条件

```sql
-- ❌ 如果 OR 两边的列不都有索引，索引可能失效
SELECT * FROM employees 
WHERE emp_id = 100 OR email = 'test@test.com';
-- 假设只有 emp_id 有索引

-- ✅ 解决方案 1：确保所有列都有索引
CREATE INDEX idx_emp_id ON employees(emp_id);
CREATE INDEX idx_email ON employees(email);

-- ✅ 解决方案 2：使用 UNION ALL
SELECT * FROM employees WHERE emp_id = 100
UNION ALL
SELECT * FROM employees WHERE email = 'test@test.com' AND emp_id != 100;
```

### 2.6 计算表达式

```sql
-- ❌ 索引失效
SELECT * FROM orders WHERE amount * 1.1 > 1000;
SELECT * FROM employees WHERE salary + bonus > 10000;
SELECT * FROM products WHERE price / 1.2 < 100;

-- ✅ 解决方案：将计算移到另一边
SELECT * FROM orders WHERE amount > 1000 / 1.1;
SELECT * FROM employees WHERE salary > 10000 - bonus;
SELECT * FROM products WHERE price < 100 * 1.2;
```

### 2.7 IS NULL / IS NOT NULL

```sql
-- ⚠️ 取决于数据分布
SELECT * FROM employees WHERE manager_id IS NULL;

-- 如果 NULL 值很少，可以使用索引
-- 如果 NULL 值很多，优化器可能选择全表扫描

-- ✅ 解决方案：创建函数索引
CREATE INDEX idx_mgr_null ON employees(NVL(manager_id, -1));
SELECT * FROM employees WHERE NVL(manager_id, -1) = -1;
```

### 2.8 复合索引跳过前导列

```sql
-- 索引：(dept_id, salary, hire_date)

-- ❌ 索引失效（跳过前导列）
SELECT * FROM employees WHERE salary > 5000;
SELECT * FROM employees WHERE hire_date > SYSDATE-30;

-- ✅ 可以使用部分索引
SELECT * FROM employees WHERE dept_id = 10;  -- 使用前导列

-- ✅ 解决方案：创建单独的索引
CREATE INDEX idx_salary ON employees(salary);
```

### 2.9 数据倾斜导致优化器选择 FTS

```sql
-- 即使有索引，如果返回行数太多，优化器可能选择全表扫描

-- 假设 90% 的员工在 dept_id = 10
SELECT * FROM employees WHERE dept_id = 10;
-- 优化器可能选择 TABLE ACCESS FULL

-- ✅ 解决方案：
-- 1. 收集统计信息
EXEC DBMS_STATS.GATHER_TABLE_STATS('HR', 'EMPLOYEES');

-- 2. 如果确实需要索引，使用 Hint
SELECT /*+ INDEX(e idx_emp_dept) */ * 
FROM employees e 
WHERE dept_id = 10;
```

### 2.10 统计信息过时

```sql
-- 统计信息过时会导致优化器做出错误决策

-- ✅ 定期收集统计信息
EXEC DBMS_STATS.GATHER_TABLE_STATS('HR', 'EMPLOYEES');

-- 查看统计信息更新时间
SELECT table_name, last_analyzed, num_rows
FROM user_tables
WHERE table_name = 'EMPLOYEES';
```

## 3. 诊断索引失效

### 3.1 使用 EXPLAIN PLAN

```sql
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE dept_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**关键指标**：
- `TABLE ACCESS FULL`：全表扫描（索引未使用）
- `INDEX RANGE SCAN`：使用了索引
- `Cost`：成本估算

### 3.2 使用 AUTOTRACE

```sql
SET AUTOTRACE ON EXPLAIN;
SELECT * FROM employees WHERE dept_id = 10;
SET AUTOTRACE OFF;
```

### 3.3 查看实际执行计划

```sql
-- 执行查询后查看实际计划
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR(NULL, NULL, 'ALL'));
```

## 4. 解决方案汇总

### 4.1 快速检查清单

| 问题 | 解决方案 |
|------|---------|
| 类型转换 | 使用正确的数据类型 |
| 函数调用 | 创建函数索引或改写 SQL |
| LIKE 左模糊 | 使用右模糊或全文索引 |
| NOT 条件 | 改写为正向条件 |
| OR 条件 | 使用 UNION ALL 或确保都有索引 |
| 计算表达式 | 将计算移到常量侧 |
| 跳过前导列 | 调整列顺序或创建单独索引 |
| 统计信息过时 | 收集统计信息 |

### 4.2 通用优化步骤

1. **分析执行计划**
2. **确认索引存在**
3. **检查统计信息**
4. **识别失效原因**
5. **应用解决方案**
6. **验证优化效果**

## 5. 实战案例

### 案例 1：慢查询诊断

```sql
-- 原始查询（慢）
SELECT * FROM orders 
WHERE TO_CHAR(order_date, 'YYYY-MM') = '2026-01';

-- 执行计划显示全表扫描

-- 问题：函数导致索引失效

-- 解决方案：改写 SQL
SELECT * FROM orders 
WHERE order_date >= DATE '2026-01-01'
AND order_date < DATE '2026-02-01';

-- 现在可以使用 order_date 的索引
```

### 案例 2：复合索引优化

```sql
-- 索引：(status, order_date)

-- ❌ 这个查询无法使用索引
SELECT * FROM orders WHERE order_date >= SYSDATE - 30;

-- ✅ 调整索引列顺序
DROP INDEX idx_orders_status_date;
CREATE INDEX idx_orders_date_status 
ON orders(order_date, status);

-- 现在查询可以使用索引
```

## 6. 最佳实践

### 6.1 预防措施

✅ **编写规范的 SQL**  
✅ **避免在索引列上使用函数**  
✅ **使用绑定变量**  
✅ **定期收集统计信息**  
✅ **监控执行计划变化**  

### 6.2 开发规范

📌 禁止在 WHERE 条件中对索引列使用函数  
📌 避免隐式类型转换  
📌 LIKE 只使用右模糊  
📌 优先使用等值查询  
📌 审查 SQL 执行计划
