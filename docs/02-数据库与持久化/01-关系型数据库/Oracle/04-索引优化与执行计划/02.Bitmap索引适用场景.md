## 1. Bitmap 索引概述

### 1.1 什么是 Bitmap 索引？

Bitmap 索引使用位图（0/1）来表示索引键值的存在性，适合低基数列（不同值少的列）。

**对比 B-Tree**：
- B-Tree：适合高基数列（如 ID、邮箱）
- Bitmap：适合低基数列（如性别、状态）

### 1.2 Bitmap 索引结构

```
列值    | Row1 | Row2 | Row3 | Row4 | Row5 | ...
--------|------|------|------|------|------|----
Male    |  1   |  0   |  1   |  0   |  1   | ...
Female  |  0   |  1   |  0   |  1   |  0   | ...
```

每个不同值对应一个位图，位图中的每一位代表一行。

## 2. 创建和管理 Bitmap 索引

### 2.1 创建 Bitmap 索引

```sql
-- 基本语法
CREATE BITMAP INDEX idx_emp_gender ON employees(gender);

-- 指定表空间
CREATE BITMAP INDEX idx_emp_status 
ON employees(status)
TABLESPACE users_idx;

-- 复合 Bitmap 索引
CREATE BITMAP INDEX idx_emp_dept_gender 
ON employees(dept_id, gender);
```

### 2.2 查看 Bitmap 索引

```sql
SELECT 
    index_name,
    table_name,
    index_type,
    num_rows,
    leaf_blocks,
    distinct_keys
FROM user_indexes
WHERE index_type = 'BITMAP';
```

### 2.3 删除 Bitmap 索引

```sql
DROP INDEX idx_emp_gender;
```

## 3. Bitmap 索引适用场景

### 3.1 ✅ 适合场景

#### （1）低基数列

```sql
-- 性别（只有 2 个值）
CREATE BITMAP INDEX idx_gender ON employees(gender);

-- 状态标志
CREATE BITMAP INDEX idx_status ON orders(status);
-- 状态值：'PENDING', 'PROCESSING', 'SHIPPED', 'DELIVERED'

-- 类型分类
CREATE BITMAP INDEX idx_category ON products(category);
-- 类别数：< 20
```

#### （2）数据仓库环境

- 读多写少
- 批量加载数据
- 复杂查询和聚合

#### （3）多个低基数列的组合查询

```sql
-- 单独每个列选择性都低，但组合起来选择性高
SELECT * FROM employees
WHERE gender = 'F'
AND status = 'ACTIVE'
AND dept_id = 10;

-- 为这些列创建 Bitmap 索引
CREATE BITMAP INDEX idx_gender ON employees(gender);
CREATE BITMAP INDEX idx_status ON employees(status);
CREATE BITMAP INDEX idx_dept ON employees(dept_id);

-- Oracle 可以使用 Bitmap Merge 高效处理
```

#### （4）COUNT 操作

```sql
-- Bitmap 索引对 COUNT 非常高效
SELECT COUNT(*) FROM employees WHERE gender = 'F';
```

### 3.2 ❌ 不适合场景

#### （1）OLTP 系统

- 频繁的 INSERT/UPDATE/DELETE
- Bitmap 索引锁粒度大（整个位图段）
- 并发性能差

#### （2）高基数列

```sql
-- ❌ 不要这样做
CREATE BITMAP INDEX idx_emp_id ON employees(employee_id);
-- employee_id 有几千个不同值，位图会很大
```

#### （3）唯一或接近唯一的列

- 使用 B-Tree 索引更好

## 4. Bitmap 索引的优势

### 4.1 空间效率

对于低基数列，Bitmap 索引比 B-Tree 索引小得多。

```sql
-- 示例：100万行，gender 列（2个值）
-- B-Tree 索引：约 20 MB
-- Bitmap 索引：约 0.5 MB
```

### 4.2 快速位运算

Oracle 可以对位图进行快速的 AND、OR、NOT 运算。

```sql
-- 这个查询非常快
SELECT * FROM employees
WHERE gender = 'F'      -- Bitmap 1
AND status = 'ACTIVE'   -- Bitmap 2
AND dept_id IN (10, 20); -- Bitmap 3

-- Oracle 执行：Bitmap 1 AND Bitmap 2 AND (Bitmap 4 OR Bitmap 5)
```

### 4.3 高效的 COUNT

```sql
-- Bitmap 索引可以快速计算满足条件的行数
SELECT COUNT(*) FROM employees WHERE gender = 'F';
-- 只需统计位图中 1 的个数，无需访问表
```

---

## 5. Bitmap 索引的劣势

### 5.1 并发问题

**问题**：Bitmap 索引锁粒度大

```sql
-- Session 1
UPDATE employees SET gender = 'M' WHERE employee_id = 1;
-- 锁定整个 "Male" 位图段

-- Session 2
UPDATE employees SET gender = 'M' WHERE employee_id = 2;
-- 等待 Session 1 释放锁（即使不是同一行）
```

**影响**：
- OLTP 系统并发性能差
- 不适合频繁更新的列

### 5.2 维护成本高

- INSERT/UPDATE/DELETE 需要更新位图
- 批量操作时可以接受
- 单行操作开销大

### 5.3 不适用于唯一约束

- Bitmap 索引不能用于主键或唯一约束
- 必须使用 B-Tree 索引

## 6. 位图连接索引（Bitmap Join Index）

### 6.1 概念

位图连接索引在事实表和维度表之间创建，预计算 JOIN 结果。

### 6.2 创建位图连接索引

```sql
-- 假设有一个销售事实表和地区维度表
CREATE BITMAP INDEX idx_sales_region
ON sales(orders.region)
FROM sales, orders, regions
WHERE sales.order_id = orders.order_id
AND orders.region_id = regions.region_id;

-- 语法说明：
-- ON sales(orders.region)：在 sales 表上创建，使用 orders.region 列
-- FROM：指定参与的表
-- WHERE：指定 JOIN 条件
```

### 6.3 使用场景

```sql
-- 查询某个地区的销售
SELECT SUM(amount)
FROM sales s
JOIN orders o ON s.order_id = o.order_id
JOIN regions r ON o.region_id = r.region_id
WHERE r.region_name = 'North';

-- 有位图连接索引时，Oracle 可以直接使用索引，无需 JOIN
```

### 6.4 优势

✅ 避免运行时 JOIN  
✅ 大幅提高查询性能  
✅ 适合星型模式的数据仓库  

### 6.5 限制

❌ 维度表必须是主键或唯一约束  
❌ 不支持外连接  
❌ 维护成本高

## 7. Bitmap 索引转换

### 7.1 Bitmap to Rowid 转换

Oracle 在执行计划中会将 Bitmap 转换为 ROWID。

```sql
EXPLAIN PLAN FOR
SELECT * FROM employees
WHERE gender = 'F' AND status = 'ACTIVE';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**执行计划可能显示**：
```
BITMAP CONVERSION TO ROWIDS
  BITMAP AND
    BITMAP INDEX SINGLE VALUE IDX_GENDER
    BITMAP INDEX SINGLE VALUE IDX_STATUS
TABLE ACCESS BY USER ROWID EMPLOYEES
```

### 7.2 Bitmap Merge

合并多个位图。

```sql
-- OR 条件
SELECT * FROM employees
WHERE dept_id = 10 OR dept_id = 20;

-- 执行计划
-- BITMAP MERGE
--   BITMAP INDEX SINGLE VALUE IDX_DEPT (10)
--   BITMAP INDEX SINGLE VALUE IDX_DEPT (20)
```

## 8. 监控和优化

### 8.1 检查 Bitmap 索引使用情况

```sql
-- 启用监控
ALTER INDEX idx_gender MONITORING USAGE;

-- 查看使用情况
SELECT * FROM v$object_usage;
```

### 8.2 分析 Bitmap 索引效率

```sql
SELECT 
    index_name,
    num_rows,
    leaf_blocks,
    distinct_keys,
    ROUND(leaf_blocks / NULLIF(distinct_keys, 0), 2) as blocks_per_key
FROM user_indexes
WHERE index_type = 'BITMAP';
```

### 8.3 重建 Bitmap 索引

```sql
-- 定期重建以优化性能
ALTER INDEX idx_gender REBUILD;

-- 在线重建（不影响查询）
ALTER INDEX idx_gender REBUILD ONLINE;
```

## 9. 实战案例

### 案例 1：数据仓库优化

```sql
-- 原始表
CREATE TABLE sales_fact (
    sale_id NUMBER,
    product_id NUMBER,
    customer_id NUMBER,
    sale_date DATE,
    amount NUMBER,
    channel VARCHAR2(20),    -- 在线、门店、电话
    region VARCHAR2(20),     -- 华北、华东、华南...
    payment_method VARCHAR2(20)  -- 现金、信用卡、支付宝...
);

-- 创建 Bitmap 索引
CREATE BITMAP INDEX idx_channel ON sales_fact(channel);
CREATE BITMAP INDEX idx_region ON sales_fact(region);
CREATE BITMAP INDEX idx_payment ON sales_fact(payment_method);

-- 复杂查询（非常快）
SELECT 
    region,
    channel,
    SUM(amount) as total_sales
FROM sales_fact
WHERE sale_date >= DATE '2026-01-01'
AND payment_method IN ('信用卡', '支付宝')
GROUP BY region, channel;
```

### 案例 2：避免并发问题

```sql
-- OLTP 系统：不要使用 Bitmap 索引
-- ❌ 错误做法
CREATE BITMAP INDEX idx_order_status ON orders(status);

-- ✅ 正确做法：使用 B-Tree 索引
CREATE INDEX idx_order_status ON orders(status);

-- 如果是只读报表库，可以使用 Bitmap
-- 在报表库创建
CREATE BITMAP INDEX idx_order_status_bitmap ON orders(status);
```

## 10. 最佳实践

### 10.1 设计原则

✅ **只在数据仓库使用**  
✅ **只用于低基数列（< 100 个不同值）**  
✅ **避免在频繁更新的列上使用**  
✅ **考虑位图连接索引优化 JOIN**  
✅ **定期重建索引**  

### 10.2 选择指南

| 场景 | 推荐索引类型 |
|------|------------|
| 主键、唯一列 | B-Tree |
| 高基数列 | B-Tree |
| 低基数列 + OLTP | B-Tree |
| 低基数列 + 数据仓库 | Bitmap |
| 多列组合查询 + DW | Bitmap |
| 星型模式 JOIN | Bitmap Join Index |

### 10.3 注意事项

⚠️ Bitmap 索引不能用于唯一约束  
⚠️ OLTP 系统避免使用  
⚠️ 注意并发锁问题  
⚠️ 批量加载后重建索引  
⚠️ 监控索引使用情况

## 11. 总结

### 11.1 核心知识点

✅ **Bitmap 结构**：位图表示存在性  
✅ **适用场景**：低基数列、数据仓库  
✅ **优势**：空间小、位运算快、COUNT 高效  
✅ **劣势**：并发差、维护成本高  
✅ **位图连接索引**：预计算 JOIN  

### 11.2 关键决策

📌 OLTP → B-Tree  
📌 数据仓库 + 低基数 → Bitmap  
📌 星型模式 → Bitmap Join Index  
📌 频繁更新 → 避免 Bitmap
