## 1. 死锁概述

### 1.1 什么是死锁？

死锁是两个或多个事务互相等待对方释放锁，导致所有事务都无法继续执行。

**经典场景**：
```
事务 A：锁定资源 1，请求资源 2
事务 B：锁定资源 2，请求资源 1

结果：两个事务永远等待
```

### 1.2 死锁的四个必要条件

| 条件 | 说明 |
|------|------|
| **互斥** | 资源一次只能被一个事务持有 |
| **持有并等待** | 事务持有资源的同时等待其他资源 |
| **不可剥夺** | 已分配的资源不能被强制收回 |
| **循环等待** | 存在循环等待链 |

**破坏任一条件即可避免死锁**。

## 2. Oracle 死锁检测

### 2.1 自动检测机制

Oracle 后台进程 PMON 定期检测死锁。

**检测频率**：
- 默认每 3 秒检测一次
- 发现死锁后立即处理

### 2.2 死锁处理方式

**牺牲者选择**：
- Oracle 选择其中一个事务作为"牺牲者"
- 回滚该事务的部分语句
- 返回 ORA-00060 错误
- 其他事务继续执行

```sql
-- 牺牲者会话收到错误
ORA-00060: deadlock detected while waiting for resource
```

### 2.3 死锁日志

**Alert Log**：
```
*** DEADLOCK DETECTED ***
Current SQL statement for this session:
UPDATE employees SET salary = 10000 WHERE employee_id = 100
```

**Trace File**：
- 包含详细的死锁图
- 显示所有涉及的会话和资源
- 位置：`$ORACLE_BASE/diag/rdbms/<db>/<instance>/trace/`

## 3. 死锁场景示例

### 3.1 典型死锁场景

#### 场景 1：交叉更新

```sql
-- Session A
UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A';
-- 锁定账户 A

-- Session B
UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'B';
-- 锁定账户 B

-- Session A（尝试锁定 B）
UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'B';
-- 等待 Session B 释放

-- Session B（尝试锁定 A）
UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'A';
-- 等待 Session A 释放

-- 死锁！Oracle 检测到并回滚其中一个
```

#### 场景 2：外键未索引

```sql
-- 父表
CREATE TABLE departments (
    dept_id NUMBER PRIMARY KEY,
    dept_name VARCHAR2(50)
);

-- 子表（外键未建索引）
CREATE TABLE employees (
    emp_id NUMBER PRIMARY KEY,
    dept_id NUMBER REFERENCES departments(dept_id),
    emp_name VARCHAR2(50)
);

-- Session A
DELETE FROM departments WHERE dept_id = 10;
-- 需要检查子表，获取子表共享锁

-- Session B
INSERT INTO employees VALUES (1, 20, 'John');
-- 需要检查父表，获取父表共享锁

-- 如果同时发生，可能死锁
```

**解决**：为外键列创建索引
```sql
CREATE INDEX idx_emp_dept ON employees(dept_id);
```

#### 场景 3：位图索引竞争

```sql
-- Bitmap 索引锁粒度大
CREATE BITMAP INDEX idx_gender ON employees(gender);

-- Session A
UPDATE employees SET gender = 'M' WHERE emp_id = 1;
-- 锁定整个 "Male" 位图段

-- Session B
UPDATE employees SET gender = 'M' WHERE emp_id = 2;
-- 也尝试锁定 "Male" 位图段

-- 虽然不是真正的死锁，但会导致严重阻塞
```

**解决**：OLTP 系统避免使用位图索引

### 3.2 ITL 不足导致的死锁

ITL（Interested Transaction List）是数据块中记录事务信息的区域。

**问题**：
- ITL 槽位不足
- 多个事务竞争同一块
- 可能导致死锁-like 行为

**解决**：
```sql
-- 增加表的 INITRANS
ALTER TABLE employees INITRANS 10;

-- 或重建表
CREATE TABLE employees_new INITRANS 10 AS SELECT * FROM employees;
```

## 4. 诊断死锁

### 4.1 查看 Alert Log

```bash
# 查找死锁记录
grep -i "deadlock" $ORACLE_BASE/diag/rdbms/*/alert*.log

# 查看最近的 trace 文件
ls -lt $ORACLE_BASE/diag/rdbms/*/trace/*.trc | head -5
```

### 4.2 分析 Trace 文件

```
*** 2026-05-19 10:30:00.123
DEADLOCK DETECTED

Current SQL statement for this session:
UPDATE accounts SET balance = :1 WHERE account_id = :2

The following deadlock is not an ORACLE error. It is a
deadlock due to user error in the design of an application
or from issuing incorrect ad-hoc SQL. The following
information may aid in determining the deadlock:

Deadlock graph:
---------Blocker(s)--------  ---------Waiter(s)---------
Resource Name          process session holds waits  process session holds waits
TX-000a000b-00001234       123     456     X             789     101           X
TX-000c000d-00005678       789     101     X             123     456           X

session 456: DID 0001-007B-00000001
session 101: DID 0001-01F5-00000001
```

**关键信息**：
- 涉及的会话 ID
- 持有的锁和等待的锁
- 执行的 SQL 语句

### 4.3 实时监控死锁

```sql
-- 查看当前等待的锁
SELECT 
    s1.username || '@' || s1.machine || ' (SID=' || s1.sid || ') is blocking ' ||
    s2.username || '@' || s2.machine || ' (SID=' || s2.sid || ')' AS blocking_status
FROM v$lock l1, v$session s1, v$lock l2, v$session s2
WHERE s1.sid = l1.sid
AND s2.sid = l2.sid
AND l1.BLOCK = 1
AND l2.request > 0
AND l1.id1 = l2.id1
AND l2.id2 = l2.id2;

-- 查看锁等待历史（AWR）
SELECT 
    snap_id,
    event,
    total_waits,
    time_waited
FROM dba_hist_system_event
WHERE event = 'enq: TX - row lock contention'
ORDER BY snap_id DESC;
```

## 5. 预防死锁

### 5.1 设计层面

✅ **按相同顺序访问资源**
```sql
-- ✅ 好：所有事务都先访问 A，再访问 B
UPDATE accounts SET balance = ... WHERE account_id = 'A';
UPDATE accounts SET balance = ... WHERE account_id = 'B';

-- ❌ 坏：不同事务访问顺序不同
-- 事务 1：A → B
-- 事务 2：B → A
```

✅ **保持事务简短**
```sql
-- ✅ 快速完成
BEGIN
    UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A';
    UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'B';
    COMMIT;
END;

-- ❌ 避免在事务中进行用户交互
BEGIN
    UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A';
    -- 不要在这里等待用户输入
    UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'B';
    COMMIT;
END;
```

✅ **为外键创建索引**
```sql
-- 所有外键都应该有索引
CREATE INDEX idx_emp_dept ON employees(dept_id);
CREATE INDEX idx_order_customer ON orders(customer_id);
```

### 5.2 代码层面

#### （1）使用 NOWAIT 或 WAIT

```sql
-- 立即返回，不等待
BEGIN
    SELECT * FROM accounts WHERE account_id = 'A' FOR UPDATE NOWAIT;
EXCEPTION
    WHEN OTHERS THEN
        IF SQLCODE = -54 THEN
            -- 资源忙，稍后重试
            DBMS_LOCK.SLEEP(1);
            -- 重试逻辑
        END IF;
END;

-- 等待指定时间
SELECT * FROM accounts WHERE account_id = 'A' FOR UPDATE WAIT 5;
```

#### （2）重试机制

```plsql
DECLARE
    l_retries NUMBER := 3;
    l_deadlock_detected BOOLEAN := FALSE;
BEGIN
    WHILE l_retries > 0 LOOP
        BEGIN
            -- 业务逻辑
            UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A';
            UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'B';
            COMMIT;
            
            l_deadlock_detected := FALSE;
            EXIT;
            
        EXCEPTION
            WHEN OTHERS THEN
                IF SQLCODE = -60 THEN  -- ORA-00060 死锁
                    ROLLBACK;
                    l_retries := l_retries - 1;
                    l_deadlock_detected := TRUE;
                    
                    IF l_retries > 0 THEN
                        DBMS_LOCK.SLEEP(1);  -- 等待 1 秒后重试
                    ELSE
                        RAISE;  -- 重试次数用完，抛出异常
                    END IF;
                ELSE
                    RAISE;  -- 其他错误，直接抛出
                END IF;
        END;
    END LOOP;
END;
```

#### （3）锁定顺序标准化

```plsql
-- 定义标准的锁定顺序
PROCEDURE transfer_money(
    p_from_account VARCHAR2,
    p_to_account VARCHAR2,
    p_amount NUMBER
) IS
    l_first_account VARCHAR2(20);
    l_second_account VARCHAR2(20);
BEGIN
    -- 按字母顺序锁定，确保一致性
    IF p_from_account < p_to_account THEN
        l_first_account := p_from_account;
        l_second_account := p_to_account;
    ELSE
        l_first_account := p_to_account;
        l_second_account := p_from_account;
    END IF;
    
    -- 按固定顺序锁定
    UPDATE accounts SET balance = balance WHERE account_id = l_first_account FOR UPDATE;
    UPDATE accounts SET balance = balance WHERE account_id = l_second_account FOR UPDATE;
    
    -- 执行转账
    UPDATE accounts SET balance = balance - p_amount WHERE account_id = p_from_account;
    UPDATE accounts SET balance = balance + p_amount WHERE account_id = p_to_account;
    
    COMMIT;
END;
```

### 5.3 数据库层面

✅ **合理设置超时**
```sql
-- DDL 锁超时
ALTER SYSTEM SET DDL_LOCK_TIMEOUT = 30;

-- 分布式锁超时
ALTER SYSTEM SET DISTRIBUTED_LOCK_TIMEOUT = 60;
```

✅ **监控和告警**
```sql
-- 定期检查死锁
SELECT 
    TO_CHAR(snapshot_time, 'YYYY-MM-DD HH24:MI') as time,
    deadlocks_detected
FROM dba_hist_deadlock
ORDER BY snapshot_time DESC;
```

## 6. 处理死锁

### 6.1 立即响应

**步骤**：
1. 识别被回滚的事务
2. 通知应用用户
3. 记录死锁信息
4. 分析根本原因

### 6.2 根本原因分析

**检查清单**：
- [ ] 是否存在交叉更新？
- [ ] 外键是否有索引？
- [ ] 事务是否过长？
- [ ] 是否使用了位图索引？
- [ ] ITL 是否充足？
- [ ] 应用是否有重试机制？

### 6.3 长期解决方案

**方案 1：重新设计事务**
- 减少事务范围
- 统一访问顺序
- 批量操作改为逐行

**方案 2：优化索引**
- 为外键添加索引
- 移除不必要的位图索引
- 调整复合索引顺序

**方案 3：应用层改进**
- 实现重试逻辑
- 使用乐观锁
- 队列化处理

## 7. 最佳实践

### 7.1 开发规范

📌 **统一资源访问顺序**  
📌 **保持事务简短**  
📌 **外键必须建索引**  
📌 **实现死锁重试机制**  
📌 **避免长事务中的用户交互**  

### 7.2 监控策略

```sql
-- 每日检查死锁
SELECT 
    TO_CHAR(begin_interval_time, 'YYYY-MM-DD') as date,
    COUNT(*) as deadlock_count
FROM dba_hist_deadlock d, dba_hist_snapshot s
WHERE d.snap_id = s.snap_id
GROUP BY TO_CHAR(begin_interval_time, 'YYYY-MM-DD')
ORDER BY date DESC;

-- 告警阈值：每天超过 5 次死锁需要调查
```

### 7.3 测试建议

✅ **并发测试**：模拟高并发场景  
✅ **压力测试**：找出潜在死锁点  
✅ **代码审查**：检查事务设计  
✅ **性能测试**：验证锁竞争情况

## 8. 总结

### 8.1 核心知识点

✅ **死锁条件**：互斥、持有并等待、不可剥夺、循环等待  
✅ **Oracle 检测**：PMON 自动检测，牺牲者回滚  
✅ **常见场景**：交叉更新、外键未索引、位图索引  
✅ **诊断方法**：Alert Log、Trace File、动态视图  
✅ **预防策略**：统一顺序、短事务、索引优化  

### 8.2 关键要点

📌 死锁是应用设计问题，不是 Oracle bug  
📌 预防胜于治疗  
📌 实现重试机制  
📌 持续监控和分析  
📌 定期代码审查
