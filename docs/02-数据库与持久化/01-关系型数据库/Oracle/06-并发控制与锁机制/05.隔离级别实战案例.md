## 1. 隔离级别回顾

### 1.1 SQL 标准隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| **READ UNCOMMITTED** | ✅ | ✅ | ✅ |
| **READ COMMITTED** | ❌ | ✅ | ✅ |
| **REPEATABLE READ** | ❌ | ❌ | ✅ |
| **SERIALIZABLE** | ❌ | ❌ | ❌ |

### 1.2 Oracle 支持的隔离级别

⚠️ **Oracle 不支持 READ UNCOMMITTED 和 REPEATABLE READ**

| Oracle 隔离级别 | 对应标准 | 特点 |
|----------------|---------|------|
| **READ COMMITTED** | READ COMMITTED | 默认，看到已提交数据 |
| **SERIALIZABLE** | SERIALIZABLE | 事务级快照 |
| **READ ONLY** | 扩展 | 只读事务级快照 |

## 2. 现象演示

### 2.1 脏读（Dirty Read）

**定义**：读取未提交的数据。

**Oracle 行为**：❌ **不会发生**（即使 READ UNCOMMITTED 也不支持）

```sql
-- Session A
UPDATE accounts SET balance = 9000 WHERE account_id = 'A';
-- 未提交

-- Session B（尝试读取）
SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 10000（旧值），不会看到未提交的 9000

-- 即使设置最低隔离级别
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
SELECT balance FROM accounts WHERE account_id = 'A';
-- 仍然返回 10000
```

**结论**：Oracle 永远不会发生脏读（MVCC 保证）。

### 2.2 不可重复读（Non-Repeatable Read）

**定义**：同一事务中多次查询返回不同结果。

**Oracle 行为**：✅ **可能发生**（在 READ COMMITTED 级别）

#### 演示

```sql
-- Session A
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 第一次查询
SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 10000

-- （此时 Session B 执行）
-- Session B
UPDATE accounts SET balance = 9000 WHERE account_id = 'A';
COMMIT;

-- Session A（第二次查询）
SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 9000（看到了 Session B 的提交）
```

**影响**：
- 报表数据不一致
- 业务逻辑错误

**解决**：使用 SERIALIZABLE 或 READ ONLY

```sql
-- Session A
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 10000

-- Session B 提交后

SELECT balance FROM accounts WHERE account_id = 'A';
-- 仍然返回 10000（事务开始时的快照）
```

### 2.3 幻读（Phantom Read）

**定义**：同一查询条件，多次查询返回的行数不同。

**Oracle 行为**：✅ **可能发生**（在 READ COMMITTED 级别）

#### 演示

```sql
-- Session A
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 第一次查询
SELECT COUNT(*) FROM employees WHERE dept_id = 10;
-- 返回 5

-- （此时 Session B 执行）
-- Session B
INSERT INTO employees VALUES (100, 'New Employee', 10, ...);
COMMIT;

-- Session A（第二次查询）
SELECT COUNT(*) FROM employees WHERE dept_id = 10;
-- 返回 6（看到了新插入的行）
```

**解决**：使用 SERIALIZABLE

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

SELECT COUNT(*) FROM employees WHERE dept_id = 10;
-- 返回 5

-- Session B 提交后

SELECT COUNT(*) FROM employees WHERE dept_id = 10;
-- 仍然返回 5
```

## 3. 实战案例

### 案例 1：银行转账（需要强一致性）

**场景**：生成账户余额报表，要求数据一致。

#### 问题：使用 READ COMMITTED

```sql
-- 报表事务
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 查询账户 A
SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 10000

-- （此时发生转账）
-- 其他会话
UPDATE accounts SET balance = 9000 WHERE account_id = 'A';
UPDATE accounts SET balance = 11000 WHERE account_id = 'B';
COMMIT;

-- 查询账户 B
SELECT balance FROM accounts WHERE account_id = 'B';
-- 返回 11000

-- 报表结果：A=10000, B=11000，总和 21000
-- 但实际总和应该是 20000！数据不一致！
```

#### 解决：使用 SERIALIZABLE

```sql
-- 报表事务
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 查询账户 A
SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 10000

-- （此时发生转账，但不影响当前事务）

-- 查询账户 B
SELECT balance FROM accounts WHERE account_id = 'B';
-- 返回 10000

-- 报表结果：A=10000, B=10000，总和 20000 ✓
```

### 案例 2：库存扣减（需要最新数据）

**场景**：电商下单，需要检查实时库存。

#### 推荐：使用 READ COMMITTED

```sql
-- 下单事务
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;  -- 默认

-- 检查库存
SELECT quantity FROM products WHERE product_id = 100 FOR UPDATE;
-- 返回 10

-- 扣减库存
UPDATE products SET quantity = quantity - 1 
WHERE product_id = 100 AND quantity >= 1;

-- 如果影响行数为 0，说明库存不足
IF SQL%ROWCOUNT = 0 THEN
    ROLLBACK;
    RAISE_APPLICATION_ERROR(-20001, '库存不足');
END IF;

COMMIT;
```

**为什么不用 SERIALIZABLE**：
- 需要最新库存数据
- SERIALIZABLE 可能看到过期数据
- 高并发下容易 ORA-08177

### 案例 3：数据统计（长时间运行）

**场景**：夜间批处理，统计全天销售数据。

#### 推荐：使用 READ ONLY

```sql
-- 统计事务
SET TRANSACTION READ ONLY;

-- 多个查询，保证看到同一时间点的数据
SELECT COUNT(*) as order_count FROM orders 
WHERE order_date >= TRUNC(SYSDATE);

SELECT SUM(amount) as total_sales FROM orders 
WHERE order_date >= TRUNC(SYSDATE);

SELECT customer_id, COUNT(*) as cnt FROM orders 
WHERE order_date >= TRUNC(SYSDATE)
GROUP BY customer_id;

-- 所有查询都基于事务开始时的快照
-- 即使白天有数据修改，也不影响统计结果
```

**优势**：
- 数据一致性保证
- 不会阻塞 DML
- 适合长时间运行的报表

### 案例 4：数据迁移（避免干扰）

**场景**：在线迁移数据，不能影响业务。

#### 策略：混合使用

```sql
-- 迁移脚本
DECLARE
    TYPE t_emp IS TABLE OF employees%ROWTYPE;
    l_emps t_emp;
BEGIN
    -- 分批读取（READ COMMITTED，看到最新数据）
    FOR batch IN 1..10 LOOP
        SELECT * BULK COLLECT INTO l_emps
        FROM employees
        WHERE employee_id BETWEEN (batch-1)*1000+1 AND batch*1000;
        
        -- 写入新表（独立事务）
        FORALL i IN 1..l_emps.COUNT
            INSERT INTO employees_new VALUES l_emps(i);
        
        COMMIT;  -- 每批提交
    END LOOP;
END;
```

## 4. Oracle vs 其他数据库

### 4.1 与 MySQL 对比

| 特性 | Oracle | MySQL (InnoDB) |
|------|--------|----------------|
| **默认隔离级别** | READ COMMITTED | REPEATABLE READ |
| **支持 READ UNCOMMITTED** | ❌ | ✅ |
| **支持 REPEATABLE READ** | ❌ | ✅ |
| **实现机制** | MVCC | MVCC + Gap Lock |
| **幻读防止** | SERIALIZABLE | REPEATABLE READ 即可 |

### 4.2 与 PostgreSQL 对比

| 特性 | Oracle | PostgreSQL |
|------|--------|------------|
| **默认隔离级别** | READ COMMITTED | READ COMMITTED |
| **MVCC 实现** | UNDO | 多版本存储 |
| **VACUUM 需求** | 自动清理 | 需要手动/自动 VACUUM |
| **长事务影响** | UNDO 空间 | 表膨胀 |

### 4.3 迁移注意事项

**从 MySQL 迁移到 Oracle**：
```sql
-- MySQL 依赖 REPEATABLE READ 的代码
-- 在 Oracle 中需要改为 SERIALIZABLE

-- MySQL
START TRANSACTION;  -- 默认 REPEATABLE READ
SELECT ...;
SELECT ...;  -- 保证看到相同数据
COMMIT;

-- Oracle
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
SELECT ...;
SELECT ...;
COMMIT;
```

**从 Oracle 迁移到 MySQL**：
```sql
-- Oracle 的 READ COMMITTED 代码
-- 在 MySQL 中可能需要调整

-- Oracle（默认）
SELECT ... FOR UPDATE;

-- MySQL（REPEATABLE READ 可能有 Gap Lock）
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
SELECT ... FOR UPDATE;
```

## 5. 常见问题

### 5.1 ORA-08177：无法串行化访问

**原因**：SERIALIZABLE 事务尝试修改其他事务已提交的数据。

**解决**：
```sql
-- 方案 1：重试
BEGIN
    SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
    -- 业务逻辑
    COMMIT;
EXCEPTION
    WHEN OTHERS THEN
        IF SQLCODE = -8177 THEN
            ROLLBACK;
            -- 重试
        ELSE
            RAISE;
        END IF;
END;

-- 方案 2：改用 READ COMMITTED
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### 5.2 报表数据不一致

**问题**：READ COMMITTED 导致报表各部分数据不一致。

**解决**：
```sql
-- 使用 READ ONLY 事务
SET TRANSACTION READ ONLY;

-- 所有查询基于同一快照
SELECT ...;
SELECT ...;
SELECT ...;
```

### 5.3 需要最新数据但又要一致性

**问题**：矛盾的需求。

**解决**：
```sql
-- 方案 1：短时间窗口
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- 快速完成查询
SELECT ...;
COMMIT;

-- 方案 2：应用层缓存
-- 第一次查询后缓存结果
-- 短时间内使用缓存数据

-- 方案 3：物化视图
CREATE MATERIALIZED VIEW mv_sales_summary
REFRESH FAST ON COMMIT
AS SELECT ...;
```

## 6. 最佳实践

### 6.1 选择指南

| 场景 | 推荐隔离级别 |
|------|------------|
| **OLTP（大多数）** | READ COMMITTED（默认） |
| **财务报表** | SERIALIZABLE 或 READ ONLY |
| **数据分析** | READ ONLY |
| **批量处理** | READ COMMITTED + 分批 |
| **数据迁移** | READ COMMITTED |
| **强一致性要求** | SERIALIZABLE |

### 6.2 开发规范

📌 **默认使用 READ COMMITTED**  
📌 **报表使用 READ ONLY**  
📌 **避免长时间 SERIALIZABLE 事务**  
📌 **实现 ORA-08177 重试逻辑**  
📌 **明确事务边界**  

### 6.3 性能考虑

✅ **READ COMMITTED**：最高并发  
✅ **SERIALIZABLE**：可能 ORA-08177，需要重试  
✅ **READ ONLY**：不阻塞 DML，适合报表  
❌ **避免不必要的高隔离级别**  

---

## 7. 总结

### 7.1 核心知识点

✅ **Oracle 不支持脏读**：MVCC 保证  
✅ **READ COMMITTED**：可能不可重复读和幻读  
✅ **SERIALIZABLE**：事务级快照，可能 ORA-08177  
✅ **READ ONLY**：只读事务级快照  
✅ **根据场景选择**：没有银弹  

### 7.2 决策要点

📌 需要最新数据 → READ COMMITTED  
📌 需要一致性 → SERIALIZABLE / READ ONLY  
📌 长时间报表 → READ ONLY  
📌 高并发 OLTP → READ COMMITTED  
📌 财务系统 → SERIALIZABLE
