## 1. MVCC 概述

### 1.1 什么是 MVCC？

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 Oracle 实现高并发的核心机制。

**核心理念**：
- **读者不阻塞写者**
- **写者不阻塞读者**
- 通过维护数据的多个版本实现

<!-- TODO: 插入图片：MVCC 工作原理图 -->

### 1.2 传统锁机制 vs MVCC

| 特性 | 传统锁机制 | Oracle MVCC |
|------|-----------|-------------|
| **读-写冲突** | 读等待写 | 读不等待写 |
| **写-读冲突** | 写等待读 | 写不等待读 |
| **实现方式** | 锁 | 多版本 + UNDO |
| **并发性能** | 较低 | 高 |
| **存储开销** | 小 | 需要 UNDO 空间 |

### 1.3 MVCC 的优势

✅ **高并发**：读写互不阻塞  
✅ **读一致性**：查询看到一致的数据快照  
✅ **简化编程**：无需担心脏读  
✅ **可扩展性**：支持大量并发用户

## 2. MVCC 工作原理

### 2.1 数据版本管理

```
时间点 T1：行数据 VERSION 1 (balance = 10000)
时间点 T2：事务 A 修改 → VERSION 2 (balance = 9000)
时间点 T3：事务 B 查询

结果：
- 事务 B 看到 VERSION 1（T1 时刻的版本）
- 通过 UNDO 数据重构历史版本
```

### 2.2 关键组件

#### （1）SCN（System Change Number）

Oracle 的内部时钟，标记每个事务的时间点。

```sql
-- 查看当前 SCN
SELECT CURRENT_SCN FROM v$database;

-- 查看事务的 SCN
SELECT 
    xidusn,
    xidslot,
    xidsqn,
    start_scn,
    commit_scn
FROM v$transaction;
```

#### （2）UNDO 表空间

存储数据的"前镜像"（修改前的值）。

```sql
-- 查看 UNDO 表空间
SELECT tablespace_name, status, contents
FROM dba_tablespaces
WHERE contents = 'UNDO';

-- 查看 UNDO 使用情况
SELECT 
    tablespace_name,
    ROUND(SUM(bytes)/1024/1024, 2) as size_mb
FROM dba_data_files
WHERE tablespace_name IN (
    SELECT value FROM v$parameter WHERE name = 'undo_tablespace'
)
GROUP BY tablespace_name;
```

#### （3）ITL（Interested Transaction List）

数据块头部的事务槽，记录哪些事务修改了该块。

```
Block Header:
  ITL Slot 1: XID = 0x000a.00b.00001234, LCK = 1
  ITL Slot 2: XID = 0x000c.00d.00005678, LCK = 1
  ...
```

### 2.3 一致性读流程

```
用户发起查询（SCN = 1000）
    ↓
读取数据块
    ↓
检查块的 SCN
    ↓
如果块 SCN > 查询 SCN
    ↓
从 UNDO 中获取旧版本
    ↓
重构 SCN = 1000 时的数据
    ↓
返回给用户
```

## 3. UNDO 管理机制

### 3.1 UNDO 数据结构

```
UNDO Segment
    ├── UNDO Extent
    │   ├── UNDO Block
    │   │   ├── UNDO Record 1（事务 1 的前镜像）
    │   │   ├── UNDO Record 2（事务 2 的前镜像）
    │   │   └── ...
    │   └── ...
    └── ...
```

### 3.2 UNDO 保留时间

```sql
-- 查看 UNDO 保留时间（秒）
SHOW PARAMETER undo_retention;

-- 默认 900 秒（15 分钟）
-- 建议设置为最长查询时间的 2-3 倍
ALTER SYSTEM SET undo_retention = 3600;  -- 1 小时
```

### 3.3 UNDO 表空间管理

#### （1）自动扩展

```sql
-- 确保 UNDO 表空间可以自动扩展
ALTER DATABASE DATAFILE '/u01/oradata/undotbs01.dbf'
AUTOEXTEND ON NEXT 100M MAXSIZE 10G;
```

#### （2）GUARANTEE 模式

```sql
-- 强制保留 UNDO，即使空间不足也不覆盖
ALTER TABLESPACE undotbs1 RETENTION GUARANTEE;

-- 检查状态
SELECT tablespace_name, retention
FROM dba_tablespaces
WHERE contents = 'UNDO';
```

**注意**：GUARANTEE 可能导致 DML 失败（空间不足时）

#### （3）监控 UNDO 使用

```sql
-- 查看 UNDO 段使用
SELECT 
    segment_name,
    tablespace_name,
    status,
    extents,
    blocks
FROM dba_undo_extents
WHERE status = 'ACTIVE';

-- 查看 UNDO 生成速率
SELECT 
    TO_CHAR(begin_time, 'HH24:MI') as time,
    undoblks,
    txncount,
    maxquerylen,
    ssolderrcnt  -- ORA-01555 错误次数
FROM v$undostat
ORDER BY begin_time DESC;
```

## 4. ORA-01555：快照过旧

### 4.1 错误原因

**ORA-01555: snapshot too old**

当查询需要的 UNDO 数据被覆盖时发生。

**场景**：
```
时间点 T1：查询 Q1 开始（需要 SCN = 1000 的数据）
时间点 T2：事务 T1 修改数据，生成 UNDO
时间点 T3：事务 T1 提交，UNDO 标记为可重用
时间点 T4：其他事务覆盖 UNDO
时间点 T5：查询 Q1 尝试访问被覆盖的 UNDO
    ↓
ORA-01555 错误
```

### 4.2 常见原因

❌ **UNDO 表空间太小**  
❌ **UNDO_RETENTION 设置过短**  
❌ **长时间运行的查询**  
❌ **频繁的 DML 操作**  
❌ **LOB 列未使用 PCTVERSION**  

### 4.3 解决方案

#### 方案 1：增加 UNDO 表空间

```sql
-- 添加数据文件
ALTER TABLESPACE undotbs1 
ADD DATAFILE '/u01/oradata/undotbs02.dbf' SIZE 5G AUTOEXTEND ON;

-- 扩展现有文件
ALTER DATABASE DATAFILE '/u01/oradata/undotbs01.dbf' RESIZE 10G;
```

#### 方案 2：增加 UNDO_RETENTION

```sql
ALTER SYSTEM SET undo_retention = 7200;  -- 2 小时
```

#### 方案 3：使用 GUARANTEE

```sql
ALTER TABLESPACE undotbs1 RETENTION GUARANTEE;
```

#### 方案 4：优化长查询

```sql
-- 分解大查询为小查询
-- 原查询：处理 1000 万行
SELECT * FROM large_table WHERE ...;

-- 改为分批处理
FOR i IN 1..100 LOOP
    SELECT * FROM large_table 
    WHERE id BETWEEN (i-1)*100000+1 AND i*100000;
END LOOP;
```

#### 方案 5：LOB 列优化

```sql
-- LOB 列使用 PCTVERSION 或 SECUREFILE
CREATE TABLE documents (
    doc_id NUMBER,
    content CLOB
) LOB (content) STORE AS SECUREFILE (
    RETENTION MIN 3600
);
```

### 4.4 诊断脚本

```sql
-- 检查 ORA-01555 历史
SELECT 
    TO_CHAR(begin_time, 'YYYY-MM-DD HH24:MI') as time,
    maxquerylen,  -- 最长查询时间（秒）
    ssolderrcnt,  -- ORA-01555 错误次数
    nospaceerrcnt -- 空间不足错误次数
FROM v$undostat
WHERE ssolderrcnt > 0
ORDER BY begin_time DESC;

-- 查找长运行查询
SELECT 
    sid,
    serial#,
    username,
    sql_id,
    elapsed_time/1000000 as elapsed_seconds
FROM v$session_longops
WHERE time_remaining > 0
ORDER BY elapsed_seconds DESC;
```

## 5. 一致性读场景

### 5.1 语句级读一致性

每条 SQL 语句看到的是执行开始时的数据快照。

```sql
-- Session A
SELECT COUNT(*) FROM employees;
-- 返回 100

-- Session B（同时执行）
INSERT INTO employees VALUES (...);
COMMIT;

-- Session A（再次查询）
SELECT COUNT(*) FROM employees;
-- 返回 101（看到了 Session B 的提交）
```

### 5.2 事务级读一致性

SERIALIZABLE 或 READ ONLY 事务看到事务开始时的快照。

```sql
-- Session A
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
SELECT COUNT(*) FROM employees;
-- 返回 100

-- Session B
INSERT INTO employees VALUES (...);
COMMIT;

-- Session A（再次查询）
SELECT COUNT(*) FROM employees;
-- 仍然返回 100（事务级一致性）
```

### 5.3 跨数据库链接的一致性

```sql
-- 分布式查询可能不一致
SELECT * FROM local_table l, remote_table@dblink r
WHERE l.id = r.id;

-- 本地表和远程表可能看到不同时间点的数据
```

## 6. MVCC 性能影响

### 6.1 优点

✅ **高并发**：读写不阻塞  
✅ **可扩展**：支持大量用户  
✅ **简化开发**：无需处理锁  

### 6.2 成本

⚠️ **UNDO 空间消耗**：需要存储多版本  
⚠️ **UNDO I/O**：重构历史版本需要读取 UNDO  
⚠️ **清理开销**：延迟块清除（Delayed Block Cleanout）  

### 6.3 延迟块清除

**问题**：事务提交后，数据块中的 ITL 未及时清理。

**影响**：
- 后续查询需要额外 I/O 清理 ITL
- 可能导致 ORA-01555

**解决**：
```sql
-- 定期执行全表扫描触发清除
SELECT COUNT(*) FROM large_table;

-- 或使用 DBMS_STATS 收集统计信息
EXEC DBMS_STATS.GATHER_TABLE_STATS('HR', 'EMPLOYEES');
```

## 7. 最佳实践

### 7.1 UNDO 配置

✅ **合理设置 UNDO_RETENTION**
```sql
-- 基于最长查询时间
SELECT MAX(maxquerylen) FROM v$undostat;

-- 设置为 2-3 倍
ALTER SYSTEM SET undo_retention = <max_query_time> * 3;
```

✅ **足够的 UNDO 空间**
```sql
-- 监控 UNDO 使用率
SELECT 
    tablespace_name,
    ROUND(used_percent, 2) as used_pct
FROM dba_tablespace_usage_metrics
WHERE tablespace_name LIKE '%UNDO%';

-- 保持使用率 < 80%
```

✅ **避免 GUARANTEE（除非必要）**
- GUARANTEE 可能导致 DML 失败
- 优先通过增加空间和 retention 解决

### 7.2 查询优化

📌 **避免长时间运行的查询**  
📌 **分批处理大数据集**  
📌 **使用合适的隔离级别**  
📌 **定期收集统计信息**  

### 7.3 应用设计

✅ **短事务原则**  
✅ **及时提交/回滚**  
✅ **避免在事务中进行用户交互**  
✅ **实现重试逻辑（针对 ORA-01555）**
