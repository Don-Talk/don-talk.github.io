## 1. 事务概述

### 1.1 什么是事务？

事务是数据库操作的逻辑单元，要么全部成功，要么全部失败。

**示例**：银行转账
```sql
-- 从账户 A 扣款
UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A';

-- 向账户 B 加款
UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'B';

-- 两个操作必须同时成功或同时失败
COMMIT;
```

### 1.2 事务控制语句

| 语句 | 作用 |
|------|------|
| **COMMIT** | 提交事务，使更改永久生效 |
| **ROLLBACK** | 回滚事务，撤销所有更改 |
| **SAVEPOINT** | 设置保存点，部分回滚 |
| **SET TRANSACTION** | 设置事务属性 |

## 2. ACID 特性

### 2.1 Atomicity（原子性）

**定义**：事务中的所有操作要么全部完成，要么全部不完成。

**实现机制**：
- UNDO 表空间记录变更前镜像
- 失败时通过 UNDO 回滚

```sql
-- 示例：原子性保证
BEGIN
    UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A';
    UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'B';
    COMMIT;  -- 要么都成功
EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;  -- 要么都回滚
END;
```

### 2.2 Consistency（一致性）

**定义**：事务执行前后，数据库必须保持一致状态。

**保证机制**：
- 约束检查（主键、外键、唯一、检查）
- 触发器
- 业务规则

```sql
-- 约束保证一致性
ALTER TABLE accounts ADD CONSTRAINT chk_balance 
CHECK (balance >= 0);

-- 如果违反约束，事务自动回滚
UPDATE accounts SET balance = -100 WHERE account_id = 'A';
-- ORA-02290: check constraint violated
```

### 2.3 Isolation（隔离性）

**定义**：并发事务之间互不干扰。

**实现机制**：
- 锁机制
- 多版本读一致性（MVCC）
- 隔离级别控制

```sql
-- 事务 A
UPDATE accounts SET balance = 9000 WHERE account_id = 'A';
-- 未提交

-- 事务 B（看到旧值）
SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 10000（事务 A 修改前的值）
```

### 2.4 Durability（持久性）

**定义**：事务提交后，更改永久保存，即使系统故障也不会丢失。

**实现机制**：
- Redo Log Buffer → Redo Log Files
- LGWR 进程确保日志写入磁盘
- 归档日志（ARCHIVELOG 模式）

```sql
-- 提交后，数据持久化
UPDATE accounts SET balance = 9000 WHERE account_id = 'A';
COMMIT;  -- LGWR 将 redo 写入磁盘

-- 即使此时断电，重启后数据仍然存在
```

## 3. Oracle 事务隔离级别

### 3.1 支持的隔离级别

Oracle 支持两种标准隔离级别和一种扩展：

| 隔离级别 | Oracle 名称 | 脏读 | 不可重复读 | 幻读 |
|---------|------------|------|-----------|------|
| **READ COMMITTED** | 默认 | ❌ | ✅ | ✅ |
| **SERIALIZABLE** | SERIALIZABLE | ❌ | ❌ | ❌ |
| **READ ONLY** | READ ONLY | ❌ | ❌ | ❌ |

### 3.2 READ COMMITTED（读已提交）

**Oracle 默认隔离级别**。

**特点**：
- 只能看到已提交的数据
- 同一事务中多次查询可能看到不同结果（不可重复读）
- 可能出现幻读

```sql
-- Session 1
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 10000

-- Session 2（另一个会话）
UPDATE accounts SET balance = 9000 WHERE account_id = 'A';
COMMIT;

-- Session 1（再次查询）
SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 9000（看到了 Session 2 的提交）
```

**适用场景**：
- ✅ 大多数 OLTP 应用
- ✅ 需要最新数据
- ✅ 高并发环境

### 3.3 SERIALIZABLE（可串行化）

**最高隔离级别**。

**特点**：
- 事务看到的数据是事务开始时的快照
- 不会出现不可重复读和幻读
- 可能发生 ORA-08177 错误

```sql
-- Session 1
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
SELECT balance FROM accounts WHERE account_id = 'A';
-- 返回 10000

-- Session 2
UPDATE accounts SET balance = 9000 WHERE account_id = 'A';
COMMIT;

-- Session 1（尝试更新）
UPDATE accounts SET balance = 8000 WHERE account_id = 'A';
-- ORA-08177: can't serialize access for this transaction
```

**适用场景**：
- ✅ 需要强一致性的报表
- ✅ 复杂的数据分析
- ❌ 高并发 OLTP（可能导致大量错误）

### 3.4 READ ONLY（只读事务）

**特殊隔离级别**。

**特点**：
- 只能读取数据，不能修改
- 看到事务开始时的快照
- 适合长时间运行的报表

```sql
-- 设置只读事务
SET TRANSACTION READ ONLY;

-- 可以查询
SELECT * FROM accounts;

-- 不能修改
UPDATE accounts SET balance = 0;
-- ORA-01456: may not perform insert/delete/update operation inside a READ ONLY transaction
```

## 4. 多版本读一致性（MVCC）

### 4.1 什么是 MVCC？

MVCC（Multi-Version Read Consistency）是 Oracle 实现隔离性的核心机制。

**原理**：
- 不锁定读者
- 为每行数据维护多个版本
- 通过 UNDO 数据构建历史版本

<!-- TODO: 插入图片：MVCC 工作原理图 -->

### 4.2 MVCC 工作流程

```
时间点 T1：事务 A 开始
时间点 T2：事务 B 修改行 X（未提交）
时间点 T3：事务 A 查询行 X

结果：
- 事务 A 看到行 X 在 T1 时刻的版本
- 通过 UNDO 数据重构历史版本
- 不受事务 B 影响
```

### 4.3 SCN（System Change Number）

SCN 是 Oracle 的内部时钟，用于版本控制。

```sql
-- 查看当前 SCN
SELECT CURRENT_SCN FROM v$database;

-- 查看事务的 SCN
SELECT 
    xidusn,
    xidslot,
    xidsqn,
    start_scn
FROM v$transaction;
```

### 4.4 MVCC 的优势

✅ **读写不阻塞**：读者不等待写者  
✅ **写者不阻塞读者**：写者不等待读者  
✅ **高并发性能**：减少锁竞争  
✅ **一致性读**：保证数据一致性  

### 4.5 MVCC 的成本

⚠️ **UNDO 空间消耗**：需要存储多版本  
⚠️ **UNDO 保留时间**：长查询需要更多 UNDO  
⚠️ **ORA-01555 错误**：UNDO 被覆盖

## 5. 事务管理最佳实践

### 5.1 事务设计原则

✅ **保持事务简短**
```sql
-- ❌ 避免长事务
BEGIN
    FOR i IN 1..1000000 LOOP
        INSERT INTO logs VALUES (...);
    END LOOP;
    COMMIT;
END;

-- ✅ 分批提交
BEGIN
    FOR i IN 1..1000000 LOOP
        INSERT INTO logs VALUES (...);
        IF MOD(i, 1000) = 0 THEN
            COMMIT;
        END IF;
    END LOOP;
END;
```

✅ **明确的提交/回滚**
```sql
-- ✅ 使用异常处理
BEGIN
    UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A';
    UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'B';
    COMMIT;
EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;
        RAISE;
END;
```

✅ **避免在循环中提交**
```sql
-- ❌ 频繁提交
FOR rec IN cursor LOOP
    UPDATE ...;
    COMMIT;  -- 每次循环都提交
END LOOP;

-- ✅ 批量提交
FOR rec IN cursor LOOP
    UPDATE ...;
END LOOP;
COMMIT;  -- 循环结束后提交
```

### 5.2 保存点使用

```sql
-- 设置保存点
SAVEPOINT sp1;

UPDATE accounts SET balance = 9000 WHERE account_id = 'A';

-- 如果出错，回滚到保存点
SAVEPOINT sp2;

UPDATE accounts SET balance = 8000 WHERE account_id = 'B';

-- 回滚到 sp1，sp2 的操作被撤销
ROLLBACK TO sp1;

-- 仍然可以提交 sp1 之前的操作
COMMIT;
```

### 5.3 分布式事务

```sql
-- 两阶段提交
UPDATE accounts@db1 SET balance = balance - 1000 WHERE account_id = 'A';
UPDATE accounts@db2 SET balance = balance + 1000 WHERE account_id = 'B';
COMMIT;  -- 自动两阶段提交
```

## 6. 常见问题诊断

### 6.1 查看活动事务

```sql
-- 查看所有活动事务
SELECT 
    s.sid,
    s.serial#,
    s.username,
    t.start_time,
    t.status,
    t.used_ublk * 8 / 1024 as undo_mb
FROM v$session s, v$transaction t
WHERE s.taddr = t.addr;

-- 查看事务持有的锁
SELECT 
    l.session_id,
    l.locked_mode,
    o.object_name,
    o.object_type
FROM v$locked_object l, dba_objects o
WHERE l.object_id = o.object_id;
```

### 6.2 ORA-01555：快照过旧

**原因**：UNDO 数据被覆盖，无法构建历史版本。

**解决**：
```sql
-- 增加 UNDO 保留时间
ALTER SYSTEM SET undo_retention = 3600;  -- 1 小时

-- 增加 UNDO 表空间大小
ALTER DATABASE DATAFILE 'undo01.dbf' RESIZE 10G;

-- 或使用 GUARANTEE
ALTER TABLESPACE undotbs1 RETENTION GUARANTEE;
```

### 6.3 ORA-08177：无法串行化访问

**原因**：SERIALIZABLE 事务尝试修改其他事务已提交的数据。

**解决**：
```sql
-- 方案 1：重试事务
BEGIN
    SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
    -- 执行操作
    COMMIT;
EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;
        -- 重试
END;

-- 方案 2：改用 READ COMMITTED
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### 6.4 长运行事务

```sql
-- 查找长时间运行的事务
SELECT 
    s.sid,
    s.username,
    t.start_time,
    ROUND((SYSDATE - TO_DATE(t.start_time, 'MM/DD/YY HH24:MI:SS')) * 24 * 60, 2) as minutes
FROM v$session s, v$transaction t
WHERE s.taddr = t.addr
ORDER BY t.start_time;

-- 考虑终止异常事务
ALTER SYSTEM KILL SESSION 'sid,serial#';
```

## 7. 总结

### 7.1 核心知识点

✅ **ACID 特性**：原子性、一致性、隔离性、持久性  
✅ **隔离级别**：READ COMMITTED、SERIALIZABLE、READ ONLY  
✅ **MVCC**：多版本读一致性，读写不阻塞

✅ **事务管理**：COMMIT、ROLLBACK、SAVEPOINT  

### 7.2 最佳实践

📌 保持事务简短  
📌 使用 READ COMMITTED（默认）  
📌 正确处理异常  
📌 监控长运行事务  
📌 合理设置 UNDO 参数
