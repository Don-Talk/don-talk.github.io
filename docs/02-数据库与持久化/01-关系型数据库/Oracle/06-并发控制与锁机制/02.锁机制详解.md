## 1. Oracle 锁概述

### 1.1 为什么需要锁？

锁是保证并发控制的核心机制，防止多个事务同时修改同一数据。

**目标**：
- 保证数据一致性
- 最大化并发性能
- 避免冲突

### 1.2 Oracle 锁的特点

✅ **自动管理**：Oracle 自动获取和释放锁  
✅ **多粒度**：行级、表级、数据库级  
✅ **无死锁检测开销**：轻量级实现  
✅ **读写不冲突**：MVCC 支持

## 2. 锁的模式

### 2.1 锁模式分类

| 模式 | 名称 | 缩写 | 说明 |
|------|------|------|------|
| **0** | None | - | 无锁 |
| **1** | Null | NULL | 空锁 |
| **2** | Row Share | RS | 行共享 |
| **3** | Row Exclusive | RX | 行排他 |
| **4** | Share | S | 共享 |
| **5** | Share Row Exclusive | SRX | 共享行排他 |
| **6** | Exclusive | X | 排他 |

### 2.2 锁兼容性矩阵

```
        NULL  RS   RX   S    SRX  X
NULL    ✓     ✓    ✓    ✓    ✓    ✓
RS      ✓     ✓    ✓    ✓    ✓    ✗
RX      ✓     ✓    ✓    ✗    ✗    ✗
S       ✓     ✓    ✗    ✓    ✗    ✗
SRX     ✓     ✓    ✗    ✗    ✗    ✗
X       ✓     ✗    ✗    ✗    ✗    ✗
```

**规则**：
- ✓ 表示兼容（可以同时持有）
- ✗ 表示冲突（不能同时持有）

## 3. DML 锁（数据锁）

### 3.1 行级锁（Row Lock）

**特点**：
- 只在 UPDATE/DELETE 时自动获取
- SELECT 不加锁（MVCC）
- 锁存储在数据块中，不在内存

```sql
-- Session 1：更新一行
UPDATE employees SET salary = 10000 WHERE employee_id = 100;
-- 自动获取该行的排他锁（X）

-- Session 2：尝试更新同一行
UPDATE employees SET salary = 11000 WHERE employee_id = 100;
-- 等待 Session 1 释放锁
```

**查看行锁**：
```sql
SELECT 
    l.session_id,
    l.locked_mode,
    o.object_name,
    ROW_WAIT_OBJ#,
    ROW_WAIT_FILE#,
    ROW_WAIT_BLOCK#,
    ROW_WAIT_ROW#
FROM v$locked_object l, dba_objects o
WHERE l.object_id = o.object_id;
```

### 3.2 表级锁（Table Lock）

#### （1）ROW SHARE（RS）- 模式 2

```sql
-- SELECT ... FOR UPDATE 自动获取
SELECT * FROM employees WHERE dept_id = 10 FOR UPDATE;

-- 或者显式锁定
LOCK TABLE employees IN ROW SHARE MODE;
```

**允许**：其他事务查询、更新其他行  
**阻止**：其他事务获取排他锁（X）

#### （2）ROW EXCLUSIVE（RX）- 模式 3

```sql
-- UPDATE、INSERT、DELETE 自动获取
UPDATE employees SET salary = 10000 WHERE employee_id = 100;

-- 或者显式锁定
LOCK TABLE employees IN ROW EXCLUSIVE MODE;
```

**允许**：其他事务查询、更新其他行  
**阻止**：其他事务获取共享锁（S）或排他锁（X）

#### （3）SHARE（S）- 模式 4

```sql
-- 显式锁定
LOCK TABLE employees IN SHARE MODE;
```

**允许**：其他事务查询、获取 RS 锁  
**阻止**：其他事务修改数据（DML）

**用途**：确保表在读取期间不被修改

#### （4）SHARE ROW EXCLUSIVE（SRX）- 模式 5

```sql
LOCK TABLE employees IN SHARE ROW EXCLUSIVE MODE;
```

**允许**：其他事务查询  
**阻止**：其他事务获取任何锁（除了 NULL）

#### （5）EXCLUSIVE（X）- 模式 6

```sql
LOCK TABLE employees IN EXCLUSIVE MODE;
```

**允许**：其他事务查询（使用 MVCC）  
**阻止**：其他事务获取任何锁

**用途**：独占访问，通常用于 DDL 操作

## 4. DDL 锁（字典锁）

### 4.1 什么是 DDL 锁？

DDL 锁保护对象定义（表、索引、视图等）在修改时不被访问。

### 4.2 DDL 锁类型

| 类型 | 说明 |
|------|------|
| **Exclusive DDL** | ALTER TABLE、DROP TABLE |
| **Share DDL** | CREATE INDEX（在线） |
| **Breakable Parse** | 解析 SQL 时获取 |

### 4.3 DDL 锁示例

```sql
-- Session 1：执行 DDL
ALTER TABLE employees ADD (new_column VARCHAR2(50));
-- 获取 Exclusive DDL 锁

-- Session 2：尝试访问表
SELECT * FROM employees;
-- 等待 DDL 完成
```

### 4.4 在线 DDL（减少锁影响）

```sql
-- 在线添加列（Oracle 11g+）
ALTER TABLE employees ADD (new_column VARCHAR2(50)) DEFAULT 'N';

-- 在线重建索引
ALTER INDEX idx_emp REBUILD ONLINE;

-- 在线移动表
ALTER TABLE employees MOVE ONLINE;
```

## 5. 闩锁（Latch）

### 5.1 什么是闩锁？

闩锁是低级别的内存结构保护机制，比锁更轻量。

**特点**：
- 保护 SGA 中的数据结构
- 自旋等待（busy-wait）
- 持有时间极短（微秒级）

### 5.2 常见闩锁类型

| 闩锁 | 保护对象 |
|------|---------|
| **Library Cache Latch** | Shared Pool 中的 SQL |
| **Cache Buffers Chains** | Buffer Cache 中的块 |
| **Redo Copy Latch** | Redo Log Buffer |
| **Shared Pool Latch** | Shared Pool 空闲列表 |

### 5.3 监控闩锁竞争

```sql
-- 查看闩锁统计
SELECT 
    name,
    gets,
    misses,
    ROUND(misses/NULLIF(gets,0)*100, 2) as miss_pct,
    sleeps,
    spin_gets
FROM v$latch
WHERE misses > 0
ORDER BY misses DESC;

-- 查看闩锁子闩
SELECT 
    latch#,
    child#,
    gets,
    misses,
    sleeps
FROM v$latch_children
WHERE misses > 0
ORDER BY misses DESC;
```

### 5.4 解决闩锁竞争

**Library Cache Latch 竞争**：
```sql

ALTER SYSTEM SET shared_pool_size = 2G;

-- 使用绑定变量减少硬解析
```

## 6. 锁监控与诊断

### 6.1 查看当前锁

```sql
-- 查看所有锁
SELECT 
    l.sid,
    s.username,
    s.program,
    l.type,
    l.lmode,
    l.request,
    o.object_name,
    l.block
FROM v$lock l
LEFT JOIN v$session s ON l.sid = s.sid
LEFT JOIN dba_objects o ON l.id1 = o.object_id
WHERE l.type IN ('TM', 'TX')
ORDER BY l.block DESC, l.request;
```

**字段说明**：
- `lmode`：当前持有的锁模式（0-6）
- `request`：请求的锁模式（0 表示未请求）
- `block`：是否阻塞其他会话（1=是）

### 6.2 查找锁等待

```sql
-- 查看被阻塞的会话
SELECT 
    w.sid as waiting_sid,
    w.event as waiting_event,
    w.seconds_in_wait as wait_seconds,
    b.sid as blocking_sid,
    b.username as blocking_user,
    b.program as blocking_program
FROM v$session w, v$session b
WHERE w.blocking_session = b.sid
AND w.event LIKE 'enq%';

-- 详细锁等待信息
SELECT 
    s1.username || '@' || s1.machine || ' (SID=' || s1.sid || ') is blocking ' ||
    s2.username || '@' || s2.machine || ' (SID=' || s2.sid || ')' AS blocking_status
FROM v$lock l1, v$session s1, v$lock l2, v$session s2
WHERE s1.sid = l1.sid
AND s2.sid = l2.sid
AND l1.BLOCK = 1
AND l2.request > 0
AND l1.id1 = l2.id1
AND l2.id2 = l2.id2;
```

### 6.3 锁等待历史

```sql
-- 从 AWR 查看锁等待
SELECT 
    event,
    total_waits,
    time_waited,
    average_wait
FROM dba_hist_system_event
WHERE event LIKE 'enq: TX%'
ORDER BY time_waited DESC;
```

## 7. 解决锁问题

### 7.1 终止阻塞会话

```sql
-- 方法 1：正常终止
ALTER SYSTEM KILL SESSION 'sid,serial#' IMMEDIATE;

-- 方法 2：操作系统层面（最后手段）
-- Linux/Unix
kill -9 <spid>

-- Windows
orakill <sid> <spid>
```

**注意**：先尝试联系用户正常提交/回滚

### 7.2 设置锁超时

```sql
-- 设置 DDL 锁超时
ALTER SESSION SET DDL_LOCK_TIMEOUT = 30;  -- 30 秒

-- 设置分布式锁超时
ALTER SYSTEM SET DISTRIBUTED_LOCK_TIMEOUT = 60;
```

### 7.3 NOWAIT 和 WAIT

```sql
-- 立即返回，不等待
SELECT * FROM employees FOR UPDATE NOWAIT;
-- 如果锁不可用，立即报错 ORA-00054

-- 等待指定时间
SELECT * FROM employees FOR UPDATE WAIT 5;
-- 最多等待 5 秒，超时则报错
```

### 7.4 SKIP LOCKED（跳过已锁定行）

```sql
-- 只锁定未被锁定的行
SELECT * FROM employees 
WHERE dept_id = 10 
FOR UPDATE SKIP LOCKED;

-- 适用场景：队列处理
```

## 8. 最佳实践

### 8.1 锁设计原则

✅ **保持事务简短**：减少锁持有时间  
✅ **按相同顺序访问资源**：避免死锁  
✅ **选择合适的隔离级别**：通常 READ COMMITTED  
✅ **使用绑定变量**：减少库缓存闩锁竞争  
✅ **批量操作分批提交**：避免长事务  

### 8.2 避免锁竞争

📌 **优化 SQL**：减少扫描行数  
📌 **合理索引**：精确定位行  
📌 **分区表**：分散热点  
📌 **队列处理**：使用 SKIP LOCKED  
📌 **读写分离**：报表查询走备库  

### 8.3 监控策略

```sql
-- 定期监控脚本
-- 每小时执行一次

-- 1. 检查长时间运行的事务
SELECT sid, username, start_time
FROM v$transaction t, v$session s
WHERE s.taddr = t.addr
AND (SYSDATE - TO_DATE(t.start_time, 'MM/DD/YY HH24:MI:SS')) * 24 > 1;

-- 2. 检查锁等待
SELECT COUNT(*) as blocked_sessions
FROM v$session
WHERE blocking_session IS NOT NULL;

-- 3. 检查闩锁竞争
SELECT name, misses
FROM v$latch
WHERE misses > 1000;
```
