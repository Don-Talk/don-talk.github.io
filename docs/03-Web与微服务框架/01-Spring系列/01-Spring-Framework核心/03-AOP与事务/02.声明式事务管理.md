## 一、事务基础概念

### 1.1 什么是事务

**事务 (Transaction)** 是数据库操作的最小工作单元,要么全部成功,要么全部失败。

**典型场景**: 银行转账

```java
// A给B转账100元
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    // 1. A账户扣款
    accountDao.decreaseBalance(fromId, amount);
    
    // 2. B账户加款
    accountDao.increaseBalance(toId, amount);
}
```

**问题**: 如果第1步成功,第2步失败,会导致数据不一致!

**解决**: 使用事务,保证两步操作要么都成功,要么都回滚。

### 1.2 ACID特性

| 特性 | 说明 | 示例 |
|------|------|------|
| 原子性 (Atomicity) | 事务是不可分割的整体 | 转账的两步操作要么都成功,要么都失败 |
| 一致性 (Consistency) | 事务前后数据保持一致 | 转账前后总金额不变 |
| 隔离性 (Isolation) | 多个事务互不干扰 | A转账时,B不能同时操作A的账户 |
| 持久性 (Durability) | 事务提交后永久保存 | 即使系统崩溃,已提交的数据也不会丢失 |

## 二、@Transactional注解使用

### 2.1 环境搭建

#### 导入依赖

```xml
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-tx</artifactId>
    <version>5.3.20</version>
</dependency>
```

#### 启用事务

**XML方式**:

```xml
<!--开启注解驱动的事务管理-->
<tx:annotation-driven/>
```

**注解方式**:

```java
@Configuration
@EnableTransactionManagement
public class TransactionConfig {
    
    @Bean
    public PlatformTransactionManager transactionManager(DataSource dataSource) {
        return new DataSourceTransactionManager(dataSource);
    }
}
```

### 2.2 基本用法

#### 方法级别事务

```java
@Service
public class AccountService {
    
    @Autowired
    private AccountDao accountDao;
    
    /**
     * 转账方法 - 添加事务
     */
    @Transactional
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        // 1. A账户扣款
        accountDao.decreaseBalance(fromId, amount);
        
        // 模拟异常
        if (amount.compareTo(new BigDecimal("10000")) > 0) {
            throw new RuntimeException("转账金额过大");
        }
        
        // 2. B账户加款
        accountDao.increaseBalance(toId, amount);
    }
}
```

**效果**:
- 正常执行: 两步操作都提交
- 抛出异常: 两步操作都回滚

#### 类级别事务

```java
@Service
@Transactional  // 类中所有public方法都有事务
public class OrderService {
    
    public void createOrder(Order order) {
        // 有事务
    }
    
    public void cancelOrder(Long orderId) {
        // 有事务
    }
}
```

### 2.3 常用属性

```java
@Transactional(
    // 事务管理器(多数据源时使用)
    transactionManager = "transactionManager",
    
    // 传播行为(默认REQUIRED)
    propagation = Propagation.REQUIRED,
    
    // 隔离级别(默认DEFAULT)
    isolation = Isolation.DEFAULT,
    
    // 超时时间(秒),-1表示不超时
    timeout = 30,
    
    // 是否只读(查询操作可优化性能)
    readOnly = false,
    
    // 哪些异常回滚(默认RuntimeException和Error)
    rollbackFor = {RuntimeException.class, Exception.class},
    
    // 哪些异常不回滚
    noRollbackFor = {BusinessException.class}
)
public void businessMethod() {
    // ...
}
```

## 三、事务传播行为

### 3.1 什么是传播行为

**传播行为 (Propagation)** 定义了多个事务方法相互调用时,事务如何传播。

**场景**: A方法调用B方法,两个方法都有事务,它们共享一个事务还是各自独立?

### 3.2 七种传播行为

| 传播行为 | 说明 | 使用场景 |
|----------|------|----------|
| REQUIRED | 如果当前有事务就加入,没有就新建(默认) | 大多数场景 |
| REQUIRES_NEW | 不管当前有没有事务,都新建事务 | 日志记录、审计 |
| SUPPORTS | 如果当前有事务就加入,没有就以非事务方式执行 | 查询操作 |
| NOT_SUPPORTED | 以非事务方式执行,如果当前有事务就挂起 | 批量操作 |
| MANDATORY | 必须在事务中执行,否则抛异常 | 强制要求事务 |
| NEVER | 必须以非事务方式执行,如果有事务就抛异常 | 不允许事务 |
| NESTED | 如果当前有事务就嵌套执行,没有就新建 | 部分回滚 |

### 3.3 REQUIRED vs REQUIRES_NEW

#### REQUIRED (默认)

```java
@Service
public class OrderService {
    
    @Autowired
    private LogService logService;
    
    @Transactional(propagation = Propagation.REQUIRED)
    public void createOrder(Order order) {
        // 保存订单
        orderDao.insert(order);
        
        // 记录日志 - 加入当前事务
        logService.saveLog("创建订单");
        
        // 如果这里抛异常,订单和日志都会回滚
        throw new RuntimeException("测试回滚");
    }
}

@Service
public class LogService {
    
    @Transactional(propagation = Propagation.REQUIRED)
    public void saveLog(String message) {
        logDao.insert(message);
    }
}
```

**结果**: 订单和日志都在同一个事务中,一起回滚。

#### REQUIRES_NEW (新建事务)

```java
@Service
public class LogService {
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void saveLog(String message) {
        logDao.insert(message);
        // 日志单独提交,不受外层事务影响
    }
}
```

**结果**: 
- 日志先提交
- 订单回滚
- 日志不会回滚(已独立提交)

**适用场景**: 审计日志需要保留,即使业务操作失败。

### 3.4 NESTED嵌套事务

```java
@Service
public class BatchService {
    
    @Transactional
    public void batchProcess(List<Order> orders) {
        for (Order order : orders) {
            try {
                processSingleOrder(order);  // 嵌套事务
            } catch (Exception e) {
                // 单个订单失败不影响其他订单
                log.error("订单处理失败: {}", order.getId());
            }
        }
    }
    
    @Transactional(propagation = Propagation.NESTED)
    public void processSingleOrder(Order order) {
        orderDao.insert(order);
        // 如果失败,只回滚这个订单,不影响批次
    }
}
```

## 四、事务隔离级别

### 4.1 并发问题

| 问题 | 说明 | 示例 |
|------|------|------|
| 脏读 (Dirty Read) | 读到未提交的数据 | A修改数据但未提交,B读到了修改后的数据 |
| 不可重复读 (Non-repeatable Read) | 两次读取结果不一致 | A第一次读取数据,B修改并提交,A第二次读取发现变了 |
| 幻读 (Phantom Read) | 两次查询记录数不一致 | A第一次查询有10条记录,B插入1条,A第二次查询有11条 |

### 4.2 五种隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|----------|------|-----------|------|------|
| DEFAULT | - | - | - | 使用数据库默认 |
| READ_UNCOMMITTED | ❌ 可能 | ❌ 可能 | ❌ 可能 | 最快 |
| READ_COMMITTED | ✅ 避免 | ❌ 可能 | ❌ 可能 | 较快 |
| REPEATABLE_READ | ✅ 避免 | ✅ 避免 | ❌ 可能(MySQL避免) | 较慢 |
| SERIALIZABLE | ✅ 避免 | ✅ 避免 | ✅ 避免 | 最慢 |

**MySQL默认**: REPEATABLE_READ  
**Oracle默认**: READ_COMMITTED

### 4.3 设置隔离级别

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public void queryData() {
    // 使用READ_COMMITTED隔离级别
}
```

## 五、事务失效场景

### 5.1 常见失效原因

#### 1. 方法不是public

```java
// ❌ 错误: private方法事务失效
@Transactional
private void save() {
    // ...
}

// ✅ 正确: public方法
@Transactional
public void save() {
    // ...
}
```

#### 2. 自调用(self-invocation)

```java
@Service
public class UserService {
    
    public void createUser(User user) {
        // ❌ 错误: 内部调用,事务失效
        this.saveUser(user);
    }
    
    @Transactional
    public void saveUser(User user) {
        userDao.insert(user);
    }
}
```

**解决**:

```java
// 方案1: 注入自己
@Service
public class UserService {
    
    @Autowired
    @Lazy
    private UserService self;
    
    public void createUser(User user) {
        // ✅ 通过代理调用
        self.saveUser(user);
    }
    
    @Transactional
    public void saveUser(User user) {
        userDao.insert(user);
    }
}

// 方案2: 使用AopContext
@Service
public class UserService {
    
    public void createUser(User user) {
        ((UserService) AopContext.currentProxy()).saveUser(user);
    }
    
    @Transactional
    public void saveUser(User user) {
        userDao.insert(user);
    }
}
```

#### 3. 异常被捕获

```java
@Transactional
public void saveUser(User user) {
    try {
        userDao.insert(user);
        // 模拟异常
        int i = 1 / 0;
    } catch (Exception e) {
        // ❌ 错误: 异常被捕获,事务不会回滚
        e.printStackTrace();
    }
}
```

**解决**:

```java
@Transactional
public void saveUser(User user) {
    try {
        userDao.insert(user);
        int i = 1 / 0;
    } catch (Exception e) {
        e.printStackTrace();
        // ✅ 手动回滚
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        // 或者重新抛出
        throw new RuntimeException(e);
    }
}
```

#### 4. 异常类型不匹配

```java
@Transactional(rollbackFor = Exception.class)
public void saveUser(User user) throws Exception {
    userDao.insert(user);
    // 抛出检查型异常
    throw new Exception("测试");
}
```

**注意**: 
- 默认只对`RuntimeException`和`Error`回滚
- 检查型异常(Exception)需要在`rollbackFor`中声明

#### 5. 数据库引擎不支持

```sql
-- MyISAM引擎不支持事务
CREATE TABLE user (...) ENGINE=MyISAM;

-- InnoDB引擎支持事务
CREATE TABLE user (...) ENGINE=InnoDB;
```

## 六、编程式事务

### 6.1 TransactionTemplate

**适用场景**: 细粒度控制事务边界

```java
@Service
public class OrderService {
    
    @Autowired
    private TransactionTemplate transactionTemplate;
    
    public void createOrder(Order order) {
        transactionTemplate.execute(status -> {
            try {
                // 1. 保存订单
                orderDao.insert(order);
                
                // 2. 扣减库存
                stockDao.decrease(order.getProductId(), order.getQuantity());
                
                return order.getId();
            } catch (Exception e) {
                // 手动回滚
                status.setRollbackOnly();
                throw e;
            }
        });
    }
}
```

### 6.2 PlatformTransactionManager

```java
@Service
public class BatchService {
    
    @Autowired
    private PlatformTransactionManager transactionManager;
    
    public void batchInsert(List<User> users) {
        // 创建事务定义
        DefaultTransactionDefinition def = new DefaultTransactionDefinition();
        def.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRED);
        def.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
        def.setTimeout(30);
        
        // 开启事务
        TransactionStatus status = transactionManager.getTransaction(def);
        
        try {
            for (User user : users) {
                userDao.insert(user);
            }
            
            // 提交事务
            transactionManager.commit(status);
        } catch (Exception e) {
            // 回滚事务
            transactionManager.rollback(status);
            throw e;
        }
    }
}
```

## 七、最佳实践

### 7.1 事务使用原则

1. **事务范围尽可能小**: 只包含必要的数据库操作
2. **避免在事务中做耗时操作**: 如HTTP请求、文件IO
3. **合理选择传播行为**: 大多数场景用REQUIRED
4. **明确指定rollbackFor**: 不要依赖默认行为
5. **查询操作设置readOnly=true**: 优化性能

### 7.2 反模式

**❌ 错误做法**:

```java
@Transactional
public void processOrder(Order order) {
    // 1. 数据库操作
    orderDao.insert(order);
    
    // 2. 发送HTTP请求(耗时)
    httpClient.post("http://external-api.com", order);
    
    // 3. 发送邮件(耗时)
    emailService.send(order.getUserEmail());
    
    // 4. 生成PDF(耗时)
    pdfGenerator.generate(order);
}
```

**✅ 正确做法**:

```java
@Transactional
public void processOrder(Order order) {
    // 只包含数据库操作
    orderDao.insert(order);
}

// 异步执行耗时操作
@Async
public void sendNotification(Order order) {
    httpClient.post("http://external-api.com", order);
    emailService.send(order.getUserEmail());
    pdfGenerator.generate(order);
}
```

### 7.3 事务与连接池

**配置合理的连接池大小**:

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20  # 最大连接数
      minimum-idle: 5        # 最小空闲连接
      connection-timeout: 30000  # 连接超时时间(毫秒)
```

**注意**: 长时间持有事务会占用连接,导致连接池耗尽。
