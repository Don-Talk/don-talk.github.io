## 一、Nacos集群部署

### 1.1 为什么需要集群?

**单机模式的问题**:
```
❌ 单点故障: Nacos宕机,整个微服务系统瘫痪
❌ 性能瓶颈: 大量服务注册/查询时响应慢
❌ 数据丢失: 重启后注册信息全部丢失
```

**集群模式的优势**:
```
✅ 高可用: 一台宕机,其他节点继续工作
✅ 高性能: 请求分散到多个节点
✅ 数据持久化: 数据存储在MySQL,不会丢失
```

## 二、健康检查深入

### 2.1 健康检查机制

Nacos采用**主动探测 + 被动上报**相结合的方式进行健康检查。

**两种检查模式**:

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **临时实例** | 客户端心跳上报 | 大多数微服务 ⭐推荐 |
| **持久化实例** | 服务端主动探测 | 数据库、中间件等 |

#### 临时实例(默认)

```
服务实例 → Nacos: 心跳包(每5秒)
Nacos: 如果15秒没收到心跳,标记为不健康
Nacos: 如果30秒没收到心跳,删除实例
```

**优点**: 
- ✅ 实时性高
- ✅ 减轻Nacos压力
- ✅ 适合频繁上下线的场景

#### 持久化实例

```
Nacos → 服务实例: HTTP GET /actuator/health(每5秒)
Nacos: 根据HTTP响应码判断健康状态
Nacos: 连续3次失败,标记为不健康
```

**优点**:
- ✅ 更准确(不依赖客户端)
- ✅ 适合非Java应用
- ✅ 实例不会自动删除

### 2.2 自定义健康检查

#### 方式1: Spring Boot Actuator

**添加依赖**:

```xml

    <groupId>org.springframework.boot</groupId>

**配置暴露端点**:

```

**自定义健康检查**:

```java
@Component
public class DatabaseHealthIndicator implements HealthIndicator {
    
    @Autowired
    private DataSource dataSource;
    
    @Override
    public Health health() {
        try {
            // 检查数据库连接
            Connection connection = dataSource.getConnection();
            if (connection.isValid(1000)) {
                return Health.up().withDetail("database", "OK").build();
            } else {
                return Health.down().withDetail("database", "Connection invalid").build();
            }
        } catch (Exception e) {
            return Health.down(e).withDetail("database", "Error").build();
        }
    }
}
```

## 三、服务优雅上下线

### 3.1 优雅下线

**问题**: 服务关闭时,如果还有请求在处理,直接关闭会导致请求失败。

**解决方案**: 优雅下线流程

```
1. 服务接收到关闭信号(Ctrl+C / kill)
2. 从Nacos注销自己(不再接收新请求)
3. 等待现有请求处理完成(最多30秒)
4. 关闭应用
```

#### 实现方式

**方式1: Spring Boot Actuator(推荐)**

**添加依赖**:
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

**配置**:
```yaml
management:
  endpoint:
    shutdown:
      enabled: true  # 启用shutdown端点
  endpoints:
    web:
      exposure:
        include: shutdown
```

**触发优雅下线**:
```bash
POST http://localhost:8081/actuator/shutdown
```

**方式2: 监听关闭事件**

```java
@Component
@Slf4j
public class GracefulShutdownListener {
    
    @Autowired
    private NacosServiceRegistry registry;
    
    @Autowired
    private NacosRegistration registration;
    
    @EventListener
    public void onContextClosedEvent(ContextClosedEvent event) {
        log.info("开始优雅下线...");
        
        // 1. 从Nacos注销
        registry.deregister(registration);
        log.info("已从Nacos注销");
        
        // 2. 等待现有请求处理完成
        try {
            Thread.sleep(5000);  // 等待5秒
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        log.info("优雅下线完成");
    }
}
```

### 3.2 优雅上线

**问题**: 服务刚启动时,Spring容器还未完全初始化,此时接收请求可能失败。

**解决方案**: 延迟注册

```yaml
spring:
  cloud:
    nacos:
      discovery:
        # 延迟注册时间(毫秒),默认0
        register-delay: 10000  # 启动10秒后再注册
```

或者使用健康检查:

```yaml
spring:
  cloud:
    nacos:
      discovery:
        # 只有健康检查通过后才注册
        service-enabled: true
```

## 四、负载均衡策略

### 4.1 Ribbon负载均衡

Ribbon是Spring Cloud的客户端负载均衡器。

**内置策略**:

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **RoundRobinRule** | 轮询(默认) | 均匀分布流量 ⭐推荐 |
| **RandomRule** | 随机 | 测试环境 |
| **WeightedResponseTimeRule** | 响应时间加权 | 实例性能不均 |
| **BestAvailableRule** | 最小并发数 | 避免过载 |
| **RetryRule** | 重试 | 不稳定网络 |

#### 配置负载均衡策略

**全局配置**:

```yaml
ribbon:
  NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RandomRule
```

**指定服务配置**:

```yaml
user-service:  # 服务名
  ribbon:
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.WeightedResponseTimeRule
```

#### 自定义负载均衡策略

```java
@Configuration
public class CustomRibbonConfig {
    
    @Bean
    public IRule customRule() {
        return new CustomRule();
    }
}

public class CustomRule extends AbstractLoadBalancerRule {
    
    @Override
    public Server choose(Object key) {
        ILoadBalancer lb = getLoadBalancer();
        List<Server> servers = lb.getReachableServers();
        
        if (servers.isEmpty()) {
            return null;
        }
        
        // 自定义选择逻辑: 优先选择元数据中version=2.0的实例
        for (Server server : servers) {
            Map<String, String> metadata = getMetadata(server);
            if ("2.0".equals(metadata.get("version"))) {
                return server;
            }
        }
        
        // 如果没有2.0版本,随机选择一个
        return servers.get(new Random().nextInt(servers.size()));
    }
    
    private Map<String, String> getMetadata(Server server) {
        if (server instanceof NacosServer) {
            return ((NacosServer) server).getMetadata();
        }
        return Collections.emptyMap();
    }
}
```

### 4.2 基于权重的负载均衡

**场景**: 不同性能的机器分配不同的流量比例。

**配置权重**:

```yaml
# 实例1(高性能机器)
spring:
  cloud:
    nacos:
      discovery:
        weight: 10  # 权重10

# 实例2(低性能机器)
spring:
  cloud:
    nacos:
      discovery:
        weight: 1   # 权重1
```

**效果**: 实例1接收的流量是实例2的10倍。

## 五、Nacos数据一致性

### 5.1 CAP理论

| 特性 | 说明 |
|------|------|
| **C**(Consistency) | 一致性: 所有节点数据一致 |
| **A**(Availability) | 可用性: 每个请求都能得到响应 |
| **P**(Partition Tolerance) | 分区容错性: 网络分区时系统仍能工作 |

**Nacos支持CP和AP切换**:

- **CP模式**: 强一致性,适合配置中心
- **AP模式**: 高可用性,适合服务注册发现(默认)

### 5.2 Raft协议(CP模式)

```
Client → Leader: 写请求
Leader → Followers: 同步数据
Followers → Leader: 确认
Leader → Client: 返回成功(超过半数确认后)
```

**特点**:
- ✅ 强一致性
- ⚠️ 写入性能较低
- ⚠️ Leader选举期间不可用

### 5.3 Distro协议(AP模式)

```
Client → Node1: 写请求
Node1 → 本地存储: 保存数据
Node1 → Client: 立即返回成功
Node1 → Node2,Node3: 异步同步数据
```

**特点**:
- ✅ 高可用性
- ✅ 写入性能高
- ⚠️ 短暂不一致(最终一致)

## 七、常见问题

### Q1: 集群节点之间数据不同步?

**A**: 
1. 检查`cluster.conf`配置是否正确
2. 检查网络是否通畅(防火墙、安全组)
3. 查看日志:`logs/nacos.log`
4. 重启所有节点

### Q2: 服务实例频繁上下线?

**A**: 
1. 检查网络稳定性
2. 增加心跳超时时间
3. 检查GC情况(频繁Full GC会导致心跳中断)
4. 调整JVM参数

### Q3: Nacos内存占用过高?

**A**: 
1. 调整JVM参数:`-Xms512m -Xmx512m`
2. 清理过期配置
3. 减少不必要的服务注册
4. 开启G1垃圾收集器

### Q4: 如何迁移Eureka到Nacos?

**A**: 
1. 替换依赖:`eureka-client` → `nacos-discovery`
2. 修改配置:`eureka.client.service-url` → `spring.cloud.nacos.discovery.server-addr`
3. 替换注解:`@EnableEurekaClient` → `@EnableDiscoveryClient`
4. 测试验证
