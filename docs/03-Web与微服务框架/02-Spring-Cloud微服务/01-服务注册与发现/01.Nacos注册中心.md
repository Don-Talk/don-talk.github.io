## 一、服务注册与发现概述

### 1.1 为什么需要服务注册与发现?

**单体应用的问题**:
```
用户请求 → 单体应用(所有功能在一起)
           ├── 用户模块
           ├── 订单模块
           ├── 商品模块
           └── 支付模块

问题: 
❌ 代码耦合严重
❌ 部署困难(改一个功能要重启整个应用)
❌ 扩展性差(只能整体扩容)
❌ 技术栈受限(只能用一种语言)
```

**微服务架构**:
```
用户请求 → API网关
           ├── 用户服务(user-service:8081)
           ├── 订单服务(order-service:8082)
           ├── 商品服务(product-service:8083)
           └── 支付服务(pay-service:8084)

优势:
✅ 服务独立部署
✅ 技术栈灵活
✅ 按需扩容
✅ 故障隔离

新问题:
❓ 订单服务如何找到用户服务的地址?
❓ 用户服务IP变了怎么办?
❓ 如何知道某个服务是否可用?
```

### 1.2 核心概念

| 概念 | 说明 | 类比 |
|------|------|------|
| **服务提供者** | 提供服务接口的应用 | 餐厅(提供菜品) |
| **服务消费者** | 调用服务接口的应用 | 顾客(点菜) |
| **注册中心** | 存储服务地址信息的中间件 | 电话簿(查号码) |
| **服务注册** | 启动时向注册中心报告自己的地址 | 餐厅开业登记 |
| **服务发现** | 从注册中心获取其他服务的地址 | 顾客查电话簿 |
| **健康检查** | 定期检查服务是否可用 | 餐厅卫生检查 |

### 1.3 工作流程

```
1. 服务启动时注册
   服务提供者 → 注册中心: "我叫user-service,地址是192.168.1.100:8081"

2. 服务消费者查询
   服务消费者 → 注册中心: "我要找user-service的地址"
   注册中心 → 服务消费者: "user-service的地址是192.168.1.100:8081"

3. 服务消费者调用
   服务消费者 → 服务提供者: HTTP请求 http://192.168.1.100:8081/api/users

4. 心跳检测
   服务提供者 → 注册中心: "我还活着"(每5秒)
   注册中心: 如果30秒没收到心跳,标记为不健康
```

## 二、Nacos简介

### 2.1 什么是Nacos?

**Nacos** = **Na**ming and **Co**nfiguration **S**ervice

> Nacos是一个更易于构建云原生应用的动态服务发现、配置管理和服务管理平台。

**核心功能**:
1. ✅ **服务注册与发现**: 替代Eureka/Consul
2. ✅ **配置中心**: 替代Spring Cloud Config
3. ✅ **服务健康监测**: 自动剔除不健康实例
4. ✅ **服务元数据管理**: 存储版本、权重等信息
5. ✅ **服务可视化**: Web控制台管理服务

**官方地址**: https://nacos.io/

### 2.2 Nacos vs Eureka对比

| 特性 | Nacos | Eureka |
|------|-------|--------|
| **CAP理论** | CP + AP可切换 | AP |
| **配置中心** | ✅ 内置 | ❌ 需额外配置 |
| **控制台** | ✅ 功能强大 | ⚠️ 功能简单 |
| **健康检查** | ✅ TCP/HTTP/MySQL | ✅ TCP |
| **跨数据中心** | ✅ 支持 | ❌ 不支持 |
| **社区活跃度** | 🔥 高(阿里维护) | ⚠️ 低(停更) |
| **学习成本** | 低 | 低 |
| **推荐程度** | ⭐⭐⭐⭐⭐ | ⭐⭐ |

**结论**: **新项目强烈推荐Nacos**,Eureka已停止更新

#### 步骤3: 配置文件

**user-service/src/main/resources/application.yml**

```yaml
server:
  port: 8081  # 服务端口

spring:
  application:
    name: user-service  # 服务名称
  
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848  # Nacos Server地址
        namespace: public  # 命名空间(默认public)
        group: DEFAULT_GROUP  # 分组(默认DEFAULT_GROUP)
        weight: 1  # 权重(负载均衡用,值越大被选中的概率越高)
        metadata:
          version: 1.0.0  # 服务版本
          author: DonTalk  # 自定义元数据
```

**关键配置说明**:
- `spring.application.name`: 服务唯一标识,消费者通过这个名字查找服务
- `server-addr`: Nacos Server的地址,集群时用逗号分隔
- `namespace`: 命名空间,用于环境隔离(dev/test/prod)
- `group`: 分组,用于服务分组管理
- `metadata`: 自定义元数据,可以存储版本号、描述等信息

#### 步骤5: 使用RestTemplate调用(基础方式)

**order-service/src/main/java/com/dontalk/order/config/RestTemplateConfig.java**

```java
package com.dontalk.order.config;

import org.springframework.cloud.client.loadbalancer.LoadBalanced;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.client.RestTemplate;

@Configuration
public class RestTemplateConfig {
    
    @Bean
    @LoadBalanced  // 启用负载均衡
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

**关键点**: `@LoadBalanced`注解让RestTemplate支持负载均衡,可以直接使用服务名代替IP地址。

**order-service/src/main/java/com/dontalk/order/controller/OrderController.java**

```java
package com.dontalk.order.controller;

import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.web.bind.annotation.*;
import org.springframework.web.client.RestTemplate;

import java.util.Map;

@Slf4j
@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
public class OrderController {
    
    private final RestTemplate restTemplate;
    
    /**
     * 创建订单(需要查询用户信息)
     */
    @PostMapping
    public Map<String, Object> createOrder(@RequestBody Map<String, Object> params) {
        log.info("创建订单: {}", params);
        
        Long userId = Long.valueOf(params.get("userId").toString());
        
        // 通过服务名调用user-service(负载均衡)
        String url = "http://user-service/api/users/" + userId;
        Map<String, Object> user = restTemplate.getForObject(url, Map.class);
        
        log.info("查询到用户信息: {}", user);
        
        // 创建订单逻辑
        Map<String, Object> order = new java.util.HashMap<>();
        order.put("orderId", System.currentTimeMillis());
        order.put("userId", userId);
        order.put("userName", user.get("username"));
        order.put("amount", params.get("amount"));
        order.put("status", "CREATED");
        
        Map<String, Object> result = new java.util.HashMap<>();
        result.put("code", 200);
        result.put("message", "订单创建成功");
        result.put("data", order);
        
        return result;
    }
}
```

**调用流程**:
```
1. order-service发起请求: http://user-service/api/users/1
2. Ribbon从Nacos获取user-service的实例列表: [192.168.1.100:8081]
3. Ribbon选择其中一个实例(负载均衡策略)
4. 实际请求: http://192.168.1.100:8081/api/users/1
5. user-service返回用户信息
6. order-service组装订单信息并返回
```

运行`OrderServiceApplication.main()`方法

**验证**: Nacos控制台应该能看到两个服务:
- user-service (1个实例)
- order-service (1个实例)

### 7.2 健康检查

Nacos会自动进行健康检查,如果服务实例宕机,会自动从列表中剔除。

**模拟服务宕机**:

停止其中一个user-service实例,等待15秒后刷新Nacos控制台,会看到该实例状态变为"不健康"。

**健康检查配置**:

```yaml
spring:
  cloud:
    nacos:
      discovery:
        # 心跳间隔(毫秒),默认5000
        heart-beat-interval: 5000
        # 心跳超时时间(毫秒),默认15000
        heart-beat-timeout: 15000
        # IP删除超时时间(毫秒),默认30000
        ip-delete-timeout: 30000
```

### 7.3 命名空间隔离

**场景**: 开发、测试、生产环境隔离

#### 创建命名空间

1. 登录Nacos控制台
2. 点击"命名空间" → "新建命名空间"
3. 填写信息:
   - 命名空间ID: `dev`(自动生成或手动填写)
   - 命名空间名: `开发环境`
   - 描述: `开发环境专用`

重复创建`test`(测试环境)和`prod`(生产环境)

#### 配置命名空间

**dev环境(application-dev.yml)**:
```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848
        namespace: dev  # 指定命名空间ID
```

**test环境(application-test.yml)**:
```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848
        namespace: test
```

**prod环境(application-prod.yml)**:
```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: 192.168.1.100:8848  # 生产环境Nacos地址
        namespace: prod
```

**效果**: 不同命名空间的服务互不可见,实现环境隔离。

### 7.4 服务元数据管理

**应用场景**:
- 灰度发布: 标记新版本服务
- 权重调整: 高性能机器分配更高权重
- 区域路由: 标记服务所在机房

**配置元数据**:

```yaml
spring:
  cloud:
    nacos:
      discovery:
        metadata:
          version: 2.0.0          # 服务版本
          region: beijing         # 所在区域
          zone: zone-a            # 可用区
          gray: true              # 是否灰度版本
          weight: 10              # 权重(值越大流量越多)
```

**获取元数据**:

```java
@Autowired
private DiscoveryClient discoveryClient;

public void getServiceMetadata() {
    List<ServiceInstance> instances = discoveryClient.getInstances("user-service");
    for (ServiceInstance instance : instances) {
        Map<String, String> metadata = instance.getMetadata();
        System.out.println("版本: " + metadata.get("version"));
        System.out.println("区域: " + metadata.get("region"));
        System.out.println("权重: " + metadata.get("weight"));
    }
}
```

## 八、常见问题

### Q1: 服务注册失败怎么办?

**A**: 检查以下几点:
1. Nacos Server是否启动(`http://localhost:8848/nacos`能否访问)
2. `server-addr`配置是否正确
3. 网络是否通畅(防火墙、安全组)
4. 查看启动日志是否有报错

### Q2: 服务找不到(NOT FOUND)?

**A**: 
1. 确认服务名`spring.application.name`是否正确
2. 确认服务和消费者在同一个命名空间
3. 确认服务已启动且健康检查通过
4. 查看Nacos控制台服务列表

### Q3: 如何实现服务优雅下线?

**A**: Spring Cloud会自动处理,服务关闭时会主动从Nacos注销。

```java
// 无需额外配置,Spring Boot Actuator会自动处理
@PreDestroy
public void onDestroy() {
    log.info("服务正在关闭...");
    // Spring Cloud会自动从Nacos注销
}
```

### Q4: Nacos集群如何配置?

**A**: 

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: 192.168.1.100:8848,192.168.1.101:8848,192.168.1.102:8848
```
