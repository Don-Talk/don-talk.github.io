## 一、微服务设计原则

### 1.1 单一职责原则 (SRP - Single Responsibility Principle)

**定义**:
> 每个服务只负责一个业务领域,只做一件事并做好。

**示例**:

❌ **错误设计**:
```java
// ecommerce-service (电商大杂烩)
@Service
public class EcommerceService {
    public void createUser() { }      // 用户管理
    public void createOrder() { }     // 订单管理
    public void createProduct() { }   // 商品管理
    public void processPayment() { }  // 支付处理
    public void sendNotification() { }// 消息通知
}

问题:
- 代码耦合严重
- 无法独立部署
- 团队职责不清
```

✅ **正确设计**:
```
user-service (用户服务)
├── 用户注册/登录
├── 个人信息管理
└── 权限认证

order-service (订单服务)
├── 创建订单
├── 订单查询
└── 订单状态流转

product-service (商品服务)
├── 商品管理
├── 库存管理
└── 分类标签

pay-service (支付服务)
├── 发起支付
├── 支付回调
└── 退款处理

notification-service (通知服务)
├── 短信发送
├── 邮件发送
└── 推送通知
```

**判断标准**:
```
问自己: "这个服务的变更原因是什么?"

如果答案是:
- "因为用户需求变化" → 用户服务
- "因为订单流程调整" → 订单服务
- "因为商品规则改变" → 商品服务

如果一个服务有多个变更原因 → 需要继续拆分!
```

---

### 1.2 自治性原则 (Autonomy)

**定义**:
> 每个服务应该尽可能独立,减少对外部服务的依赖。

**实现要点**:

#### 1. 独立数据库
```yaml
# ✅ 每个服务有自己的数据库
user-service:
  datasource:
    url: jdbc:mysql://localhost:3306/user_db
    username: user_admin
    password: ***

order-service:
  datasource:
    url: jdbc:mysql://localhost:3306/order_db
    username: order_admin
    password: ***

# ❌ 不要共享数据库
shared-database:
  url: jdbc:mysql://localhost:3306/ecommerce_db
```

#### 2. 独立部署
```bash
# ✅ 可以单独重启某个服务
docker restart order-service

# 其他服务不受影响
user-service: 正常运行 ✅
product-service: 正常运行 ✅
```

#### 3. 独立技术栈
```
user-service:    Java 17 + Spring Boot 3.0
ai-service:      Python 3.10 + TensorFlow
realtime-service: Go 1.21
frontend-bff:    Node.js 18
```

### 1.3 去中心化数据管理

**定义**:
> 每个服务管理自己的数据,其他服务只能通过API访问。

**反模式**:

❌ **直接跨库查询**:
```sql
-- 订单服务直接查询用户表
SELECT o.*, u.name, u.email
FROM order_db.orders o
JOIN user_db.users u ON o.user_id = u.id;

问题:
- 紧耦合(订单服务依赖用户表结构)
- 无法独立演进
- 性能瓶颈(跨库JOIN慢)
```

✅ **通过API获取**:
```java
// 订单服务通过用户服务API获取数据
@RestController
public class OrderController {
    
    @Autowired
    private UserFeignClient userFeignClient;
    
    @GetMapping("/order/{id}")
    public Result getOrderDetail(@PathVariable Long id) {
        // 1. 查询订单
        Order order = orderService.getById(id);
        
        // 2. 调用用户服务获取用户信息
        User user = userFeignClient.getById(order.getUserId());
        
        // 3. 组装返回
        OrderDetailVO vo = new OrderDetailVO();
        vo.setOrder(order);
        vo.setUserName(user.getName());
        
        return Result.success(vo);
    }
}
```

**数据冗余策略**:
```java
// 适当冗余,减少远程调用
@Entity
public class Order {
    private Long id;
    private Long userId;
    
    // 冗余用户名(避免每次都要调用用户服务)
    private String userName;
    
    // 冗余用户手机号
    private String userPhone;
}

注意:
- 冗余的是展示字段,不是核心数据
- 通过异步消息保持最终一致性
```

### 1.4 接口隔离原则

**定义**:
> 服务间通过稳定的API通信,隐藏内部实现细节。

**API设计规范**:

#### 1. RESTful风格
```java
// ✅ 好的API设计
GET    /api/users/{id}          // 查询用户
POST   /api/users               // 创建用户
PUT    /api/users/{id}          // 更新用户
DELETE /api/users/{id}          // 删除用户
GET    /api/users/{id}/orders   // 查询用户订单

// ❌ 坏的API设计
POST   /api/getUser
POST   /api/updateUser
POST   /api/deleteUser
GET    /api/queryOrdersByUserId
```

#### 2. 版本管理
```java
// API版本化
/api/v1/users/{id}
/api/v2/users/{id}

@FeignClient(name = "user-service", path = "/api/v1")
public interface UserFeignClient {
    @GetMapping("/users/{id}")
    User getById(@PathVariable("id") Long id);
}
```

#### 3. 向后兼容
```java
// ✅ 添加新字段(兼容旧客户端)
@Data
public class UserVO {
    private Long id;
    private String name;
    private String email;
    
    // 新增字段,设置默认值
    private String avatar = ""; 
}

// ❌ 删除或修改已有字段(破坏兼容性)
@Data
public class UserVO {
    private Long id;
    // private String name;  ← 删除了,旧客户端会报错!
    private String userName;  // ← 改名了,旧客户端会报错!
}
```

### 1.5 容错设计原则

**定义**:
> 服务必须具备故障处理能力,不能假设依赖的服务总是可用。

**实现方式**:

#### 1. 超时控制
```yaml
feign:
  client:
    config:
      default:
        connectTimeout: 3000  # 连接超时3秒
        readTimeout: 5000     # 读取超时5秒
```

#### 2. 熔断降级
```java
@FeignClient(name = "user-service", fallback = UserFeignFallback.class)
public interface UserFeignClient {
    
    @GetMapping("/api/users/{id}")
    User getById(@PathVariable("id") Long id);
}

@Component
public class UserFeignFallback implements UserFeignClient {
    
    @Override
    public User getById(Long id) {
        // 降级: 返回缓存数据或默认值
        User user = new User();
        user.setId(id);
        user.setName("未知用户");
        return user;
    }
}
```

#### 3. 重试机制
```yaml
spring:
  cloud:
    openfeign:
      retryer:
        enabled: true
        maxAttempts: 3      # 最多重试3次
        backoffInterval: 1000 # 间隔1秒
```

## 二、领域驱动设计(DDD)基础

### 2.1 什么是DDD?

**定义**:
> Domain-Driven Design,是一种软件设计方法,强调将业务逻辑放在核心位置,通过领域模型解决复杂业务问题。

**核心概念**:

```
战略设计 (Strategic Design)
├── 领域 (Domain)
├── 子域 (Subdomain)
├── 限界上下文 (Bounded Context)
└── 上下文映射 (Context Map)

战术设计 (Tactical Design)
├── 实体 (Entity)
├── 值对象 (Value Object)
├── 聚合根 (Aggregate Root)
├── 仓储 (Repository)
└── 领域服务 (Domain Service)
```

### 2.2 领域划分

**电商系统领域划分**:

```
电商领域 (E-commerce Domain)
│
├── 核心域 (Core Domain) - 核心竞争力
│   ├── 订单子域 (Order Subdomain)
│   ├── 支付子域 (Payment Subdomain)
│   └── 库存子域 (Inventory Subdomain)
│
├── 支撑域 (Supporting Domain) - 必要但非核心
│   ├── 用户子域 (User Subdomain)
│   ├── 商品子域 (Product Subdomain)
│   └── 促销子域 (Promotion Subdomain)
│
└── 通用域 (Generic Domain) - 可购买现成方案
    ├── 消息通知子域 (Notification Subdomain)
    ├── 日志审计子域 (Audit Subdomain)
    └── 权限认证子域 (Auth Subdomain)
```

### 2.3 限界上下文 (Bounded Context)

**定义**:
> 每个领域概念只在特定的上下文中有效,不同上下文中的同一概念可能有不同含义。

**示例**:

```
用户上下文 (User Context):
User {
    id: Long
    username: String
    password: String
    email: String
}

订单上下文 (Order Context):
User {
    id: Long
    name: String      // 只需要姓名
    phone: String     // 只需要手机号
    address: String   // 收货地址
}

商品上下文 (Product Context):
User {
    id: Long
    nickname: String  // 评价显示的昵称
    level: String     // 会员等级
}

同一个"User"概念,在不同上下文中有不同的属性和行为!
```

### 2.4 聚合根 (Aggregate Root)

**定义**:
> 聚合是一组相关对象的集合,聚合根是聚合的入口点,外部只能通过聚合根访问聚合内的对象。

**示例**:

```java
// 订单聚合
public class Order {  // ← 聚合根
    private Long id;
    private Long userId;
    private List<OrderItem> items;  // 订单项(内部对象)
    private OrderStatus status;
    
    // 只能通过Order操作OrderItem
    public void addItem(Product product, int quantity) {
        OrderItem item = new OrderItem(product, quantity);
        this.items.add(item);
        this.calculateTotal();  // 重新计算总价
    }
    
    public void removeItem(Long itemId) {
        this.items.removeIf(item -> item.getId().equals(itemId));
        this.calculateTotal();
    }
    
    private void calculateTotal() {
        // 计算订单总价
    }
}

// 订单项(不能作为聚合根被外部直接访问)
public class OrderItem {
    private Long id;
    private Long productId;
    private String productName;
    private BigDecimal price;
    private Integer quantity;
}
```

**规则**:
```
✅ 外部只能持有聚合根的引用
✅ 聚合内的一致性由聚合根保证
✅ 聚合根是事务的边界
❌ 不要让聚合太大(影响性能)
```

## 三、服务拆分策略

### 3.1 基于业务领域拆分

**最常见也是最推荐的拆分方式**。

**步骤**:

1. **识别业务领域**
```
电商系统:
- 用户管理
- 商品管理
- 订单管理
- 支付管理
- 库存管理
- 物流管理
```

2. **划分限界上下文**
```
用户上下文 → user-service
商品上下文 → product-service
订单上下文 → order-service
支付上下文 → pay-service
库存上下文 → inventory-service
物流上下文 → logistics-service
```

3. **定义服务边界**
```
user-service:
├── 用户注册/登录
├── 个人信息管理
├── 地址管理
└── 会员等级

不包含:
❌ 订单查询(属于order-service)
❌ 购物车(可以单独拆为cart-service)
```

---

### 3.2 基于动词拆分(CQRS)

**CQRS**: Command Query Responsibility Segregation (命令查询职责分离)

**适用场景**: 读写比例悬殊(如读多写少)

**示例**:

```
传统方式:
order-service (读写在一起)
├── 创建订单 (写)
├── 更新订单 (写)
├── 查询订单 (读)
└── 订单统计 (读)

CQRS方式:
order-command-service (写服务)
├── 创建订单
├── 更新订单
└── 取消订单

order-query-service (读服务)
├── 查询订单列表
├── 订单详情查询
└── 订单统计分析
```

**优势**:
```
✅ 读写优化(读服务可以用缓存/ES)
✅ 独立扩展(读服务可以多实例)
✅ 性能提升(互不干扰)
```

**劣势**:
```
❌ 复杂度增加(需要数据同步)
❌ 最终一致性(读写可能有延迟)
```

---

### 3.3 基于数据拆分

**根据数据访问模式拆分**。

**示例**:

```
热点数据 vs 冷数据:

product-hot-service (热门商品)
├── Redis缓存
├── 高频访问商品
└── 快速响应

product-cold-service (冷门商品)
├── MySQL存储
├── 低频访问商品
└── 正常响应
```

---

### 3.4 基于流量拆分

**根据访问量拆分**。

**示例**:

```
秒杀场景:

seckill-service (秒杀服务)
├── 高并发优化
├── 限流熔断
├── 异步处理
└── 独立部署

normal-order-service (普通订单服务)
├── 正常业务流程
└── 常规部署
```

## 五、拆分反模式(避免踩坑)

### 5.1 分布式单体 (Distributed Monolith)

**症状**:
```
❌ 服务间紧密耦合
❌ 必须同时部署所有服务
❌ 一个服务宕机,全部不可用
❌ 远程调用比本地调用还多
```

**原因**:
```
- 拆分粒度太细
- 服务边界不清
- 过度追求微服务
```

**解决方案**:
```
✅ 合并相关服务
✅ 明确服务边界
✅ 减少同步调用,改用异步消息
```

---

### 5.2 共享数据库

**症状**:
```
❌ 多个服务共用一个数据库
❌ 直接JOIN其他服务的表
❌ 数据库成为瓶颈
```

**解决方案**:
```
✅ 每个服务独立数据库
✅ 通过API获取数据
✅ 适当数据冗余
```

---

### 5.3 智能管道, dumb端点

**症状**:
```
❌ ESB总线过于复杂
❌ 服务只是简单的CRUD
❌ 业务逻辑都在ESB中
```

**解决方案**:
```
✅ 智能端点, dumb管道
✅ 业务逻辑在服务内部
✅ 轻量级通信(HTTP/gRPC)
```

---

### 5.4 过度拆分

**症状**:
```
❌ 一个CRUD功能拆成3个服务
❌ 服务数量 > 50个
❌ 调用链深度 > 5层
```

**示例**:
```
user-create-service (只负责创建用户)
user-update-service (只负责更新用户)
user-delete-service (只负责删除用户)

正确做法:
user-service (统一管理用户)
```

**解决方案**:
```
✅ 合并小服务
✅ 保持合理的服务粒度
✅ 调用链不超过3层
```

## 六、服务边界划分技巧

### 6.1 康威定律 (Conway's Law)

**定律**:
> 系统设计受限于产生设计的组织的沟通结构。

**应用**:
```
团队结构决定服务边界:

团队A (5人) → user-service + auth-service
团队B (8人) → order-service + pay-service
团队C (6人) → product-service + inventory-service

原则:
- 一个团队负责2-3个相关服务
- 团队间通过API协作
- 团队内部自主决策
```

---

### 6.2 变更频率分析

**方法**:
```
分析代码变更历史:

user-module: 每周变更3次
order-module: 每周变更5次
product-module: 每月变更1次

决策:
- user和order频繁变更 → 可能需要拆分
- product很少变更 → 可以保持现状
```

---

### 6.3 数据关联性分析

**方法**:
```
分析数据访问模式:

经常一起查询的数据 → 放在同一个服务
很少一起访问的数据 → 可以拆分

示例:
订单 + 订单项 → 总是在一起 → order-service
用户 + 权限   → 总是在一起 → user-service
商品 + 订单   → 偶尔关联   → 可以分开
```

## 七、实战案例: 电商系统拆分

### 7.1 初始状态(单体应用)

```
ecommerce-monolith (单体应用)
├── user模块 (10万行代码)
├── product模块 (15万行代码)
├── order模块 (20万行代码)
├── pay模块 (8万行代码)
├── inventory模块 (5万行代码)
└── notification模块 (2万行代码)

总代码量: 60万行
团队规模: 50人
问题: 代码冲突频繁,部署困难
```

### 7.2 第一次拆分(垂直拆分)

```
拆分为5个服务:

user-service (10万行)
├── 用户管理
└── 权限认证

product-service (20万行)
├── 商品管理
└── 库存管理

order-service (20万行)
├── 订单管理
└── 购物车

pay-service (8万行)
└── 支付管理

notification-service (2万行)
└── 消息通知

效果:
✅ 团队可以并行开发
✅ 部署时间从10分钟降到2分钟
❌ product和order仍然较大
```

### 7.3 第二次拆分(细化拆分)

```
进一步拆分:

product-service (10万行)
├── 商品管理
└── 分类标签

inventory-service (10万行)
├── 库存管理
└── 仓库管理

order-service (12万行)
├── 订单管理
└── 订单统计

cart-service (8万行)
├── 购物车
└── 收藏夹

最终服务列表(8个):
1. user-service
2. auth-service
3. product-service
4. inventory-service
5. order-service
6. cart-service
7. pay-service
8. notification-service

效果:
✅ 服务粒度合理
✅ 团队职责清晰
✅ 独立部署灵活
```

- [ ] 这个服务有明确的业务边界吗?
- [ ] 团队可以独立开发和部署吗?
- [ ] 数据库可以独立管理吗?
- [ ] 与其他服务的耦合度低吗?
- [ ] 服务粒度合适吗?(不太大也不太小)
- [ ] 调用链深度合理吗?(≤3层)

如果以上都是YES → 可以拆分! ✅
