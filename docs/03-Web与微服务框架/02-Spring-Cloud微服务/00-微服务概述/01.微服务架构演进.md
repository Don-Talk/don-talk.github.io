## 二、单体架构 (Monolithic Architecture)

### 2.1 什么是单体架构?

**定义**:
> 所有功能模块打包在一个应用中,共享同一个数据库,部署在单个服务器上。

**典型架构**:
```
┌─────────────────────────────┐
│      单体应用 (WAR/JAR)      │
├─────────────────────────────┤
│  ├── 用户模块                │
│  ├── 订单模块                │
│  ├── 商品模块                │
│  ├── 支付模块                │
│  └── 通知模块                │
├─────────────────────────────┤
│      单一数据库 (MySQL)      │
└─────────────────────────────┘
```

### 2.2 单体架构优势

✅ **开发简单**
- 一个IDE打开整个项目
- 代码跳转方便,易于理解
- 无需考虑分布式问题

✅ **测试容易**
- 单元测试直接运行
- 集成测试启动一个应用
- 端到端测试简单

✅ **部署方便**
- 打包一个JAR/WAR文件
- 上传到服务器
- 重启应用即可

✅ **性能优秀**
- 本地方法调用,无网络开销
- 事务管理简单(本地事务)
- 内存共享,无序列化成本

### 2.3 单体架构劣势

❌ **代码耦合严重**
```java
// 订单服务直接调用用户服务
public class OrderService {
    @Autowired
    private UserService userService;  // 强耦合
    
    public void createOrder(Long userId) {
        User user = userService.getById(userId);
        // ...
    }
}

问题: 
- 修改UserService可能影响OrderService
- 无法独立编译和部署
- 代码库越来越大,难以维护
```

❌ **扩展性差**
```
场景: 订单模块负载高,需要扩容

单体架构: 
❌ 只能整体扩容(复制整个应用)
❌ 浪费资源(用户模块不需要扩容)

微服务架构:
✅ 只扩容订单服务
✅ 节省资源,按需扩展
```

❌ **技术栈受限**
```
问题:
- 整个应用只能用一种语言(Java)
- 无法使用最适合的技术(如Python做AI)
- 升级框架困难(需要整体升级)
```

❌ **启动缓慢**
```
大型单体应用:
- 代码量: 100万+行
- 启动时间: 5-10分钟
- 内存占用: 4GB+

影响:
- 开发效率低(每次重启等很久)
- 部署风险高(停机时间长)
```

❌ **可靠性差**
```
单点故障:
某个模块内存泄漏 → 整个应用崩溃
              ↓
      所有功能不可用
```

### 2.4 适用场景

✅ **适合单体架构**:
- 小型项目(< 10人团队)
- 业务简单,变化少
- 快速原型验证(MVP)
- 初创公司早期阶段

❌ **不适合单体架构**:
- 大型项目(> 50人团队)
- 业务复杂,频繁迭代
- 高并发场景(双11秒杀)
- 多团队协作开发

## 三、SOA面向服务架构

### 3.1 SOA核心概念

**定义**:
> Service-Oriented Architecture,将系统拆分为多个粗粒度服务,通过ESB企业服务总线通信。

**架构图**:
```
                    ┌──────────────┐
                    │   ESB总线     │
                    └──┬───┬───┬───┘
                       │   │   │
              ┌────────┘   │   └────────┐
              ↓            ↓            ↓
    ┌─────────────┐ ┌──────────┐ ┌──────────┐
    │ 用户服务     │ │订单服务   │ │支付服务   │
    └─────────────┘ └──────────┘ └──────────┘
```

### 3.2 SOA特点

| 特性 | 说明 |
|------|------|
| **粗粒度服务** | 服务粒度较大(如整个CRM系统) |
| **ESB总线** | 统一消息路由和转换 |
| **WS-*标准** | SOAP协议,WSDL描述 |
| **XML通信** | 消息体大,解析慢 |
| **中心化** | ESB是中心枢纽 |

### 3.3 SOA vs 单体

**改进点**:
```
✅ 服务可复用(多个系统共用一个服务)
✅ 异构系统集成(不同语言的服务)
✅ 业务流程编排(BPEL)

❌ 仍然较重(ESB复杂)
❌ XML性能差
❌ 服务粒度粗
```

## 四、微服务架构 (Microservices Architecture)

### 4.1 微服务定义

**Martin Fowler定义**:
> 微服务架构是一种将单个应用程序开发为一组小型服务的方法,每个服务运行在自己的进程中,并通过轻量级机制(通常是HTTP API)通信。

**核心特征**:
```
1. 单一职责: 每个服务只做一件事
2. 独立部署: 可以单独发布更新
3. 去中心化: 无中央控制点
4. 轻量通信: HTTP/REST或gRPC
5. 技术多样: 不同服务可用不同技术栈
```

### 4.2 微服务架构

**典型架构**:
```
                    ┌──────────────┐
                    │  API Gateway  │
                    └──┬───┬───┬───┘
                       │   │   │
              ┌────────┘   │   └────────┐
              ↓            ↓            ↓
    ┌─────────────┐ ┌──────────┐ ┌──────────┐
    │user-service │ │order-ser.│ │pay-ser.  │
    │   :8081     │ │  :8082   │ │  :8083   │
    ├─────────────┤ ├──────────┤ ├──────────┤
    │  MySQL      │ │  MySQL   │ │  MySQL   │
    └─────────────┘ └──────────┘ └──────────┘
```

### 4.3 微服务优势

✅ **服务解耦**
```java
// 服务间通过HTTP API通信
@RestController
public class OrderController {
    
    @Autowired
    private RestTemplate restTemplate;
    
    @GetMapping("/order/{userId}")
    public Result getOrder(@PathVariable Long userId) {
        // 远程调用用户服务
        String url = "http://user-service/api/users/" + userId;
        User user = restTemplate.getForObject(url, User.class);
        
        return Result.success(user);
    }
}

优势:
- 服务独立开发,互不影响
- 可以独立编译和部署
- 接口清晰,便于协作
```

✅ **独立部署**
```
场景: 订单服务需要紧急修复Bug

微服务架构:
✅ 只重新部署order-service
✅ 其他服务不受影响
✅ 停机时间< 1分钟

单体架构:
❌ 需要重新部署整个应用
❌ 停机时间5-10分钟
❌ 影响所有功能
```

✅ **技术多样性**
```
不同服务可以选择最适合的技术:

user-service:    Java + Spring Boot (成熟稳定)
ai-service:      Python + TensorFlow (AI生态好)
realtime-service: Go (高性能)
frontend:        Node.js (前后端同构)
```

✅ **弹性伸缩**
```
双11场景:
- 订单服务: 扩容10个实例
- 用户服务: 扩容5个实例
- 商品服务: 扩容20个实例
- 支付服务: 扩容8个实例

按需扩容,节省成本!
```

✅ **容错性强**
```
某个服务故障:
order-service宕机
    ↓
其他服务仍可正常工作
    ↓
用户可以浏览商品、查看个人信息
    ↓
只是不能下单(降级处理)
```

### 4.4 微服务挑战

⚠️ **分布式复杂性**
```
问题:
- 网络延迟(远程调用比本地慢10-100倍)
- 部分失败(某个服务超时)
- 数据一致性(分布式事务)
- 服务发现(如何找到服务地址)

解决方案:
✅ 熔断降级(Sentinel)
✅ 异步消息(RabbitMQ)
✅ 最终一致性(Saga)
✅ 服务注册(Nacos)
```

⚠️ **运维成本高**
```
单体架构:
- 部署1个应用
- 监控1个日志文件
- 配置1台服务器

微服务架构:
- 部署10+个服务
- 聚合10+个日志
- 管理50+台服务器

解决方案:
✅ Docker容器化
✅ Kubernetes编排
✅ ELK日志聚合
✅ Prometheus监控
```

⚠️ **测试复杂**
```
单体测试:
- 启动1个应用
- 运行测试用例

微服务测试:
- 启动10+个服务
- Mock外部依赖
- 集成测试困难

解决方案:
✅ 契约测试(Pact)
✅ 服务虚拟化(WireMock)
✅ 端到端测试环境
```

⚠️ **数据一致性**
```
分布式事务问题:

订单服务: 创建订单 ✅
库存服务: 扣减库存 ✅
支付服务: 扣款失败 ❌

如何回滚?

解决方案:
✅ Saga模式(补偿事务)
✅ TCC模式(Try-Confirm-Cancel)
✅ 最终一致性(消息队列)
```

## 五、微服务设计原则

### 5.1 单一职责原则 (SRP)

**原则**:
> 每个服务只负责一个业务领域,只做一件事并做好。

**示例**:
```
❌ 错误设计:
ecommerce-service (电商服务)
├── 用户管理
├── 订单管理
├── 商品管理
└── 支付管理

✅ 正确设计:
user-service (用户服务) - 只负责用户相关
order-service (订单服务) - 只负责订单相关
product-service (商品服务) - 只负责商品相关
pay-service (支付服务) - 只负责支付相关
```

### 5.2 自治性原则

**原则**:
> 每个服务应该尽可能独立,减少对外部服务的依赖。

**实现**:
```yaml
# 每个服务有自己的数据库
user-service:
  datasource:
    url: jdbc:mysql://localhost:3306/user_db

order-service:
  datasource:
    url: jdbc:mysql://localhost:3306/order_db

# 而不是共享同一个数据库
```

### 5.3 去中心化数据管理

**原则**:
> 每个服务管理自己的数据,其他服务只能通过API访问。

**反模式**:
```sql
-- ❌ 错误: 订单服务直接查询用户表
SELECT * FROM user_db.users WHERE id = 1;

-- ✅ 正确: 通过用户服务API获取
GET http://user-service/api/users/1
```

### 5.4 基础设施自动化

**原则**:
> 微服务数量多,必须依靠自动化工具管理。

**工具链**:
```
CI/CD: Jenkins / GitLab CI
容器化: Docker
编排: Kubernetes
配置: Nacos / Apollo
监控: Prometheus + Grafana
日志: ELK Stack
```

## 六、云原生架构

### 6.1 云原生定义

**CNCF定义**:
> 云原生技术有利于各组织在公有云、私有云和混合云等新型动态环境中,构建和运行可弹性扩展的应用。

**四大要素**:
```
1. 容器化 (Container)
2. 动态编排 (Orchestration)
3. 微服务 (Microservices)
4. DevOps (持续交付)
```

### 6.2 云原生技术栈

```
应用层:    Spring Cloud / Dubbo
    ↓
编排层:    Kubernetes / Docker Swarm
    ↓
容器层:    Docker / containerd
    ↓
基础设施:  AWS / Azure / 阿里云
```

## 七、Service Mesh服务网格

### 7.1 什么是Service Mesh?

**定义**:
> 服务网格是一个基础设施层,用于处理服务间通信,包括流量管理、安全、可观测性等。

**架构演进**:
```
第一代: 客户端负载均衡 (Ribbon)
    ↓
第二代: API网关 (Spring Cloud Gateway)
    ↓
第三代: Service Mesh (Istio)
```

### 7.2 Istio架构

```
                    ┌──────────────┐
                    │  Control Plane│
                    │  (Pilot/Mixer)│
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
```

## 八、架构选型建议

### 8.1 何时使用微服务?

✅ **适合微服务**:
- 团队规模 > 20人
- 代码行数 > 50万行
- 需要快速迭代(每周发布)
- 高并发场景(QPS > 10000)
- 多业务线并行

❌ **不适合微服务**:
- 初创公司(< 10人)
- 简单管理系统
- 内部工具
- 原型验证阶段

### 8.2 渐进式演进

```
推荐路径:
单体应用 (MVP阶段)
    ↓ 业务增长
模块化单体 (清晰分层)
    ↓ 团队扩大
垂直拆分 (按业务域)
    ↓ 复杂度增加
微服务架构 (完全拆分)
    ↓ 规模扩大
Service Mesh (服务网格)
```

## 九、总结

### 核心知识点

✅ **架构演进**: 单体 → SOA → 微服务 → 云原生  
✅ **单体架构**: 简单但耦合,适合小项目  
✅ **微服务**: 解耦但复杂,适合大项目  
✅ **设计原则**: SRP、自治性、去中心化  
✅ **技术选型**: 根据团队和业务决定
