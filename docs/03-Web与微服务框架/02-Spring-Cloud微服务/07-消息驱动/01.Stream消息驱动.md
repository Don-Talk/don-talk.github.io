## 一、消息驱动概述

### 1.1 为什么需要消息中间件?

**同步调用的问题**:
```
用户下单 → order-service (同步)
           ↓
       inventory-service (同步,耗时2秒)
           ↓
       payment-service (同步,耗时3秒)
           ↓
       notification-service (同步,耗时1秒)
           
总耗时: 6秒! ❌
用户体验: 等待太久
```

TODO: [此处需要添加异步架构对比图]

**消息驱动的解决方案**:
```
用户下单 → order-service (立即返回)
           ↓ 发送消息
       Message Broker (RabbitMQ/Kafka)
           ↓ 异步消费
       ├── inventory-service (扣减库存)
       ├── payment-service (处理支付)
       └── notification-service (发送通知)
           
总耗时: 200ms! ✅
用户体验: 响应迅速
```

### 1.2 Spring Cloud Stream优势

**传统方式 vs Stream**:

#### 传统方式(直接集成MQ)

```java
// RabbitMQ原生API
@Autowired
private RabbitTemplate rabbitTemplate;

public void sendMessage(Order order) {
    rabbitTemplate.convertAndSend("order.exchange", "order.create", order);
}

// 切换Kafka时需要重写代码
@Autowired
private KafkaTemplate<String, Order> kafkaTemplate;

public void sendMessage(Order order) {
    kafkaTemplate.send("order-topic", order);
}
```

#### Stream方式(统一抽象)

```java
// 定义消息通道
@EnableBinding(Source.class)
public class OrderMessageProducer {
    
    @Autowired
    private MessageChannel output;
    
    public void sendMessage(Order order) {
        output.send(MessageBuilder.withPayload(order).build());
    }
}

// 切换MQ只需修改配置,代码不变!
spring.cloud.stream.binders: rabbit -> kafka
```

**核心优势**:
```
✅ 统一编程模型: 屏蔽MQ差异
✅ 简化开发: 注解驱动
✅ 内置功能: 分组、分区、重试
✅ 易于测试: 支持嵌入式Binder
```

### 1.3 核心概念

| 概念 | 说明 | 类比 |
|------|------|------|
| **Binder** | 消息中间件抽象 | JDBC驱动 |
| **Input** | 消息输入通道 | 消费者 |
| **Output** | 消息输出通道 | 生产者 |
| **Message** | 消息对象 | 快递包裹 |
| **Group** | 消费者组 | 工作组 |
| **Partition** | 消息分区 | 快递分拣 |

```bash
# 用户名/密码: guest/guest
```

### 2.2 消息生产者

#### 步骤1: 添加依赖

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-stream-rabbit</artifactId>
</dependency>
```

#### 步骤2: 配置连接

```yaml
# application.yml
spring:
  cloud:
    stream:
      binders:
        defaultRabbit:
          type: rabbit
          environment:
            spring:
              rabbitmq:
                host: localhost
                port: 5672
                username: guest
                password: guest
      bindings:
        order-output:
          destination: orders
          binder: defaultRabbit
```

#### 步骤3: 定义消息通道

```java
public interface OrderChannels {
    
    String ORDER_OUTPUT = "order-output";
    
    @Output(ORDER_OUTPUT)
    MessageChannel orderOutput();
}
```

#### 步骤4: 发送消息

```java
@Component
@EnableBinding(OrderChannels.class)
public class OrderMessageProducer {
    
    @Autowired
    private MessageChannel orderOutput;
    
    /**
     * 发送订单消息
     */
    public void sendOrderMessage(Order order) {
        Message<Order> message = MessageBuilder
            .withPayload(order)
            .setHeader("messageType", "ORDER_CREATE")
            .setHeader("timestamp", System.currentTimeMillis())
            .build();
        
        boolean sent = orderOutput.send(message);
        if (sent) {
            log.info("订单消息发送成功: {}", order.getOrderId());
        } else {
            log.error("订单消息发送失败: {}", order.getOrderId());
        }
    }
}
```

## 三、消息分组

### 3.1 为什么需要分组?

**问题场景**:
```
订单服务启动了3个实例:
                
Message Broker
    ↓
├── order-service:8081 (实例1)
├── order-service:8082 (实例2)
└── order-service:8083 (实例3)

没有分组: 每条消息被所有实例消费 ❌
期望结果: 每条消息只被一个实例消费 ✅
```

### 3.2 配置分组

```yaml
# 实例1
spring:
  cloud:
    stream:
      bindings:
        order-input:
          destination: orders
          group: order-group  # 同一组

# 实例2 (配置相同)
spring:
  cloud:
    stream:
      bindings:
        order-input:
          destination: orders
          group: order-group  # 同一组

# 实例3 (配置相同)
spring:
  cloud:
    stream:
      bindings:
        order-input:
          destination: orders
          group: order-group  # 同一组
```

**结果**:
```
消息1 → order-service:8081
消息2 → order-service:8082
消息3 → order-service:8083
消息4 → order-service:8081 (轮询分配)
```

### 3.3 不同组独立消费

```yaml
# 订单服务
spring:
  cloud:
    stream:
      bindings:
        order-input:
          destination: orders
          group: order-group

# 库存服务 (独立组)
spring:
  cloud:
    stream:
      bindings:
        order-input:
          destination: orders
          group: inventory-group
```

**结果**:
```
消息1 → order-group (订单服务处理)
消息1 → inventory-group (库存服务也处理)

每个组都会收到消息!
```

## 四、消息分区

### 4.1 为什么需要分区?

**问题场景**:
```
同一个订单的消息可能被不同实例处理:

消息A(订单1-创建) → 实例1
消息B(订单1-支付) → 实例2 ❌ (实例2没有订单1的缓存)
```

**分区解决方案**:
```
消息A(订单1-创建) → 分区1 → 实例1
消息B(订单1-支付) → 分区1 → 实例1 ✅ (保证同一订单由同一实例处理)
```

### 4.2 配置分区

#### 生产者配置

```yaml
spring:
  cloud:
    stream:
      bindings:
        order-output:
          destination: orders
          producer:
            partition-key-expression: payload.orderId  # 按订单ID分区
            partition-count: 3  # 3个分区
```

#### 消费者配置

```yaml
spring:
  cloud:
    stream:
      bindings:
        order-input:
          destination: orders
          group: order-group
          consumer:
            partitioned: true  # 启用分区
            
      instance-count: 3  # 3个实例
      instance-index: 0  # 当前实例索引(0/1/2)
```

### 4.3 自定义分区策略

```java
@Configuration
public class PartitionConfig {
    
    @Bean
    public PartitionKeyExtractorStrategy partitionKeyExtractor() {
        return message -> {
            if (message.getPayload() instanceof Order) {
                Order order = (Order) message.getPayload();
                // 按用户ID分区
                return order.getUserId();
            }
            return null;
        };
    }
    
    @Bean
    public PartitionSelectorStrategy partitionSelector() {
        return key -> {
            if (key instanceof Long) {
                return (int) ((Long) key % 3);
            }
            return 0;
        };
    }
}
```

## 五、消息可靠性

### 5.1 消息确认机制

#### 手动确认

```yaml
spring:
  cloud:
    stream:
      rabbit:
        bindings:
          order-input:
            consumer:
              acknowledge-mode: MANUAL  # 手动确认
```

```java
@StreamListener(OrderChannels.ORDER_INPUT)
public void handleOrderMessage(
    Message<Order> message,
    @Header(AmqpHeaders.CHANNEL) Channel channel,
    @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag) {
    
    try {
        Order order = message.getPayload();
        
        // 业务处理
        processOrder(order);
        
        // 手动确认
        channel.basicAck(deliveryTag, false);
        log.info("消息处理成功: {}", order.getOrderId());
        
    } catch (Exception e) {
        log.error("消息处理失败", e);
        
        // 拒绝消息,requeue=true重新入队
        try {
            channel.basicNack(deliveryTag, false, true);
        } catch (IOException ioException) {
            log.error("拒绝消息失败", ioException);
        }
    }
}
```

### 5.2 消息重试

```yaml
spring:
  cloud:
    stream:
      bindings:
        order-input:
          consumer:
            max-attempts: 3  # 最大重试次数
          back-off:
            initial-interval: 1000  # 初始间隔1秒
            multiplier: 2  # 倍增因子
            max-interval: 10000  # 最大间隔10秒
```

### 5.3 死信队列

```yaml
spring:
  cloud:
    stream:
      rabbit:
        bindings:
          order-input:
            consumer:
              auto-bind-dlq: true  # 自动绑定死信队列
              dead-letter-exchange: orders.dlx
              dead-letter-routing-key: orders.dead
```

**死信消费者**:
```java
@Component
@EnableBinding(OrderChannels.class)
public class DeadLetterConsumer {
    
    @StreamListener("orders.dlx")
    public void handleDeadLetter(Message<Order> message) {
        Order order = message.getPayload();
        
        log.error("消息进入死信队列: {}", order.getOrderId());
        
        // 记录日志,人工处理
        deadLetterService.save(order);
    }
}
```

## 六、函数式编程模型(新)

### 6.1 Supplier(生产者)

```java
@SpringBootApplication
public class ProducerApplication {
    
    public static void main(String[] args) {
        SpringApplication.run(ProducerApplication.class, args);
    }
    
    @Bean
    public Supplier<Message<Order>> orderSupplier() {
        return () -> {
            Order order = new Order();
            order.setOrderId(System.currentTimeMillis());
            order.setAmount(new BigDecimal("99.99"));
            
            return MessageBuilder
                .withPayload(order)
                .setHeader("contentType", "application/json")
                .build();
        };
    }
}
```

**配置**:
```yaml
spring:
  cloud:
    stream:
      function:
        definition: orderSupplier
      bindings:
        orderSupplier-out-0:
          destination: orders
```

```java
            // 业务处理
            processOrder(order);
        };
    }

```

**配置**:

### 6.3 Function(转换)

```java
@SpringBootApplication
public class FunctionApplication {
    
    public static void main(String[] args) {
        SpringApplication.run(FunctionApplication.class, args);
    }
    
    @Bean
    public Function<Message<OrderDTO>, Message<Order>> orderFunction() {
        return message -> {
            OrderDTO dto = message.getPayload();
            
            // DTO转Entity
            Order order = new Order();
            BeanUtils.copyProperties(dto, order);
            order.setCreateTime(new Date());
            
            return MessageBuilder
                .withPayload(order)
                .copyHeaders(message.getHeaders())
                .build();
        };
    }
}
```

### 7.2 消息转换器

```java
@Component
public class OrderMessageConverter implements MessageConverter {
    
    @Override
    public Object fromMessage(Message<?> message, Class<?> targetClass) {
        if (targetClass == Order.class) {
            String json = new String((byte[]) message.getPayload());
            return JSON.parseObject(json, Order.class);
        }
        return message.getPayload();
    }
    
    @Override
    public Message<?> toMessage(Object payload, MessageHeaders headers) {
        if (payload instanceof Order) {
            String json = JSON.toJSONString(payload);
            return MessageBuilder
                .withPayload(json.getBytes())
                .copyHeaders(headers)
                .build();
        }
        return MessageBuilder.withPayload(payload).build();
    }
}
```

## 九、最佳实践

### 9.1 消息幂等性

```java
@Component
public class IdempotentConsumer {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @StreamListener(OrderChannels.ORDER_INPUT)
    public void handleMessage(Message<Order> message) {
        String messageId = message.getHeaders().getId().toString();
        
        // 检查是否已处理
        Boolean exists = redisTemplate.hasKey("msg:" + messageId);
        if (Boolean.TRUE.equals(exists)) {
            log.warn("消息已处理,跳过: {}", messageId);
            return;
        }
        
        // 处理消息
        Order order = message.getPayload();
        processOrder(order);
        
        // 标记已处理(过期时间24小时)
        redisTemplate.opsForValue().set("msg:" + messageId, "1", 24, TimeUnit.HOURS);
    }
}
```

## 十、常见问题

### 10.1 消息丢失

**问题**: 消息发送后消费者没收到

**排查**:
```yaml
# 1. 检查目的地是否正确
spring:
  cloud:
    stream:
      bindings:
        order-output:
          destination: orders  # 确保一致

# 2. 检查消费者组
spring:
  cloud:
    stream:
      bindings:
        order-input:
          destination: orders
          group: order-group  # 确保组名正确
```

### 10.2 重复消费

**问题**: 同一条消息被多次处理

**解决**: 实现幂等性
```java
// 使用唯一ID去重
String messageId = message.getHeaders().getId().toString();
if (alreadyProcessed(messageId)) {
    return;
}
```

### 10.3 消息积压

**问题**: 消息堆积,消费不过来

**解决**:
```yaml
# 1. 增加消费者实例
# 2. 优化消费逻辑
# 3. 增加并发度
spring:
  cloud:
    stream:
      rabbit:
        bindings:
          order-input:
            consumer:
              concurrency: 5  # 5个并发线程
```
